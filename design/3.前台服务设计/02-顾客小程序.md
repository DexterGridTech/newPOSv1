## 一、产品定位与边界

### 1.1 定位

购物中心统一顾客小程序，是**顾客与购物中心内所有餐饮商户交互的统一数字入口**。

- 一个小程序覆盖购物中心内所有餐饮商户
- 支持堂食扫码点餐、外卖配送、到店自取三种就餐模式
- 承载购物中心统一会员体系（积分、优惠券、储值）
- 微信小程序 + 支付宝小程序双端（uni-app 实现）

### 1.2 业务边界

**小程序负责：**
- 商户/菜单浏览（含搜索、筛选）
- 购物车管理
- 订单提交与支付（微信支付/支付宝）
- 订单状态跟踪
- 堂食加菜（同桌会话下的增量订单）
- 预订（餐段查询、创建预订、订金支付、预订详情、取消）
- 会员中心（积分、优惠券、历史订单）
- 堂食扫码入口（扫桌台码后跳转 H5 或小程序内点餐）

**小程序不负责：**
- 门店运营管理（商户后台负责）
- 收银操作（POS 负责）
- 外卖骑手端（三方平台负责）

## 三、核心页面详细设计

### 3.1 商户广场（首页）

**页面目标：** 帮助顾客快速找到想去的商户。

**布局结构：**
```
┌─────────────────────────────┐
│  [搜索框] "搜索餐厅或菜品"   │
├─────────────────────────────┤
│  [Banner 轮播] 营销活动      │
├─────────────────────────────┤
│  [楼层筛选] B1 | 1F | 2F    │
│  [分类筛选] 全部|快餐|火锅…  │
├─────────────────────────────┤
│  [排序] 综合|距离|评分|销量  │
├─────────────────────────────┤
│  商户卡片列表（瀑布流）       │
│  ┌──────────────────────┐   │
│  │ [封面图]              │   │
│  │ 商户名称    ⭐4.8     │   │
│  │ 快餐 · 1F · 营业中   │   │
│  │ 人均 ¥35  月售 1200  │   │
│  └──────────────────────┘   │
└─────────────────────────────┘
```

**商户卡片数据字段：**
```typescript
interface MerchantCard {
  storeId: string;
  name: string;
  coverImage: string;
  category: string;          // 快餐、正餐、饮品等
  floor: string;             // B1、1F、2F
  rating: number;            // 4.8
  monthSales: number;        // 月销量
  avgPrice: number;          // 人均消费
  status: 'OPEN' | 'CLOSED' | 'BUSY'; // 营业中/休息中/繁忙
  tags: string[];            // ['新店', '必吃榜']
  deliveryInfo?: {           // 外卖信息（如支持外卖）
    deliveryFee: number;
    estimatedMinutes: number;
    minOrderAmount: number;
  };
}
```

**状态处理：**
- `CLOSED`：卡片置灰，显示"休息中"，仍可点击进入查看菜单（不可下单）
- `BUSY`：正常显示，加"繁忙"标签，提示等待时间可能较长
- 营业时间外：自动切换为 CLOSED 状态

**API 调用：**
```
GET /api/v1/stores?projectId={projectId}&floor={floor}&category={category}&sort={sort}&page={page}&size=20
```

**性能要求：**
- 首屏展示前 10 个商户，滚动加载更多
- 商户封面图使用 CDN，WebP 格式，懒加载
- 搜索防抖 300ms

**商品卡片数据字段：**
```typescript
interface ProductCard {
  productId: string;
  name: string;
  description: string;
  image: string;
  basePrice: number;         // 基础价格（无规格时即售价）
  originalPrice?: number;    // 划线价
  monthSales: number;
  tags: string[];            // ['招牌', '新品', '辣']
  status: 'AVAILABLE' | 'SOLD_OUT' | 'OFF_SHELF';

  specs?: ProductSpec[];
  stock?: number;            // 剩余库存（仅限量商品展示）
}

interface ProductSpec {
  specGroupName: string;     // 规格组名，如"辣度"
  required: boolean;
  options: {
    optionId: string;
    name: string;            // "微辣"
    priceAdjust: number;     // 价格调整，0 表示不加价
    available: boolean;
  }[];
}
```

**规格选择弹层：**
```
┌─────────────────────────────┐
│  红烧肉                      │
│  ¥38                        │
├─────────────────────────────┤
│  份量（必选）                │
│  [小份¥28] [中份¥38✓] [大份¥58]│
├─────────────────────────────┤
│  辣度（必选）                │
│  [不辣] [微辣✓] [中辣] [重辣]│
├─────────────────────────────┤

├─────────────────────────────┤
│  [加入购物车  ¥38]           │
└─────────────────────────────┘
```

**购物车逻辑：**
- 购物车数据存储在前端（sessionStorage），不调用后台接口
- 切换就餐方式时，清空购物车并提示
- 切换商户时，清空购物车并提示
- 同一商品不同规格视为不同 SKU，分别计数
- 购物车最多 50 个 SKU

**菜单数据加载：**
```
GET /api/v1/stores/{storeId}/menu?scene={dine_in|delivery|takeout}
```
- 菜单数据按就餐场景返回（不同场景可能有不同价格/商品）
- 菜单数据缓存 5 分钟（staleTime），后台菜单变更通过 WebSocket 推送失效信号

**售罄处理：**
- 商品 `status = SOLD_OUT`：显示"售罄"标签，不可加购，不置灰（仍可查看）
- 规格选项 `available = false`：置灰不可选
- 加购时后台返回售罄错误（22003）：提示"该商品已售罄，请选择其他商品"

**提交订单 API：**
```
POST /api/v1/orders
{
  "storeId": "store_001",
  "scene": "DINE_IN",
  "tableCode": "A05",
  "items": [
    {
      "productId": "prod_001",
      "skuId": "sku_001",
      "quantity": 1,
      "specSnapshot": { "份量": "中份", "辣度": "微辣" },
      "unitPrice": 38.00
    }
  ],
  "couponId": "coupon_001",
  "remark": "",
  "paymentMethod": "WECHAT_PAY",
  "clientOrderId": "cli_uuid_001"   // 幂等键
}
```

**响应处理：**
```typescript
// 成功：跳转支付
if (response.code === 200) {
  const { orderId, paymentParams } = response.data;
  // 调用微信支付 SDK
  wx.requestPayment({
    ...paymentParams,
    success: () => navigateTo('/order/result?orderId=' + orderId),
    fail: (err) => {
      if (err.errMsg.includes('cancel')) {
        // 用户主动取消，跳转订单详情（待支付状态）
        navigateTo('/order/detail?orderId=' + orderId);
      } else {
        // 支付失败，展示错误
        showPaymentError(err);
      }
    }
  });
}

// 商品售罄（22003）
if (response.code === 22003) {
  // 标记购物车中售罄商品，提示用户修改
  markSoldOutItems(response.data.soldOutItems);
}
```

**幂等处理：**
- `clientOrderId` 由前端生成（UUID），进入确认页时生成，支付失败重试时保持不变
- 后台根据 `clientOrderId` 去重，重复提交返回已有订单

### 3.5 订单详情页

**页面目标：** 展示订单全生命周期状态，提供操作入口。

**订单状态进度条：**
```
堂食订单：
[下单] → [支付] → [接单] → [制作中] → [已上菜] → [完成]

外卖订单：
[下单] → [支付] → [接单] → [制作中] → [配送中] → [完成]

自取订单：
[下单] → [支付] → [接单] → [制作中] → [待取餐] → [完成]
```

**各状态对应操作按钮：**

| 订单状态 | 可用操作 | 备注 |
|----------|----------|------|
| PENDING_PAYMENT | 去支付、取消订单 | 15分钟超时自动关闭 |
| PAID | 取消订单（需商户同意） | 已支付，取消需退款 |
| ACCEPTED | 催单 | 商户已接单 |
| IN_PRODUCTION | 催单 | 制作中 |
| PRODUCTION_DONE | — | 等待配送/取餐 |
| FULFILLED | 确认收货（外卖/自取） | 堂食自动完成 |
| COMPLETED | 再来一单、申请售后 | — |
| CANCELLED | 再来一单 | — |

**实时状态更新：**
- 订单详情页打开时，监听 WebSocket 推送的 `ORDER_STATUS_CHANGED` 消息（过滤 `data.orderId` 匹配当前订单）
- 收到状态变更推送后，刷新页面状态（不需要重新请求完整订单）
- 页面不可见时（切后台），降级为 30s 轮询

**取消订单流程：**
```
点击"取消订单"
  → 弹出确认框（展示：已支付金额、退款方式、预计到账时间）
  → 确认取消
  → POST /api/v1/orders/{orderId}/cancel
  → 成功：更新状态为 CANCELLED，展示退款信息
  → 失败（20003 订单已开始制作）：提示"商家已开始制作，无法取消，如需退款请联系商家"
```

## 四、关键交互规范

### 4.1 加载状态

- 页面首次加载：骨架屏（Skeleton），不用 loading spinner
- 操作提交（下单、支付）：按钮 loading 状态 + 禁用，防止重复点击
- 列表加载更多：底部 loading 指示器
- 图片加载：占位色块，加载完成后淡入

### 4.2 错误提示规范

| 错误类型 | 展示方式 | 示例 |
|----------|----------|------|
| 表单校验错误 | 字段下方红色提示 | "请选择就餐方式" |
| 业务错误（可恢复） | Toast（3s 自动消失） | "该商品已售罄" |
| 业务错误（需操作） | 弹窗（需用户确认） | "订单已开始制作，无法取消" |
| 网络错误 | 全屏错误页 + 重试按钮 | "网络连接失败，请检查网络" |
| 系统错误 | Toast + 错误码 | "系统繁忙，请稍后重试（10001）" |

### 4.3 支付流程异常处理

```
发起支付
  ├── 支付成功 → 跳转成功页
  ├── 用户取消 → 跳转订单详情（待支付）
  ├── 支付失败 → 展示失败原因 + 重试入口
  └── 支付结果未知（网络超时）
        → 展示"支付结果确认中"
        → 轮询查询支付状态（每 3s 查一次，最多 10 次）
        → 确认成功 → 跳转成功页
        → 确认失败 → 展示失败 + 重试
        → 仍未知 → 提示"请在订单列表查看支付结果"
```

### 4.4 桌台扫码入口

顾客扫描桌台二维码后：
1. 解析二维码参数：`storeId`、`tableCode`、`scene=DINE_IN`
2. 检查是否已有进行中的桌台会话（`GET /api/v1/table-sessions/active?tableCode={tableCode}`）
3. 如有进行中会话：提示"该桌台已有订单，是否加入？"
4. 跳转商户详情页，自动设置就餐方式为堂食，桌台号已确认

### 3.7 堂食加菜（同桌会话增量订单）

堂食首单支付后，顾客可在小程序继续加菜。加菜创建的是**同一桌台会话下的新增量订单**，不修改原订单。

**加菜入口：** 订单详情页 → "继续加菜"按钮（仅堂食且订单状态为 ACCEPTED/IN_PRODUCTION 时显示）

**加菜流程：**
```
点击"继续加菜"
  → 进入商户菜单页（加菜模式，显示当前桌台已点菜品）
  → 选择新增菜品
  → 进入加菜确认页：
    ┌─────────────────────────────┐
    │  加菜确认                    │
    │  桌台：A区05号桌             │
    ├─────────────────────────────┤
    │  新增菜品                   │
    │  甜品×1                ¥8  │
    ├─────────────────────────────┤
    │  服务费  ¥0.80（10%）       │
    │  本次实付  ¥8.80            │
    ├─────────────────────────────┤
    │  [立即支付 ¥8.80]           │
    └─────────────────────────────┘
  → POST /api/v1/orders（携带 tableSessionId）
  → 支付成功：展示"加菜成功，厨房已收到"
  → 支付失败：加菜单保留为 PENDING_PAYMENT，可在订单列表重新支付
```

**同桌多单汇总展示：**

订单列表页中，同一桌台会话的多个订单聚合展示：
```
┌─────────────────────────────┐
│  A区05号桌  就餐中           │
│  首单 #001  ¥44  已支付     │
│  加菜 #002  ¥8.80 已支付    │
│  合计消费：¥52.80            │
│  [查看详情]  [继续加菜]      │
└─────────────────────────────┘
```

**加菜支付失败处理：**
- 加菜单状态保持 `PENDING_PAYMENT`，在订单列表显示"待支付"
- 顾客可在离店前重新发起支付
- 若离店时仍未支付，门店收银员可在 POS 端处理

### 3.8 顾客预订

**预订入口：** 商户详情页顶部操作栏，与"点餐""外卖"并列。

**餐段查询页：**
```
┌─────────────────────────────┐
│  选择就餐时间                │
├─────────────────────────────┤
│  [< 4月5日(今天) >]          │
├─────────────────────────────┤
│  午市                        │
│  ┌──────┐ ┌──────┐ ┌──────┐ │
│  │11:00 │ │11:30 │ │12:00 │ │
│  │可预订│ │可预订│ │剩2位 │ │
│  └──────┘ └──────┘ └──────┘ │
│  ┌──────┐                    │
│  │12:30 │                    │
│  │已满  │                    │
│  └──────┘                    │
│  晚市                        │
│  ┌──────┐ ┌──────┐           │
│  │17:30 │ │18:00 │           │
│  │可预订│ │可预订│           │
│  └──────┘ └──────┘           │
└─────────────────────────────┘
```

餐段状态：`可预订` / `剩N位`（N≤5时展示）/ `已满`（不可点击）

**创建预订页：**
```
┌─────────────────────────────┐
│  确认预订信息                │
│  时间：4月5日 12:00  午市    │
├─────────────────────────────┤
│  就餐人数  [-] 2 [+]         │
│  联系姓名  [输入姓名]         │
│  联系电话  [输入手机号]       │
│  备注      [特殊需求...]      │
├─────────────────────────────┤
│  订金说明                    │
│  需支付订金 ¥50，可抵扣消费  │
│  24小时前取消全额退还        │
├─────────────────────────────┤
│  [确认预订]                  │
└─────────────────────────────┘
```

提交后若需订金，跳转订金支付页（微信/支付宝）；无需订金则直接进入预订详情页。

**预订详情页：**
```
┌─────────────────────────────┐
│  预订详情  #BK20240405001    │
│  状态：已确认 ✅              │
├─────────────────────────────┤
│  XX火锅  4月5日 12:00  2人   │
├─────────────────────────────┤
│  订金：¥50（已支付）         │
│  到店后可抵扣消费            │
├─────────────────────────────┤
│  [取消预订]  [联系门店]      │
└─────────────────────────────┘
```

| 状态 | 展示文案 | 可操作 |
|------|----------|--------|
| PENDING | 待门店确认 | 取消 |
| CONFIRMED | 已确认 ✅ | 取消（24h前） |
| ARRIVED | 已到店 | — |
| COMPLETED | 已完成 | 再次预订 |
| CANCELLED | 已取消 | 再次预订 |
| OVERDUE | 逾期未到 | 再次预订 |

**取消预订：** 展示退款规则 → 选择取消原因 → 确认 → `POST /api/v1/bookings/{bookingId}/cancel` → 符合条件自动退还订金。

**到店提醒：** 预订前 2 小时推送小程序服务通知；逾期 15 分钟未到店再次提醒。

**预订相关 API：**

| 接口 | 方法 | 路径 |
|------|------|------|
| 查询可预订餐段 | GET | `/api/v1/booking/periods?storeId=&date=` |
| 创建预订 | POST | `/api/v1/bookings` |
| 预订详情 | GET | `/api/v1/bookings/{bookingId}` |
| 取消预订 | POST | `/api/v1/bookings/{bookingId}/cancel` |
