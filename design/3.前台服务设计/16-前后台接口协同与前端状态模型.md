## 一、API 调用规范

### 1.1 请求规范

**必须携带的请求头：**
```
Authorization: Bearer <jwt_token>
Content-Type: application/json
```

> Token 中已包含 tenant_id、store_id、role、device_id 等上下文，无需额外请求头。

**幂等请求额外携带：**
```
Idempotency-Key: <uuid_v4>
```

**幂等键使用规则：**
- 每次用户发起的操作生成一个新 UUID
- 同一操作的重试（网络超时后重试）必须使用相同 UUID
- 不同操作必须使用不同 UUID
- 幂等键有效期：24 小时

### 1.2 响应规范

**统一响应结构（以后台确认稿为准）：**
```typescript
interface ApiResponse<T> {
  code: number;        // 200 = 成功，非200 = 错误
  message: string;
  data: T;
  timestamp: number;   // 服务端时间戳（毫秒）
  traceId: string;
}
```

**分页响应结构：**
```typescript
interface PageResponse<T> {
  items: T[];
  total: number;
  pageNum: number;     // 当前页码（从1开始）
  pageSize: number;
  totalPages: number;
}
```

### 1.3 错误码规范（以后台确认稿为准）

## 一补、并发请求去重机制

同一操作在网络慢时可能被用户重复触发（连点、重试）。前端必须在发起层去重，避免重复提交。

```typescript
// 请求去重 Map：key = 操作标识，value = 进行中的 Promise
const inflightRequests = new Map<string, Promise<any>>();

async function deduplicatedRequest<T>(
  key: string,
  requestFn: () => Promise<T>
): Promise<T> {
  if (inflightRequests.has(key)) {
    // 相同操作正在进行中，复用同一个 Promise
    return inflightRequests.get(key) as Promise<T>;
  }
  const promise = requestFn().finally(() => {
    inflightRequests.delete(key);
  });
  inflightRequests.set(key, promise);
  return promise;
}
```

**使用规则：**

| 操作 | 去重 key 构成 | 说明 |
|------|--------------|------|
| 提交订单 | `submit-order-${clientOrderId}` | clientOrderId 由用户操作时生成 |
| 接单/拒单 | `accept-${orderId}` | 同一订单只允许一次进行中请求 |
| 工单完成 | `complete-wu-${workUnitId}` | KDS 防止双击 |
| 支付发起 | `pay-${orderId}` | 支付中禁止重复发起 |

**与幂等键的关系：** 去重在前端内存层防止重复发出请求；幂等键（`Idempotency-Key` 请求头，见接口规范第2节）在服务端防止网络重传导致的重复执行。两者互补，缺一不可。

## 二、前端状态模型

### 2.1 状态分层

```
┌─────────────────────────────────────────┐
│  服务端状态（Server State）              │
│  订单、菜单、库存、桌台                  │
│  → 使用 React Query / SWR 管理          │
│  → 不存入全局 store                     │
├─────────────────────────────────────────┤
│  全局 UI 状态（Global UI State）         │
│  购物车、当前操作员、当前桌台上下文      │
│  → 使用 Zustand / Pinia 管理            │
├─────────────────────────────────────────┤
│  本地 UI 状态（Local UI State）          │
│  弹窗开关、loading、表单值              │
│  → 使用组件 useState                   │
└─────────────────────────────────────────┘
```

### 2.2 服务端状态缓存策略

| 数据类型 | staleTime | gcTime | 说明 |
|----------|-----------|--------|------|
| 菜单数据 | 5 分钟 | 30 分钟 | 菜单变更通过 WS 主动失效 |
| 订单列表 | 30 秒 | 5 分钟 | 频繁变化，短缓存 |
| 订单详情 | 0（始终刷新） | 5 分钟 | 详情页每次打开强制刷新 |
| 桌台列表 | 1 分钟 | 10 分钟 | WS 推送时主动失效 |
| 商户信息 | 10 分钟 | 60 分钟 | 变化少 |
| 权限数据 | 登录期间不失效 | — | 登出时清除 |

## 三、订单状态机（前端视角）

### 3.1 订单状态流转

```
PENDING_PAYMENT
    │ 支付成功
    ▼
PAID
    │ 门店接单
    ▼
ACCEPTED
    │ 开始制作
    ▼
IN_PRODUCTION
    │ 制作完成
    ▼
PRODUCTION_DONE
    │ 配送/上菜/取餐
    ▼
FULFILLED
    │ 确认完成
    ▼
COMPLETED

任意状态 ──取消──► CANCELLED
> 退款（部分或全额）不改变订单主状态，退款情况由独立的 `refundStatus` 字段表达。
> 全额退款完成后主状态仍为 CANCELLED，前端通过 `refundStatus == FULLY_REFUNDED` 展示"已退款"标签。
```

### 3.2 各状态前端展示规则


### 3.3 退款状态机

```
PARTIAL_REFUNDING  ──成功──► PARTIALLY_REFUNDED
                   ──失败──► REFUND_FAILED → 人工介入

FULL_REFUNDING     ──成功──► FULLY_REFUNDED
                   ──失败──► REFUND_FAILED → 人工介入
```

| 退款状态 | 前端展示 | 可执行操作 |
|----------|----------|-----------|
| PARTIAL_REFUNDING | "部分退款处理中…" | 禁止重复发起 |
| PARTIALLY_REFUNDED | "已部分退款 ¥X" | 可继续发起剩余退款 |
| FULL_REFUNDING | "退款处理中…" | 禁止重复发起 |
| FULLY_REFUNDED | "已全额退款" | 只读 |
| REFUND_FAILED | "退款失败，请联系客服" | 提供客服入口 |

### 3.4 桌台会话状态机

```
OPEN（开台）
  │ 所有订单完成/结账
  ▼
PENDING_CLEAR（待清台）
  │ 服务员确认清台
  ▼
CLOSED（已关闭）

任意状态 ──转台──► 原会话 CLOSED，目标桌台新建/合并 OPEN
```

### 3.5 预订状态机

```
PENDING（待确认）
  │ 门店确认
  ▼
CONFIRMED（已确认）
  │ 顾客到店
  ▼
ARRIVED（已到店）
  │ 完成就餐
  ▼
COMPLETED（已完成）

PENDING/CONFIRMED ──取消──► CANCELLED
CONFIRMED ──逾期未到──► OVERDUE
```

## 四、乐观更新策略

对于低风险操作，使用乐观更新提升体验：

**适合乐观更新的操作：**
- 购物车增减数量
- 工单标记完成（KDS）
- 上菜确认（PDA）

**不适合乐观更新的操作：**
- 支付（金额相关，必须等后台确认）
- 退款（金额相关）
- 订单取消（状态不可逆）

```typescript
// KDS 工单完成的乐观更新示例
const completeWorkUnit = useMutation({
  mutationFn: (workUnitId: string) =>
    api.post(`/work-units/${workUnitId}/complete`),

  onMutate: async (workUnitId) => {
    // 取消正在进行的查询（避免覆盖乐观更新）
    await queryClient.cancelQueries(['work-units']);

    // 保存当前状态（用于回滚）
    const previous = queryClient.getQueryData(['work-units']);

    // 乐观更新：立即从列表移除
    queryClient.setQueryData(['work-units'], (old: WorkUnit[]) =>
      old.filter(wu => wu.workUnitId !== workUnitId)
    );

    return { previous };
  },

  onError: (err, workUnitId, context) => {
    // 失败时回滚
    queryClient.setQueryData(['work-units'], context?.previous);
    showError('操作失败，请重试');
  },
});
```

## 五、接口版本管理

所有 API 路径包含版本号：`/api/v1/...`

**版本升级原则：**
- 向后兼容的变更（新增字段）：不升级版本
- 破坏性变更（删除/重命名字段）：升级到 v2
- 旧版本保留至少 6 个月后下线

**前端版本检测：**
```typescript
// 响应头中包含 API 版本
// X-Api-Version: 1.2.3
// 当 minor 版本升级时，前端提示刷新
```
