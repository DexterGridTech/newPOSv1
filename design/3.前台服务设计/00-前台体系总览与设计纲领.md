## 二、系统定位

### 2.1 平台定位

本平台是**购物中心餐饮数字化中台**，核心价值：

- **对购物中心**：统一管理所有餐饮商户的经营数据，提供跨商户会员体系、联合营销能力
- **对商户**：提供开箱即用的全渠道收单、生产调度、结算对账能力，替代传统 POS 孤岛
- **对顾客**：统一的购物中心小程序入口，跨商户积分、优惠券、历史订单

### 2.2 前台在整体架构中的位置

```
┌─────────────────────────────────────────────────────────────┐
│                        前台服务层                            │
│  顾客侧          门店执行侧              管理侧              │
│  小程序/H5/      POS/PDA/KDS/           商户后台/           │
│  自助机/叫号屏   打包台/打印机           平台后台            │
└──────────────────────────┬──────────────────────────────────┘
                           │ REST API / WebSocket
┌──────────────────────────▼──────────────────────────────────┐
│                      后台服务层（BFF + 微服务）               │
│  交易域  商品域  履约域  渠道域  结算域  租户域  运营域       │
└─────────────────────────────────────────────────────────────┘
```

前台服务层是**业务状态的消费者**，所有持久化业务状态以后台领域模型为准。前台可持有以下三类本地态：

| 本地态类型 | 示例 | 说明 |
|-----------|------|------|
| 会话态 | 当前桌台上下文、当前操作员 | 登录/扫码后建立，退出/关闭后清除 |
| 草稿态 | POS 收银草稿单（未提交） | 仅存内存，不落库，不影响业务状态 |
| 缓存态 | 菜单数据、商户信息 | 有 TTL，以后台推送失效为准 |

前台的职责是：
1. 将业务意图转化为 API 调用
2. 将后台状态渲染为用户界面
3. 处理网络异常、设备异常的本地降级

## 四、核心设计原则

### 原则一：后台是唯一真相源

所有业务状态（订单状态、库存、菜单）以后台领域模型为准。前台**不得**在本地维护业务状态副本并以此做决策。

**实践要求：**
- 订单状态展示必须来自 API 返回，不得依赖本地推断
- 库存判断必须调用后台可售性接口，不得前端自行计算
- 支付结果必须以后台回调为准，不得以支付 SDK 回调为最终态

### 原则二：操作必须显式确认

涉及金额、退款、取消、改单的操作，必须有明确的二次确认步骤，且确认信息中需展示变更摘要。

**实践要求：**
- 退款操作：展示退款金额、退款原因、影响的订单项
- 取消订单：展示已生产工单状态（是否已开始制作）
- 改单加菜：展示新增金额、当前支付状态

### 原则三：异常是主流程的一部分

网络超时、支付失败、设备离线不是边缘情况，是高峰期的常态。每个操作都必须有明确的异常处理路径。

**实践要求：**
- 所有提交操作必须有 loading 状态和超时提示
- 支付类操作必须有"查询支付结果"的补偿入口
- 设备离线时必须有本地缓存或降级提示

### 原则四：高峰期优先

门店高峰期（午餐 11:30-13:30，晚餐 17:30-20:30）是设计的压力场景。

**实践要求：**
- 核心操作路径（点餐→支付→接单）不超过 3 步
- 列表页必须支持分页/虚拟滚动，不得一次性加载全量数据
- KDS、POS 的关键操作必须支持快捷键/手势，减少触屏次数

### 原则五：操作可审计

所有涉及金额变动、订单状态变更的操作，必须记录操作人、操作时间、操作前后状态。

**实践要求：**
- 退款、改单、取消操作需传递操作员 ID
- 前台日志需记录关键操作的请求/响应摘要
- 异常操作需在商户后台可查询

## 五、通用技术规范

### 5.1 API 调用规范

> **唯一契约来源：** 所有前台终端的接口规范以 `final-后台设计-claude/11-前台业务端API详细设计.md` 为准。本节仅列出核心约定摘要，详见文档 16。

**请求头（必须）：**
```
Authorization: Bearer <jwt_token>
Content-Type: application/json
```

> Token 中已包含 tenant_id、store_id、role 等上下文，无需额外请求头。设备标识通过 Token 中的 device_id 字段传递。幂等请求使用 `Idempotency-Key: <uuid_v4>` 请求头。

**统一响应格式：**
```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "timestamp": 1712345678000,
  "traceId": "abc123"
}
```

**分页响应格式：**
```json
{
  "code": 200,
  "data": {
    "items": [],
    "total": 100,
    "pageNum": 1,
    "pageSize": 20,
    "totalPages": 5
  }
}
```

**错误码分段（以后台确认稿为准）：**
| 段 | 范围 | 含义 |
|----|------|------|
| 系统级 | 10000-19999 | 参数错误、未授权、服务不可用 |
| 订单域 | 20000-20999 | 订单不存在、状态无效、金额异常 |
| 支付/退款 | 21000-21999 | 支付失败、退款失败 |
| 商品域 | 22000-22999 | 商品不存在、售罄、不可售 |
| 门店域 | 23000-23999 | 门店不存在、未营业、已停用 |

### 5.2 实时通信规范

前台终端通过 WebSocket 接收后台推送，连接地址：
```
wss://<gateway>/ws/store?token=<jwt>&storeId=<store_id>
```

**消息类型清单（UPPER_SNAKE_CASE）：**
| 消息类型 | 推送时机 | 接收终端 |
|----------|----------|----------|
| `ORDER_NEW` | 新订单到达 | POS、KDS |
| `ORDER_STATUS_CHANGED` | 订单状态变更 | POS、小程序 |
| `WORK_UNIT_STATUS_CHANGED` | 工单状态变更 | KDS、打包台 |
| `PICKUP_CALL` | 叫号触发 | 叫号屏 |
| `TABLE_STATUS_CHANGED` | 桌台状态变更 | PDA、POS |
| `INVENTORY_ALERT` | 库存预警 | POS |

**断线重连策略：**
- 初始重连间隔：1s
- 指数退避，最大间隔：30s
- 重连成功后，主动拉取断线期间的增量数据（通过 `GET /api/v1/events/missed?since=<timestamp>`）

### 5.3 认证与会话

**门店终端（POS/PDA/KDS）：**
- 设备级 Token：设备注册后颁发，长期有效（90天），用于设备身份
- 操作员 Token：员工登录后颁发，短期有效（8小时），用于操作审计
- 双 Token 并存：请求头同时携带设备 Token 和操作员 Token

**顾客侧（小程序/H5）：**
- 微信 openid 换取平台 Token，有效期 2 小时，自动刷新
- 匿名访问：浏览菜单、查看商品不需要登录
- 登录触发点：加入购物车、提交订单、查看历史订单

### 5.4 前端状态管理原则

- **服务端状态**（订单、菜单、库存）：使用 SWR/React Query 管理，设置合理的 staleTime
- **UI 状态**（弹窗、loading、选中项）：使用组件本地 state
- **跨组件共享状态**（购物车、当前桌台、当前操作员）：使用全局 store（Zustand/Pinia）
- **禁止**将服务端状态存入全局 store 后手动维护，必须通过缓存失效触发重新请求

### 5.5 离线与降级策略

| 场景 | 降级策略 |
|------|----------|
| 网络完全断开 | 展示离线提示，禁用提交类操作，允许查看本地缓存的菜单 |
| API 超时（>5s） | 展示超时提示，提供"重试"按钮，不自动重试（避免重复提交） |
| WebSocket 断开 | 自动重连，同时降级为轮询（30s 间隔）获取订单状态 |
| 打印机离线 | 提示打印机离线，允许跳过打印继续流程，记录待打印队列 |
| 支付结果未知 | 展示"支付结果查询中"，提供手动查询入口，禁止重复发起支付 |

## 六、业务场景全景

### 6.1 核心业务链路

**链路一：堂食扫码点餐（最高频）**
```
顾客扫桌台码 → H5/小程序加载菜单 → 选菜加购物车
→ 提交订单 → 在线支付 → 后台接单确认
→ KDS 显示工单 → 厨房制作 → 服务员上菜确认
→ 订单完成 → 结算
```

**链路二：POS 堂食点单（服务员代点）**
```
服务员开桌 → POS/PDA 选菜 → 提交订单
→ 顾客扫码支付 / 现金支付 → 接单
→ KDS 工单 → 制作 → 上菜 → 结算
```

**链路三：外卖订单（三方平台）**
```
美团/饿了么推单 → 渠道集成域接收 → 翻译为内部订单
→ POS 弹出新订单提醒 → 接单/拒单
→ KDS 工单 → 制作 → 打包台核验
→ 骑手取餐 → 完成
```

**链路四：自助点餐（自助机）**
```
顾客选择就餐方式 → 浏览菜单 → 加购
→ 提交 → 扫码/刷脸支付 → 打印取餐号
→ 叫号屏叫号 → 顾客取餐
```

### 6.2 异常场景清单

| 场景 | 触发条件 | 处理终端 | 处理方式 |
|------|----------|----------|----------|
| 支付超时 | 顾客发起支付后 5 分钟未完成 | 小程序/POS | 订单自动关闭，提示重新下单 |
| 支付成功但接单超时 | 支付后 15 分钟未接单 | POS 强提醒 | 自动接单或触发平台介入 |
| 菜品售罄 | 下单时库存为 0 | 小程序/POS | 提示售罄，引导替换 |
| 打印机离线 | 出单时打印机无响应 | POS/打包台 | 记录待打印队列，提示人工处理 |
| 外卖超时未接单 | 三方平台订单 10 分钟未接 | POS 强提醒 | 自动接单（可配置）|
| 改单后支付差额失败 | 加菜后补差价支付失败 | POS | 保留改单记录，标记待收款 |
| 网络断开期间有新订单 | WebSocket 断开后重连 | POS/KDS | 重连后拉取增量，补显工单 |

## 七、非功能性要求

### 7.1 性能指标

| 终端 | 指标 | 目标值 |
|------|------|--------|
| 顾客小程序 | 首屏加载时间（4G） | ≤ 2s |
| 顾客小程序 | 提交订单响应时间 | ≤ 1s |
| POS | 菜单加载时间 | ≤ 1s（本地缓存） |
| POS | 接单操作响应时间 | ≤ 500ms |
| KDS | 新工单推送延迟 | ≤ 3s（P99） |
| 自助机 | 支付完成到出票 | ≤ 2s |

### 7.2 可用性要求

- 顾客侧终端：99.9%（允许每月 43 分钟不可用）
- 门店执行侧：99.95%（允许每月 22 分钟不可用）
- 高峰期（午/晚餐时段）：不得进行发布或维护操作

### 7.3 安全要求

- 所有 API 调用必须携带有效 JWT，无 Token 的请求一律返回 401
- 顾客侧不得暴露其他顾客的订单信息（按 openid 隔离）
- 门店终端操作必须记录操作员 ID，不得以设备身份执行业务操作
- 退款、取消等敏感操作需要操作员具备对应权限（见文档 13）
- 前端不得存储支付卡号、密码等敏感信息

## 八、开发约定

### 8.1 技术栈约定

| 终端类型 | 推荐技术栈 |
|----------|------------|
| 顾客小程序 | uni-app（微信/支付宝双端）或 Taro |
| 堂食 H5 | Vue 3 + Vite，移动端适配 |
| POS 工作台 | React + Electron（桌面）或 React Web（平板） |
| PDA | React Native 或 uni-app |
| KDS/打包台 | React Web（Android 浏览器全屏） |
| 自助机 | React + Electron（Android Kiosk） |
| 商户/平台后台 | React + Ant Design Pro |

### 8.2 组件设计约定

- 业务组件按领域划分目录：`/components/order/`、`/components/menu/`、`/components/payment/`
- 每个终端有独立的路由配置，不共享路由
- 公共 UI 组件（Button、Modal、Toast）抽取为 design-system 包
- API 调用统一封装在 `/services/` 目录，组件不直接调用 fetch/axios

### 8.3 错误处理约定

```typescript
// 统一错误处理模式
try {
  const result = await orderService.submit(payload);
  // 成功处理
} catch (error) {
  if (error.code >= 13000 && error.code < 14000) {
    // 交易域业务错误，展示具体提示
    showBusinessError(error.message);
  } else if (error.code === 10001) {
    // 网络超时，展示重试
    showRetryDialog(() => submitOrder(payload));
  } else {
    // 未知错误，上报并展示通用提示
    reportError(error);
    showGenericError();
  }
}
```

## 九、全局参数定义（规范基准）

> **重要：** 本章节是全文档体系的参数基准。所有终端文档中涉及以下参数时，必须以本章节定义为准，不得自行设定不同值。

### 9.1 支付结果查询策略

适用场景：顾客小程序支付、H5 支付、POS 扫码收款、自助机支付，任何支付发起后结果未知时。

| 参数 | 值 | 说明 |
|------|----|------|
| 首次查询延迟 | 2s | 支付发起后等待 2s 再开始查询 |
| 查询间隔 | 3s | 每次查询间隔 3 秒 |
| 最大查询次数 | 10 次 | 共查询约 30 秒 |
| 超时后处理 | 展示"结果未知"提示，提供手动查询入口 | 不自动重试，不自动关闭订单 |
| 手动查询接口 | `GET /api/v1/orders/{orderId}/payment-status` | — |

```typescript
// 标准支付结果查询实现
async function pollPaymentStatus(orderId: string): Promise<'PAID' | 'UNPAID' | 'UNKNOWN'> {
  await sleep(2000); // 首次延迟
  for (let i = 0; i < 10; i++) {
    const res = await api.get(`/orders/${orderId}/payment-status`);
    if (res.data.status === 'PAID') return 'PAID';
    if (res.data.status === 'UNPAID') return 'UNPAID';
    if (i < 9) await sleep(3000);
  }
  return 'UNKNOWN';
}
```

### 9.2 菜单缓存策略

| 参数 | 值 | 说明 |
|------|----|------|
| staleTime | 5 分钟 | 5 分钟内不重新请求 |
| gcTime | 30 分钟 | 30 分钟后从内存清除 |
| 主动失效触发 | 收到 `MENU_INVALIDATED` WebSocket 事件 | 立即失效，下次访问重新请求 |
| 离线 TTL | 10 分钟 | 未收到 WS 推送时，10 分钟后自动失效 |

> 说明：staleTime=5min 是正常联网状态下的缓存时间；离线 TTL=10min 是 WebSocket 断线时的兜底失效时间，两者不冲突。

### 9.5 KDS 断线降级策略

KDS 设备离线时，后台自动切换该工作站的出单方式：

| 阶段 | 触发条件 | 处理方式 |
|------|----------|----------|
| 离线检测 | KDS 心跳超时（30s + 10s = 40s 无响应） | 后台标记该设备离线 |
| 降级触发 | 离线持续 2 分钟 | 该工作站工单改为厨房打印机出单 |
| 恢复检测 | KDS 重新连接 WebSocket | 后台标记设备在线 |
| 恢复处理 | 恢复后 | 切回 KDS 推送模式；断线期间工单在 KDS 补显（标记"补显"） |

KDS 端本地处理：
- 断线时顶部显示"网络断开，工单可能不完整"警告条
- 重连后主动调用 `GET /api/v1/work-units?since={lastEventTimestamp}` 拉取增量工单
