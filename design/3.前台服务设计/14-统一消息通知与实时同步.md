# 统一消息通知与实时同步详细设计

> 版本：2.0 | 状态：正式稿 | 日期：2026-04-05

---

## 重要说明

**本文档描述终端实时数据分发的整体方案，核心基于终端数据平面服务（TDP）。**

### 架构定位

餐饮中台从第一期开始就采用**终端数据平面服务（Terminal Data Plane, TDP）**作为统一的终端数据分发基础设施，提供：

- 统一的 Projection Store 与 Revision 机制
- 可靠的增量追平与全量对账
- L1/L2 联邦架构支持边缘接入
- 平滑会话迁移与降级容错
- 统一的 Topic 治理体系

**TDP 完整设计详见 `4.终端数据平面服务/` 目录。**

### 本文档范围

本文档聚焦于**前台终端侧的集成实现**，包括：

1. 终端如何连接 TDP 服务
2. 终端如何处理实时推送
3. 终端如何实现离线恢复
4. 终端侧的 Topic Handler 实现
5. 各终端类型的订阅配置

**后台 TDP 服务的架构、协议、API 设计请参考 `4.终端数据平面服务/` 目录。**

## 一、体系概述

### 1.1 消息通知分类

系统消息分为两大类：

**实时推送（WebSocket）：** 需要终端立即响应的业务事件
- 新订单到达（POS/KDS）
- 订单状态变更（顾客小程序）
- 退款状态变更（POS、顾客小程序）
- 工单推送（KDS/打包台）
- 叫号触发（叫号屏）
- 催单（POS/KDS）
- 预订新增/取消（POS）
- 打印任务失败（POS）
- 渠道推单失败（POS）
- 设备离线告警（POS）

**异步通知（短信/微信模板消息）：** 不需要立即响应，允许延迟
- 顾客支付成功通知
- 顾客订单完成通知
- 顾客预订确认/拒绝/提醒通知
- 商户新订单提醒（备用，WebSocket 断线时）
- 退款到账通知

### 1.2 架构

```
业务领域事件（RocketMQ）
      │
      ▼
Projection Adapter（事件转投影）
      │
      ▼
终端数据平面服务（TDP）
  ├─ L1 中心节点（权威写入、Revision 分配）
  └─ L2 边缘节点（就近接入、边缘推送）
      │
      ├── WebSocket 实时推送 ──► 在线终端（POS/KDS/PDA/KIOSK）
      │
      ├── HTTP 增量追平 ──► 离线恢复终端
      │
      └── HTTP 全量对账 ──► 状态可疑终端

异步通知（独立通道）
      │
      ├── 微信模板消息 ──► 顾客微信
      │
      └── 短信 ──► 商户手机
```

**说明：**
- **实时推送**：通过 TDP 的 WebSocket 连接推送到企业终端（POS/KDS/PDA 等）
- **异步通知**：通过微信/短信推送到顾客和商户（不经过 TDP）

## 二、终端实时推送设计（基于 TDP）

### 2.1 连接管理

**连接地址：**
```
wss://<l2-gateway>/tdp/ws/connect?terminalId={id}&token={jwt}
```

**握手消息（客户端 → TDP）：**
```json
{
  "type": "HANDSHAKE",
  "data": {
    "terminalId": "pos-001",
    "tenantId": "t-001",
    "orgScopeId": "store:s-001",
    "profileScopeId": "device-model:pos-standard",
    "appVersion": "2.1.0",
    "lastCursor": 10020,
    "capabilities": ["print", "scan"],
    "subscribedTopics": [
      "menu.catalog",
      "menu.availability",
      "order.pending",
      "store.config",
      "config.system-params",
      "table.status",
      "booking.today",
      "order.urge",
      "print.command",
      "remote.control"
    ]
  }
}
```

**握手响应（TDP → 客户端）：**
```json
{
  "type": "SESSION_READY",
  "data": {
    "sessionId": "sess_xxx",
    "nodeId": "l2-beijing-01",
    "nodeState": "healthy",
    "highWatermark": 10042,
    "syncMode": "incremental",
    "alternativeEndpoints": [
      "wss://l2-beijing-02.example.com/tdp/ws/connect",
      "wss://l1-central.example.com/tdp/ws/connect"
    ]
  }
}
```

**syncMode 说明：**
- `incremental`：增量追平（lastCursor 有效且未过期）
- `full`：全量对账（首次连接或 lastCursor 过期）

### 2.2 心跳机制

**客户端 → TDP（每 30 秒）：**
```json
{ "type": "PING" }
```

**TDP → 客户端：**
```json
{ "type": "PONG" }
```

**心跳超时处理：**
- 超时阈值：60 秒
- 超时后触发重连
- 记录 `heartbeatTimeoutCount`

**实现示例：**
```typescript
class WsHeartbeat {
  private timer: ReturnType<typeof setInterval> | null = null;
  private pongTimeout: ReturnType<typeof setTimeout> | null = null;

  start(ws: WebSocket, onDead: () => void) {
    this.timer = setInterval(() => {
      ws.send(JSON.stringify({ type: 'PING' }));
      this.pongTimeout = setTimeout(() => {
        // 60s 未收到 PONG，判定断线
        ws.close();
        onDead();
      }, 60_000);
    }, 30_000); // 每 30s 发送一次 PING
  }

  onPong() {
    if (this.pongTimeout) clearTimeout(this.pongTimeout);
  }

  stop() {
    if (this.timer) clearInterval(this.timer);
    if (this.pongTimeout) clearTimeout(this.pongTimeout);
  }
}
}
```

服务端收到 `PING` 消息后，立即回复 `{ "type": "PONG" }`。客户端在 `onmessage` 中检测到 `PONG` 时调用 `heartbeat.onPong()`。

### 2.3 消息格式

所有 WebSocket 消息统一格式：
```typescript
interface WsMessage {
  type: string;          // 消息类型
  eventId: string;       // 唯一事件 ID（用于去重）
  timestamp: number;     // 服务端时间戳
  data: any;             // 业务数据
}
```

### 2.4 客户端处理规范

```typescript
// 统一消息处理器
class WsMessageHandler {
  private processedEvents = new Set<string>(); // 去重

  handle(message: WsMessage) {
    // 去重：同一 eventId 只处理一次
    if (this.processedEvents.has(message.eventId)) return;
    this.processedEvents.add(message.eventId);

    // 清理过期 eventId（保留最近1000条）
    if (this.processedEvents.size > 1000) {
      const oldest = [...this.processedEvents].slice(0, 100);
      oldest.forEach(id => this.processedEvents.delete(id));
    }

    switch (message.type) {
      case 'ORDER_NEW':
        this.handleNewOrder(message.data);
        break;
      case 'MENU_INVALIDATED':
        // 使菜单缓存失效，触发重新请求
        queryClient.invalidateQueries(['menu', message.data.storeId]);
        break;
      // ...
    }
  }
}
```

### 2.5 断线重连策略

```typescript
class WsConnection {
  private retryCount = 0;
  private maxRetry = 10;

  connect() {
    this.ws = new WebSocket(url);
    this.ws.onclose = () => this.scheduleReconnect();
  }

  scheduleReconnect() {
    if (this.retryCount >= this.maxRetry) {
      // 降级为轮询
      this.startPolling();
      return;
    }
    // 指数退避：1s, 2s, 4s, 8s, ... 最大 30s
    const delay = Math.min(1000 * Math.pow(2, this.retryCount), 30000);
    this.retryCount++;
    setTimeout(() => this.connect(), delay);
  }

  onReconnected() {
    this.retryCount = 0;
    // 拉取断线期间的增量数据
    this.fetchMissedEvents();
  }

  fetchMissedEvents() {
    // 用断线前最后一条消息的 timestamp 作为 since 参数
    api.get(`/api/v1/events/missed?since=${this.lastEventTimestamp}`);
  }
}
```

## 四、增量数据同步

### 4.1 断线补偿接口

```
GET /api/v1/events/missed?since={timestamp}&deviceType={type}&storeId={storeId}
```

返回断线期间该设备类型关心的所有事件，前端按事件类型分别处理。

### 4.2 菜单变更同步

菜单发布后，所有终端需要刷新菜单缓存：
1. 后台推送 `MENU_INVALIDATED` 事件
2. 终端收到后，使本地菜单缓存失效
3. 下次进入菜单页时重新请求

对于离线终端（未收到推送），菜单缓存设置 TTL = 10 分钟，超时自动失效。
