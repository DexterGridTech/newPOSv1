## 一、权限模型

### 1.1 RBAC 模型

系统采用 **RBAC（基于角色的访问控制）** 模型：

```
用户（User）
  └── 分配角色（Role）
        └── 包含权限（Permission）
              └── 对应资源操作（Resource + Action）
```

权限粒度到**接口级别**，每个 API 端点对应一个权限点。

### 1.2 角色层级

```
平台层角色（跨所有商户）
├── PLATFORM_SUPER_ADMIN    平台超级管理员
├── PLATFORM_OPERATOR       平台运营
└── PLATFORM_FINANCE        平台财务

商户层角色（限定在商户/门店范围内）
├── BRAND_ADMIN             品牌管理员（管理品牌下所有门店）
├── STORE_MANAGER           门店店长
├── CASHIER                 收银员
├── WAITER                  服务员
├── KITCHEN_STAFF           厨房员工
└── PACKER                  打包员
```

### 1.3 数据权限范围

角色除了操作权限，还有数据范围限制：

| 角色 | 数据范围 |
|------|----------|
| PLATFORM_SUPER_ADMIN | 全平台所有数据 |
| PLATFORM_OPERATOR | 全平台，部分敏感数据只读 |
| PLATFORM_FINANCE | 全平台结算数据 |
| BRAND_ADMIN | 品牌下所有门店数据 |
| STORE_MANAGER | 本门店数据 |
| CASHIER | 本门店当班数据 |
| WAITER | 本门店桌台/订单（只读） |
| KITCHEN_STAFF | 本门店工单（只读） |
| PACKER | 本门店打包工单 |

## 三、敏感操作二次授权

以下操作需要**额外授权**（店长扫码或输入授权码）：

| 操作 | 触发条件 | 授权方式 |
|------|----------|----------|
| 大额退款 | 退款金额 > ¥200（默认，可配置） | 店长授权码（见文档 04 §4.4） |
| 免费加菜 | 加菜不收费 | 店长扫码授权 |
| 手动折扣 | 给订单打折 | 店长扫码授权 |
| 删除员工 | — | 品牌管理员操作 |
| 修改费率 | — | 平台超管操作 |

**授权码机制：**
```
收银员发起需授权操作
  → 弹出授权框："此操作需要店长授权"
  → 店长扫描自己的工牌二维码（或输入6位授权码）
  → 后台验证店长身份和权限
  → 授权成功，操作继续执行
  → 记录授权人和被授权操作到审计日志
```

## 四、Token 与会话管理

### 4.1 Token 类型

| Token 类型 | 有效期 | 用途 | 存储位置 |
|------------|--------|------|----------|
| 设备 Token | 90 天 | 设备身份认证 | 设备本地存储 |
| 操作员 Token | 8 小时 | 员工操作审计 | 内存（不持久化） |
| 顾客 Token | 2 小时（自动刷新） | 顾客身份 | 小程序 storage |
| 平台管理员 Token | 4 小时 | 后台管理 | 浏览器 sessionStorage |

### 4.2 设备注册流程

```
门店管理员在商户后台创建设备
  → 生成设备注册码（6位，有效期24小时）
  → 在设备上输入注册码
  → 设备调用注册接口，获取设备 Token
  → 设备 Token 存储在本地，后续请求携带
```

### 4.3 操作员登录

POS/PDA 上的员工登录：
```
员工扫描工牌二维码（或输入工号+密码）
  → POST /api/v1/auth/staff-login
    { deviceToken, staffId, credential }
  → 返回操作员 Token（8小时有效）
  → 后续请求同时携带设备 Token 和操作员 Token
  → 8小时后自动退出，需重新登录
```
