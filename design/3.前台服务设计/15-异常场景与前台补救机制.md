# 异常场景与前台补救机制详细设计

> 版本：1.0 | 状态：正式稿 | 日期：2026-04-05

---

## 一、异常分类

系统异常分为四类：

| 类别 | 示例 | 影响范围 | 处理优先级 |
|------|------|----------|------------|
| 支付异常 | 支付超时、结果未知、重复扣款 | 顾客资金 | P0 |
| 网络/设备异常 | WebSocket 断线、打印机离线 | 门店操作 | P1 |
| 业务状态异常 | 订单状态不一致、工单丢失 | 履约准确性 | P1 |
| 外部渠道异常 | 美团推单失败、骑手取消 | 外卖订单 | P2 |

## 二、支付异常处理

### 2.1 支付结果未知

**触发场景：** 顾客发起支付后，网络中断，前端未收到支付回调。

**处理流程：**
```
支付 SDK 回调超时（10s）
  → 展示"支付结果确认中"页面
  → 前端轮询支付状态（每3s，最多10次）
    GET /api/v1/orders/{orderId}/payment-status
  → 查询结果：
    ├── PAID：跳转支付成功页，正常流程
    ├── UNPAID：展示"支付未完成"，提供重新支付入口
    └── 仍未知（10次后）：
          展示"请在订单列表查看支付结果"
          提供"联系商家"入口
```

**前端状态机：**
```typescript
type PaymentCheckState =
  | 'CHECKING'      // 轮询中
  | 'CONFIRMED_PAID'
  | 'CONFIRMED_UNPAID'
  | 'UNKNOWN';      // 超过重试次数仍未知

// 禁止在 CHECKING 状态下重新发起支付
// 禁止在 UNKNOWN 状态下自动重试
```

### 2.2 重复支付

**触发场景：** 顾客支付成功，但因网络问题以为失败，再次支付。

**防护机制：**
- 订单状态为 `PAID` 时，支付按钮不可点击
- 后台幂等校验：同一 `clientOrderId` 只创建一个订单
- 如确实发生重复扣款，后台自动退还多余款项，并推送通知

### 2.3 POS 扫码收款超时

**触发场景：** 收银员扫描顾客付款码后，支付平台响应超时。

```
扫码后 30s 无结果
  → 展示"支付结果查询中"
  → 后台异步查询支付结果（最多查询 5 分钟）
  → 查询到结果后，通过 WebSocket 推送给 POS
  → POS 展示最终结果
  → 若 5 分钟后仍未知：
      展示"支付结果未知，请手动核查"
      提供"标记为已收款"（需店长授权）
      提供"取消此次收款"
```

## 三、网络与设备异常

### 3.1 POS 网络断开

| 功能 | 断网处理 |
|------|----------|
| 查看已有订单 | 展示本地缓存（最后同步状态） |
| 新建订单 | 禁用，提示"网络断开，无法新建订单" |
| 接单/拒单 | 禁用，提示"网络断开，无法操作" |
| 打印 | 本地打印机仍可打印（通过本地 Agent） |

断网期间顶部显示红色提示条："网络已断开，部分功能不可用"。

恢复后：
1. 自动重连 WebSocket
2. 拉取断线期间的增量事件
3. 刷新订单列表
4. 提示条消失

### 3.2 打印机离线

**检测机制：** 本地 Agent 每 30s 心跳检测，离线后推送 `PRINTER_STATUS_CHANGED` 事件。

**处理策略：**
```
打印任务创建时，打印机离线
  → 任务进入待打印队列（本地 Agent 持久化）
  → 前端展示"打印机离线，已加入待打印队列"
  → 打印机恢复后，Agent 自动按顺序补打
  → 超过 2 小时的厨房工单不补打（已过时）
  → 顾客小票、退款凭证始终补打
```

**手动处理：**
- POS 端可查看待打印队列
- 可手动触发补打或清除队列

### 3.3 KDS 断线

KDS 断线期间，新工单通过厨房打印机出单（降级方案）：
- 后台检测到 KDS 设备离线超过 2 分钟
- 自动将该工作站的工单改为打印机出单模式
- KDS 重连后，自动恢复推送模式
- 断线期间的工单在 KDS 重连后补显（标记为"补显"）

## 四、业务状态异常

### 4.1 订单状态不一致

**触发场景：** 前端展示的订单状态与后台实际状态不符（通常因缓存问题）。

**处理：**
- 所有订单状态展示必须来自 API，不依赖本地推断
- 订单详情页每次打开时强制刷新（不使用缓存）
- 发现状态异常时，提供"刷新"按钮

### 4.2 工单丢失

**触发场景：** 订单已接单，但 KDS 未收到工单推送。

**补救：**
- KDS 重连后自动拉取当前所有进行中工单
- POS 端可手动触发"重新推送工单"
  ```
  POST /api/v1/orders/{orderId}/resend-work-units
  ```
- 商户后台可查看工单推送记录，手动补推

### 4.3 外卖订单接单后骑手取消

```
骑手取消配送
  → 后台收到三方平台回调
  → 推送 WebSocket 事件给 POS：
    { type: 'RIDER_CANCELLED', orderId, message: '骑手取消，请重新呼叫' }
  → POS 展示弹窗提醒
  → 收银员操作：
    ├── 重新呼叫骑手（POST /api/v1/orders/{orderId}/recall-rider）
    └── 联系顾客改为自取
```

## 五、异常操作审计

所有异常处理操作必须记录：

```typescript
interface ExceptionRecord {
  recordId: string;
  orderId: string;
  exceptionType: string;     // 'PAYMENT_UNKNOWN' | 'PRINTER_OFFLINE' | ...
  description: string;
  handledBy: string;         // 操作员 ID
  handledAt: string;
  resolution: string;        // 处理结果描述
  requiresFollowUp: boolean; // 是否需要后续跟进
}
```

异常记录可在商户后台"异常记录"页查询，平台后台可查看全平台异常统计。

## 六、前端错误上报

所有未预期的前端错误需上报：

```typescript
// 全局错误捕获
window.addEventListener('unhandledrejection', (event) => {
  reportError({
    type: 'UNHANDLED_PROMISE',
    message: event.reason?.message,
    stack: event.reason?.stack,
    url: window.location.href,
    deviceId: getDeviceId(),
    userId: getCurrentUserId(),
    timestamp: Date.now(),
  });
});

// API 错误上报（5xx 错误）
axios.interceptors.response.use(null, (error) => {
  if (error.response?.status >= 500) {
    reportError({
      type: 'API_ERROR',
      status: error.response.status,
      url: error.config.url,
      traceId: error.response.data?.traceId,
    });
  }
  return Promise.reject(error);
});
```
