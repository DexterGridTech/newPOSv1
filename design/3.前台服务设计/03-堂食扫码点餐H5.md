# 堂食扫码点餐 H5 详细设计

> 版本：1.0 | 状态：正式稿 | 日期：2026-04-05

---

## 一、产品定位与边界

### 1.1 定位

堂食扫码点餐 H5 是**顾客到店堂食时，通过扫描桌台二维码进入的轻量化点餐入口**。

与顾客小程序的区别：
- H5 是**单次会话**，专注于当前桌台的点餐流程，无需下载安装
- 小程序是**持久化入口**，支持历史订单、会员中心等完整功能
- H5 运行在微信/支付宝内置浏览器中，通过 JSAPI 完成支付
- H5 可以引导用户关注小程序（非强制）

### 1.2 业务边界

**H5 负责：**
- 桌台识别与就餐会话建立
- 菜单浏览与商品选择
- 购物车管理
- 订单提交与 JSAPI 支付
- 订单状态跟踪（当前桌台订单）
- 加菜（已支付后继续点餐）

**H5 不负责：**
- 会员中心（引导至小程序）
- 历史订单（引导至小程序）
- 外卖/自取（仅堂食场景）

### 1.3 H5 与小程序功能对比

| 功能 | 堂食 H5 | 顾客小程序 | 说明 |
|------|---------|------------|------|
| 菜单浏览 | ✓ | ✓ | — |
| 堂食点餐 | ✓ | ✓ | — |
| 外卖/自取点餐 | — | ✓ | H5 仅限堂食 |
| JSAPI 支付 | ✓ | ✓ | 均在微信/支付宝内完成 |
| 加菜（已支付后） | ✓ | ✓ | — |
| 订单状态跟踪 | ✓（当前桌台） | ✓（全部历史） | H5 仅当次会话 |
| 历史订单 | — | ✓ | 引导至小程序 |
| 会员中心/积分 | — | ✓ | 引导至小程序 |
| 优惠券领取 | — | ✓ | H5 可使用已有券 |
| 催单 | ✓ | ✓ | — |
| 登录要求 | 匿名可浏览，支付时授权 | 同左 | — |
| 安装要求 | 无需安装 | 需关注/安装小程序 | H5 更低门槛 |

## 二、页面流程

### 2.1 完整流程

```
扫描桌台二维码
      │
      ▼
解析二维码参数
(storeId, tableCode, scene=DINE_IN)
      │
      ├── 参数无效 ──► 错误页（二维码已失效）
      │
      ▼
检查桌台会话状态
GET /api/v1/table-sessions/active?tableCode={tableCode}
      │
      ├── 有进行中会话 ──► 提示"加入当前桌台订单？"
      │                        │
      │                        ├── 确认 ──► 加入会话，进入菜单页
      │                        └── 取消 ──► 停留在提示页
      │
      ▼
获取微信/支付宝用户身份（静默授权）
      │
      ▼
进入菜单页（主页面）
      │
      ▼
选菜 → 购物车 → 提交订单 → 支付 → 订单跟踪
      │
      └── 已有订单时：加菜流程（追加到已有订单）
```

### 2.2 页面清单

| 页面 | 路径 | 说明 |
|------|------|------|
| 菜单页 | `/menu` | 主页面，选菜 + 购物车 |
| 规格选择弹层 | 菜单页内弹层 | 选择商品规格 |
| 购物车弹层 | 菜单页内弹层 | 查看/修改购物车 |
| 订单确认页 | `/order/confirm` | 确认订单信息 |
| 支付结果页 | `/order/result` | 支付成功/失败 |
| 订单跟踪页 | `/order/track` | 当前桌台订单状态 |
| 加菜确认页 | `/order/add-items` | 追加菜品确认 |
| 错误页 | `/error` | 二维码失效、门店关闭等 |

## 三、核心页面详细设计

### 3.1 菜单页（主页面）

H5 的核心页面，承载 90% 的用户操作时间。

**布局结构：**
```
┌─────────────────────────────┐
│ [商户名] [桌台:A05] [营业中] │  ← 顶部状态栏（固定）
├──────────┬──────────────────┤
│          │  [分类标题]       │
│  分类    │  ┌─────────────┐ │
│  导航    │  │[图] 商品名   │ │
│  （左侧  │  │ 描述文字     │ │
│   固定）  │  │ ¥38  [+]    │ │
│          │  └─────────────┘ │
│  > 热销  │  ┌─────────────┐ │
│   招牌菜 │  │[图] 商品名   │ │
│   主食   │  │ 售罄        │ │
│   小吃   │  └─────────────┘ │
│   饮品   │                  │
│          │                  │
├──────────┴──────────────────┤
│  [购物车图标] 已选3件  ¥114  │  ← 底部购物车栏（固定）
│                   [去结算]  │
└─────────────────────────────┘
```

**分类导航联动：**
- 左侧分类列表固定，右侧商品列表滚动
- 右侧滚动时，左侧自动高亮当前分类
- 点击左侧分类，右侧平滑滚动到对应位置
- 使用 IntersectionObserver 实现，不用 scroll 事件监听

**商品列表渲染：**
- 虚拟滚动（商品数量 > 50 时启用）
- 图片懒加载，使用 IntersectionObserver
- 商品图片统一 1:1 比例，WebP 格式

**顶部状态栏信息：**
```typescript
interface TableSessionInfo {
  storeId: string;
  storeName: string;
  tableCode: string;          // "A05"
  tableLabel: string;         // "A区 05号桌"
  sessionId: string;
  status: 'ACTIVE' | 'ORDERING'; // 就餐中
  guestCount?: number;        // 就餐人数（可选填）
  existingOrders: {           // 当前桌台已有订单
    orderId: string;
    status: string;
    totalAmount: number;
  }[];
}
```

**已有订单时的加菜入口：**
- 顶部状态栏显示"查看订单"按钮
- 底部购物车栏显示"加菜"标签（区分首次点餐和加菜）
- 加菜提交时走加菜接口，不是新建订单

### 3.4 订单确认页

**布局：**
```
┌─────────────────────────────┐
│  确认订单                    │
├─────────────────────────────┤
│  🍽 堂食  A区 05号桌         │
├─────────────────────────────┤
│  商品清单                   │
│  红烧肉（中份/微辣）× 1  ¥38│
│  白米饭 × 2              ¥6 │
├─────────────────────────────┤
│  优惠券  [暂不使用 >]        │
│  备注    [有什么要备注的...]  │
├─────────────────────────────┤
│  合计  ¥44                  │
├─────────────────────────────┤
│  [微信支付  ¥44]             │
└─────────────────────────────┘
```

**支付方式：**
- H5 内只支持当前容器的支付方式
  - 微信内置浏览器：微信支付（JSAPI）
  - 支付宝内置浏览器：支付宝支付（JSAPI）
  - 其他浏览器：展示二维码让顾客扫码支付

**提交订单：**
```typescript
// 提交订单（与后台 CreateOrderRequest 字段对齐）
const submitOrder = async () => {
  const payload = {
    storeId,
    channelType: 'H5',                    // H5 扫码场景（canonical 渠道类型）
    orderScene: 'DINE_IN',                // 三维度：消费场景
    fulfillmentType: 'TABLE_SERVICE',     // 三维度：履约方式
    tableCode,
    sessionId,                            // 桌台会话 ID
    items: cart.items.map(item => ({
      productId: item.productId,
      skuId: item.skuId,
      quantity: item.quantity,
      specSnapshot: item.specs,
      unitPrice: item.unitPrice,
    })),
    couponId: cart.selectedCouponId,
    remark: cart.remark,
    paymentMethod: detectPaymentMethod(), // 根据 UA 判断
    clientOrderId: generateUUID(),        // 幂等键，对应 Idempotency-Key 请求头
  };

  const res = await orderApi.submit(payload);
  if (res.code === 200) {                 // 统一响应码：200 表示成功
    await invokeJsapiPay(res.data.paymentParams);
  }
};
```

### 3.5 订单跟踪页

支付成功后自动跳转，也可从顶部"查看订单"入口进入。

**布局：**
```
┌─────────────────────────────┐
│  A区 05号桌                  │
├─────────────────────────────┤
│  订单状态：制作中 🍳          │
│  ████████░░░░░░░░░░         │
│  下单 → 支付 → 接单 → 制作中 │
├─────────────────────────────┤
│  订单 #20240405001           │
│  红烧肉（中份/微辣）× 1  ¥38│
│  白米饭 × 2              ¥6 │
│  实付 ¥44                   │
├─────────────────────────────┤
│  [催单]  [加菜]              │
└─────────────────────────────┘
```

**实时状态更新：**
- 通过 WebSocket 订阅订单状态变更
- H5 中 WebSocket 连接在页面可见时保持，切后台时断开
- 重新进入页面时，先请求最新状态，再重建 WebSocket 连接

**状态变更动效：**
- 状态推进时，进度条动画过渡
- 新状态到达时，顶部 Toast 提示（如"商家已接单，正在制作中"）

### 3.6 加菜流程

已有进行中订单时，顾客可以继续点餐追加菜品。

**触发条件：**
- 当前桌台有状态为 `ACCEPTED` 或 `IN_PRODUCTION` 的订单
- 顾客在菜单页选菜后点击"去结算"

**加菜确认页：**
```
┌─────────────────────────────┐
│  加菜确认                    │
├─────────────────────────────┤
│  追加菜品                   │
│  可乐 × 2               ¥12│
├─────────────────────────────┤
│  原订单（已支付）            │
│  红烧肉 × 1             ¥38│
│  白米饭 × 2              ¥6│
├─────────────────────────────┤
│  本次追加  ¥12              │
│  [微信支付  ¥12]            │
└─────────────────────────────┘
```

**加菜 API：**

> **注意：** 堂食加菜的业务语义是"同一桌台会话下创建新增量订单"，与小程序保持一致。不使用 `add-items` 补丁接口，而是携带 `sessionId` 创建新订单。

```
POST /api/v1/orders
Idempotency-Key: {clientOrderId}
{
  "storeId": "...",
  "channelType": "H5",
  "orderScene": "DINE_IN",
  "fulfillmentType": "TABLE_SERVICE",
  "sessionId": "{当前桌台会话ID}",
  "paymentMethod": "WECHAT_PAY",
  "clientOrderId": "cli_uuid_002"
}
```

## 四、性能优化要求

### 4.1 首屏性能目标

| 指标 | 目标 | 说明 |
|------|------|------|
| FCP（首次内容绘制） | ≤ 1.5s | 4G 网络 |
| LCP（最大内容绘制） | ≤ 2.5s | 4G 网络 |
| TTI（可交互时间） | ≤ 3s | 4G 网络 |
| 菜单数据加载 | ≤ 1s | WiFi 网络 |

### 4.2 优化措施

**资源优化：**
- JS Bundle 分包，核心包 < 200KB（gzip）
- 图片全部 WebP，缩略图 200×200，详情图 800×800
- 字体使用系统字体，不引入自定义字体

**数据预加载：**
- 扫码后立即并行请求：桌台会话状态 + 菜单数据 + 用户身份
- 菜单数据缓存到 sessionStorage，同一会话内不重复请求

**渲染优化：**
- 商品列表使用虚拟滚动（vue-virtual-scroller）
- 分类切换时不重新渲染整个列表，只滚动位置

## 五、特殊场景处理

### 5.1 多人同桌点餐

同一桌台多人同时扫码点餐时：
- 各自独立维护购物车（前端本地）
- 提交订单时，后台合并到同一桌台会话
- 订单跟踪页展示桌台所有订单（不区分谁点的）

### 5.2 二维码过期

桌台二维码有效期为当日营业时间内。过期后：
- 展示错误页："二维码已过期，请联系服务员"
- 不提供重新扫码入口（防止顾客扫到其他桌台的码）

### 5.3 门店已打烊

扫码时门店已关闭：
- 展示错误页："该餐厅今日已打烊，营业时间 10:00-22:00"
- 提供"查看其他餐厅"按钮，跳转小程序商户广场

### 5.4 网络断开

- 菜单页：展示离线提示条，已加载的菜单仍可浏览
- 提交订单时：提示"网络连接失败，请检查网络后重试"，不自动重试
- 支付结果未知：展示查询入口，引导顾客手动查询
