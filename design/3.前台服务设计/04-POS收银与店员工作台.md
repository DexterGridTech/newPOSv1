## 一、产品定位与边界

### 1.1 定位

POS 收银工作台是**门店最核心的操作终端**，承载收银员和店长的全部日常工作。

核心职责：
- 堂食/外带/自取的人工点单与收银
- 接收并处理所有渠道的新订单（在线、三方外卖）
- 订单全生命周期管理（改单、退款、取消）
- 桌台状态管理
- 班次管理与日结

### 1.2 使用角色

| 角色 | 主要操作 | 权限范围 |
|------|----------|----------|
| 收银员 | 点单、收款、接单、改单 | 本班次订单操作 |
| 店长 | 全部收银员操作 + 退款审批 + 报表 | 本店全部 |
| 品牌管理员 | 查看报表、配置管理 | 品牌内所有门店 |

### 1.3 技术形态

- **Web 版**（推荐）：React + Ant Design，运行在 Chrome 浏览器，适配 15-21 寸触摸屏
- **Electron 版**（可选）：封装 Web 版，增加本地打印机直连能力
- 分辨率：1920×1080（主屏），800×480（副屏/顾客展示屏）

```

│  实付金额：¥43.40            │
├─────────────────────────────┤
│  支付方式                   │
│  [微信扫码] [支付宝扫码]     │
│  [现金]     [会员余额]       │
│  [银行卡]   [记账]           │
├─────────────────────────────┤
│  优惠券  [选择优惠券 >]      │
│  折扣    [输入折扣...]       │  ← 需店长权限
│  附加费  [减免服务费]        │  ← 需店长权限
├─────────────────────────────┤
│  [确认收款  ¥43.40]         │
└─────────────────────────────┘
```

**附加费减免流程：**
```
收银员点击"减免服务费"
  → 弹出授权框（需店长授权码，见文档 00 §9.3）
  → 店长授权后，服务费从账单中移除
  → 记录减免原因和授权人到审计日志
```

**扫码收款流程（主流）：**
```
收银员点击"微信扫码"
  → 提示"请顾客出示付款码"
  → 收银员用扫码枪扫描顾客付款码
  → POST /api/v1/payments/scan-pay
    { orderId, authCode, amount }
  → 后台调用微信/支付宝主扫接口
  → 轮询支付结果（每 3s 一次，最多 10 次，见文档 00 §9.1）
  → 支付成功：播放提示音，打印小票，跳转成功页
  → 支付失败：展示失败原因，提供重试入口
```

**现金收款流程：**
```
收银员输入收到现金金额（如 ¥50）
  → 系统显示找零金额（¥6）
  → 点击"确认收款"
  → POST /api/v1/payments/cash-pay
    { orderId, receivedAmount: 50, changeAmount: 6 }
  → 打开钱箱（通过打印机指令）
  → 打印小票
```

### 3.6 收银草稿单（临时暂存）

> **命名说明：** 此处的"草稿暂存"是**前端本地临时草稿**，与后台业务动作 `hang-bill`（桌台会话挂单）不同：
>
> | 对比项 | 收银草稿单 | 桌台会话挂单（hang-bill） |
> |--------|-----------|--------------------------|
> | 是否落库 | 否，仅存前端内存 | 是，后台有记录 |
> | 影响桌台状态 | 否 | 是 |
> | 跨设备恢复 | 否 | 是 |
> | 参与审计 | 否 | 是 |
> | 关闭浏览器后 | 丢失 | 保留 |

收银员可将当前未完成的点单暂存，切换处理其他顾客：
- 最多同时暂存 10 单
- 草稿列表在点单页顶部显示
- 草稿数据存储在前端本地（不提交后台），关闭浏览器后丢失

### 3.7 桌台会话挂单（hang-bill）

挂单是**后台业务动作**，将桌台会话暂时挂起，适用于顾客临时离席、等待补菜等场景。与收银草稿单不同，挂单数据落库，跨设备可恢复。

**触发入口：** 桌台详情页 → 「挂单」按钮（需店长权限或配置开放）

**挂单流程：**
```
收银员点击「挂单」
  → 弹出确认框：「挂起桌台 A05 会话？」
  → 确认：POST /api/v1/table-sessions/{sessionId}/hang-bill
  → 桌台状态变为「挂单中」（橙色虚线边框）
  → 已点菜品保留，待支付金额保留
  → POS 订单列表显示「挂单」标签
```

**恢复挂单：**
```
点击「挂单中」桌台卡片 → 「恢复挂单」
  → POST /api/v1/table-sessions/{sessionId}/resume-hang
  → 桌台状态恢复「就餐中」
  → 进入点单页，显示已挂单菜品
```

### 3.8 预结账（pre-bill）

顾客提出结账意向时，收银员可先打印预结单供顾客核对，再正式收款。

**触发入口：** 桌台详情页 → 「预结」按钮，或订单详情页 → 「打印预结单」

**预结流程：**
```
收银员点击「预结」
  → 系统汇总桌台会话所有订单（含待付加菜单）
  → 展示预结账单预览：
    ┌─────────────────────────────┐
    │  预结账单（非正式收据）       │
    │  A区05号桌  2026-04-05 13:20 │
    ├─────────────────────────────┤
    │  红烧肉×1            ¥38    │
    │  白米饭×2             ¥6    │
    │  可乐×2              ¥12    │
    │  ─────────────────────────  │
    │  小计                ¥56    │
    │  服务费(10%)          ¥5.6  │
    │  预订订金抵扣        -¥50   │
    │  应付合计             ¥11.6 │
    └─────────────────────────────┘
  → [打印预结单]  [直接收款]
  → 打印后桌台状态不变，顾客核对后再正式收款
```

**注意：** 预结单不锁定订单，顾客仍可继续加菜；正式收款时重新计算金额。

## 四、订单管理功能

### 4.0 同桌多单与桌台会话模型

堂食场景下，一张桌台可能产生多个订单（首单 + 多次加菜单）。前台以**桌台会话（TableSession）**为聚合视图：

```
TableSession（桌台会话）
  ├── 首单 SaleOrder #001（已支付 ¥44）
  ├── 加菜单 SaleOrder #002（已支付 ¥12）
  └── 加菜单 SaleOrder #003（待支付 ¥8）
  ─────────────────────────────
  汇总消费：¥64  待收：¥8
```

**桌台汇总账单视图（点击桌台卡片 → "查看账单"）：**
```
┌─────────────────────────────┐
│  A区05号桌  汇总账单         │
│  会话开始：12:30             │
├─────────────────────────────┤
│  首单 #001  12:30  ¥44 已付 │
│  红烧肉×1  白米饭×2          │
├─────────────────────────────┤
│  加菜 #002  13:05  ¥12 已付 │
│  可乐×2                     │
├─────────────────────────────┤
│  加菜 #003  13:20  ¥8  待付 │
│  甜品×1                     │
├─────────────────────────────┤
│  合计消费：¥64               │
│  已付：¥56  待收：¥8         │
├─────────────────────────────┤
│  [收取待付款 ¥8]  [整单退款] │
└─────────────────────────────┘
```

**加菜流程（首单支付后）：**
```
收银员点击"加菜"
  → 进入点单页（加菜模式，显示当前桌台已点菜品）
  → 选择新增菜品
  → 弹出确认框：
    "追加菜品：甜品×1 ¥8"
    [收取差价 ¥8] [免费加菜（需店长授权）]
  → 收取差价：创建新的加菜 SaleOrder，发起支付
  → 免费加菜：创建加菜单，金额为0，记录授权人
  → POST /api/v1/orders（关联 tableSessionId）
```

**加菜补差价支付失败处理：**
```
差价支付失败
  → 加菜单保留，状态为 PENDING_PAYMENT
  → 桌台汇总账单显示"待收 ¥8"
  → 收银员可在顾客离店前再次发起收款
  → 若顾客已离店，标记为"坏账"，记录到异常日志
```

**退菜触发部分退款：**
```
退菜（已支付订单中的菜品）
  → 选择退菜商品和数量
  → 系统计算退款金额
  → 若退款金额 ≤ ¥200：收银员直接操作
  → 若退款金额 > ¥200：需店长授权码（见文档 00 §9.3）
  → POST /api/v1/orders/{orderId}/items/{itemId}/remove
  → 退款自动原路退回
  → 订单主状态不变（仍为 ACCEPTED/IN_PRODUCTION 等）
  → 退款状态独立更新：refundStatus = PARTIAL_REFUNDING → PARTIALLY_REFUNDED
  → 前台展示：订单卡片右上角显示"部分退款"标签
```

**某订单行不可退的原因提示：**
| 原因 | 提示文案 |
|------|----------|
| 已开始制作 | "该菜品已开始制作，不可退" |
| 已完成制作 | "该菜品已制作完成，退菜需人工处理" |
| 已上桌 | "该菜品已上桌，不可退" |
| 已退过 | "该菜品已退款" |




### 4.3 改单流程（加菜/退菜）

**加菜：**
```
点击"加菜"
  → 进入点单页（加菜模式，显示原订单商品）
  → 选择新增菜品
  → 点击"提交加菜"
  → 弹出确认框：
    "追加菜品：可乐×2 ¥12，是否向顾客收取差价？"
    [收取差价 ¥12] [免费加菜（需店长权限）]
  → 收取差价：发起支付流程
  → 免费加菜：直接提交，记录操作员
  → POST /api/v1/orders/{orderId}/add-items
```

**退菜：**
```
点击"退菜"
  → 展示可退菜品列表（已制作完成的不可退）
  → 选择退菜商品和数量
  → 弹出确认框：
    "退菜：白米饭×1 ¥3，退款方式：原路退回"
    [确认退菜]
  → POST /api/v1/orders/{orderId}/items/{itemId}/remove（每件菜品逐项调用）
  → 退款自动发起（原路退回）
```

### 4.4 退款流程

```
点击"退款"
  → 选择退款原因（必填）：
    [顾客不满意] [菜品质量问题] [下单错误] [其他]
  → 选择退款范围：
    [全单退款] [部分退款（选择商品）]
  → 展示退款金额和退款方式
  → 需要店长审批（金额 > 配置阈值时，阈值见文档 00 §9.3）：
    → 弹出授权框，店长有两种授权方式：
      方式一：店长扫描自己工牌上的授权二维码
              二维码内容：staffId + 时间戳 + HMAC 签名（每次扫码生成新码，有效期 60s）
      方式二：店长输入 6 位动态授权码（TOTP，30s 刷新）
    → POST /api/v1/auth/authorize
      { "action": "REFUND", "targetId": orderId, "credential": "<扫码内容或6位码>" }
    → 后台验证店长身份和权限，返回授权 token（单次有效）
    → 携带授权 token 继续提交退款请求
  → 确认提交
  → POST /api/v1/orders/{orderId}/refund
  → 退款成功：更新订单状态，打印退款凭证
```

### 5.2 接单倒计时

三方外卖订单有接单时限（美团/饿了么通常 5 分钟）：
- 倒计时显示在接单按钮旁
- 剩余 1 分钟时，卡片红色闪烁 + 提示音加强
- 超时未接单：自动接单（可配置）或标记为"超时未接"

### 5.3 拒单流程

```
点击"拒单"
  → 选择拒单原因（必填，三方平台要求）：
    [门店已打烊] [商品售罄] [配送范围外] [其他]
  → 确认拒单
  → POST /api/v1/orders/{orderId}/reject
    { reason, reasonCode }
  → 后台通知三方平台
  → 订单状态更新为 REJECTED
```

### 5.4 催单处理

顾客在小程序催单后，POS 端展示：
- 对应订单卡片顶部显示"顾客催单"标签
- 播放催单提示音
- 收银员可点击"已知晓"消除提醒
