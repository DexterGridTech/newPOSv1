# 多端数据裁剪策略

> 版本：1.0 | 状态：正式稿 | 日期：2026-04-05

---

## 一、背景与问题

后台 API 返回的完整业务对象（如 `SaleOrder`、`MenuItem`）字段丰富，但不同终端只需其中一部分：

- **顾客小程序**：关注商品展示、价格、订单状态，不需要内部成本字段
- **KDS**：只需工单相关字段，不需要支付信息
- **PDA**：只需桌台状态和上菜任务，不需要财务明细
- **商户后台**：需要完整报表字段，但不需要实时推送字段

若不裁剪，存在以下问题：
1. 移动端流量浪费（4G 网络下菜单数据过大）
2. 敏感字段（成本价、内部备注）泄露到顾客端
3. 前端状态管理复杂度增加

---

## 二、裁剪策略

### 2.1 场景参数（scene）驱动裁剪

菜单接口通过 `scene` 参数返回不同字段集：

| scene 值 | 使用终端 | 裁剪说明 |
|----------|----------|----------|
| `DINE_IN` | 顾客小程序、堂食 H5 | 返回展示字段，去除成本价、内部标签 |
| `POS` | POS 收银 | 返回完整字段，含库存预警、内部备注 |
| `PDA` | 服务员 PDA | 同 `DINE_IN`，额外含上菜优先级 |
| `KDS` | KDS | 只返回商品名、规格、制作时长，无价格 |
| `KIOSK` | 自助点餐机 | 同 `DINE_IN`，含大图 URL |

**示例：**
```
GET /api/v1/stores/{storeId}/menu?scene=KDS
```

### 2.2 订单对象按终端裁剪

`SaleOrder` 完整对象约 40+ 字段，各终端按需取用：

| 字段组 | 顾客端 | POS | KDS | PDA | 商户后台 |
|--------|--------|-----|-----|-----|----------|
| 基础信息（orderId, orderNo, status） | ✓ | ✓ | ✓ | ✓ | ✓ |
| 商品清单（items） | ✓ | ✓ | ✓ | ✓ | ✓ |
| 支付信息（paymentMethod, paidAmount） | ✓ | ✓ | — | — | ✓ |
| 退款信息（refundStatus, refundAmount） | ✓ | ✓ | — | — | ✓ |
| 桌台信息（tableCode, sessionId） | ✓ | ✓ | ✓ | ✓ | ✓ |
| 渠道信息（channelType, channelOrderNo） | — | ✓ | — | — | ✓ |
| 成本字段（costPrice） | — | — | — | — | ✓（店长+） |
| 操作历史（operationLogs） | — | ✓ | — | — | ✓ |

### 2.3 WebSocket 推送裁剪

推送消息的 `data` 字段按接收终端裁剪，不推送无关字段：

| 消息类型 | 顾客端 data | POS data | KDS data |
|----------|-------------|----------|----------|
| `ORDER_STATUS_CHANGED` | `{ orderId, newStatus }` | `{ orderId, oldStatus, newStatus, tableCode }` | `{ orderId, newStatus }` |
| `WORK_UNIT_NEW` | — | `{ workUnitId, orderNo }` | 完整 `WorkUnit` 对象 |
| `TABLE_STATUS_CHANGED` | — | `{ tableCode, oldStatus, newStatus }` | — |

## 三、实现方式

### 3.1 BFF（Backend for Frontend）层

推荐在 BFF 层（或 API Gateway）实现裁剪，而非在各终端自行过滤：

```
顾客小程序 ──► BFF/顾客端 ──► 后台 API（完整对象）
POS        ──► BFF/POS端  ──► 后台 API（完整对象）
KDS        ──► BFF/KDS端  ──► 后台 API（完整对象）
```

BFF 层职责：
- 字段裁剪（移除无关字段）
- 字段脱敏（手机号、地址）
- 数据聚合（合并多个接口结果）

### 3.2 字段脱敏规则

| 字段 | 顾客端展示 | POS 展示 | 商户后台 |
|------|-----------|----------|----------|
| 手机号 | `138****0001` | `138****0001` | 完整（需权限） |
| 地址 | 完整 | 完整 | 完整 |
| 支付流水号 | 后4位 | 完整 | 完整 |

### 3.3 菜单缓存与版本控制

菜单数据按 `scene` 分别缓存，避免不同终端缓存互相污染：

```
缓存 Key：menu:{storeId}:{scene}:{version}
TTL：5 分钟（后台发布菜单后通过 MENU_INVALIDATED 事件主动失效）
```

## 四、敏感字段保护

以下字段**禁止**出现在顾客端响应中：

| 字段 | 原因 |
|------|------|
| `costPrice`（成本价） | 商业机密 |
| `internalRemark`（内部备注） | 可能含员工操作记录 |
| `staffId`（操作员 ID） | 隐私保护 |
| `channelCommission`（渠道佣金） | 商业机密 |

后台 API 网关应在响应出口统一过滤，不依赖各终端自觉遵守。
