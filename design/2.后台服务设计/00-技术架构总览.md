# 00-技术架构总览

## 1. 文档目的

本文档是餐饮中台后台服务的技术设计总览，基于 DDD 业务分析文档，提供完整的技术架构设计，支持从第一期 1 万家店铺平滑演进到第三期 100 万家店铺的超大规模场景。

### 2.2 架构演进路线

**第一期（1 万家店铺，日均 50 万单）**
- 单数据中心部署
- PostgreSQL + Citus（4 节点）
- Redis Cluster（3 主 3 从）
- RocketMQ（3 节点）
- 不引入搜索引擎和离线分析

**第二期（10 万家店铺，日均 500 万单）**
- 多数据中心部署（华北、华东、华南）
- PostgreSQL + Citus 扩展到 16 节点
- 引入 Elasticsearch（商品搜索、订单搜索）
- 引入 ClickHouse（离线数据分析）
- Redis Cluster 扩展到 10 主 10 从
- RocketMQ 扩展到 10+ 节点

**第三期（100 万家店铺，日均 5000 万单）**
- 全球多数据中心部署
- PostgreSQL + Citus 扩展到 100+ 节点
- 引入数据归档（冷数据到对象存储）
- 引入 Flink（实时流计算）
- 引入 Presto/Trino（跨数据源联邦查询）

### 3.2 微服务划分

**按限界上下文拆分**：每个限界上下文对应一个微服务，确保业务边界清晰

| transaction-service | 订单管理、支付管理、购物车 | SaleOrder, Payment, ShoppingCart | transaction_db | 8082 |
| fulfillment-service | 履约编排、生产计划、交付任务 | FulfillmentOrder, ProductionPlan, DeliveryTask | fulfillment_db | 8083 |
| channel-service | 渠道对接、订单翻译、状态同步 | ChannelShopBinding, ChannelOrder | channel_db | 8084 |
| settlement-service | 结算事实、对账任务、账单生成 | SettlementFact, SettlementCycle, ReconciliationTask | settlement_db | 8085 |
| tenant-service | 租户管理、组织架构、权限管理 | Tenant, Brand, Store, Staff | tenant_db | 8086 |
| booking-service | 预订单管理、餐段配置、订金支付 | BookingOrder, BookingPeriod | booking_db | 8089 |
| external-service | 会员集成、配送集成、ERP集成 | MemberIntegration, DeliveryIntegration | external_db | 8087 |

> **终端数据平面服务（TDP）** 是统一的终端数据分发基础设施，负责接收业务系统的数据变更并推送到各类终端（POS/KDS/PDA/小程序等）。详见 `4.终端数据平面服务/` 目录。

### 3.3 BFF 层（Backend for Frontend）

BFF 层位于 API 网关与领域微服务之间，承担**面向终端的接口聚合与裁剪**职责，使各终端只收到它所需的数据，降低前端适配成本。

**定位示意：**
```
顾客小程序 / POS / KDS / 商户后台
           │ HTTPS / WSS
   ┌───────▼────────┐
   │  Spring Cloud   │  路由、鉴权、限流、熔断
   │  Gateway        │
   └───────┬─────────┘
           │
   ┌───────▼─────────────────────────────┐
   │  BFF 层（内嵌于 API Gateway filter  │
   │  或独立 bff-service，port 8090）    │
   │  ● 字段裁剪（按 scene 参数过滤）    │
   │  ● 字段脱敏（手机号、地址掩码）     │
   │  ● 多接口聚合（减少前端 RTT）       │
   │  ● 终端差异适配（POS vs 小程序）    │
   └───────┬─────────────────────────────┘
           │  内部 Feign / gRPC
   ┌───────▼──────────────────────────────┐
   │  领域微服务层（返回完整领域对象）    │
   │  transaction / product / fulfillment │
   └──────────────────────────────────────┘
```

**第一期部署方案：** BFF 职责以 Spring Cloud Gateway 的自定义 `GatewayFilter` 实现（字段裁剪、脱敏在响应后置过滤器中执行），无需独立进程，降低运维复杂度。

**第二期扩展：** 若聚合逻辑增多（如首页一次性拉取订单 + 菜单 + 桌台），可拆分为独立 `bff-service`（port 8090），仍通过 Gateway 统一入口对外暴露。

**BFF 职责边界（不属于 BFF 的能力）：**
- 业务规则校验 → 留在领域服务
- 数据持久化 → 留在领域服务
- 消息发布 → 留在领域服务

> 详细的字段裁剪规则与脱敏策略见 `3.前台服务设计/21-多端数据裁剪策略.md §3.1`。

### 4.3 分布式事务策略

**Seata AT 模式（仅限内部受控数据库资源）**：
1. **订单创建 + 库存扣减**（order_db + product_db，均为内部 PostgreSQL）

**本地事务 + 可靠消息 + 补偿（涉及外部系统时）**：
1. **订单支付 + 优惠券核销**：本地事务写 outbox，异步通知外部会员系统；会员系统超时/失败走补偿队列
2. **订单取消 + 退款**：本地事务更新订单状态，通过 RocketMQ 事务消息触发支付网关退款；退款结果异步回调，失败进入人工处理队列

> **为什么不用 Seata AT 包住外部系统？** Seata AT 依赖数据库 undo log 回滚，无法协调外部会员系统、支付平台等不受控资源。涉及外部系统的一致性必须通过幂等回调 + 补偿 + 最终一致性实现。

**示例代码（内部 Seata AT）**：
```java
@GlobalTransactional(name = "create-order-deduct-inventory", rollbackFor = Exception.class)
public void createOrderAndDeductInventory(CreateOrderRequest request) {
    // 创建订单（order_db）
    orderService.createOrder(request);
    // 扣减库存（product_db）
    inventoryService.deductInventory(request);
}
```

### 5.2 分片策略

**Citus 分片配置**：

```sql
-- 设置分片键（所有表按 store_id 分片）
SELECT create_distributed_table('sale_orders', 'store_id');
SELECT create_distributed_table('order_items', 'store_id');
SELECT create_distributed_table('payments', 'store_id');
SELECT create_distributed_table('products', 'store_id');
SELECT create_distributed_table('fulfillment_orders', 'store_id');
SELECT create_distributed_table('production_plans', 'store_id');
SELECT create_distributed_table('work_units', 'store_id');

-- 第一期：32个分片，4个节点，每个节点8个分片
ALTER SYSTEM SET citus.shard_count = 32;

-- 第二期：128个分片，16个节点，每个节点8个分片
ALTER SYSTEM SET citus.shard_count = 128;

-- 第三期：1024个分片，128个节点，每个节点8个分片
ALTER SYSTEM SET citus.shard_count = 1024;

-- 设置副本数量
ALTER SYSTEM SET citus.shard_replication_factor = 1;
```

**分片优势**：
1. **数据隔离**：每家门店的数据在同一个分片，查询不跨分片
2. **水平扩展**：增加节点即可扩展容量和性能
3. **故障隔离**：单个分片故障不影响其他分片
4. **负载均衡**：Hash 分片自动均衡负载

**查询优化**：
- **单店查询**：带 `store_id` 的查询直接路由到对应分片，性能极高
- **跨店查询**：平台级查询（如运营报表）通过 Elasticsearch 或 ClickHouse 实现

### 5.5 缓存一致性方案

```
数据库更新
    ↓
PostgreSQL Logical Replication 监听 WAL
    ↓
发送 RocketMQ 消息 (Topic: cache-invalidation)
    ↓
各服务节点消费消息
    ↓
失效本地缓存 (Caffeine)
    ↓
失效 Redis 缓存
    ↓
下次查询时重新加载
```

**代码实现**：

```java
@Service
public class CacheInvalidationService {

    @Autowired
    private Cache<String, Object> localCache;

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @RocketMQMessageListener(
        topic = "cache-invalidation",
        consumerGroup = "cache-invalidation-consumer"
    )
    public class CacheInvalidationListener implements RocketMQListener<CacheInvalidationMessage> {

        @Override
        public void onMessage(CacheInvalidationMessage message) {
            String cacheKey = message.getCacheKey();

            // 失效本地缓存
            localCache.invalidate(cacheKey);

            // 失效 Redis 缓存
            redisTemplate.delete(cacheKey);

            log.info("Cache invalidated: {}", cacheKey);
        }
    }
}
```

### 5.6 数据一致性策略

**强一致性场景**（使用 Seata AT 模式，仅限内部受控数据库）：
- 订单创建 + 库存扣减（order_db + product_db，均为内部受控 DB）

> Seata AT 依赖对数据源的直接 UNDO_LOG 控制，**不得**跨越外部系统边界。支付网关、优惠券服务若属于外部系统或独立服务，必须使用下方的可靠消息 + 补偿方案。

**可靠消息 + 补偿场景**（涉及外部系统或跨服务）：
- 订单支付 + 优惠券核销（支付回调后通过 Outbox 事件驱动核销；核销失败走补偿）
- 订单取消 + 退款（退款请求通过 Outbox 发布，退款结果由支付平台异步回调；失败走对账补偿）

**最终一致性场景**（使用事件驱动）：
- 订单完成 → 积分赠送
- 订单完成 → 结算事实生成
- 订单状态变更 → 外部平台同步

## 11. 读写分离（第二期）

### 11.1 PostgreSQL 主从复制

```
┌─────────────────────────────────────────────────────────┐
│  Citus 协调器节点 (Coordinator)                          │
│  ├─ 主节点 (Master): 写操作                              │
│  └─ 从节点 (Slave): 读操作                               │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Citus 工作节点 (Worker)                                 │
│  ├─ Worker 1: 主节点 + 2个从节点                         │
│  ├─ Worker 2: 主节点 + 2个从节点                         │
│  ├─ Worker 3: 主节点 + 2个从节点                         │
│  └─ Worker 4: 主节点 + 2个从节点                         │
└─────────────────────────────────────────────────────────┘
```

### 11.2 应用层读写分离

```java
@Configuration
public class DataSourceConfig {

    @Bean
    @ConfigurationProperties("spring.datasource.master")
    public DataSource masterDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @ConfigurationProperties("spring.datasource.slave")
    public DataSource slaveDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    public DataSource routingDataSource() {
        Map<Object, Object> targetDataSources = new HashMap<>();
        targetDataSources.put("master", masterDataSource());
        targetDataSources.put("slave", slaveDataSource());

        DynamicRoutingDataSource routingDataSource = new DynamicRoutingDataSource();
        routingDataSource.setTargetDataSources(targetDataSources);
        routingDataSource.setDefaultTargetDataSource(masterDataSource());

        return routingDataSource;
    }
}

@Service
public class OrderService {

    // 写操作：使用主库
    @Transactional
    @DataSource("master")
    public void createOrder(Order order) {
        orderRepository.save(order);
    }

    // 读操作：使用从库
    @Transactional(readOnly = true)
    @DataSource("slave")
    public Order getOrder(String orderId) {
        return orderRepository.findById(orderId);
    }
}
```
