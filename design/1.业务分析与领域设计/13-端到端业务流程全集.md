# 13-端到端业务流程全集

## 1. 堂食扫码点单完整流程

### 1.1 文档目的

详细描述堂食扫码点单场景的完整端到端流程，展示各域如何协作完成从顾客扫码到订单完成的全过程。这是购物中心一方渠道的核心场景。

### 1.2 场景定义

**业务场景：** 顾客在购物中心餐饮门店就餐，扫描桌台二维码进入小程序点单，支付后等待上菜，用餐完毕离开。

**三维度标识：**
```
channel_type: MINI_PROGRAM
order_scene: DINE_IN
fulfillment_type: TABLE_SERVICE
```

**参与角色：** 顾客、门店服务员、门店厨师（各岗位）、购物中心平台系统

### 1.3 前置条件

**门店准备：**
- 门店已在平台完成入驻和审核（租户域）
- 门店菜单已发布且有可售商品（商品域）
- 门店营业状态为 OPEN（商品域）
- 门店已配置桌台信息（商品域）
- 门店厨房工作站已配置（履约域）
- 门店 KDS 设备已绑定（租户域）

**顾客准备：** 顾客已进入门店并就座、手机可扫描二维码、微信已授权小程序

### 1.4 完整流程

#### 阶段一：顾客扫码进入

**步骤 1.1：扫描桌台二维码**

```
顾客扫描桌台二维码
    ↓
小程序启动，解析二维码参数
    ↓
提取参数：store_id / table_id / table_no（如"A01"）
    ↓
调用商品域查询接口
GET /api/v1/stores/{store_id}/menu?table_id={table_id}
```

商品域处理：1.验证门店营业状态 2.验证桌台状态（是否可用）3.查询门店当前可售菜单 4.返回菜单投影（含商品、规格、加料、价格）

**步骤 1.2：顾客浏览菜单**

```
小程序展示菜单 → 顾客选择商品 → 选择规格（中杯/大杯）
    ↓
选择加料（加奶/加糖） → 设置数量 → 加入购物车
```

#### 阶段二：提交订单

**步骤 2.1：顾客提交订单**

```
顾客点击"提交订单"
    ↓
小程序构造订单创建请求
    ↓
POST /api/v1/orders
```

请求载荷：
```json
{
  "channel_type": "MINI_PROGRAM",
  "order_scene": "DINE_IN",
  "fulfillment_type": "TABLE_SERVICE",
  "store_id": "store_123",
  "customer_id": "cust_456",
  "service_context": {"table_id": "table_001", "table_no": "A01", "diner_count": 2},
  "items": [{"product_id": "prod_001", "product_name": "美式咖啡",
    "selected_variant": {"variant_id": "v2", "name": "大杯"},
    "selected_modifiers": [{"modifier_id": "m1", "name": "加奶", "price": 200}],
    "quantity": 2, "unit_price": 3100, "subtotal": 6200}],
  "total_amount": 6200, "remark": "少冰"
}
```

**步骤 2.2：交易域创建订单**

```
交易域接收创建命令
    ↓
1. 生成订单ID和订单号
2. 调用商品域验证商品可售性
3. 创建 SaleOrder 聚合（状态：PENDING_PAYMENT）
4. 持久化订单
5. 发布 OrderCreated 事件
6. 返回订单ID和支付参数
```

OrderCreated 事件消费方：结算域（登记订单，准备生成结算事实）、运营治理域（记录审计日志）

#### 阶段三：支付确认

**步骤 3.1：顾客支付**

```
小程序调起微信支付 → 顾客完成支付 → 微信支付回调平台
    ↓
外部集成域接收支付回调
    ↓
1.验证签名 2.解析支付结果 3.幂等检查（防重复）4.发布 PaymentConfirmed 事件
```

**步骤 3.2：交易域更新订单状态**

```
交易域消费 PaymentConfirmed 事件
    ↓
1. 更新订单状态：PENDING_PAYMENT → PAID
2. 记录支付信息
3. 检查门店自动接单配置
    ↓
if (store.auto_accept_enabled) { 立即执行 AcceptOrder 命令 }
else { 等待门店人工接单 }
```

#### 阶段四：门店接单

**步骤 4.1：自动接单（假设门店开启自动接单）**

```
交易域执行 AcceptOrder 命令
    ↓
1. 更新订单状态：PAID → ACCEPTED
2. 记录接单时间
3. 发布 OrderAccepted 事件
```

OrderAccepted 事件消费方：履约域（触发生产计划生成）、运营治理域（记录审计日志）

#### 阶段五：生产编排

**步骤 5.1：履约域生成生产计划**

```
履约域消费 OrderAccepted 事件
    ↓
1. 创建 FulfillmentOrder 聚合（状态：PLAN_PENDING）
2. 调用生产计划生成服务
3. 查询商品生产画像
4. 拆分工作单，生成 ProductionPlan
```

生产画像示例：
```
product_id: prod_001, primary_station: COFFEE_BAR,
estimated_duration: 180, can_batch: true, dependencies: []
```

工作单拆分：
```
WorkUnit 1: work_unit_id: wu_001, station_type: COFFEE_BAR,
product_name: "美式咖啡（大杯加奶）", quantity: 2,
estimated_duration: 180秒, status: PENDING
```

**步骤 5.2：派发工作单**

```
履约域派发工作单
    ↓
1. 更新 FulfillmentOrder 状态：PLAN_PENDING → PRODUCTION_PLANNED
2. 将工作单推送到 KDS 设备
3. 发布 ProductionPlanCreated 事件（跨域标准事件）
4. 更新状态：PRODUCTION_PLANNED → IN_PRODUCTION
5. 发布 OrderInProduction 事件
```

OrderInProduction 事件消费方：交易域（更新订单状态为 IN_PRODUCTION）

#### 阶段六：厨房制作

**步骤 6.1：厨师接收工作单**

```
KDS 设备显示工作单 → 厨师点击"开始制作"
    ↓
POST /api/work-units/{work_unit_id}/start
    ↓
履约域：更新工作单状态 PENDING → IN_PROGRESS，记录开始时间，发布 WorkUnitStarted 事件
```

**步骤 6.2：厨师完成制作**

```
厨师制作完成 → 点击"完成"
    ↓
POST /api/work-units/{work_unit_id}/complete
    ↓
履约域：更新工作单状态 IN_PROGRESS → COMPLETED，记录完成时间，发布 WorkUnitCompleted 事件
    ↓
检查是否所有工作单都已完成
    ↓
if (所有工作单已完成) {
    更新 FulfillmentOrder 状态：IN_PRODUCTION → PRODUCTION_DONE
    发布 ProductionCompleted 事件
}
```

ProductionCompleted 事件消费方：履约域自身（触发交付编排）

#### 阶段七：交付编排

**步骤 7.1：履约域准备交接**

```
履约域消费 ProductionCompleted 事件
    ↓
根据 fulfillment_type 决定交付方式
    ↓
if (fulfillment_type == TABLE_SERVICE) { 触发堂食上菜流程 }
    ↓
1. 更新 FulfillmentOrder 状态：PRODUCTION_DONE → HANDOFF_READY
2. 发布 HandoffPrepared 事件
3. 推送通知到服务员设备（PDA/小程序）
```

**步骤 7.2：服务员确认上菜**

```
服务员收到通知 → 取餐 → 送到桌台 → 服务员在 PDA 点击"已上菜"
    ↓
POST /api/production-orders/{production_order_id}/handoff-complete
    ↓
履约域：更新 FulfillmentOrder 状态 HANDOFF_READY → FULFILLED，记录交付时间，发布 FulfillmentCompleted 事件
```

FulfillmentCompleted 事件消费方：交易域（更新订单状态为 FULFILLED）

#### 阶段八：订单完成

**步骤 8.1：交易域关闭订单**

```
交易域消费 FulfillmentCompleted 事件
    ↓
1. 更新订单状态：FULFILLED → COMPLETED
2. 记录完成时间
3. 发布 OrderCompleted 事件
```

OrderCompleted 事件消费方：结算域（生成结算事实）、运营治理域（记录审计日志）

#### 阶段九：结算处理

**步骤 9.1：结算域生成结算事实**

```
结算域消费 OrderCompleted 事件
    ↓
1. 提取订单金额信息
2. 查询适用的佣金规则
3. 调用清分计算服务
4. 创建 SettlementFact 聚合
```

清分计算：
```
订单金额：6200分，渠道类型：MINI_PROGRAM（一方小程序），适用规则：一方渠道基础费率 3%
gross_revenue: 6200
platform_commission: 186（6200 * 3%）
channel_commission: 0（一方渠道无渠道佣金）
delivery_fee: 0（堂食无配送费）
payment_fee: 12（6200 * 0.2%）
merchant_receivable: 6002（6200 - 186 - 12）
```

**步骤 9.2：持久化结算事实**

```
结算域持久化 SettlementFact
    ↓
1. 保存清分结果
2. 关联到当前开放的结算周期
3. 发布 SettlementFactCreated 事件
```

### 1.5 关键设计点

**桌台上下文传递：** 二维码参数 → 订单创建请求 → SaleOrder.service_context → OrderAccepted 事件 → FulfillmentOrder.service_context → HandoffPrepared 事件 → 服务员通知

**自动接单策略：**
```
if (channel_type == MINI_PROGRAM && order_scene == DINE_IN) {
    if (store.auto_accept_enabled) { 立即自动接单 }
    else { 等待门店人工接单（超时告警）}
}
```

**生产计划拆分特点：** 可以分批上菜（不同商品可以分开制作）、工作单按岗位拆分（咖啡吧台、甜品吧台等）、支持并行制作（多个岗位同时工作）

**交付确认机制：** 制作完成后通知服务员 → 服务员取餐并送到桌台 → 服务员在 PDA 确认已上菜 → 确认后订单才进入 FULFILLED 状态

### 1.6 异常处理

**支付超时：** 订单创建后 15 分钟未支付 → 定时任务检测到超时 → 交易域执行 CancelOrder 命令 → 更新状态 PENDING_PAYMENT → CANCELLED → 发布 OrderCancelled 事件 → 释放库存（如有）

**生产超时：** 订单进入生产后超过预计时长 → 履约域检测到超时 → 发布 ProductionOverdue 事件 → 运营治理域创建异常记录 → 触发告警通知门店

**顾客取消订单：**
```
if (status == PENDING_PAYMENT || status == PAID) { 允许取消，全额退款 }
else if (status == ACCEPTED || status == IN_PRODUCTION) { 需要门店确认，可能部分退款 }
else { 不允许取消 }
```

### 1.7 性能指标

| 阶段 | 目标时效 |
|---|---|
| 菜单加载 | < 1秒 |
| 订单创建 | < 2秒 |
| 支付确认 | < 5秒 |
| 自动接单 | < 1秒 |
| 工作单派发 | < 3秒 |
| 上菜时长 | < 2分钟 |

成功率要求：订单创建成功率 > 99.9%、支付成功率 > 99.5%、工作单派发成功率 > 99.9%、订单完成率 > 98%

## 2. 三方外卖订单完整流程

### 2.1 文档目的

详细描述三方外卖订单（美团/饿了么）的完整端到端流程，展示渠道集成域如何作为防腐层隔离外部平台差异，以及各域如何协作完成从接收外部订单到配送完成的全过程。

### 2.2 场景定义

**业务场景：** 顾客在美团 App 下单，美团通过 ISV 授权推送订单到购物中心平台，门店接单制作，美团骑手配送。

**三维度标识：**
```
channel_type: MEITUAN
order_scene: DELIVERY_PLATFORM
fulfillment_type: PLATFORM_DELIVERY
```

**参与角色：** 顾客（在美团 App 下单）、门店厨师（各岗位）、美团骑手、美团平台、购物中心平台系统

### 2.3 前置条件

**渠道绑定：** 门店已在美团开店并获得 POI 编码、门店已完成 ISV 授权绑定（渠道集成域）、授权 Token 有效且未过期、商品已同步到美团平台

**商品映射：** 内部商品与美团商品已建立映射关系、商品价格/规格/加料已同步、商品库存状态已同步

### 2.4 完整流程

#### 阶段一：接收外部订单

**步骤 1.1：美团推送订单**

```
顾客在美团 App 下单 → 美团平台生成订单
    ↓
美团通过 ISV 接口推送订单
POST https://platform.mall.com/api/v1/channel/callback/meituan/order
```

**步骤 1.2：渠道集成域接收订单**

```
渠道集成域接收推送
    ↓
1. 验证签名（防伪造）
2. 幂等检查（基于 wm_order_id_view）
    ↓
if (已处理过) { 返回成功响应（幂等）; return }
    ↓
3. 保存原始订单数据（完整保留）
4. 创建 ChannelOrder 聚合（状态：RECEIVED）
5. 记录操作日志
6. 返回接收成功响应给美团（{"data": "ok"}）
7. 异步触发订单解析
```

#### 阶段二：订单解析与翻译

**步骤 2.1：解析美团订单**

```
渠道集成域异步解析订单
    ↓
1. 根据 channel_type 选择美团解析器
2. 提取订单基本信息
3. 翻译订单状态码（2 → 待接单）
4. 解析顾客信息
5. 解析订单行（商品映射）
6. 解析金额信息（元转分）
7. 解析配送地址
8. 解析美团优惠券
9. 生成 unified_order_payload
```

商品映射查询：根据 app_food_code 查询 ChannelProductMapping，external_product_id → internal_product_id

**步骤 2.2：更新解析状态**

```
解析完成 → 更新 ChannelOrder 状态：RECEIVED → PARSED
    ↓
记录解析时间 → 发布 ChannelOrderParsed 事件
```

#### 阶段三：创建内部订单

**步骤 3.1：渠道集成域触发订单创建**

```
渠道集成域消费 ChannelOrderParsed 事件
    ↓
调用交易域创建订单
POST /api/v1/orders-from-channel
```

**步骤 3.2：交易域创建订单**

```
交易域接收创建命令
    ↓
1. 生成订单 ID 和订单号
2. 验证商品可售性
3. 创建 SaleOrder 聚合
   状态：PAID（三方外卖已支付）
   channel_type: MEITUAN / order_scene: DELIVERY_PLATFORM / fulfillment_type: PLATFORM_DELIVERY
4. 持久化订单
5. 发布 OrderCreated 事件
6. 返回订单 ID
```

**步骤 3.3：渠道集成域更新关联**

```
渠道集成域接收订单创建成功响应
    ↓
更新 ChannelOrder: internal_order_id: "order_789", processing_status: ORDER_CREATED
    ↓
发布 ChannelOrderReceived 事件
```

**步骤 3.4：回传接收确认给美团**

```
渠道集成域调用美团接口
POST https://waimai.meituan.com/api/order/confirm
{ "app_poi_code": "store_poi_123", "wm_order_id_view": "1234567890123456" }
    ↓
记录操作日志
```

#### 阶段四：门店接单

**步骤 4.1：等待门店接单**

```
订单进入 PAID 状态
    ↓
if (channel_type == MEITUAN || channel_type == ELEME) {
    // 三方外卖必须人工接单或快速自动接单
    if (store.auto_accept_enabled && store.auto_accept_timeout < 30) {
        延迟自动接单（如 10 秒后）
    } else { 等待人工接单 }
}
    ↓
启动接单超时监控（美团要求 30-90 秒内接单）
```

**步骤 4.2：门店接单**

```
门店在 POS 或小程序点击"接单"
POST /api/v1/orders/{order_id}/accept
    ↓
交易域执行 AcceptOrder 命令
    ↓
更新订单状态：PAID → ACCEPTED → 发布 OrderAccepted 事件
```

**步骤 4.3：回传接单给美团**

```
渠道集成域消费 OrderAccepted 事件
    ↓
调用美团接口回传接单
POST https://waimai.meituan.com/api/order/confirm
{ "app_poi_code": "store_poi_123", "wm_order_id_view": "...", "confirm_type": 2, "expected_delivery_time": ... }
    ↓
记录操作日志
```

#### 阶段五：生产编排

**步骤 5.1：履约域生成生产计划**

```
履约域消费 OrderAccepted 事件
    ↓
创建 FulfillmentOrder 聚合 → 查询商品生产画像 → 拆分工作单
注意：三方外卖需要整单打包，不能分批
```

工作单示例：
```
WorkUnit 1: station_type: COFFEE_BAR, product_name: "美式咖啡（大杯加奶少冰）",
quantity: 2, estimated_duration: 180秒, packaging_required: true
```

**步骤 5.2：派发工作单**

```
派发工作单到 KDS → 更新状态：PLAN_PENDING → IN_PRODUCTION → 发布 OrderInProduction 事件
```

#### 阶段六：厨房制作

**步骤 6.1：厨师制作**

```
厨师在 KDS 看到工作单 → 开始制作 → 完成制作 → 点击"完成"
    ↓
更新工作单状态：COMPLETED → 发布 WorkUnitCompleted 事件
```

**步骤 6.2：检查全部完成**

```
if (所有工作单已完成) {
    更新状态：IN_PRODUCTION → PRODUCTION_DONE
    发布 ProductionCompleted 事件
}
```

**步骤 6.3：回传备餐完成给美团**

```
渠道集成域消费 ProductionCompleted 事件
    ↓
POST https://waimai.meituan.com/api/order/preparing
{ "app_poi_code": "store_poi_123", "wm_order_id_view": "..." }
    ↓
记录操作日志
```

#### 阶段七：等待骑手取餐

**步骤 7.1：履约域准备交接**

```
if (fulfillment_type == PLATFORM_DELIVERY) {
    // 三方外卖，等待平台骑手取餐
    更新状态：PRODUCTION_DONE → HANDOFF_READY
    发布 HandoffPrepared 事件
}
```

**步骤 7.2：美团骑手到店**

```
美团骑手到店 → 门店员工确认取餐码 → 门店员工在 POS 点击"已出餐"
POST /api/production-orders/{production_order_id}/handoff-complete
    ↓
履约域：更新状态 HANDOFF_READY → FULFILLED → 发布 FulfillmentCompleted 事件
```

#### 阶段八：配送与完成

**步骤 8.1：美团骑手配送**

```
骑手取餐后配送 → 骑手送达顾客 → 顾客确认收货
    ↓
美团平台更新订单状态为"已完成"
    ↓
美团通过回调通知平台
POST https://platform.mall.com/api/v1/channel/callback/meituan/status
{ "wm_order_status": 6, "status_change_time": ... }
```

**步骤 8.2：渠道集成域处理状态回调**

```
渠道集成域接收回调 → 验证签名 → 解析状态码（6 → 已完成）
    ↓
发布 ChannelOrderStatusChanged 事件
```

**步骤 8.3：交易域完成订单**

```
if (external_status == "已完成") {
    执行 CloseOrder 命令
    更新状态：FULFILLED → COMPLETED
    发布 OrderCompleted 事件
}
```

### 2.5 关键设计点

**原始数据完整保留：** ChannelOrder.raw_order_data 字段完整保存美团原始 JSON，目的：支持问题排查和对账、支持订单重放和补偿、防止信息丢失

**幂等处理：** 基于 external_order_no 的幂等检查，已存在相同 external_order_no 的 ChannelOrder 则返回成功响应不重复处理

**状态回传重试策略：** 立即重试1次 → 1分钟后重试 → 5分钟后重试 → 15分钟后重试 → 超过最大重试次数后转入异常台账

**接单超时处理：**
```
if (距离订单创建时间 > 30秒 && 订单状态仍为 PAID) {
    触发告警
    if (store.auto_accept_on_timeout) { 自动接单 }
    else { 通知门店紧急处理 }
}
```

### 2.6 异常处理

**商品映射缺失：** 记录错误 PRODUCT_MAPPING_NOT_FOUND → 更新 ChannelOrder 状态 PARSE_FAILED → 创建异常台账 → 通知运营人员建立映射 → 支持手动重试解析

**门店拒单：** 门店在 POS 点击"拒单" → 交易域执行 RejectOrder 命令 → 更新状态 PAID → REJECTED → 发布 OrderRejected 事件 → 渠道集成域消费事件 → 调用美团接口回传拒单

**状态回传失败：** 记录操作日志（状态：FAILED）→ 加入重试队列 → 重试次数 < 5 则按指数退避策略重试 → 超过最大重试次数创建异常台账转入人工处理

### 2.7 对账处理

**订单对账（每日凌晨）：** 查询昨日所有美团订单 → 调用美团 API 拉取订单列表 → 按订单号匹配 → 检查差异（内部有但美团没有、美团有但内部没有、状态不一致、金额不一致）→ 生成对账报告 → 发现差异时创建异常台账

**金额对账：** 查询结算事实中的美团订单金额 → 调用美团 API 查询结算账单 → 比对金额（订单总金额、平台佣金、商户应收）→ 发现差异时记录并告警
