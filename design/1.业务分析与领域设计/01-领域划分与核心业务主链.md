# 01-领域划分与核心业务主链

## 1. DDD核心概念在本平台的应用

### 1.1 限界上下文的本质

限界上下文是业务概念和语言的边界。同一个词在不同上下文中有不同的含义，必须在各自上下文内保持一致，跨上下文需要显式翻译。

本平台的典型例子：

| 上下文 | "订单"的含义 | 核心关注点 |
|---|---|---|
| 交易域 | `sale_order`（顾客买了什么） | 金额、状态、改退 |
| 履约域 | `fulfillment_order`（怎么做、怎么交） | 生产计划、工作单、交付 |
| 结算域 | `settlement_fact`（经营事实和清分依据） | 收入、佣金、对账 |

三者都关联同一笔经营行为，但代表不同的业务语义，不能混用。

### 1.2 领域划分原则

- **真相源集中**：每类业务对象有唯一的主权域，其他域只读引用
- **变化频率相近**：变化频率相近的对象放在同一上下文
- **对外语义稳定**：上下文对外输出的接口语义不因内部实现变化
- **不因页面或终端拆域**：应用层不是核心域的划分依据
- **不因外部平台字段拆域**：美团/饿了么的字段差异在渠道域消化，不向核心域渗透

## 2. 核心域详细划分

### 2.1 商品与门店经营域

**核心职责：** 维护"门店当前能卖什么、怎么卖、卖多少钱"的经营真相。

**主要概念：**
- `product`（商品主数据）：商品的名称、描述、分类、规格
- `menu`（门店菜单）：门店当前生效的菜单版本
- `price_rule`（价格规则）：不同渠道/时段的定价策略
- `availability_rule`（可售规则）：商品的上下架、售罄规则
- `channel_catalog_projection`（渠道商品投影）：各渠道的商品映射

**边界说明：**
- 向交易域输出：下单时的商品快照（价格、名称、规格），快照一旦写入不再变化
- 向渠道域输出：各渠道的商品映射投影
- 不接收外部状态修改其主数据

**为何是核心域：** 交易成立的前提，多业态统一的关键。商品主数据的质量直接决定交易链路的正确性。

### 2.2 交易域

**核心职责：** 形成并维护统一、稳定、可审计的订单事实。

**主要概念：**
- `sale_order`（统一订单）：所有渠道订单的内部统一表示
- `sale_order_item`（订单行快照）：下单时的商品快照，不随商品变更而变化
- `sale_order_status_line`（状态线记录）：订单状态变更的完整历史
- `sale_order_relation`（订单关系）：原始订单与补单/退单的关联关系
- `sale_order_action_log`（操作日志）：所有操作的审计记录

**边界说明：**
- 接收来自渠道域的来源事实（原始订单号、来源渠道）
- 向履约域发出"可执行订单事实"
- 向结算域输出"经营事实起点"
- 不直接对接外部平台

**为何是核心域：** 平台所有经营行为的事实中枢，是结算、履约、治理的数据起点。

### 2.3 履约与生产域

**核心职责：** 将确认的订单转化为可执行的生产和交付任务，管理门店执行链路。

**主要概念：**
- `fulfillment_order`（履约订单）：对应一笔交易订单的履约执行对象
- `production_plan`（生产计划）：将订单行拆解为各工作站的生产任务
- `work_unit`（工作单）：推送到KDS/打印机的最小执行单元
- `handoff_task`（交接任务）：堂食上菜、自取核销的交付任务
- `delivery_task`（配送任务）：外卖配送的调度任务

**边界说明：**
- 接收交易域的订单确认事件（`OrderAccepted`）
- 向设备执行层推送工作单指令
- 向渠道域反馈生产和交付状态（用于回传外部平台）
- 向结算域输出履约完成事实

**为何是核心域：** 餐饮区别于纯交易系统的核心差异，门店执行链路的管理是平台的核心价值之一。

### 2.4 渠道集成域

**核心职责：** 隔离外部渠道差异，代理商户完成渠道侧操作，向内输出统一的渠道事实。

**主要概念：**
- `channel_shop_binding`（渠道店铺绑定）：门店与外部平台店铺的映射关系
- `channel_order`（渠道原始订单）：外部平台推入的原始订单数据（保留不修改）
- `channel_product_mapping`（渠道商品映射）：内部商品与渠道商品的对应关系
- `channel_coupon_record`（渠道优惠券核销记录）：渠道侧优惠券的核销状态
- `channel_operation_log`（渠道操作日志）：所有渠道操作的记录

**边界说明：**
- 接收外部平台的回调推送
- 向交易域输入统一格式的订单创建命令
- 向外部平台发出代理操作（接单/拒单/菜品更新等）
- 保留外部平台原始数据（不修改，仅翻译）

**为何是核心域：** 购物中心对商户的核心价值载体，ISV代理模式的技术实现基础。

### 2.5 结算与对账域

**核心职责：** 管理经营事实的清分逻辑和各渠道账单的对账核验。

**主要概念：**
- `settlement_fact`（结算事实）：每笔订单的清分结果
- `bill_record`（账单记录）：从外部平台拉取的渠道账单
- `reconciliation_task`（对账任务）：定期执行的对账作业
- `reconciliation_diff`（对账差异）：平台流水与渠道账单的差异记录
- `coupon_settlement`（优惠券结算记录）：优惠券补贴的清分记录

**边界说明：**
- 接收交易域和履约域的经营完成事件
- 定期从外部平台拉取账单数据
- 不修改交易域订单，只生成结算投影
- 向运营治理域输出差异记录

**为何是核心域：** 经营闭环的最后一环，财务真相源，直接影响商户收益。

## 3. 支撑域详细划分

### 3.1 租户与组织域

**核心职责：** 接收并维护从外部系统同步的组织主数据，提供组织层级查询和 Platform 隔离服务。

**主要概念：**
- `platform`（平台/集团，如万达集团、大悦城集团）
- `project`（购物中心项目，归属 Platform）
- `tenant`（租户/商户，按统一社会信用代码识别，Platform 隔离）
- `brand`（品牌，Platform 隔离）
- `store`（门店，包含 unit_no 铺位号）
- `contract`（合同，连接 Project + Tenant + Brand + Store）
- `table`（桌台）

**注意：** 组织域是主数据接收方，不负责商户入驻审核、合同签署等前置业务。其他域在处理业务时，以 `platform_id`/`store_id`/`tenant_id` 等标识引用组织对象，但不跨域直接操作组织数据。

### 3.2 外部集成域

**核心职责：** 管理平台与外部系统（会员系统、品牌ERP、第三方配送）的集成。

**主要概念：**
- `member_integration`（会员系统集成）
- `erp_data_export`（ERP数据输出任务）
- `delivery_provider_account`（配送账户）
- `delivery_dispatch`（配送调度记录）

### 3.3 运营治理域

**核心职责：** 管理异常处理、补偿机制、人工介入流程和审计日志。

**主要概念：**
- `exception_record`（异常台账）
- `compensation_task`（补偿任务）
- `manual_intervention`（人工介入记录）
- `audit_log`（审计日志）

### 3.4 预订域

**核心职责：** 管理正餐预订单完整生命周期，从顾客预订到到店开台。

**主要概念：**
- `booking_order`（预订单）
- `booking_period`（餐段配置）

**上下游关系：**
- 向租户与组织域发事件：更新桌台状态（RESERVED/OCCUPIED/AVAILABLE）
- 向交易域发事件：`BookingArrived` → 触发开台创建 TableSession
- 向外部集成域发事件：通知顾客确认/取消短信

## 4. 上下文协作关系地图

```
                    ┌──────────────────┐
                    │  商品与门店经营域  │
                    │  (商品主数据、菜单) │
                    └────────┬─────────┘
                             │ 商品快照
                             ▼
┌──────────────┐    ┌──────────────────┐    ┌──────────────────┐
│  渠道集成域   │───▶│     交易域        │───▶│  履约与生产域     │
│ (ISV代理、   │    │  (统一订单、状态线) │    │ (工作单、交付链路) │
│  渠道映射)   │    └────────┬─────────┘    └────────┬─────────┘
└──────────────┘             │                       │
       ▲                     │ 经营事实               │ 履约完成
       │ 状态回传              ▼                       ▼
       │              ┌──────────────────┐    ┌──────────────────┐
       │              │   结算与对账域    │    │  设备执行层（应用）│
       │              │ (清分、对账、差异) │    │ (POS/KDS/打印机) │
       │              └────────┬─────────┘    └──────────────────┘
       │                       │
       │              ┌────────▼─────────┐
       └──────────────│   运营治理域      │
                      │ (异常、补偿、审计) │
                      └──────────────────┘
```

**关键协作路径：**

1. **商品域 → 交易域**：下单时提供商品快照（价格、规格、名称），快照一旦写入订单不再变化
2. **渠道域 → 交易域**：将外部平台推入的原始订单翻译为统一格式，发起订单创建命令
3. **交易域 → 履约域**：订单接单后（`OrderAccepted`），通过领域事件触发履约订单和生产计划创建
4. **履约域 → 设备层**：工作单推送到KDS/打印机，设备反馈通过命令接口回传
5. **交易域 → 结算域**：订单完成后触发结算事实生成
6. **履约域 → 渠道域**：门店生产/交付状态变更，通过渠道域回传给外部平台
7. **结算域 → 运营治理域**：对账差异触发异常台账创建
8. **渠道域 ← 运营治理域**：异常补偿操作通过渠道域向外部平台发出补偿请求

## 5. 不共表红线

以下对象对不允许共表存储：

| 对象A | 对象B | 原因 |
|---|---|---|
| 内部门店 (store) | 渠道店铺 (channel_shop) | 生命周期不同，变化原因不同 |
| 内部商品主数据 (product) | 渠道商品投影 (channel_product) | 主数据与派生投影不能混存 |
| 统一订单 (sale_order) | 渠道原始订单 (channel_order) | 业务语义不同，翻译关系需显式表达 |
| 履约工作单 (work_unit) | 订单主表 (sale_order) | 执行细节不能污染交易事实 |
| 结算事实 (settlement_fact) | 订单主表 (sale_order) | 结算是派生视图，不是订单本身 |
| 异常台账 (exception_record) | 主业务对象 | 异常是治理关注点，不是业务主流程字段 |

## 6. 三条核心业务主链

平台的业务主链分为三条，并行运转，通过领域事件互相触发，各自有独立边界。

### 6.1 交易主链

**职责：** 管理"顾客买了什么"

```
交易发起 → 交易确认 → 交易变更（改单/加菜） → 交易取消 → 交易完成
```

**详细流程：**

```
[渠道订单到达 / 顾客下单]
        ↓
  商品快照锁定（从商品域获取当前价格和规格）
        ↓
  订单创建（sale_order 初始化，状态 = PENDING_PAYMENT）
        ↓
  支付处理（微信/支付宝/现金/会员钱包）
        ↓
  支付确认（状态 = PAID，与支付确认同一事务）
        ↓
  接单决策（自动接单 or 人工接单）
        ↓
  订单接受（状态 = ACCEPTED）→ 触发履约主链
        ↓
  [可选] 订单变更（加菜/退菜/换菜）
        ↓
  履约完成回调（来自履约主链）
        ↓
  订单完成（状态 = COMPLETED）→ 触发结算主链
```

**关键状态机：**
```
PENDING_PAYMENT → PAID → ACCEPTED → IN_PRODUCTION → READY → COMPLETED
                    ↓         ↓
                REJECTED   CANCELLED
```

**交易主链的核心约束：**
- 支付确认与订单状态更新必须强一致（同一事务）
- 优惠券核销与订单折扣必须强一致（同一事务）
- 订单行快照一旦创建不可修改（改单通过新增/删除行实现）
- 退款发起与退款记录创建必须强一致（同一事务）

### 6.2 履约主链

**职责：** 管理"门店怎么做、怎么交"

```
接单确认 → 生产编排 → 工作单执行 → 交付准备 → 交付完成
```

**详细流程：**

```
[接收 OrderAccepted 事件]
        ↓
  创建履约订单（fulfillment_order）
        ↓
  生成生产计划（production_plan）
  ├── 按工作站拆分工作单（work_unit）
  ├── 设置优先级和制作顺序
  └── 计算预计出餐时间
        ↓
  推送工作单到设备（KDS/打印机）
        ↓
  工作站执行（KDS操作员确认开始/完成）
        ↓
  所有工作单完成 → 生产完成（ProductionCompleted）
        ↓
  根据 fulfillment_type 进入不同交付流程：
  ├── TABLE_SERVICE：上菜确认（堂食）
  ├── SELF_PICKUP：取餐码核销（自取）
  └── THIRD_PARTY_DELIVERY：派单配送（外卖）
        ↓
  交付完成 → FulfillmentCompleted 事件 → 通知交易域
```

**履约主链的核心约束：**
- 履约启动的唯一触发点是 `OrderAccepted` 事件
- 工作单是推送到设备的最小单元，不可再拆分
- 取消订单时必须撤回未完成的工作单
- 配送任务通过外部集成域调用第三方配送平台

### 6.3 治理主链

**职责：** 管理"经营数据如何收口"

```
经营事实记录 → 渠道对账 → 差异识别 → 补偿/处理 → 结算关账
```

**详细流程：**

```
[接收 OrderCompleted 事件]
        ↓
  生成结算事实（settlement_fact）
  ├── 计算各方分成（平台佣金/渠道佣金/门店收入）
  ├── 记录优惠券补贴明细
  └── 关联支付流水
        ↓
  [定时任务] 拉取渠道账单
        ↓
  执行对账（平台流水 vs 渠道账单）
        ↓
  差异识别（MISSING_IN_PLATFORM / MISSING_IN_CHANNEL / AMOUNT_DIFF）
        ↓
  差异处理：
  ├── 自动补偿（可自动处理的差异）
  └── 人工介入（需要人工确认的差异）
        ↓
  结算关账（月结/日结）
```

**治理主链的核心约束：**
- 结算事实是派生视图，不修改交易域的订单数据
- 对账差异必须关联具体的订单ID，可追溯
- 人工介入记录必须完整保留，支持审计

## 7. 跨域通信原则

### 7.1 三种通信模式

**模式一：领域事件（异步通知）**
- 适用：通知"某件事已发生"，消费方对发布方没有强依赖
- 特征：发布方不等待消费方处理结果，允许多个消费方订阅
- 典型用例：`OrderCompleted` → 结算域生成结算事实

**模式二：命令接口（同步请求）**
- 适用：请求另一个域"执行某个动作"，需要立即知道结果
- 特征：同步调用，调用方等待结果，必须有超时和降级策略
- 典型用例：创建订单时，商品域提供"商品快照查询"接口

**模式三：查询投影（跨域只读查询）**
- 适用：需要聚合多个域的数据展示，不需要修改其他域的状态
- 特征：只读，通过事件订阅维护查询投影数据
- 典型用例：门店后台的经营报表

### 7.2 依赖方向规则

```
允许的依赖方向（单向）：
  应用层 → 核心域
  渠道域 → 交易域（通过命令接口）
  履约域 → 商品域（通过查询接口，仅查生产画像）
  结算域 → 交易域（通过事件订阅）

禁止的依赖方向：
  核心域 → 应用层
  交易域 → 渠道域（反方向）
  商品域 → 交易域（循环）
```

### 7.3 跨域数据共享规则

**允许共享（只读引用）：**
- `store_id`：门店标识（所有域都可引用）
- `order_id`：订单标识（交易域主权，其他域只读引用）
- 枚举和编码定义：状态码、渠道类型等（统一定义，各域引用）

**禁止跨域共享：**
- 禁止直接访问其他域的内部数据表
- 禁止在一个域内维护另一个域业务对象的完整副本
- 禁止通过数据库外键关联跨域数据（可以有业务ID引用，但不建物理外键）

**反模式示例：**
```
// 错误：在履约域里 JOIN 交易域的表
SELECT fo.*, so.customer_name
FROM fulfillment_order fo
JOIN sale_order so ON fo.order_id = so.id  -- 跨域JOIN，禁止！

// 正确：交易域在事件 payload 中携带足够信息
// 履约域从 OrderAccepted 事件中获取所需数据，不回查交易域
```

## 8. 应用上下文

以下是应用层上下文，它们是核心域的**视图投影**，不是业务真相源：

| 应用上下文 | 主要用户 | 主要消费的核心域 |
|---|---|---|
| 消费者小程序 | 消费者 | 商品域（菜单展示）、交易域（下单）、履约域（进度查询） |
| 门店POS | 店员/收银 | 交易域（录单/改单）、商品域（菜单查询）、履约域（状态查看） |
| KDS厨显 | 厨房/吧台岗位 | 履约域（工作单接收和反馈） |
| 打包台 | 打包岗位 | 履约域（打包任务查看、核对、完成确认） |
| 叫号屏 | 自取取餐区 | 履约域（自取叫号展示） |
| 门店后台 | 店长 | 商品域（菜单管理）、结算域（营业报表）、交易域（历史查询） |
| 平台运营后台 | 购物中心运营团队 | 所有域的聚合视图（治理、对账、监控） |

**关键原则：** 应用上下文只消费核心域的数据，不创造业务真相。设备（POS/KDS）是执行终端，不能直接修改核心域的业务对象状态。

---

## 9. 本文结论

平台的领域划分遵循一个核心原则：**每个域有唯一的业务真相主权，跨域通信通过事件、命令、查询三种方式，不直接共享数据表。**

五个核心域的分工：
- **商品域**：经营配置的真相源（卖什么、卖多少钱）
- **交易域**：经营事实的真相源（卖了什么、收了多少）
- **履约域**：执行过程的真相源（怎么做、怎么交）
- **渠道域**：外部世界与内部核心域之间的防腐层
- **结算域**：财务事实的真相源（实际收入是多少）

三条业务主链（交易/履约/治理）并行运转，通过领域事件互相触发，各自有独立边界，互不越权。这是平台能够支持多业态、多渠道、多门店的架构基础。
