# 04-履约与生产域

## 1. 域定位与职责边界

履约与生产域是平台的**执行引擎**，负责将确认的订单转化为可执行的生产和交付任务，管理门店执行链路。它是连接"交易承诺"和"顾客收餐"之间的桥梁。

### 1.1 核心职责

- 接收交易域的订单接单事件（OrderAccepted），创建履约订单
- 生成生产计划，拆分工作单
- 派发工作单到各工作站（KDS/打印机）
- 跟踪生产进度
- 协调交付流程（堂食上菜/自取叫号/配送派单）
- 管理配送任务（一方外卖）
- 反馈履约状态给交易域

### 1.2 不负责的事项

| 事项 | 归属域 | 为什么排除 |
|------|--------|-----------|
| 订单金额和支付 | 交易域 | 履约域只关注"做什么、怎么送"，不关注"值多少钱" |
| 外部渠道API对接 | 渠道域 | 防腐层隔离外部平台差异，让履约域专注于生产执行 |
| 商品主数据 | 商品域 | 履约域只消费商品的"生产画像"（工作站类型、预估时长），不管理商品定义 |
| 配送平台账户 | 外部集成域 | 配送平台的账号授权、签名验证等属于集成层关注点 |

## 2. 核心聚合根设计

### 2.1 履约订单聚合 `FulfillmentOrder`

**聚合根：** `fulfillment_order`

**设计意图：** FulfillmentOrder 是履约域的核心锚点。它从交易域的 OrderAccepted 事件中创建，冗余了顾客信息和服务上下文（而非引用交易订单），确保履约域可以独立运行，不依赖跨域查询。

**完整字段定义：**

```
fulfillment_order_id    履约订单ID（UUID）
order_id                关联交易订单ID
store_id                门店ID
fulfillment_type        履约方式（TABLE_SERVICE|SELF_PICKUP|THIRD_PARTY_DELIVERY|PLATFORM_DELIVERY|STORE_DELIVERY）
order_scene             业务场景（DINE_IN|TAKEAWAY_SELF|DELIVERY_ONESELF|DELIVERY_PLATFORM）
priority                优先级（HIGH|NORMAL|LOW）

顾客信息（冗余，用于展示）：
customer_name           顾客姓名
customer_phone_suffix   手机号后4位

服务上下文：
service_context         服务上下文（JSON）
  table_id              桌台ID（堂食）
  table_name            桌台名称
  pickup_code           取餐码（自取）
  delivery_address      配送地址（外卖）
  delivery_note         配送备注

履约状态：
status                  履约状态（见状态机）
production_status       生产状态（NOT_STARTED|IN_PROGRESS|COMPLETED）
handoff_status          交付状态（PENDING|READY|IN_DELIVERY|COMPLETED）

时间信息：
created_at              创建时间
accepted_at             接单时间（来自交易域）
production_started_at   开始生产时间
production_completed_at 生产完成时间
handoff_ready_at        交付准备完成时间
handoff_completed_at    交付完成时间
expected_ready_time     预计出餐时间
expected_delivery_time  预计送达时间（外卖）

SLA管理：
production_sla_seconds  生产SLA（秒）
handoff_sla_seconds     交付SLA（秒）
is_overdue              是否超时
overdue_seconds         超时秒数

其他：
remark                  备注
tags                    标签（急单/VIP等）
version                 乐观锁版本号
```

**为什么冗余顾客信息和服务上下文？** 履约过程中需要频繁展示顾客信息（如 KDS 显示、叫号屏幕、配送单据），如果每次都跨域查询交易订单会产生不必要的耦合和延迟。冗余快照在订单创建时一次性复制，之后履约域自给自足。

**为什么有 `fulfillment_type` 和 `order_scene` 两个字段？** `order_scene` 描述业务场景（堂食、自取、外卖），`fulfillment_type` 描述执行路径（上菜、自取柜台、一方配送、平台配送）。同一个 order_scene 可能对应不同的 fulfillment_type（例如外卖可以是一方配送或平台配送）。

**实体：FulfillmentOrderItem（履约订单行）**

```
FulfillmentOrderItem（履约订单行）
  item_id               履约订单行ID
  fulfillment_order_id  履约订单ID
  order_item_id         关联交易订单行ID
  product_snapshot      商品快照（来自交易域）
  quantity              数量
  production_status     生产状态（PENDING|IN_PROGRESS|COMPLETED|CANCELLED）
  assigned_work_units   分配的工作单ID列表
  remark                备注
```

**FulfillmentOrder 状态机：**

```
PLAN_PENDING（待生成计划）
    ↓ [GenerateProductionPlan 命令]
PRODUCTION_PLANNED（计划已生成）
    ↓ [WorkUnitStarted 事件 → 首个工作单开始制作]
IN_PRODUCTION（生产中）
    ↓ [ProductionCompleted 事件 → 全部工作单完成]
PRODUCTION_DONE（生产完成）
    ↓ [PrepareHandoff 命令]
HANDOFF_READY（待交接）
    ↓ [StartHandoff 命令]
IN_HANDOFF（交接中）
    ↓ [CompleteHandoff 命令]
FULFILLED（已完成交付）

异常路径：
任意状态 → CANCELLED（订单取消）
```

> **事件命名规范说明：**
> - `WorkUnitStarted`、`WorkUnitCompleted`、`ProductionCompleted` 是**跨域发布事件**（标准名，与事件全集一致）
> - 状态机内部的"首个工作单开始"和"全部工作单完成"是触发条件描述，不是独立事件名
> - 禁止在代码或文档中使用 `WorkUnitsDispatched` / `AllWorkUnitsCompleted` 作为对外事件名

**为什么状态机分为"生产"和"交接"两个阶段？** 因为生产和交付是两个独立的关注点：生产关注"厨房做好了没"，交付关注"顾客拿到了没"。分离后可以独立追踪，也便于对堂食"边做边上"这种场景提供支持（一个工作单完成即可触发部分交付）。

### 2.2 生产计划聚合 `ProductionPlan`

**聚合根：** `production_plan`

**设计意图：** 生产计划是将一个履约订单转换为多个可并行执行的工作单的编排层。它的存在使得"拆单"逻辑有了明确的归属，也便于统计和监控整体生产进度。

**完整字段定义：**

```
production_plan_id      生产计划ID
fulfillment_order_id    履约订单ID
store_id                门店ID
plan_type               计划类型（IMMEDIATE|BATCH|SCHEDULED）
plan_status             计划状态（PENDING|ACTIVE|COMPLETED|CANCELLED）
total_work_units        工作单总数
completed_work_units    已完成工作单数
estimated_duration      预计总时长（秒）
actual_duration         实际总时长（秒）
created_at              创建时间
started_at              开始时间
completed_at            完成时间
```

**实体：WorkUnit（工作单）**

工作单是派发到具体工作站（热饮站、冷饮站、甜品站等）的最小执行单元。

```
WorkUnit（工作单）
  work_unit_id          工作单ID
  production_plan_id    生产计划ID
  station_id            工作站ID
  station_type          工作站类型（HOT_DRINK|COLD_DRINK|DESSERT|MAIN_COURSE|PACKAGE）
  station_name          工作站名称
  work_unit_type        工作单类型（PRODUCTION|PACKAGE）
  priority              优先级（HIGH|NORMAL|LOW）
  sequence              建议制作顺序
  status                工作单状态（见状态机）
  items                 工作单项列表（JSON）
    product_id          商品ID
    product_name        商品名称
    variant_name        规格名称
    modifiers           加料列表
    quantity            数量
    specifications      规格描述（完整的口味要求）
  estimated_duration    预计制作时长（秒）
  actual_duration       实际制作时长（秒）
  operator_id           操作员ID
  started_at            开始时间
  completed_at          完成时间
  device_id             推送设备ID（KDS/打印机）
  print_count           打印次数
```

**为什么 WorkUnit 的 items 包含完整的商品描述（名称、规格、加料、口味要求）？** 因为工作单是推送到 KDS/打印机的最终展示内容，制作人员需要看到完整信息。这些信息在工作单创建时从商品快照中提取，之后不再依赖商品域。

**为什么有 PRODUCTION 和 PACKAGE 两种工作单类型？** PRODUCTION 是常规制作工作单，PACKAGE 是打包工作单。外卖场景需要在所有商品制作完成后进行统一打包，打包本身也是一个需要派发和追踪的工作环节。

**WorkUnit 状态机：**

```
PENDING（待派发）
    ↓ [DispatchWorkUnit 命令]
DISPATCHED（已派发到设备）
    ↓ [StartWorkUnit 命令]
IN_PROGRESS（制作中）
    ↓ [CompleteWorkUnit 命令]
COMPLETED（已完成）

异常路径：
PENDING → CANCELLED（订单取消）
DISPATCHED → RECALLED（撤回重做）
```

**为什么有 RECALLED 状态？** 实际运营中，已经派发到 KDS 的工作单可能因为退菜、顾客修改等原因需要撤回。RECALLED 状态标记原工作单作废，系统会重新生成新的工作单，保证历史可追溯。

### 2.3 交接任务聚合 `HandoffTask`

**聚合根：** `handoff_task`

**设计意图：** 交接任务统一建模了堂食上菜、自取叫号、配送交接三种交付方式。虽然三种方式的业务流程差异很大，但它们共享同一个生命周期（准备 → 进行中 → 完成），因此用同一个聚合根加类型区分的方式建模，简化了履约订单对交付状态的追踪。

**完整字段定义：**

```
handoff_task_id         交接任务ID
fulfillment_order_id    履约订单ID
handoff_type            交接类型（TABLE_SERVICE|PICKUP|DELIVERY）
handoff_status          交接状态（见状态机）

堂食上菜：
table_id                桌台ID
table_name              桌台名称
server_id               服务员ID
served_at               上菜时间

自取叫号：
pickup_code             取餐码
pickup_counter          取餐柜台
call_number_at          叫号时间
pickup_at               取餐时间

配送交接：
delivery_task_id        配送任务ID
handoff_to_rider_at     交给骑手时间
rider_name              骑手姓名
rider_phone             骑手电话

时间信息：
created_at              创建时间
ready_at                准备完成时间
completed_at            交接完成时间
```

**HandoffTask 状态机：**

```
PENDING（待准备）
    ↓ [MarkReady 命令]
READY（已准备好）
    ↓ [StartHandoff 命令]
IN_PROGRESS（交接中）
    ↓ [CompleteHandoff 命令]
COMPLETED（已完成）
```

### 2.4 配送任务聚合 `DeliveryTask`

**聚合根：** `delivery_task`（仅一方外卖场景使用）

**设计意图：** 一方外卖（商户自营外卖）需要调用第三方配送平台（达达/顺丰/闪送）派单。DeliveryTask 独立建模配送生命周期，包括骑手分配、实时位置追踪、费用结算等。三方外卖（美团/饿了么平台配送）不使用此聚合，因为配送由平台自行管理。

**完整字段定义：**

```
delivery_task_id        配送任务ID
fulfillment_order_id    履约订单ID
order_id                交易订单ID
store_id                门店ID
delivery_provider       配送平台（DADA|SHUNFENG|FLASHEX）
external_delivery_id    配送平台订单ID
delivery_status         配送状态（见状态机）

配送信息：
pickup_address          取货地址（门店地址）
delivery_address        送货地址
estimated_distance      预计距离（米）
estimated_duration      预计时长（分钟）

骑手信息：
rider_id                骑手ID
rider_name              骑手姓名
rider_phone             骑手电话（加密）
rider_location          骑手位置（实时更新）
  latitude              纬度
  longitude             经度
  updated_at            更新时间

费用信息：
delivery_fee            配送费（分）
distance_fee            距离费
time_fee                时长费
extra_fee               额外费用

时间信息：
created_at              创建时间
dispatched_at           派单时间
accepted_at             骑手接单时间
picked_up_at            取货时间
delivered_at            送达时间
estimated_pickup_at     预计取货时间
estimated_delivery_at   预计送达时间

异常信息：
exception_type          异常类型（RIDER_CANCEL|TIMEOUT|ADDRESS_ERROR）
exception_reason        异常原因
exception_at            异常时间
```

**为什么骑手电话加密存储？** 骑手电话属于个人隐私数据，加密存储满足数据安全合规要求。展示时通过解密服务获取，通常以虚拟号方式提供给顾客。

**DeliveryTask 状态机：**

```
PENDING（待派单）
    ↓ [DispatchDelivery 命令]
DISPATCHED（已派单）
    ↓ [RiderAccepted 事件]
RIDER_ASSIGNED（骑手已接单）
    ↓ [RiderArrived 事件]
RIDER_ARRIVED（骑手已到店）
    ↓ [PickedUp 事件]
PICKED_UP（已取货）
    ↓ [Delivered 事件]
DELIVERED（已送达）

异常路径：
DISPATCHED → DISPATCH_FAILED（派单失败）
RIDER_ASSIGNED → RIDER_CANCELLED（骑手取消）
PICKED_UP → DELIVERY_EXCEPTION（配送异常）
```

### 2.5 取餐叫号实体 `PickupCall`

**设计意图：** 自取场景下，生产完成后需要通知顾客取餐。PickupCall 管理叫号的完整生命周期，包括重复叫号和超时过期，避免顾客错过取餐或餐品长时间无人领取。

```
call_id                 叫号ID（UUID）
order_id                订单ID
store_id                门店ID
pickup_code             取餐码
call_status             叫号状态（WAITING|CALLED|PICKED_UP|EXPIRED）
called_at               叫号时间
picked_up_at            取餐时间
call_count              叫号次数（支持重复叫号）
expired_at              过期时间
```

**叫号触发规则：**

```
ProductionCompleted事件 → 自动触发叫号（call_status=CALLED）
叫号后N分钟未取餐 → 自动重复叫号（最多3次）
超时未取餐 → call_status=EXPIRED，通知门店
```

**为什么最多重复叫号3次？** 超过3次仍未取餐，说明顾客可能已经离开，继续叫号会干扰其他顾客。此时转为通知门店人工处理（如电话联系顾客），避免餐品浪费。

### 2.6 打印任务实体 `PrintTask`

**设计意图：** 餐饮场景涉及多种票据（后厨联、客看联、汇总单、标签、外卖小票、预结单、结账单），不同票据发送到不同设备。PrintTask 统一管理打印任务的派发和状态追踪。

```
task_id                 任务ID（UUID）
order_id                订单ID
store_id                门店ID
receipt_type            票据类型（KITCHEN_GATE|GUEST_WATCH|KITCHEN_SUMMARY|LABEL|TAKEOUT|PRE_BILL|RECEIPT）
target_device_id        目标设备ID（打印机/KDS）
print_items             指定打印菜品行（可选，为空则全单）
  item_id / quantity
trigger_type            触发类型（AUTO|REPRINT）
status                  状态（PENDING|SENT|PRINTED|FAILED）
created_at / sent_at / printed_at
```

**票据类型说明：**

| receipt_type | 名称 | 用途 |
|---|---|---|
| KITCHEN_GATE | 后厨联 | 发送到对应工作站的打印机，指导制作 |
| GUEST_WATCH | 客看联 | 放在餐品旁边给顾客核对 |
| KITCHEN_SUMMARY | 后厨汇总单 | 汇总一单所有菜品，用于出餐核对 |
| LABEL | 标签 | 贴在外卖包装上 |
| TAKEOUT | 外卖小票 | 外卖订单的完整小票 |
| PRE_BILL | 预结单 | 堂食顾客结账前的账单 |
| RECEIPT | 结账单 | 最终结账凭证 |

**KDS工作单推送机制：**

> **三层边界说明（P1-2）：**
> - `Device / DeviceCommand`：归属**租户与组织域**，负责设备注册、绑定、状态监控和命令下发
> - `PrintTask / PickupCall / HandoffTask`：归属**履约与生产域**，是履约业务对象，描述"需要打印什么"和"需要叫号什么"
> - WebSocket 长连接、设备接入网关：归属**应用层/基础设施层**，不属于任何业务域
>
> 履约域创建 PrintTask/PickupCall 后，通过发布领域事件通知租户与组织域的设备管理能力，由设备管理能力创建 DeviceCommand 并通过应用层推送到物理设备。履约域不直接操作设备。

```
WorkUnit创建后：
  → 查询门店KDS设备（按workstation_type匹配）
  → 创建 DeviceCommand（type=DISPLAY_ORDER，payload含工作单详情）
  → 通过WebSocket推送到对应KDS设备
  → 等待KDS回执确认（DeviceCommandAcknowledged）
  → 超时（>30s）则告警并记录，重试最多3次
```

**为什么用 WebSocket 而不是轮询？** KDS 是实时显示设备，工作单需要即时出现在屏幕上。WebSocket 提供毫秒级推送能力，远优于轮询的延迟。超时重试机制确保网络波动时工作单不会丢失。

## 3. 生产计划生成逻辑

### 3.1 生产计划生成服务 `ProductionPlanService`

**输入：** `FulfillmentOrder`

**输出：** `ProductionPlan`（包含多个WorkUnit）

**生成步骤：**

```
1. 读取履约订单的所有订单行
    ↓
2. 提取每个商品的生产画像（station_type, estimated_duration）
    ↓
3. 按工作站类型分组商品
    ↓
4. 为每个工作站生成一个工作单
    ↓
5. 根据fulfillment_type决定是否需要打包工作单
    ↓
6. 计算工作单优先级和顺序
    ↓
7. 计算预计出餐时间
    ↓
8. 返回生产计划
```

**为什么按工作站类型分组而不是按商品逐个生成工作单？** 同一工作站的商品由同一组人员制作，合并为一个工作单可以减少 KDS 上的显示条目，制作人员只需关注一张单就能完成该站的所有出品，提高效率。

### 3.2 工作单拆分规则

**规则1：按工作站拆分**

```
订单包含：
  - 2杯热饮（制作岗位：HOT_DRINK）
  - 1杯冷饮（制作岗位：COLD_DRINK）
  - 1份甜品（制作岗位：DESSERT）

拆分为3个工作单：
  - WorkUnit1: 热饮工作站，2杯热饮
  - WorkUnit2: 冷饮工作站，1杯冷饮
  - WorkUnit3: 甜品工作站，1份甜品
```

**规则2：外卖需要打包工作单**

```
if (fulfillment_type == THIRD_PARTY_DELIVERY || fulfillment_type == PLATFORM_DELIVERY) {
    // 所有商品制作完成后，生成打包工作单
    WorkUnit_Package: 打包工作站，打包所有商品
}
```

**为什么外卖需要单独的打包工作单？** 外卖需要统一打包（放入袋中、封口、贴标签），这是一个独立的操作环节，需要等待所有制作工作单完成后才能开始。将打包建模为工作单可以追踪打包进度，也可以派发到专门的打包工位。

**规则3：堂食可分批上菜**

```
if (fulfillment_type == TABLE_SERVICE) {
    // 每个工作单完成后可以立即上菜，不需要等待所有工作单完成
    allow_partial_handoff = true
}
```

**为什么堂食允许分批上菜？** 堂食顾客期望"做好一道上一道"，而不是等所有菜品全部完成后一次性上齐。分批上菜提升了顾客体验，也减少了出餐口的积压。

### 3.3 工作单优先级计算

```
priority = base_priority + urgency_factor + vip_factor

base_priority:
  - 外卖订单：HIGH
  - 堂食订单：NORMAL
  - 自取订单：NORMAL

urgency_factor:
  - 超过预计出餐时间：+2
  - 接近预计出餐时间（5分钟内）：+1
  - 正常：0

vip_factor:
  - VIP顾客：+1
  - 普通顾客：0
```

**为什么外卖订单默认 HIGH 优先级？** 外卖有配送时效要求（平台考核准时率），超时会导致罚款和差评。堂食和自取的顾客已经在店内，容忍度相对较高。

## 5. 交付流程编排

### 5.1 交付决策逻辑

```
生产完成
    ↓
根据 fulfillment_type 决策交付方式：

if (fulfillment_type == TABLE_SERVICE) {
    // 堂食上菜
    创建 HandoffTask（type=TABLE_SERVICE）
    通知服务员上菜
    等待服务员确认上菜完成
}

if (fulfillment_type == SELF_PICKUP) {
    // 自取叫号
    创建 HandoffTask（type=PICKUP）
    触发叫号系统
    等待顾客扫码取餐
}

if (fulfillment_type == THIRD_PARTY_DELIVERY) {
    // 一方外卖，调用第三方配送
    创建 DeliveryTask
    调用配送平台派单API
    等待骑手取货和送达
}

if (fulfillment_type == PLATFORM_DELIVERY) {
    // 三方外卖，平台自己配送
    创建 HandoffTask（type=DELIVERY）
    通知渠道域"餐已备好"
    渠道域回传给外部平台
    等待平台骑手取货
}
```

**为什么一方外卖用 DeliveryTask 而三方外卖用 HandoffTask？** 一方外卖的配送由商户自己发起（需要调用达达/顺丰等配送平台），配送的完整生命周期由本域管理。三方外卖的配送由美团/饿了么平台管理，本域只需要告知平台"餐已备好"，然后等待骑手来取，本质上是一个"交接"动作。

### 5.2 堂食上菜流程

```
1. 生产完成，创建HandoffTask
    ↓
2. 通知服务员（推送到服务员APP或POS）
    ↓
3. 服务员确认取餐
    ↓
4. 服务员送到桌台
    ↓
5. 服务员确认上菜完成
    ↓
6. 更新HandoffTask状态为COMPLETED
    ↓
7. 发布 FulfillmentCompleted 事件
```

### 5.3 自取叫号流程

```
1. 生产完成，创建HandoffTask
    ↓
2. 生成取餐码（如果未生成）
    ↓
3. 触发叫号系统（语音播报/屏幕显示）
    ↓
4. 发送取餐通知给顾客（小程序推送）
    ↓
5. 顾客到前台扫码或报取餐码
    ↓
6. 店员核销取餐码
    ↓
7. 更新HandoffTask状态为COMPLETED
    ↓
8. 发布 FulfillmentCompleted 事件
```

### 5.4 一方外卖配送流程

```
1. 生产完成，创建DeliveryTask
    ↓
2. 调用配送平台派单API（达达/顺丰/闪送）
    ↓
3. 配送平台返回派单结果
    ↓
4. 更新DeliveryTask状态为DISPATCHED
    ↓
5. 接收配送平台回调（骑手接单/到店/取货/送达）
    ↓
6. 更新DeliveryTask状态
    ↓
7. 骑手送达后，更新状态为DELIVERED
    ↓
8. 发布 FulfillmentCompleted 事件
```

### 5.5 三方外卖交接流程

```
1. 生产完成，创建HandoffTask
    ↓
2. 通知渠道域"餐已备好"
    ↓
3. 渠道域回传给美团/饿了么
    ↓
4. 外部平台派骑手到店
    ↓
5. 骑手到店，店员确认交接
    ↓
6. 更新HandoffTask状态为COMPLETED
    ↓
7. 外部平台回调送达状态
    ↓
8. 发布 FulfillmentCompleted 事件
```

## 6. 配送平台集成（一方外卖）

### 6.1 配送平台防腐层

履约域通过外部集成域的配送防腐层调用第三方配送平台：

```
履约域
    ↓ 调用统一接口
外部集成域（配送防腐层）
    ↓ 适配器模式
┌────┴────┬────────┬────────┐
达达适配器  顺丰适配器  闪送适配器
```

**为什么不让履约域直接对接配送平台？** 每家配送平台的 API 格式、签名方式、回调协议都不同。如果履约域直接对接，每增加一个配送平台就要修改履约域代码。通过防腐层隔离，履约域只需调用统一接口，新增配送平台只需添加一个适配器。

### 6.2 统一配送接口

```
CreateDeliveryOrder(
  order_id: string,
  pickup_address: Address,
  delivery_address: Address,
  goods_info: GoodsInfo,
  expected_pickup_time: DateTime
) → DeliveryOrderResult {
  delivery_order_id: string,
  estimated_fee: number,
  estimated_pickup_at: DateTime,
  estimated_delivery_at: DateTime
}

CancelDeliveryOrder(
  delivery_order_id: string,
  reason: string
) → Result

QueryDeliveryStatus(
  delivery_order_id: string
) → DeliveryStatus
```

### 6.3 配送平台回调处理

```
接收配送平台回调
    ↓
验证签名
    ↓
解析回调内容
    ↓
根据回调类型更新DeliveryTask状态：
  - 骑手接单 → RIDER_ASSIGNED
  - 骑手到店 → RIDER_ARRIVED
  - 已取货 → PICKED_UP
  - 已送达 → DELIVERED
  - 骑手取消 → RIDER_CANCELLED
    ↓
发布相应的领域事件
    ↓
返回成功响应给配送平台
```

## 9. 与其他域的交互

### 9.1 接收交易域事件：OrderAccepted

**处理：** 创建履约订单，生成生产计划

**为什么由 OrderAccepted 而非 OrderCreated 触发？** 因为订单创建时可能还未确认接单（尤其是外卖订单需要商户确认），只有接单后才应该开始生产准备。

### 9.2 接收交易域事件：OrderCancelled

**处理：** 取消履约订单，撤回工作单

### 9.3 调用商品域：GetProductionProfile

**时机：** 生成生产计划时

**用途：** 获取商品的生产画像（工作站类型、预估制作时长等）

### 9.4 调用外部集成域：CreateDeliveryOrder

**时机：** 一方外卖生产完成后

**用途：** 派发配送任务

### 9.5 通知渠道域：ProductionCompleted

**用途：** 渠道域回传给外部平台"餐已备好"

### 9.6 通知交易域：FulfillmentCompleted

**用途：** 交易域将订单推进到完成状态

## 10. 数据一致性保障

### 10.1 生产计划一致性

- 生产计划一旦生成，不可修改
- 订单变更（加菜/退菜）触发计划重算
- 工作单撤回后重新生成新工作单
- 所有工作单完成才触发生产完成事件

**为什么生产计划不可修改而是重新生成？** 生产计划的工作单可能已经派发到 KDS 并在制作中，直接修改会导致制作人员看到的信息和实际不一致。重新生成可以撤回未开始的工作单，同时保留已完成的记录。

### 10.2 工作单状态一致性

- 工作单状态转换使用乐观锁
- 并发完成多个工作单时，只有最后一个触发生产完成事件
- 工作单撤回后，原工作单标记为RECALLED，不可再操作

### 10.3 配送任务一致性

- 配送派单失败时，自动重试或切换配送平台
- 配送平台回调幂等处理
- 骑手取消后，自动重新派单
- 配送异常时，记录异常原因，通知门店处理

## 11. 核心设计原则总结

履约与生产域的设计质量直接影响生产效率、交付准确性、顾客体验和门店运营。

| 原则 | 含义 | 为什么重要 |
|------|------|-----------|
| 生产计划驱动 | 所有生产活动由生产计划统一编排 | 确保拆单、排序、优先级计算有统一的决策点 |
| 工作单隔离 | 每个工作站独立接收和完成工作单 | 各工位并行工作互不干扰，最大化厨房产能 |
| 交付编排 | 根据履约方式自动选择交付流程 | 堂食、自取、外卖的交付逻辑差异大，统一编排降低复杂度 |
| 配送防腐 | 通过防腐层隔离不同配送平台差异 | 新增/更换配送平台不影响履约域核心逻辑 |
| 状态同步 | 实时同步履约状态给交易域和渠道域 | 顾客和平台能实时看到订单进度，减少催单和投诉 |
