# 08-外部集成域

## 1. 域定位与设计意图

外部集成域是平台与外部系统之间的**集成层与隔离层**。购物中心餐饮中台需要对接大量外部系统——购物中心会员系统、第三方配送平台、品牌ERP系统、支付网关等——这些系统各有不同的 API 协议、数据格式和业务规则。如果让各业务域直接对接，外部系统的变化将传导到整个平台内部，造成高度耦合。

因此，外部集成域的核心设计意图是：

1. **防腐层（ACL）隔离外部差异**：所有外部系统集成都通过防腐层接口暴露给内部域，内部域只依赖统一接口，不依赖具体外部实现。当外部系统 API 变更时，只需修改对应适配器，内部业务逻辑无需改动。
2. **适配器模式支持可扩展性**：每个外部系统有独立的适配器实现，新增外部系统（如新增一个配送平台）只需新增一个适配器，不影响已有集成。
3. **完整日志支持可监控性**：所有外部调用记录完整日志，出现问题时可快速定位是内部逻辑还是外部系统的问题。
4. **健康监控保障系统稳定性**：定期检查外部系统健康状态，在外部系统出现故障时能及时发现并触发降级策略。

## 2. 域职责边界

### 2.1 核心职责

- 集成购物中心会员系统（积分、优惠券）
- 集成第三方配送平台（达达、顺丰、闪送）
- 集成品牌ERP系统（订单数据输出、库存同步）
- 集成支付网关（微信、支付宝）
- 管理外部系统调用日志和监控
- 提供统一的防腐层接口

### 2.2 不负责的事项

- 不管理订单业务逻辑（交易域职责）——外部集成域只负责将会员优惠券/积分等信息桥接到交易流程中，订单本身的创建、状态流转由交易域负责
- 不管理会员主数据（会员系统职责）——本域只是会员系统的集成层，不持有会员数据
- 不管理配送任务业务逻辑（履约域职责）——本域只负责将配送请求转发给第三方平台，配送任务的跟踪和管理由履约域负责
- 不管理结算清分（结算域职责）

## 3. 核心聚合根设计

### 3.1 会员集成聚合 `MemberIntegration`

**聚合根：** `member_integration`

**设计意图：** 每个购物中心有自己的会员系统，API 端点、认证方式、接口格式各不相同。MemberIntegration 聚合将某个购物中心会员系统的全部集成配置封装在一起，使得对接一个新购物中心只需创建一条新的集成记录并配置接口地址，业务代码无需变更。SPI 反向回调配置的存在是因为部分商户收银系统需要主动调用中台来识别会员或扣减权益，中台需要暴露 SPI 端点给这些外部系统。

**完整字段定义：**

```
integration_id          集成ID
mall_id                 购物中心ID
integration_type        集成类型（MEMBER_SYSTEM）
system_name             系统名称
api_endpoint            API端点
auth_type               认证类型（API_KEY|OAUTH2|BASIC）
auth_config             认证配置（加密存储）

接口配置：
member_query_api        会员查询接口
coupon_query_api        优惠券查询接口
coupon_lock_api         优惠券锁定接口
coupon_redeem_api       优惠券核销接口
points_query_api        积分查询接口
points_deduct_api       积分扣减接口

超时配置：
connection_timeout      连接超时（毫秒）
read_timeout            读取超时（毫秒）
retry_count             重试次数
retry_interval          重试间隔（毫秒）

SPI反向回调配置（P0-5）：
spi_config              SPI配置（JSON）
  identify_url          会员识别回调URL（我方提供给商户收银系统）
  deduct_url            权益扣减回调URL
  redeem_coupon_url     券核销回调URL
  release_coupon_url    券释放回调URL
  accumulate_url        积分累积回调URL
  timeout_ms            SPI超时时间（建议300ms）
  sign_secret           回调签名密钥

状态：
integration_status      集成状态（ACTIVE|INACTIVE|FAULT）
last_health_check_at    最后健康检查时间
health_check_result     健康检查结果

其他：
created_at              创建时间
updated_at              更新时间
```

**SPI 反向回调配置说明：** 购物中心会员系统或商户收银系统可能需要主动调用中台来完成会员识别、权益扣减等操作。`spi_config` 定义了中台暴露给外部系统的回调端点。`timeout_ms` 建议设为 300ms，是因为 SPI 调用处于收银结账的关键路径上，超时时间过长会阻塞收银流程，影响顾客体验。`sign_secret` 用于回调请求的签名验证，防止伪造调用。

### 3.3 ERP集成聚合 `ERPIntegration`

**聚合根：** `erp_integration`

**设计意图：** 不同餐饮品牌使用不同的 ERP 系统（金蝶、用友、SAP 或自研），数据结构差异很大。ERPIntegration 聚合通过字段映射配置（field_mapping）将内部数据格式与 ERP 格式解耦，使得对接新 ERP 系统只需配置映射规则，不需要编写新代码。双向同步（BIDIRECTIONAL）支持既将中台订单推送到 ERP，又从 ERP 拉取库存和员工数据。

**完整字段定义：**

```

brand_id                品牌ID
erp_system_type         ERP系统类型（CUSTOM|KINGDEE|YONYOU|SAP）
erp_system_name         ERP系统名称
api_endpoint            API端点
auth_config             认证配置（加密存储）

数据同步配置：
sync_direction          同步方向（OUTBOUND|INBOUND|BIDIRECTIONAL）
sync_frequency          同步频率（REALTIME|HOURLY|DAILY）
sync_entities           同步实体列表（JSON）
  order_sync_enabled    订单同步
  inventory_sync_enabled 库存同步
  staff_sync_enabled    员工同步
  product_sync_enabled  商品同步

字段映射配置：
field_mapping           字段映射配置（JSON）
  order_mapping         订单字段映射
  product_mapping       商品字段映射
  inventory_mapping     库存字段映射

同步状态：
last_sync_at            最后同步时间
last_sync_status        最后同步状态（SUCCESS|FAILED|PARTIAL）
last_sync_error         最后同步错误信息

状态：
integration_status      集成状态（ACTIVE|INACTIVE|FAULT）
created_at              创建时间
```

### 3.4 外部调用日志聚合 `ExternalCallLog`

**聚合根：** `external_call_log`

**设计意图：** 外部系统调用是平台故障的高发区。完整记录每次外部调用的请求、响应、耗时和结果，使得运维人员能够快速定位问题——是请求参数错误、网络超时、还是外部系统返回错误。correlation_id 支持链路追踪，可以将一次业务操作涉及的多次外部调用串联起来分析。

**完整字段定义：**

```
log_id                  日志ID
integration_id          集成ID
integration_type        集成类型（MEMBER|DELIVERY|ERP|PAYMENT）
call_type               调用类型（API|WEBHOOK|BATCH）
api_name                API名称
api_method              HTTP方法
api_url                 API URL

请求信息：
request_headers         请求头（JSON）
request_body            请求体（JSON）
request_time            请求时间

响应信息：
response_status_code    响应状态码
response_headers        响应头（JSON）
response_body           响应体（JSON）
response_time           响应时间
duration_ms             耗时（毫秒）

执行结果：
execution_status        执行状态（SUCCESS|FAILED|TIMEOUT）
error_code              错误码
error_message           错误信息
retry_count             重试次数

关联信息：
correlation_id          关联ID（链路追踪）
business_id             业务ID（订单ID等）
business_type           业务类型

其他：
created_at              创建时间
```

### 3.5 SPI调用日志实体 `SpiInvokeLog`（P0-5）

**设计意图：** SPI 调用与普通外部 API 调用方向相反——是外部系统调用中台。由于 SPI 调用处于收银关键路径上，对延迟极其敏感，需要单独记录每次 SPI 调用的响应时间，以便监控 SPI 性能并及时发现超时问题。幂等键确保外部系统重试时不会产生重复扣减。

**完整字段定义：**

```
log_id                  日志ID（UUID）
store_id                门店ID
spi_type                SPI类型（MEMBER_IDENTIFY|DEDUCT|REDEEM|RELEASE|ACCUMULATE）
request_body            请求内容（JSON）
response_body           响应内容（JSON）
status                  状态（SUCCESS|TIMEOUT|FAILED）
latency_ms              响应时间（毫秒）
idempotency_key         幂等键
created_at              调用时间
```

**SPI超时降级策略：**

超时降级策略的核心原则：**不能因为外部系统故障阻塞核心收银流程**。根据不同 SPI 类型的业务影响程度，采取不同的降级方案。

```
会员识别超时（>300ms）→ 允许匿名下单，不阻塞收银
积分扣减超时 → 拒绝积分支付，提示顾客重试
券核销超时 → 拒绝券支付，提示顾客重试
积分累积超时 → 异步补偿任务，不影响支付完成
```

**策略说明：**
- 会员识别是非关键操作，识别失败不影响下单，因此超时时直接降级为匿名下单
- 积分扣减和券核销涉及金额变动，不能在不确定结果的情况下放行，因此超时时拒绝并提示重试
- 积分累积发生在支付完成后，不影响当前交易结果，因此超时时创建异步补偿任务后续处理

## 4. 会员系统集成

### 4.1 会员防腐层接口

**设计意图：** 定义统一的会员服务接口，屏蔽不同购物中心会员系统的 API 差异。内部域（如交易域）只依赖这些统一接口，不需要知道底层对接的是哪个购物中心的会员系统。

```
MemberService:
  VerifyMember(member_id: string) → MemberInfo {
    member_id: string,
    member_name: string,
    member_level: string,
    available_points: number,
    is_active: boolean
  }

  QueryAvailableCoupons(
    member_id: string,
    store_id: string
  ) → Coupon[] {
    coupon_id: string,
    coupon_code: string,
    coupon_type: string,
    discount_amount: number,
    min_order_amount: number,
    expire_date: Date
  }

  LockCoupon(
    coupon_id: string,
    order_id: string
  ) → LockResult {
    lock_id: string,
    locked_at: Date,
    expire_at: Date
  }

  RedeemCoupon(
    coupon_id: string,
    order_id: string
  ) → RedeemResult {
    redemption_id: string,
    redeemed_at: Date,
    discount_amount: number
  }

  CancelCouponRedemption(
    coupon_id: string,
    order_id: string
  ) → Result

  QueryPoints(member_id: string) → PointsInfo {
    available_points: number,
    frozen_points: number,
    expire_points: number,
    expire_date: Date
  }

  DeductPoints(
    member_id: string,
    points: number,
    order_id: string
  ) → DeductResult {
    deduction_id: string,
    deducted_points: number,
    remaining_points: number
  }

  RefundPoints(
    member_id: string,
    points: number,
    original_order_id: string
  ) → Result
```

### 4.2 会员系统适配器实现

**设计意图：** 适配器将统一接口调用翻译为具体购物中心会员系统的 API 格式。每个购物中心对接一个适配器实例。适配器负责协议转换、数据格式转换和错误码映射，最后一步记录调用日志用于监控和问题排查。

```
购物中心会员系统适配器
    ↓
1. 接收统一接口调用
    ↓
2. 转换为购物中心API格式
    ↓
3. 调用购物中心会员API
    ↓
4. 解析响应结果
    ↓
5. 转换为统一格式返回
    ↓
6. 记录调用日志
```

## 5. 配送平台集成

### 5.1 配送防腐层接口

**设计意图：** 屏蔽达达、顺丰同城、闪送等不同配送平台的 API 差异，为履约域提供统一的配送服务接口。履约域不需要关心具体使用哪个配送平台，由外部集成域根据策略自动选择。

```
DeliveryService:
  CreateDeliveryOrder(
    delivery_request: DeliveryRequest
  ) → DeliveryOrderResult {
    delivery_order_id: string,
    external_order_id: string,
    estimated_fee: number,
    estimated_pickup_at: Date,
    estimated_delivery_at: Date
  }

  CancelDeliveryOrder(
    delivery_order_id: string,
    reason: string
  ) → Result

  QueryDeliveryStatus(
    delivery_order_id: string
  ) → DeliveryStatus {
    status: string,
    rider_name: string,
    rider_phone: string,
    rider_location: Location,
    estimated_arrival: Date
  }

  EstimateDeliveryFee(
    from: Address,
    to: Address,
    weight: number
  ) → FeeEstimate {
    estimated_fee: number,
    distance: number,
    estimated_duration: number
  }
```

### 5.2 配送平台适配器

**设计意图：** 每个配送平台一个适配器，实现统一的 DeliveryService 接口。适配器内部封装了各平台特有的业务规则（如达达的小费机制、顺丰的重量分段计费），对外统一暴露标准接口。新增配送平台只需新增适配器，无需修改业务逻辑。

**达达适配器：**

```
DadaDeliveryAdapter:
  - 实现统一配送接口
  - 转换为达达API格式
  - 处理达达特有的业务规则
  - 解析达达回调格式
```

**顺丰同城适配器：**

```
ShunfengDeliveryAdapter:
  - 实现统一配送接口
  - 转换为顺丰API格式
  - 处理顺丰特有的业务规则
  - 解析顺丰回调格式
```

**闪送适配器：**

```
FlashexDeliveryAdapter:
  - 实现统一配送接口
  - 转换为闪送API格式
  - 处理闪送特有的业务规则
  - 解析闪送回调格式
```

### 5.3 配送平台选择策略

**设计意图：** 多平台对接的价值不仅在于冗余备份，更在于动态选择最优平台。选择策略支持按费用最低、速度最快或门店默认等维度决策，先并发调用各平台费用预估接口获取实时报价，再根据策略择优。这样可以为商户节省配送成本或提升配送时效。

```
配送平台选择服务
    ↓
1. 获取配送地址和商品信息
    ↓
2. 查询可用的配送平台
    ↓
3. 调用各平台费用预估接口
    ↓
4. 比较费用和预计送达时间
    ↓
5. 根据策略选择平台：
    - 优先选择费用最低
    - 或优先选择速度最快
    - 或按门店配置的默认平台
    ↓
6. 返回选中的平台
```

## 6. ERP系统集成

### 6.1 订单数据输出

**设计意图：** 品牌方需要在自己的 ERP 系统中管理财务数据和经营分析。订单完成后自动同步到 ERP，避免人工导出。失败时加入重试队列并触发告警，确保数据最终一致。字段映射配置使得不同 ERP 系统的数据格式差异在配置层面解决。

```
订单完成事件
    ↓
1. 监听 OrderCompleted 事件
    ↓
2. 查询ERP集成配置
    ↓
3. 提取订单数据
    ↓
4. 根据字段映射配置转换格式
    ↓
5. 调用ERP订单接口
    ↓
6. 记录同步结果
    ↓
if (失败) {
    加入重试队列
    触发告警
}
```

### 6.2 库存同步

**设计意图：** 部分品牌在 ERP 中管理库存（如中央厨房模式），中台需要定时从 ERP 拉取库存数据来更新商品可售状态，防止超卖。库存变更触发事件通知商品域更新售卖状态。

```
定时任务触发
    ↓
1. 查询启用库存同步的ERP集成
    ↓
2. 调用ERP库存查询接口
    ↓
3. 解析库存数据
    ↓
4. 转换为内部格式
    ↓
5. 更新商品库存状态
    ↓
6. 触发库存变更事件
    ↓
7. 记录同步日志
```

### 6.3 员工主数据同步

**设计意图：** 品牌方在 ERP 中管理员工入离职，中台需要与之同步以确保员工账号的及时启用和停用，避免离职员工仍能操作系统。

```
定时任务触发
    ↓
1. 查询启用员工同步的ERP集成
    ↓
2. 调用ERP员工查询接口
    ↓
3. 解析员工数据
    ↓
4. 比对内部员工数据
    ↓
5. 新增/更新/停用员工账号
    ↓
6. 记录同步日志
```

## 7. 支付网关集成

### 7.1 支付防腐层接口

**设计意图：** 屏蔽微信支付和支付宝的 API 差异，为交易域提供统一的支付服务接口。交易域发起支付时不需要关心用的是微信还是支付宝，由外部集成域根据用户选择的支付方式路由到对应支付网关。

```
PaymentService:
  CreatePayment(
    payment_request: PaymentRequest
  ) → PaymentResult {
    payment_id: string,
    payment_url: string,
    qr_code: string,
    expire_at: Date
  }

  QueryPaymentStatus(
    payment_id: string
  ) → PaymentStatus {
    status: string,
    paid_amount: number,
    paid_at: Date,
    transaction_no: string
  }

  RefundPayment(
    payment_id: string,
    refund_amount: number,
    reason: string
  ) → RefundResult {
    refund_id: string,
    refund_status: string,
    refunded_at: Date
  }
```

### 7.2 支付回调处理

**设计意图：** 支付平台通过回调通知支付结果。回调处理必须先验证签名防止伪造，再做幂等检查防止重复处理（支付平台可能多次发送同一回调）。支付状态确认后发布事件通知交易域推进订单流程。

```
接收支付平台回调
    ↓
1. 验证签名
    ↓
2. 解析回调内容
    ↓
3. 幂等检查
    ↓
4. 更新支付状态
    ↓
5. 发布 PaymentConfirmed 事件
    ↓
6. 返回成功响应
```

## 9. 关键域事件

### 9.1 ExternalCallFailed（外部调用失败）

**触发时机：** 外部系统调用失败且重试耗尽时发布。

**消费方：**
- 运营治理域（创建异常台账，触发告警）——外部调用失败是常见的生产问题来源，需要及时感知和处理

### 9.2 ERPSyncCompleted（ERP同步完成）

**触发时机：** ERP 数据同步完成时发布，无论成功还是失败。

**消费方：**
- 商品域（更新库存状态）——库存同步完成后商品域需要更新可售数量
- 租户域（更新员工信息）——员工同步完成后租户域需要更新员工账号状态

## 10. 与其他域的交互

### 10.1 向交易域提供服务

**接口：** `VerifyAndLockCoupon`, `RedeemCouponAndPoints`

**时机：** 创建订单和支付确认时

**用途：** 会员优惠券和积分处理。交易域在创建订单时调用锁定优惠券，支付确认时调用核销优惠券和扣减积分。

### 10.2 向履约域提供服务

**接口：** `CreateDeliveryOrder`

**时机：** 一方外卖生产完成后

**用途：** 派发配送任务。履约域确认生产完成后调用外部集成域向第三方配送平台下单。

### 10.3 接收交易域事件

**事件：** `OrderCompleted`

**处理：** 同步订单数据到ERP。订单完成是一个稳定的业务状态点，此时同步订单数据可以确保数据完整性。

### 10.4 接收商品域事件

**事件：** `ProductAvailabilityChanged`

**处理：** 同步库存状态到ERP。中台侧的库存变动需要反向同步到 ERP 以保持双方数据一致。

## 11. 数据一致性保障

### 11.1 会员操作一致性

**设计意图：** 优惠券和积分涉及金额变动，必须与订单状态保持一致，否则会出现"优惠券已核销但订单取消了"的不一致问题。

- 优惠券锁定与订单创建在同一事务
- 优惠券核销与支付确认在同一事务
- 订单取消时自动释放优惠券
- 所有会员操作记录完整日志

### 11.2 配送任务一致性

**设计意图：** 配送平台可能出现网络波动导致派单失败，需要自动重试和平台切换能力保障配送不中断。

- 配送派单失败自动重试
- 配送平台回调幂等处理
- 配送异常时记录详细日志
- 支持手动切换配送平台

### 11.3 ERP同步一致性

**设计意图：** ERP 同步是异步的最终一致性模型，通过持久化记录和全量同步能力来保障数据最终一致。

- 同步失败自动重试
- 同步记录持久化存储
- 支持手动触发全量同步
- 同步差异记录完整日志

## 12. 核心设计原则总结

外部集成域是平台与外部系统的桥梁，其设计质量直接影响：

- **系统稳定性**：通过防腐层隔离外部系统变化
- **集成效率**：通过统一接口简化集成复杂度
- **可监控性**：通过调用日志支持问题排查
- **可扩展性**：通过适配器模式支持新系统接入

核心设计原则：

1. **防腐层必须**：所有外部系统集成都必须有防腐层
2. **统一接口**：内部域只依赖统一接口，不依赖具体实现
3. **适配器模式**：每个外部系统有独立的适配器实现
4. **完整日志**：所有外部调用记录完整日志
5. **健康监控**：定期检查外部系统健康状态
