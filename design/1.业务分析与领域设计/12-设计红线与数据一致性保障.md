# 12-设计红线与数据一致性保障

## 1. 设计红线的意义

本文档通过真实或典型的餐饮系统设计失败案例，提炼出平台设计必须坚守的红线原则。每条红线都对应一个真实的系统崩溃模式。

理解失败比理解成功更重要——知道哪些路走不通，才能更坚定地走正确的路。这些红线不是建议，而是强制约束。任何违反红线的设计方案，无论短期看起来多么简单，都必须被拒绝。

---

## 2. 典型失败案例与设计红线

### 案例一：用单个status字段表达订单全状态

**场景描述：**
某餐饮系统设计时用一个 `order_status` 字段表达订单的所有状态，枚举值从 `CREATED` 到 `COMPLETED`，包含支付、接单、制作、配送等所有阶段。

**崩溃过程：**
- 第一个月：系统正常运行
- 第三个月：出现部分退款需求，不知道怎么表达（订单整体是 COMPLETED 但某一行被退了）
- 第六个月：接入美团后，美团的订单状态与内部 status 映射出现歧义（美团说配送完成但内部还没关单）
- 第十个月：出现同桌加菜，加菜单与原单状态互相影响，无法独立推进
- 最终：不得不在 `order_status` 旁边加了 `payment_status`、`production_status`、`delivery_status` 等字段，互相之间缺少一致性约束，对账数据混乱

**红线 R01：**
> **禁止用单个状态字段覆盖订单全生命周期。** 必须设计多条独立状态线（交易状态、支付状态、履约状态等），各条状态线独立推进，通过领域事件协调联动。

---

### 案例二：订单行只存商品ID，不存快照

**场景描述：**
为了"省空间"，设计订单行时只保存 `product_id` 和 `quantity`，商品详情在需要时实时查询商品表。

**崩溃过程：**
- 商家修改菜品价格后，历史订单显示的金额变成了新价格，客诉出现
- 商家下架某道菜后，历史订单中该菜品无法显示名称
- 退款时无法确认退款金额（商品价格已变）
- 对账时历史账单金额与当前商品单价不一致，财务无法核实

**红线 R02：**
> **订单行必须保存下单时刻的完整快照**，包括：商品名称、规格选项名称、单价、折扣、加价项、备注。商品数据变更不能影响历史订单。

---

### 案例三：让KDS直接修改订单状态

**场景描述：**
为了简化架构，KDS 厨显完成制作后，直接向订单表发送 UPDATE 语句将订单状态改为 `COOKING_DONE`。

**崩溃过程：**
- KDS 网络抖动时，订单状态更新遗失，订单卡在制作中无法推进
- 多个 KDS 工作站并发更新同一订单状态时，出现竞争条件
- 运营后台显示的"制作完成"订单数与实际不符
- 无法追溯是哪个工作站在什么时间完成了制作
- POS 端查询订单状态时，频繁看到不一致的中间态

**红线 R03：**
> **设备只能提交动作指令和执行反馈，不能直接修改业务订单状态。** 设备的操作通过领域命令进入业务层，业务层决定是否接受、如何推进状态，并产生领域事件。

---

### 案例四：用平台商品直接等于内部商品

**场景描述：**
接入美团时，为了快速上线，直接把美团的商品数据同步到内部商品表，以美团商品ID作为主键。

**崩溃过程：**
- 美团和内部小程序的商品不一致，美团改价不影响内部，反之亦然
- 接入饿了么时，同一道菜有了三个ID（内部/美团/饿了么），维护混乱
- 美团调整分类结构时，内部商品分类被迫同步调整
- 商品下架时需要分别在三个地方操作，遗漏导致渠道仍在售卖已停售商品
- 对账时无法确认哪笔渠道流水对应哪个内部商品

**红线 R04：**
> **内部商品主数据必须独立存在，渠道商品是内部商品的投影映射，不是替代。** 主数据变更后通过同步机制推送到各渠道，不允许渠道数据反向覆盖主数据。

---

### 案例五：渠道状态直接等于内部状态

**场景描述：**
接收到美团推送的订单状态变更回调后，直接用美团的状态码更新内部订单状态。

**崩溃过程：**
- 美团的"已送达"状态并不等于内部的"订单完成"（内部还需要确认核销）
- 美团取消语义与内部取消语义不同（美团取消有平台责任和商户责任之分）
- 回调重复推送时，同一状态变更被处理了多次
- 回调乱序时（先收到"配送中"再收到"已取件"），内部状态被错误覆盖
- 渠道回调失败时（网络超时），内部状态停留在错误位置

**红线 R05：**
> **渠道状态必须在渠道集成域内完成语义翻译后，再通过命令或事件推进内部状态。** 渠道原始状态和内部业务状态之间必须有明确的映射层，支持幂等处理和乱序容错。

---

### 案例六：对账逻辑放到最后才考虑

**场景描述：**
平台上线时优先实现了点餐、支付、履约功能，对账功能"等上线稳定后再做"。

**崩溃过程：**
- 上线三个月后，美团账单与门店流水差异已经积累了数百笔
- 由于没有设计对账数据结构，历史数据无法追溯差异原因
- 优惠券核销数据没有完整记录，无法确认哪些券被核销过
- 财务对账需要人工逐笔比对，工作量无法持续
- 最终不得不停服半个月重新设计数据结构

**红线 R06：**
> **结算与对账是一等公民，必须在系统设计阶段同步考虑。** 每笔交易从创建开始就应携带结算相关的基础数据（金额拆分、渠道归属、优惠来源），而不是事后补算。

---

### 案例七：同桌多单使用同一个订单关联关系

**场景描述：**
正餐场景下，同一桌先后点了三次菜（先点、加菜、再加菜），系统用一个 `parent_order_id` 字段把后两单关联到第一单。

**崩溃过程：**
- 退菜时不知道应该退哪一单上的哪一行
- 统一结账时不确定该汇总哪些单
- 加菜单的支付状态变更时，不知道是否需要更新"主单"状态
- 翻台时清除桌台关联，导致加菜单变成"孤立订单"
- 对账时同桌三单的金额加起来与收银台金额不一致

**红线 R07：**
> **订单关系必须用显式关系对象建模（`sale_order_relation`），而不是用单一外键字段。** 关系类型（加菜关系、退款关系、合单关系）、方向、有效期都必须正式建模，支持复杂的多单协同。

---

### 案例八：ISV Token失效不告警

**场景描述：**
美团 ISV 授权 Token 有效期为 30 天，需要定期刷新。某次刷新任务因后台服务异常失败，但没有告警机制。

**崩溃过程：**
- Token 失效后，所有美团操作（接单、状态同步、菜品更新）全部静默失败
- 门店仍在正常接收美团订单的推送（因为是平台主动推入），但回调响应失败
- 三小时后，美团平台因超时未响应开始对商户评分降权
- 运营人员发现时，已有 47 笔订单处于异常状态
- 恢复 Token 后，历史状态需要手动补偿同步

**红线 R08：**
> **ISV Token 状态是一级监控指标。** Token 刷新失败必须立即告警，Token 临近过期必须提前预警。Token 失效期间的操作必须有缓存和补偿机制，恢复后可自动重放。

---

### 案例九：高峰期不做削峰直接透传

**场景描述：**
午餐高峰期，大量订单同时进入，系统直接将所有订单同步写入数据库并同步发送 KDS 指令。

**崩溃过程：**
- 数据库连接池耗尽，新订单开始报错
- KDS 收到大量并发消息，显示混乱
- POS 收款后订单写入超时，收款成功但订单未创建
- 回调处理积压，美团订单超时未响应被取消
- 服务恢复后，积压的操作集中爆发，再次压垮数据库

**红线 R09：**
> **峰值流量必须通过队列和异步机制承接。** 订单进入消息队列后立即响应成功，实际处理在消费端控速完成。KDS 指令推送必须有背压机制，防止设备端积压过多。

---

### 案例十：运营人工修单代替系统补偿

**场景描述：**
当出现回调失败、状态不一致等异常时，系统没有自动补偿机制，完全依赖运营人员登录后台手动修改订单状态。

**崩溃过程：**
- 运营人员工作时间外的异常无法及时处理
- 人工修改没有校验规则，经常出现状态跳转不合法的情况
- 修改历史没有留痕，无法追溯是谁在什么时间做了什么操作
- 随着异常量增加，运营人员疲于应对，开始批量操作，遗漏增多
- 最终大量"修复"操作本身引入了新的数据不一致

**红线 R10：**
> **系统必须有自动补偿机制（重试、状态对账、补偿任务）。** 人工介入是兜底手段，不是主要处理路径。人工操作必须有权限控制、合法性校验和完整操作日志。

## 3. 设计红线总表

| 编号 | 红线 | 对应反模式 |
|---|---|---|
| R01 | 订单必须有多条独立状态线 | 单状态字段覆盖全生命周期 |
| R02 | 订单行必须保存下单快照 | 仅存商品ID、实时查询 |
| R03 | 设备只能提交指令，不能直接改状态 | KDS/POS直接写订单表 |
| R04 | 商品主数据独立，渠道商品是投影 | 用渠道商品替代内部主数据 |
| R05 | 渠道状态必须经语义翻译后再推进内部状态 | 直接用渠道状态更新内部状态 |
| R06 | 对账数据从交易创建时就必须随行 | 对账逻辑延后设计 |
| R07 | 订单关系必须用显式关系对象建模 | 用单一外键表达复杂关系 |
| R08 | ISV Token 是一级监控指标 | Token失效无告警无补偿 |
| R09 | 峰值流量必须通过队列异步承接 | 高并发直接同步透传 |
| R10 | 系统必须有自动补偿机制 | 完全依赖人工修单 |
| R11 | 不让外部渠道直接定义内部主数据 | 外部系统字段倒推内部模型 |
| R12 | 不让单店试点模型直接扩展为平台模型 | 以单店系统做多租户改造 |

## 4. 不共表红线

以下对象对必须分表存储，不允许共用同一张数据库表：

| 对象对 | 禁止共表的原因 |
|---|---|
| `store`（门店）vs `channel_shop_binding`（渠道绑定） | 生命周期不同：门店长期存在，渠道绑定可多次创建/解绑；数据所有权不同：门店属于租户与组织域，渠道绑定属于渠道集成域 |
| `product`（内部商品）vs `channel_product`（渠道商品） | 主数据 vs 投影：内部商品是真相源，渠道商品是派生映射；变更频率不同：渠道商品随平台结构频繁变化，不能污染主数据 |
| `sale_order`（统一订单）vs `channel_order`（渠道原始订单） | 语义不同：统一订单是内部业务真相，渠道原始订单是外部平台的原始数据快照；修改权不同：统一订单由交易域管理，渠道原始订单只读保存 |
| `work_unit`（工作单）vs `sale_order`（销售订单） | 域边界：工作单属于履约与生产域，销售订单属于交易域；生命周期不同：一个订单可产生多个工作单，工作单完成后订单仍可继续 |
| `settlement_fact`（结算事实）vs `sale_order`（销售订单） | 派生关系：结算事实是从订单派生的财务视图，不是订单本身；修改规则不同：结算事实一旦生成不可修改，订单可以变更 |
| `exception_record`（异常记录）vs 业务对象 | 治理域独立：异常记录属于运营治理域，不能与业务对象耦合；审计要求：异常记录需要独立的保留策略和访问控制 |

## 5. 数据一致性保障模式

### 5.1 事件存储与至少一次投递

**模式：** 业务操作与事件写入在同一本地事务中完成，异步投递到消息队列。

```
BEGIN TRANSACTION
  1. 执行业务操作（更新聚合状态）
  2. 写入事件存储表（event_store）
COMMIT

异步任务：
  3. 扫描未投递事件
  4. 投递到消息队列
  5. 标记为已投递
```

**保障：** 即使消息队列暂时不可用，事件不会丢失。服务重启后可从事件存储表重新投递。

### 5.2 事件消费失败处理

**策略：**
- 消费失败进入死信队列（DLQ）
- 指数退避重试（1s → 2s → 4s → 8s → 最大30s）
- 超过最大重试次数 → 转入人工处理队列
- 所有重试记录可查询，支持手动触发重试

### 5.3 幂等处理

**原则：** 同一事件被消费多次，产生的副作用与消费一次相同。

**实现：**
```sql
-- 消费前检查
SELECT id FROM processed_events WHERE event_id = ? AND consumer = ?
-- 如果已存在，跳过处理
-- 处理完成后记录
INSERT INTO processed_events (event_id, consumer, processed_at) VALUES (?, ?, NOW())
```

**适用场景：** 所有事件消费者都必须实现幂等，特别是涉及金额变更、状态变更的操作。

### 5.4 乐观锁

**原则：** 聚合根上维护 `version` 字段，并发修改时检测冲突。

```java
// 更新时携带版本号
UPDATE sale_order SET status = ?, version = version + 1
WHERE order_id = ? AND version = ?
// 影响行数为0则表示并发冲突，抛出异常重试
```

**适用场景：** 订单状态变更、库存扣减、结算事实生成等高并发写操作。

### 5.5 Saga补偿模式

**原则：** 跨域的多步骤操作，每一步都有对应的补偿动作。

```
正向流程：
  Step1: 交易域 - 锁定库存
  Step2: 渠道域 - 回传接单确认
  Step3: 履约域 - 创建生产计划

补偿流程（Step3失败时）：
  Compensate Step2: 渠道域 - 回传拒单
  Compensate Step1: 交易域 - 释放库存
```

**实现：** 通过 `compensation_task` 表记录每个步骤的补偿任务，失败时自动触发。

### 5.6 跨域数据所有权

**原则：** 每个域只拥有自己的数据，跨域访问只通过以下方式：
- **事件**：异步通知，消费方自行维护本地投影
- **命令接口**：同步调用，有明确的请求/响应契约
- **查询接口**：只读访问，不允许通过查询接口修改数据

**禁止：**
- 跨域直接访问数据库表
- 绕过领域层直接操作聚合
- 在多个域中维护同一业务对象的副本并各自修改

## 6. 失败案例共同规律

这些失败案例的共同规律是：

- **短期便捷导致长期崩溃**：每个反模式在最初都"看起来简单"，但随着业务复杂度增加，技术债务呈指数级增长
- **状态是最脆弱的地方**：状态设计不合理，整个系统的数据一致性都会受影响，且修复成本极高
- **边界模糊必然产生耦合**：设备/渠道/业务三层不清晰，最终相互侵入，任何一层的变化都会引发连锁反应
- **异常和补偿是必须设计的主流程**：不是事后加工的边角，餐饮系统的异常率远高于普通电商，必须作为一等公民设计

后续所有域设计必须以本文的红线为约束，在方案选择上主动避开已知的崩溃模式。
