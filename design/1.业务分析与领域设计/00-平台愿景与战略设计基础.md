# 00-平台愿景与战略设计基础

## 1. 平台定位与核心价值主张

### 1.1 平台是什么

本平台是**购物中心餐饮数字化经营中台**，服务于购物中心内的餐饮商户群体。

平台不是一个简单的点餐系统，也不是一个外卖聚合工具。它的本质是：**以购物中心为运营主体，以餐饮商户为服务对象，以数字化手段统一管理多业态、多渠道、多门店的餐饮经营全链路**。

### 1.2 核心价值主张

**对购物中心（平台运营方）：**
- 统一管理旗下所有餐饮商户的数字化经营
- 以ISV身份代理商户接入美团/饿了么，掌握渠道数据主权
- 跨商户、跨渠道的经营数据统一治理和对账
- 为商户提供增值服务（会员、营销、数据分析），形成平台粘性

**对餐饮商户（租户）：**
- 一套系统管理堂食/自取/外卖全渠道订单
- 无需分别对接各外卖平台，降低运营复杂度
- 统一的门店执行网络（POS/KDS/打印/打包）
- 跨渠道的经营数据汇总和对账

**对消费者：**
- 购物中心自有小程序提供统一的点餐体验
- 支持堂食扫码、自取预点、外卖配送多种场景

### 1.3 平台的差异化竞争力

平台的核心差异化不在于功能多，而在于**统一**：

| 统一维度 | 具体含义 |
|---|---|
| 统一入口 | 堂食/自取/小程序外卖/美团/饿了么，进入同一条处理链路 |
| 统一执行 | 所有渠道的订单，在同一套门店执行网络中处理 |
| 统一治理 | 跨渠道、跨门店、跨业态的经营数据，进入统一的治理和对账体系 |
| 统一扩展 | 新增门店、新接渠道、新增业态时，不需要重新设计核心模型 |

## 2. 餐饮行业经营本质

### 2.1 餐饮经营的六条核心链路

理解餐饮经营的本质，是系统设计的起点。餐饮经营不是简单的"下单-做菜-送餐"，而是六条并行运转的链路：

**链路一：商品经营链路**
```
商品研发 → 定价 → 上架 → 渠道同步 → 动态调整（沽清/补货/改价）
```
核心对象：商品主数据、菜单、价格规则、可售规则

**链路二：交易链路**
```
顾客下单 → 支付 → 商户接单 → 改单/加菜 → 取消/退款 → 订单完成
```
核心对象：统一订单（sale_order）、订单行快照、状态线

**链路三：履约链路**
```
接单确认 → 生产编排 → 工作单执行 → 交付准备 → 交付完成
```
核心对象：履约订单（fulfillment_order）、生产计划、工作单

**链路四：渠道链路**
```
渠道绑定 → 商品同步 → 订单接收 → 状态回传 → 账单对账
```
核心对象：渠道绑定关系、渠道商品映射、渠道原始订单

**链路五：结算链路**
```
经营事实记录 → 渠道账单拉取 → 差异识别 → 补偿处理 → 结算关账
```
核心对象：结算事实、账单记录、对账差异

**链路六：治理链路**
```
异常发现 → 台账创建 → 人工介入 → 补偿执行 → 审计记录
```
核心对象：异常台账、补偿任务、审计日志

> **六条链路与三条主链的关系（P1-4）：**
>
> 六条核心链路描述的是**业务经营全景**（餐饮经营需要做哪些事），三条核心业务主链描述的是**系统运行主干**（系统如何组织这些能力）。两者的映射关系如下：
>
> | 六条经营链路 | 映射到三条系统主链 |
> |---|---|
> | 商品经营链路 + 渠道链路 | 支撑交易主链（商品快照、渠道防腐） |
> | 交易链路 | 交易主链 |
> | 履约链路 | 履约主链 |
> | 结算链路 + 治理链路 | 治理与结算主链 |
>
> 六条链路是业务视角的完整描述，三条主链是系统架构的核心骨干。阅读本文档时，先理解六条链路建立业务认知，再看三条主链理解系统结构。

### 2.2 餐饮经营的五个核心对象

| 核心对象 | 业务含义 | 主权域 |
|---|---|---|
| 商品（Product） | 门店能卖什么、卖多少钱 | 商品与门店经营域 |
| 订单（SaleOrder） | 顾客买了什么、付了多少 | 交易域 |
| 工作单（WorkUnit） | 厨房做什么、怎么做 | 履约与生产域 |
| 结算事实（SettlementFact） | 这笔交易实际收了多少 | 结算与对账域 |
| 异常台账（ExceptionRecord） | 哪里出了问题、怎么处理 | 运营治理域 |

### 2.3 时间维度的重要性

餐饮经营对时间极度敏感，系统设计必须将时间作为一等公民：

- **承诺时间**：顾客下单时承诺的出餐时间（影响顾客满意度）
- **实际时间**：工作单实际完成时间（影响绩效评估）
- **超时阈值**：不同业态有不同的超时定义（快餐5分钟 vs 正餐30分钟）
- **业务时间 vs 技术时间**：事件的 `occurred_at`（业务发生时间）和 `published_at`（系统发布时间）必须分开记录

### 2.4 设备是执行终端，不是真相源

POS、KDS、打印机等设备是展示指令和接受反馈的终端：
- 设备**接收**来自系统的工作单指令
- 设备**反馈**执行结果（完成/异常）给系统
- 设备**不能**直接修改业务核心对象的状态
- 设备离线时，系统必须有降级处理机制

## 3. 业态差异与系统影响

### 3.1 三种核心业态

平台需要同时支持三种差异显著的餐饮业态：

**茶饮/快餐业态**
- 特征：标准化程度高、出餐快（3-8分钟）、翻台率高
- 订单场景：堂食自助点餐、自取预点、外卖为主
- 系统重点：高并发处理、快速出单、自动接单
- 典型挑战：高峰期订单积压、沽清实时同步

**正餐业态**
- 特征：人工服务为主、出餐慢（20-40分钟）、客单价高
- 订单场景：堂食为主、预订为核心
- 系统重点：桌台管理、预订系统、加菜改单
- 典型挑战：预订与到店的衔接、多轮加菜的账单管理

**连锁品牌**
- 特征：多门店统一管理、品牌标准化
- 系统重点：总部菜单下发、门店个性化配置、跨店数据汇总
- 典型挑战：总部权限与门店权限的边界、多店对账

### 3.2 业态差异对系统的核心影响

| 差异点 | 茶饮/快餐 | 正餐 | 系统应对 |
|---|---|---|---|
| 接单方式 | 自动接单为主 | 人工接单为主 | 接单策略可配置 |
| 出餐时间 | 3-8分钟 | 20-40分钟 | 超时阈值可配置 |
| 改单频率 | 低（下单即做） | 高（多轮加菜） | 订单行变更事件 |
| 预订需求 | 无 | 强需求 | 预订域独立设计 |
| 外卖比例 | 高 | 低 | 渠道权重可配置 |
| 桌台管理 | 简单 | 复杂 | 桌台状态机 |

### 3.3 单店 vs 连锁的系统差异

**单店模式：**
- 菜单由门店自主管理
- 数据只在门店层面汇总
- 权限结构简单

**连锁模式：**
- 菜单分层：总部标准菜单 + 门店个性化配置
- 数据需要跨店汇总到品牌层
- 权限结构复杂：总部权限 > 区域权限 > 门店权限
- 系统设计必须支持"总部下发，门店微调"的菜单管理模式

## 4. 购物中心平台的商业模式与价值链

### 4.1 购物中心作为ISV的战略意义

购物中心以**ISV（独立软件服务商）身份**代理旗下餐饮商户接入美团/饿了么，这是平台最重要的商业模式创新：

**传统模式（商户直连）：**
```
商户 → 直接注册美团/饿了么商家账号 → 各自管理
```
问题：购物中心无法获取渠道数据，无法统一管理，商户各自为战

**ISV模式（平台代理）：**
```
购物中心（ISV）→ 统一授权 → 代理旗下所有商户 → 统一管理渠道数据
```
优势：
- 购物中心掌握渠道数据主权
- 商户无需单独对接渠道，降低运营成本
- 平台可以提供跨渠道的数据分析服务
- 形成平台对商户的服务粘性

### 4.2 ISV授权模型

```
购物中心（ISV主体）
    ├── 美团ISV授权
    │   ├── 商户A的美团店铺（通过ISV代理）
    │   ├── 商户B的美团店铺（通过ISV代理）
    │   └── ...
    └── 饿了么ISV授权
        ├── 商户A的饿了么店铺（通过ISV代理）
        └── ...
```

**关键约束：**
- ISV Token有有效期，必须定期刷新
- Token过期会导致所有代理商户的渠道功能失效
- 系统必须有Token即将过期的预警机制（提前7天告警）

### 4.3 平台价值链

```
消费者 → 渠道（小程序/美团/饿了么）
    ↓ 订单流
购物中心平台（中台）
    ↓ 执行指令
餐饮商户（门店执行）
    ↓ 经营数据
购物中心运营团队（治理与分析）
    ↓ 数据输出
品牌总部ERP（经营报表）
```

## 5. 渠道生态全景

### 5.1 渠道类型

平台需要支持四类渠道，每类渠道有不同的技术接入方式：

| 渠道类型 | 代表 | 接入方式 | 特点 |
|---|---|---|---|
| 一方渠道 | 购物中心小程序 | 直接集成 | 完全自主，数据主权在平台 |
| 门店直连 | POS机 | 直接集成 | 线下收银，无渠道费用 |
| 三方外卖（ISV代理） | 美团、饿了么 | ISV API | 需要ISV授权，有渠道佣金 |
| 三方外卖（商户直连） | 美团、饿了么 | 商户API | 商户自有账号，平台辅助管理 |

### 5.2 统一三维度模型

这是平台最重要的设计决策之一。所有订单必须用三个维度来完整描述：

**维度一：渠道类型（channel_type）**
描述订单从哪个渠道进来（自有渠道含终端形态，三方渠道含平台来源）：
- `MINI_PROGRAM`：购物中心小程序
- `H5`：桌台扫码 H5（微信/支付宝内嵌浏览器）
- `POS_DIRECT`：门店 POS 直接录单
- `KIOSK`：自助点餐机
- `MEITUAN`：美团外卖平台
- `ELEME`：饿了么平台
- `DOUYIN`：抖音团购/外卖平台

**维度二：订单场景（order_scene）**
描述顾客的消费场景：
- `DINE_IN`：堂食（在店内用餐）
- `TAKEAWAY_SELF`：自取（到店自取）
- `DELIVERY_ONESELF`：商家自配送
- `DELIVERY_PLATFORM`：平台配送（美团/饿了么配送）

**维度三：履约类型（fulfillment_type）**
描述订单如何完成交付：
- `TABLE_SERVICE`：桌台服务（堂食上菜）
- `SELF_PICKUP`：自取核销
- `THIRD_PARTY_DELIVERY`：第三方配送（达达/顺丰同城等）
- `PLATFORM_DELIVERY`：平台自有配送
- `STORE_DELIVERY`：门店自配送（商家自行安排配送，对应 DELIVERY_ONESELF 场景）

**为什么需要三个维度？**

因为渠道、场景、履约三者并非一一对应：
- 小程序下单（MINI_PROGRAM）可以是堂食（DINE_IN + TABLE_SERVICE）
- 小程序下单（MINI_PROGRAM）也可以是外卖（DELIVERY_ONESELF + THIRD_PARTY_DELIVERY）
- 美团下单（MEITUAN）通常是渠道平台配送（DELIVERY_PLATFORM + THIRD_PARTY_DELIVERY）——由美团/饿了么骑手完成配送，属于第三方配送履约
- POS录单（POS_DIRECT）可以是堂食，也可以是自取

三维度的组合覆盖了所有真实业务场景，且每个维度的变化不影响其他维度的处理逻辑。

### 5.3 渠道差异的消化原则

外部渠道（美团/饿了么）的差异性必须在**渠道集成域内消化**，不允许渗透到核心域：

- 美团的订单字段格式 ≠ 饿了么的订单字段格式
- 美团的状态码 ≠ 饿了么的状态码
- 美团的取消规则 ≠ 饿了么的取消规则

这些差异全部由渠道防腐层翻译为内部统一格式，交易域只处理统一格式的订单。

## 6. 典型失败案例与设计红线

### 6.1 为什么需要设计红线

在没有明确设计约束的情况下，系统往往会在迭代中逐渐退化：
- 为了快速实现某个功能，直接在A域访问B域的数据表
- 为了减少代码量，把渠道特有字段直接加到核心订单表
- 为了方便查询，在多个域各自维护同一份数据的副本

这些"捷径"在短期内有效，但会导致系统长期难以维护和扩展。设计红线是防止这些退化的护栏。

### 6.2 十二条设计红线

**R01：渠道字段不穿透核心域**
- 违反示例：在 `sale_order` 表中加 `meituan_order_id`、`eleme_shop_id` 字段
- 正确做法：渠道原始数据保留在 `channel_order` 表，通过关联关系引用

**R02：设备不直接改订单状态**
- 违反示例：KDS设备直接调用接口将订单状态改为"已完成"
- 正确做法：KDS反馈工作单完成 → 履约域处理 → 通过事件通知交易域更新订单状态

**R03：跨域不共表**
- 违反示例：`sale_order` 和 `fulfillment_order` 存在同一张表，用字段区分
- 正确做法：各域有独立的数据表，通过业务ID关联

**R04：支付与订单状态必须强一致**
- 违反示例：支付成功后异步更新订单状态，中间有窗口期
- 正确做法：支付确认和订单状态更新在同一事务中完成

**R05：优惠券核销必须幂等**
- 违反示例：网络重试导致同一张券被核销两次
- 正确做法：核销操作以 `order_id` 为幂等键，重复请求返回相同结果

**R06：退款必须有记录**
- 违反示例：退款操作成功但未创建退款记录
- 正确做法：退款发起和退款记录创建在同一事务中完成

**R07：菜单变更不影响历史订单**
- 违反示例：商品改价后，历史订单的价格也跟着变了
- 正确做法：下单时创建商品快照，快照一旦写入不再变化

**R08：渠道账单差异必须可追溯**
- 违反示例：对账差异只记录金额，不记录涉及的订单ID
- 正确做法：差异记录必须关联 `affected_order_ids`

**R09：异常不能静默失败**
- 违反示例：渠道同步失败后只打日志，没有告警和重试机制
- 正确做法：所有异常必须进入异常台账，有重试和告警机制

**R10：Token过期必须提前预警**
- 违反示例：ISV Token过期后才发现，导致所有渠道功能失效
- 正确做法：Token有效期不足7天时触发告警，提前刷新

**R11：多租户数据必须严格隔离**
- 违反示例：查询时忘记加 `tenant_id` 过滤条件，导致数据泄露
- 正确做法：所有查询必须携带 `tenant_id`，框架层强制校验

**R12：事件必须持久化后再发布**
- 违反示例：直接发布消息到MQ，服务宕机导致事件丢失
- 正确做法：事件先写入事件表（与业务操作同一事务），再异步投递到MQ

## 7. 参与方、租户与组织模型

### 7.1 四层组织结构

平台的组织结构分为四层，每层有不同的权限和职责：

```
平台层（Platform）
    └── 购物中心项目（CommercialProject）
            └── 项目租户（ProjectTenant）
                    └── 品牌（Brand）
                            └── 门店（Store）
```

**平台层：** 平台运营方，管理所有购物中心项目

**购物中心项目（CommercialProject）：**
- 代表一个具体的购物中心（如"XX广场"）
- 是ISV授权的主体（以项目名义申请美团/饿了么ISV资质）
- 管理项目内所有商户的渠道接入

**项目租户（ProjectTenant）：**
- 代表入驻购物中心的一个商户主体（法人实体）
- 是结算的主体（收款方）
- 一个商户主体可以在同一购物中心经营多个品牌

**品牌（Brand）：**
- 代表一个餐饮品牌（如"XX奶茶"）
- 管理品牌级别的菜单和营销策略
- 一个品牌可以有多个门店

**门店（Store）：**
- 代表一个具体的经营门店
- 是订单处理的最小单元
- 有独立的菜单、营业时间、设备配置

### 7.2 关键设计约束

- `store_id` 是所有业务对象的基础关联键，几乎所有业务操作都以门店为单位
- `tenant_id` 是多租户隔离的关键，所有查询必须携带
- 组织结构变更（如品牌迁移、门店转让）不影响历史订单数据
- 渠道绑定关系（channel_shop_binding）在门店层面管理，一个门店可以绑定多个渠道

### 7.3 桌台模型

桌台是正餐业态的核心对象，有独立的状态机：

```
AVAILABLE（空闲）
    ↓ 预订确认
RESERVED（已预订）
    ↓ 顾客到店 / 直接开台
OCCUPIED（使用中）
    ↓ 结账完成
CLEANING（待清台）
    ↓ 服务员确认清台
AVAILABLE（空闲）
```

桌台状态由**租户与组织域**管理，是桌台状态的唯一真相源。状态变更由其他域的事件触发：
- `BookingConfirmed` → 桌台变为 RESERVED
- `BookingArrived` / 直接开台 → 桌台变为 OCCUPIED
- `TableSessionClosed` / 结账完成 → 桌台变为 CLEANING
- 服务员确认清台 → 桌台变为 AVAILABLE

> 注意：结账完成后桌台不直接回到 AVAILABLE，必须经过 CLEANING 状态由服务员确认清台后才可再次使用。对于不需要清台确认的轻餐业态，可在业态裁剪规则中配置跳过 CLEANING 直接回到 AVAILABLE，但不允许在总纲中省略该状态。

## 8. 系统设计的四条主线

### 8.1 单一真相源

每个业务对象有且只有一个真相源，其他系统的视图都从真相源派生：

- 商品主数据的真相源：商品与门店经营域
- 订单的真相源：交易域
- 履约执行的真相源：履约与生产域
- 结算事实的真相源：结算与对账域

**反模式：** 多个系统各自维护同一对象的"自己的版本"，导致数据不一致。

### 8.2 渠道差异在边界消化

外部渠道的差异性必须在渠道集成域内消化，不允许渗透到核心域：

- 渠道特有字段只在渠道域的数据表中存储
- 渠道状态码在防腐层翻译为内部枚举
- 核心域的代码中不出现任何渠道特有的条件判断

### 8.3 设备是执行终端而非真相源

POS、KDS、打印机等设备：
- 只能**接收**系统下发的指令
- 只能**反馈**执行结果
- 不能直接修改业务核心对象的状态

### 8.4 异常是主流程的组成部分

取消、退款、缺货、漏单、对账差异等"异常"都是设计必须覆盖的主流程：
- 每种异常场景都有明确的处理路径
- 异常处理结果可追溯
- 不能作为事后补丁处理

## 9. 本文结论

平台的战略设计基础可以用一句话概括：

**以购物中心为运营主体，以统一三维度（渠道类型/订单场景/履约类型）为核心模型，以事件驱动架构为协作骨架，构建一个能够统一管理多业态、多渠道、多门店餐饮经营全链路的数字化中台。**

这个定位决定了后续所有设计决策的方向：
- 为什么需要渠道防腐层（因为渠道差异不能污染核心域）
- 为什么需要统一三维度（因为渠道/场景/履约三者独立变化）
- 为什么需要事件驱动（因为各域需要松耦合协作）
- 为什么需要设计红线（因为没有约束的系统会在迭代中退化）

后续各域的详细设计，都是在这个战略基础上的具体展开。
