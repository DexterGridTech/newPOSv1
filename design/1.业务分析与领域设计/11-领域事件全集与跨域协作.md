# 11-领域事件全集与跨域协作

## 1. 领域事件的战略意义

领域事件是平台跨域协作的核心骨架。通过事件驱动架构，各限界上下文之间保持松耦合：发布方只负责描述"已发生的事实"，消费方自主决定如何响应。这使得新功能可以通过订阅已有事件来实现，而无需修改发布方代码。

事件设计的质量直接决定域间耦合程度和系统的可扩展性。

---

## 2. 事件设计原则

1. **事件描述事实，不描述意图**：用 `OrderAccepted`（订单已接单），而不是 `StartCooking`（开始做菜）
2. **事件必须携带足够的上下文**：消费方不需要再回查发布方的数据就能处理事件
3. **事件必须幂等处理**：同一事件重复到达，消费方的处理结果相同
4. **事件必须携带业务时间和技术时间**：`occurred_at`（事件在业务上发生的时间）和 `published_at`（事件被发布的时间）
5. **事件版本管理**：事件结构变更时，通过版本号区分，消费方向前兼容

---

## 3. 事件格式标准

所有事件遵循统一的 envelope 格式：

```json
{
  "event_id": "uuid",
  "event_type": "OrderAccepted",
  "event_version": "1.0",
  "aggregate_type": "SaleOrder",
  "aggregate_id": "order_id",
  "occurred_at": "ISO8601时间戳（业务时间）",
  "published_at": "ISO8601时间戳（发布时间）",
  "source_domain": "trading",
  "correlation_id": "链路追踪ID",
  "payload": { ... }
}
```

## 4. 商品与门店经营域事件

### 4.1 `MenuPublished`（菜单已发布）

**触发条件：** 门店菜单经审核后正式发布生效

**发布方：** 商品与门店经营域

**消费方：**
- 渠道集成域（触发渠道商品同步）
- 运营治理域（记录审计日志）

**Payload：**
```json
{
  "store_id": "string",
  "menu_id": "string",
  "menu_version": "string",
  "effective_date": "ISO8601",
  "affected_channels": ["MINI_PROGRAM", "MEITUAN", "ELEME"],
  "changed_items": [
    {
      "product_id": "string",
      "change_type": "ADD|UPDATE|REMOVE",
      "channel_sync_required": true
    }
  ]
}
```

**补偿策略：** 若渠道同步失败，在异常台账中创建"菜单同步异常"记录，由渠道域定期重试。

---

### 4.2 `ProductAvailabilityChanged`（商品可售状态变更）

**触发条件：** 商品上架/下架/售罄/补货/改价

**发布方：** 商品与门店经营域

**消费方：**
- 渠道集成域（实时同步各渠道库存状态）

**Payload：**
```json
{
  "store_id": "string",
  "product_id": "string",
  "variant_id": "string",
  "previous_status": "AVAILABLE|UNAVAILABLE|SOLD_OUT",
  "current_status": "AVAILABLE|UNAVAILABLE|SOLD_OUT",
  "effective_from": "ISO8601",
  "reason": "MANUAL|AUTO_STOCK|SCHEDULE|PRICE_CHANGE"
}
```

> **统一事件策略（P2-1）：** `ProductAvailabilityChanged` 是商品可售状态变更的**唯一标准事件**，通过 `reason` 字段区分触发原因（售罄/补货/改价/手动调整）。第11节中的 `ProductSoldOut`、`ProductStockRestored`、`PriceChanged` 是早期专用事件，已被本事件统一替代，不再新增消费方，存量消费方应迁移到 `ProductAvailabilityChanged`。

---

### 4.3 `StoreOperatingStatusChanged`（门店营业状态变更）

> **边界说明：** `Store.store_status`（归属租户与组织域）描述门店实体的生命周期状态（`PREPARING | OPERATING | SUSPENDED`），由平台管理员操作；`StoreOperatingConfig.operating_status`（归属商品与门店经营域）描述门店每日营业状态（`OPERATING | PAUSED | CLOSED`），由店长每日操作。本事件由**商品与门店经营域**发布，仅反映 `StoreOperatingConfig.operating_status` 的变化。

**触发条件：** 门店开店/暂停/打烊

**发布方：** 商品与门店经营域

**消费方：**
- 渠道集成域（同步各渠道店铺营业状态）
- 交易域（停止接受新订单）

**Payload：**
```json
{
  "store_id": "string",
  "previous_status": "OPERATING|PAUSED|CLOSED",
  "current_status": "OPERATING|PAUSED|CLOSED",
  "affected_channels": ["ALL"],
  "operator_id": "string",
  "reason": "MANUAL_PAUSE|AUTO_SCHEDULE|EMERGENCY"
}
```

## 13. 事件补偿与容错策略

### 13.1 事件发布失败处理

事件发布采用"事件存储 + 至少一次投递"模式：
1. 业务操作和事件写入在同一个本地事务中完成（写入事件表）
2. 异步进程从事件表读取并投递到消息队列
3. 投递成功后标记为已投递
4. 未投递的事件定期重试

### 13.2 事件消费失败处理

消费方处理失败时：
1. 进入消费失败队列（Dead Letter Queue）
2. 定期重试（指数退避策略）
3. 超过最大重试次数后，转入运营治理域创建人工处理任务
4. 所有重试记录可查

### 13.3 事件幂等处理

所有消费方必须实现幂等：
- 通过 `event_id` 去重
- 重复事件不产生重复副作用
- 记录已处理事件的 `event_id`，检查前先查重

## 14. 跨域协作模式

### 14.1 三种协作模式

**模式一：领域事件（异步通知）**

适用场景：通知"某件事已发生"，消费方对发布方没有强依赖

特征：
- 发布方不等待消费方处理结果
- 消费方可以后续订阅，不影响发布方
- 允许多个消费方订阅同一事件

典型用例：
- `OrderCompleted` → 结算域生成结算事实
- `ProductionCompleted` → 渠道域回传备餐状态
- `MenuPublished` → 渠道域触发商品同步

实现要求：
- 事件必须持久化存储再发布（防止服务宕机导致事件丢失）
- 消费方必须实现幂等
- 消费失败必须有重试和告警机制

**模式二：命令接口（同步请求）**

适用场景：请求另一个域"执行某个动作"，需要立即知道结果

典型用例：
- 创建订单时，商品域提供"商品快照查询"接口（同步）
- 支付确认时，会员系统提供"积分核销"接口（同步）
- 小程序外卖时，配送平台提供"派单"接口（同步）

实现要求：
- 必须定义明确的超时时间
- 必须有降级策略（如接口超时时的兜底处理）
- 不允许形成循环调用链

**模式三：查询投影（跨域只读查询）**

适用场景：需要聚合多个域的数据展示，但不需要修改其他域的状态

典型用例：
- 平台运营后台的多维度数据汇总
- 门店后台的经营报表
- 消费者小程序的订单历史查询

实现要求：
- 通过事件订阅维护查询投影数据（不实时查询各域主表）
- 明确标注数据的时效性（实时/准实时/T+1）
- 查询失败不影响主业务流程

### 14.2 关键防腐层设计

**渠道防腐层（最重要的防腐层）**

```
外部渠道侧                  防腐层                  内部侧
                    ┌─────────────────────┐
美团订单格式   ───▶  │  渠道订单适配器       │───▶  统一订单命令
饿了么订单格式 ───▶  │  (ChannelAdapter)    │
美团状态码     ───▶  │  状态码翻译器         │───▶  内部状态枚举
饿了么状态码   ───▶  │  (StatusTranslator)  │
                    │  幂等处理器           │      幂等保障
                    │  (IdempotencyHandler) │
                    │  签名验证器           │      安全校验
                    │  (SignatureVerifier)  │
                    └─────────────────────┘
```

防腐层规则：
1. 外部渠道的字段名、枚举值、状态码只在防腐层内出现，不穿透到核心域
2. 防腐层负责翻译、验证、去重三项职责
3. 防腐层保留外部原始数据（不丢弃），同时生成内部统一格式
4. 各渠道有独立的适配器实现，新增渠道只需新增适配器

渠道状态码翻译示例：
```
美团订单状态 → 内部动作
  WM_POI_RECEIVED(20) → AcceptOrderCommand
  CANCEL_BY_USR(30)   → CancelOrderCommand(initiator=CUSTOMER)
  CANCEL_BY_SYS(35)   → CancelOrderCommand(initiator=PLATFORM)
  INCOME_DISPATCH(50) → 更新配送任务，不影响订单主状态
  INCOME_COMPLETE(80) → FulfillmentCompletedCommand

饿了么订单状态 → 内部动作
  NEW(1)              → ChannelOrderReceived事件
  ACCEPTED(3)         → 已由我们主动回传，无需处理
  CANCEL_BY_BUYER(7)  → CancelOrderCommand(initiator=CUSTOMER)
  COMPLETED(10)       → FulfillmentCompletedCommand
```

**会员系统防腐层**

统一内部抽象接口：
```
MemberService:
  - verifyMember(member_id) → MemberInfo
  - queryAvailableCoupons(member_id, store_id) → List<Coupon>
  - lockCoupon(coupon_id, order_id) → LockResult
  - redeemCoupon(coupon_id, order_id) → RedeemResult
  - cancelCouponRedemption(coupon_id) → Result
  - queryPoints(member_id) → PointsInfo
  - deductPoints(member_id, points, order_id) → DeductResult
  - refundPoints(member_id, points, original_order_id) → Result
```

**配送平台防腐层**

统一配送接口：
```
DeliveryService:
  - createDeliveryOrder(deliveryRequest) → DeliveryOrderId
  - cancelDeliveryOrder(deliveryOrderId, reason) → Result
  - queryDeliveryStatus(deliveryOrderId) → DeliveryStatus
  - estimateDeliveryFee(from, to, weight) → FeeEstimate
```

### 14.3 最终一致性策略

| 场景 | 容忍延迟 | 处理策略 |
|---|---|---|
| 渠道状态回传 | 秒级 | 异步事件+重试 |
| 菜单同步到渠道 | 分钟级 | 异步事件+重试 |
| 结算事实生成 | 分钟级 | 事件触发+补偿对账 |
| 品牌ERP数据输出 | 小时级（日结为主） | 批量推送 |

**不可接受最终一致性的场景（必须强一致性）：**
- 支付确认与订单状态更新（支付了但订单没更新=资损）
- 优惠券核销与订单折扣（券已消费但订单没减=资损）
- 退款发起与退款记录创建（退款了但没记录=无法对账）

---

### 14.4 依赖方向规则

```
允许的依赖方向（单向）：
  应用层 → 核心域
  渠道域 → 交易域（通过命令接口）
  履约域 → 商品域（通过查询接口，仅查生产画像）
  结算域 → 交易域（通过事件订阅）

禁止的依赖方向：
  核心域 → 应用层
  交易域 → 渠道域（反方向）
  商品域 → 交易域（循环）
```

## 15. 本文结论

领域事件是平台跨域协作的核心骨架。通过完整定义每个事件的 payload、触发条件、发布方、消费方和补偿策略，确保：

- 域间耦合只通过事件传递，不直接共享数据
- 系统具备应对事件失败的自动补偿能力
- 每个状态转换都可溯源和审计
- 新增功能通过订阅已有事件实现，不需要修改事件发布方

跨域协作核心原则：
1. **事件优先**：能用事件解决的，不用同步调用
2. **防腐层必须**：所有外部系统集成都必须有防腐层，外部变化不穿透内部
3. **ID引用，不表引用**：跨域只传递业务 ID，不访问对方的数据表
4. **最终一致性有边界**：接受延迟一致，不接受永久不一致，强一致性场景必须用事务保障
