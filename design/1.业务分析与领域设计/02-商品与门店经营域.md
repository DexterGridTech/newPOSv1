# 商品与门店经营域

## 1. 域定位

商品与门店经营域是平台的**经营真相源**，维护"门店当前能卖什么、怎么卖、卖多少钱"的完整事实。

本域的设计质量直接影响四个下游关切：

| 下游关切 | 本域保障机制 | 设计决策理由 |
|---|---|---|
| 交易域的订单准确性 | 商品快照机制 | 快照一旦生成不可变，订单商品信息不受后续商品变更影响。如果不快照，商品改价后历史订单金额会失真 |
| 渠道域的同步效率 | 投影机制 | 内部主数据与外部渠道格式通过投影隔离，避免核心域被美团/饿了么等外部格式污染 |
| 履约域的生产准确性 | 生产画像（ProductionProfile） | 商品天然携带制作岗位和时长信息，履约域不需要二次维护 |
| 门店的经营灵活性 | 可售规则引擎 | 统一规则引擎处理时段/渠道/库存/限量判断，避免散落各处的if-else |

---

## 2. 域职责边界

### 2.1 核心职责

- 维护商品主数据（品牌商品库、门店商品库）
- 管理门店菜单及其版本演进
- 控制商品可售规则（时段、库存、渠道）
- 生成各渠道的商品投影（美团/饿了么/小程序）
- 提供下单时的商品快照服务
- 管理门店营业状态和营业时段

### 2.2 不负责的事项

| 事项 | 归属域 | 边界划分理由 |
|---|---|---|
| 订单管理 | 交易域 | 订单是独立聚合，有自己的生命周期和状态机 |
| 库存实物流转 | ERP | 本域只管"可售库存"这个抽象概念，不管仓库实物进出 |
| 直接对接外部渠道API | 渠道域 | 防腐层职责，外部API变化不应渗入商品核心模型 |
| 商品销售统计 | 结算域 | 统计是下游消费行为，不是商品本身的属性 |

## 3. 核心聚合根设计

### 3.1 商品聚合 `Product`

**聚合根：** `product`

**设计决策：** Product 作为聚合根而非实体，因为商品是经营域的第一公民，所有可售判断、快照生成、渠道投影都以商品为锚点。规格（Variant）、加料（Modifier）、套餐组成（ComboItem）作为值对象内嵌在商品聚合内，保证商品信息的一致性边界。

#### 3.1.1 基础信息字段

```
product_id              商品唯一标识（UUID）
product_code            商品业务编码（品牌内唯一）
product_name            商品名称
product_name_en         英文名称（可选）
product_type            商品类型（SINGLE|COMBO|MODIFIER）
category_id             商品分类ID
brand_id                归属品牌（连锁场景）
store_id                归属门店（单店场景，连锁为null）
ownership_scope         归属范围（BRAND|STORE）
base_price              基础价格（分）
price_unit              计价单位（ITEM|WEIGHT）
product_image_url       商品主图
product_images          商品图片列表（JSON）
description             商品描述
allergen_info           过敏原信息
nutrition_info          营养成分信息（JSON）
tags                    标签（热销/新品/推荐等）
sort_order              排序权重
status                  状态（DRAFT|ACTIVE|INACTIVE|ARCHIVED）
created_at              创建时间
updated_at              更新时间
created_by              创建人
updated_by              更新人
version                 乐观锁版本号
```

**字段设计说明：**

- `ownership_scope` 区分 BRAND 与 STORE，是为了支持连锁品牌"统一商品库+门店覆盖"的模式。品牌级商品的 `store_id` 必须为 null，门店级商品的 `brand_id` 必须为 null，二者互斥，保证归属清晰。
- `base_price` 使用**分**为单位（整数），避免浮点运算误差。所有金额字段统一用分。
- `price_unit` 区分 ITEM（按份）和 WEIGHT（按重量），因为餐饮有称重类商品（如沙拉吧）。
- `version` 乐观锁版本号，用于并发控制。

#### 3.1.2 值对象：ProductVariant（商品规格变体）

```
variant_id            规格ID
variant_name          规格名称（大杯/中杯/小杯）
price_delta           价格增量（相对base_price）
sku_code              SKU编码
is_default            是否默认规格
status                规格状态
```

**设计决策：** 规格使用 `price_delta`（增量）而非绝对价格，因为当商品调价时只需改 `base_price`，所有规格的最终价格自动跟随变化，避免逐一修改每个规格价格。

#### 3.1.3 值对象：ProductionProfile（生产画像）

```
station_type          制作岗位（HOT_DRINK|COLD_DRINK|DESSERT|MAIN_COURSE）
estimated_duration    标准制作时长（秒）
complexity_level      复杂度（SIMPLE|MEDIUM|COMPLEX）
parallel_capable      是否可并行制作
equipment_required    所需设备列表
```

**设计决策：** 生产画像内嵌在商品聚合中而非放在履约域，因为制作岗位、时长、设备等信息是商品的固有属性，在商品定义时就确定了。履约域通过快照获取这些信息来排产，不需要自己维护一份映射关系。

#### 3.1.4 值对象：ModifierGroup（加料组）

```
group_id              加料组ID
group_name            加料组名称（甜度/温度/配料）
selection_type        选择类型（SINGLE|MULTIPLE）
is_required           是否必选
min_selections        最少选择数
max_selections        最多选择数
modifiers             加料项列表
  modifier_id         加料项ID
  modifier_name       加料项名称
  price_delta         价格增量
  is_default          是否默认选中
```

**设计决策：** 加料组用 `selection_type` 区分单选和多选，用 `min_selections` / `max_selections` 控制选择范围，因为不同加料组的业务语义不同：甜度是单选必选（必须选一个），配料是多选可选（可选0-N个）。`is_required` 标记是否必选，下单时前端据此做校验。

#### 3.1.5 值对象：ComboItem（套餐组成）

```
combo_item_id         套餐项ID
item_product_id       子商品ID
quantity              数量
is_optional           是否可选
is_replaceable        是否可替换
replace_options       可替换选项列表
  product_id          可替换商品ID
  extra_price         额外加价
sort_order            排序
```

**套餐业务规则：**

- **套餐定价**：套餐 `base_price` 为整体价格，不是子商品价格之和。这样做是因为套餐通常有整体折扣，用整体价更直观，也避免了子商品调价导致套餐价格意外变化。
- **子商品数量**：每个子商品可指定数量（如2杯咖啡+1份甜点）。
- **可选项**：标记为可选的子商品可由顾客选择是否要。
- **替换规则**：可替换商品可指定额外加价（如美式换拿铁+5元）。
- **库存检查**：下单时需检查所有子商品库存。
- **制作拆分**：履约时将套餐拆分为独立制作任务，因为子商品可能属于不同制作岗位。

#### 3.1.6 Product 状态机

```
DRAFT（草稿）
    | [PublishProduct 命令]
    v
ACTIVE（上架可售）
    | [DeactivateProduct 命令]
    v
INACTIVE（下架不可售）
    | [ReactivateProduct 命令]
    v
ACTIVE
    | [ArchiveProduct 命令]
    v
ARCHIVED（归档）
```

**状态设计理由：**

- DRAFT 与 ACTIVE 分离：新建商品不应立即可售，需要确认信息完整后才发布。
- INACTIVE 可回到 ACTIVE：下架是临时操作（如季节性商品暂停销售），不应丢失商品数据。
- ARCHIVED 是终态：归档后不可恢复，用于清理已彻底淘汰的商品，防止误操作重新上架。

#### 3.1.7 Product 不变式（Invariants）

- 商品名称不能为空
- 基础价格必须大于0
- COMBO类型商品必须关联至少2个子商品
- 套餐子商品不能是另一个套餐（防止循环依赖）
- 可替换商品必须与原商品属于同一品类
- 加料组的 `min_selections` 不能大于 `max_selections`
- 品牌级商品的 `store_id` 必须为null
- 门店级商品的 `brand_id` 必须为null

### 3.2 菜单聚合 `Menu`

**聚合根：** `menu`

**设计决策：** 菜单独立为聚合根而非 Product 的附属，因为菜单有自己的生命周期（草稿→审核→发布→停用）、版本管理、审核流程，这些都是菜单自身的领域行为。菜单通过引用关联商品，而非包含商品。

#### 3.2.1 完整字段定义

```
menu_id                 菜单唯一标识
store_id                归属门店
menu_name               菜单名称（早餐菜单/午餐菜单/全天菜单）
menu_type               菜单类型（FULL_DAY|TIME_SLOT|CHANNEL_SPECIFIC）
effective_date          生效日期
expire_date             失效日期（可选）
status                  状态（DRAFT|PENDING_REVIEW|ACTIVE|INACTIVE）
version                 菜单版本号（递增）
parent_menu_id          父菜单ID（用于版本继承）
created_at              创建时间
published_at            发布时间
published_by            发布人
审核相关字段：
review_status           审核状态（NOT_SUBMITTED|PENDING|APPROVED|REJECTED）
submitted_at            提交审核时间
reviewed_at             审核完成时间
reviewed_by             审核人
review_comment          审核意见
```

**字段设计说明：**

- `menu_type` 区分 FULL_DAY / TIME_SLOT / CHANNEL_SPECIFIC，因为不同场景的菜单生效逻辑不同：全天菜单始终生效，时段菜单只在指定时间段生效，渠道菜单只对特定渠道生效。
- `parent_menu_id` 用于版本继承：菜单一旦发布不可直接修改，修改需创建新版本（新版本的 `parent_menu_id` 指向旧版本），保留完整修改历史。
- 审核字段独立列出，因为连锁品牌场景下门店修改菜单需要总部审批。

#### 3.2.2 实体：MenuSection（菜单分区）

```
section_id            分区ID
section_name          分区名称（热销推荐/主食/饮品/甜品）
section_type          分区类型（FEATURED|CATEGORY|CUSTOM）
sort_order            排序
display_style         展示样式（GRID|LIST|CAROUSEL）
```

#### 3.2.3 实体：MenuProduct（菜单商品）

```
menu_product_id       菜单商品ID
product_id            关联商品ID
section_id            所属分区
override_price        覆盖价格（可选，覆盖商品base_price）
override_name         覆盖名称（可选）
override_image        覆盖图片（可选）
sort_order            排序
is_featured           是否推荐
daily_quota           每日限量（可选）
sold_count            已售数量（实时更新）
availability_rules    可售规则列表（JSON）
  rule_type           规则类型（TIME_SLOT|CHANNEL|STOCK）
  rule_config         规则配置
```

**设计决策：** MenuProduct 支持 `override_price` / `override_name` / `override_image`，而非直接修改商品本身，因为同一商品可能出现在多个菜单中且定价不同（如午餐菜单和晚餐菜单价格不同），覆盖机制保证商品主数据不被菜单场景污染。

#### 3.2.4 Menu 状态机

```
DRAFT（草稿编辑中）
    | [SubmitForReview 命令]
    v
PENDING_REVIEW（待审核）
    | [ApproveMenu 命令]
    v
APPROVED（审核通过）
    | [PublishMenu 命令]
    v
ACTIVE（已发布生效）
    | [DeactivateMenu 命令]
    v
INACTIVE（已停用）

异常路径：
PENDING_REVIEW → DRAFT（审核拒绝，退回修改）
```

**状态设计理由：**

- 增加 PENDING_REVIEW 和 APPROVED 两个中间状态：连锁品牌场景下，门店不能随意修改菜单，需要总部审批。APPROVED 到 ACTIVE 分离是因为审批通过后可以选择发布时机（如选择在次日凌晨生效）。
- 审核拒绝退回 DRAFT 而非独立的 REJECTED 状态：被拒绝的菜单本质上需要继续编辑，回到 DRAFT 更语义化。

#### 3.2.5 Menu 不变式

- 同一门店同一时间只能有一个ACTIVE状态的全天菜单
- 菜单必须至少包含一个分区和一个商品
- 菜单商品必须引用ACTIVE状态的商品
- 时段菜单的生效时间不能与同门店其他时段菜单重叠
- 审核通过后的菜单不能直接修改，只能创建新版本

### 3.3 门店营业配置聚合 `StoreOperatingConfig`

**聚合根：** `store_operating_config`

**设计决策：** 门店营业配置独立为聚合根，因为营业状态、接单策略、营业时段等信息的变更频率和变更原因与商品/菜单不同，它们有自己的生命周期。例如门店暂停营业不应影响商品数据，商品调价也不应触及营业配置。

#### 3.3.1 完整字段定义

```
config_id               配置ID
store_id                门店ID
operating_status        营业状态（PREPARING|OPERATING|PAUSED|CLOSED）
auto_accept_enabled     是否自动接单
accept_timeout_seconds  接单超时时间（秒）
preparation_buffer_minutes  备餐缓冲时间（分钟）
max_concurrent_orders   最大并发订单数
pause_reason            暂停原因（MANUAL|EMERGENCY|SYSTEM）
paused_at               暂停时间
paused_by               暂停操作人
resume_scheduled_at     计划恢复时间
```

**字段设计说明：**

- `auto_accept_enabled`：控制是否自动接单。POS直录单不需要接单流程，但小程序和外卖订单需要。自动接单减少门店操作负担，手动接单给门店拒单权。
- `max_concurrent_orders`：限制并发订单数，防止高峰期门店产能不足导致出餐延迟。
- `preparation_buffer_minutes`：备餐缓冲时间，用于计算预计出餐时间，避免给顾客过于乐观的等待预期。
- `pause_reason` 区分 MANUAL / EMERGENCY / SYSTEM：手动暂停是店员主动操作，紧急暂停是突发情况（如设备故障），系统暂停是系统自动触发（如并发订单超限）。

#### 3.3.2 值对象：ExtraChargeRule（附加费规则）

```
extra_charge_id       附加费ID
charge_name           名称（服务费/包间费/外卖包装费/茶位费）
charge_type           类型（SERVICE_FEE|ROOM_FEE|TAKEOUT_FEE|TEA_FEE|PACKAGING_FEE）
calc_way              计算方式（PERCENT|PER_PERSON|FIXED）
calc_amount           计算数额（比例时0.1=10%，固定时为分）
apply_scenes          适用场景（DINE_IN|TAKEAWAY|SELF_PICKUP，多选）
auto_add_to_order     是否自动加入订单
calc_after_discount   是否优惠后计算（先打折再收服务费）
allow_discount        是否参与折扣
enabled               是否启用

不变式：
  PERCENT类型的calc_amount必须在0-1之间
  PER_PERSON类型必须有就餐人数上下文
```

**设计决策：** 附加费规则放在 StoreOperatingConfig 而非订单域，因为附加费策略（收不收服务费、怎么收）是门店经营决策，不是订单行为。交易域创建订单时查询门店的附加费规则并计入订单金额。

#### 3.3.3 值对象：OperatingHours（营业时段）

```
day_of_week           星期（1-7，1为周一）
time_slots            时段列表
  slot_id             时段ID
  slot_name           时段名称（早餐/午餐/下午茶/晚餐）
  start_time          开始时间（HH:mm）
  end_time            结束时间（HH:mm）
  menu_id             关联菜单ID
  accept_order_before_minutes  提前接单时间（分钟）
  stop_order_before_minutes    停止接单时间（分钟）
```

**设计决策：** 营业时段与菜单通过 `menu_id` 关联，因为不同时段可能使用不同菜单（早餐菜单 vs 午餐菜单）。`accept_order_before_minutes` 和 `stop_order_before_minutes` 控制时段前后的接单窗口，解决"午餐时段11:30开始，但11:00就可以下预约单"和"晚餐时段20:30结束，但20:00就不再接单"的实际业务需求。

#### 3.3.4 值对象：SpecialOperatingDay（特殊营业日）

```
special_date          特殊日期
day_type              日期类型（HOLIDAY|FESTIVAL|CLOSED）
override_hours        覆盖营业时段（可选）
is_closed             是否全天关闭
reason                原因说明
```

#### 3.3.5 StoreOperatingConfig 状态机

```
PREPARING（筹备中）
    | [OpenStore 命令]
    v
OPERATING（营业中）
    | [PauseStore 命令]
    v
PAUSED（暂停营业）
    | [ResumeStore 命令]
    v
OPERATING
    | [CloseStore 命令]
    v
CLOSED（已关闭）
```

### 3.4 可售库存聚合 `SaleableStock`（P1-3）

**用途：** 管理门店商品的实时可售库存，支持沽清、库存变更日志和价格变更任务。

**设计决策：** 可售库存独立为聚合，而非内嵌在商品聚合中，因为库存的变更频率远高于商品本身（每次下单都会扣减库存），如果放在商品聚合中会导致商品聚合的乐观锁频繁冲突。分离后库存可以独立并发控制。

#### 3.4.1 完整字段定义

```
stock_id                库存ID（UUID）
store_id                门店ID
product_id              商品ID
sku_id                  SKU ID（可选）
stock_type              库存类型（DAILY|PERIOD|TOTAL）
period_id               餐段ID（PERIOD类型时）
total_quantity          总库存数
sold_quantity           已售数量
remaining_quantity      剩余数量（计算字段）
sold_out_threshold      沽清阈值（剩余<=此值时自动沽清）
status                  状态（AVAILABLE|LOW_STOCK|SOLD_OUT）
reset_policy            重置策略（DAILY_RESET|MANUAL）
last_reset_at           上次重置时间
updated_at              更新时间
```

**字段设计说明：**

- `stock_type` 区分 DAILY / PERIOD / TOTAL：DAILY 每日重置（如每天限量100份），PERIOD 按餐段重置（如午餐限量50份），TOTAL 不重置（如限量版甜品总共200份）。
- `sold_out_threshold`：不等剩余为0才沽清，而是剩余量低于阈值就提前沽清。这是餐饮行业实际需求——留出缓冲，避免最后几份库存因并发下单导致超卖。

#### 3.4.2 沽清触发规则

```
订单支付成功 → 扣减库存 → 检查remaining_quantity
remaining_quantity <= sold_out_threshold → 自动设置status=SOLD_OUT
发布 ProductSoldOut 事件
消费方：渠道集成域（外卖平台下架）、设备管理（菜单刷新命令到POS/KDS）
```

#### 3.4.3 SaleableStock 领域事件

```
ProductSoldOut（商品已沽清）
  消费方：渠道集成域（外卖平台下架）、设备管理（菜单刷新）

ProductStockRestored（商品库存已恢复）
  消费方：渠道集成域（外卖平台上架）、设备管理（菜单刷新）

PriceChanged（商品价格已变更）
  消费方：渠道集成域（同步外卖平台价格）、设备管理（菜单刷新）
```

## 4. 渠道商品投影机制

### 4.1 投影生成原理

**设计决策：** 为什么不直接把商品数据推送给各渠道，而要引入"投影"概念？因为每个渠道（美团/饿了么/小程序/抖音）要求的商品数据格式不同，字段映射关系不同，如果在商品主数据上直接适配各渠道格式，会导致核心模型被外部格式污染。投影机制将"数据转换"这个关注点从核心模型中分离出来。

```
内部商品主数据（Product）
        | 投影规则
    +---+---+-------+-------+
    |       |       |       |
    v       v       v       v
小程序投影  美团投影  饿了么投影  抖音投影
```

### 4.2 渠道商品映射表 `channel_product_mapping`（ChannelCatalogProjection）

```
mapping_id              映射ID
store_id                门店ID
product_id              内部商品ID
channel_type            渠道类型（MEITUAN|ELEME|MINI_PROGRAM）
external_product_id     外部平台商品ID
external_sku_id         外部平台SKU ID
mapping_status          映射状态（PENDING|ACTIVE|FAILED|UNMAPPED）
sync_status             同步状态（NOT_SYNCED|SYNCING|SYNCED|SYNC_FAILED）
last_sync_at            最后同步时间
sync_error_message      同步失败原因
field_mapping_config    字段映射配置（JSON）
  name_mapping          名称映射规则
  price_mapping         价格映射规则
  image_mapping         图片映射规则
  modifier_mapping      加料映射规则
created_at              创建时间
updated_at              更新时间
```

**字段设计说明：**

- `mapping_status` 与 `sync_status` 分离：映射状态表示"这个商品是否已经和外部平台商品建立了对应关系"，同步状态表示"最近一次数据同步是否成功"。两者独立变化——一个商品可以已映射但同步失败（如外部API暂时不可用）。
- `field_mapping_config` 用 JSON 存储各字段的映射规则，因为不同渠道的字段映射逻辑差异大，用结构化配置比硬编码更灵活。

### 4.3 投影同步策略

| 触发条件 | 同步方式 | 同步内容 | 设计理由 |
|---|---|---|---|
| 菜单发布 | 异步批量 | 全量商品信息 | 菜单发布是低频操作，全量同步保证一致性 |
| 商品价格变更 | 实时单个 | 价格字段 | 价格敏感，需要实时同步避免顾客在外部平台看到旧价格 |
| 商品上下架 | 实时单个 | 可售状态 | 已下架的商品继续展示在外部平台会导致无效订单 |
| 商品图片更新 | 异步单个 | 图片URL | 图片变更不影响交易，异步处理即可 |
| 加料组变更 | 异步单个 | 规格和加料 | 加料变更频率低，异步处理减少系统压力 |

## 5. 可售规则引擎

### 5.1 设计决策

为什么需要统一的可售规则引擎？因为"商品能不能卖"的判断涉及多个维度（门店营业状态、时段、菜单、商品状态、渠道、库存、限量），如果这些判断散落在各个业务流程中，会导致判断逻辑不一致、遗漏检查、难以维护。统一规则引擎提供单一入口，所有可售判断都走这里。

### 5.2 可售判断流程

```
下单请求
    |
    v
1. 检查门店营业状态（OPERATING?）
    |
    v
2. 检查当前时段是否在营业时段内
    |
    v
3. 检查商品是否在当前菜单中
    |
    v
4. 检查商品状态（ACTIVE?）
    |
    v
5. 检查渠道可售规则（该渠道是否可售?）
    |
    v
6. 检查库存规则（是否售罄?）
    |
    v
7. 检查每日限量（是否达到限量?）
    |
    v
返回可售结果（可售/不可售+原因）
```

**流程顺序设计理由：** 从粗到细排列——先检查门店级别（成本最低），再检查商品级别，最后检查库存和限量（需要查询实时数据，成本最高）。前面任一步骤不通过就直接返回，避免不必要的后续查询。

### 5.3 可售规则配置 `availability_rule`

```
rule_id                 规则ID
product_id              商品ID（可选，为空表示门店级规则）
store_id                门店ID
rule_type               规则类型（TIME_SLOT|CHANNEL|STOCK|QUOTA）
rule_config             规则配置（JSON）
priority                优先级（数字越小优先级越高）
is_active               是否启用
effective_from          生效开始时间
effective_to            生效结束时间
```

**设计决策：** `product_id` 可选——为空时表示门店级规则（对该门店所有商品生效），有值时表示商品级规则。这样可以设置"门店不对抖音渠道开放"这种全局规则，也可以设置"某商品只在午餐时段售卖"这种商品级规则。`priority` 控制规则冲突时的优先级。

### 5.4 规则类型详解

```
TIME_SLOT（时段规则）
  allowed_time_slots    允许售卖的时段列表
    start_time          开始时间
    end_time            结束时间
    days_of_week        星期限制

CHANNEL（渠道规则）
  allowed_channels      允许售卖的渠道列表
  blocked_channels      禁止售卖的渠道列表

STOCK（库存规则）
  stock_type            库存类型（UNLIMITED|LIMITED|SOLD_OUT）
  current_stock         当前库存数（LIMITED时）
  low_stock_threshold   低库存阈值

QUOTA（限量规则）
  quota_type            限量类型（DAILY|WEEKLY|TOTAL）
  quota_limit           限量数量
  quota_used            已使用数量
  reset_time            重置时间（DAILY时为每日重置时间）
```

## 6. 商品快照服务

### 6.1 设计决策

**为什么需要快照？** 商品信息（价格、名称、规格等）会随时间变化，但订单中记录的商品信息必须反映下单那一刻的状态。如果订单直接引用商品ID并在展示时查商品实时数据，那么商品改名或调价后，历史订单的商品信息会失真。快照机制将"下单时刻的商品真相"冻结在订单中，与后续商品变更完全隔离。

### 6.2 快照生成时机

交易域在创建订单时，调用商品域的快照服务获取商品信息快照，快照一旦生成不再变化。

### 6.3 快照内容 `ProductSnapshot`

```json
{
  "snapshot_id": "uuid",
  "product_id": "string",
  "product_code": "string",
  "product_name": "string",
  "product_type": "SINGLE|COMBO|MODIFIER",
  "selected_variant": {
    "variant_id": "string",
    "variant_name": "string",
    "sku_code": "string"
  },
  "selected_modifiers": [
    {
      "group_name": "string",
      "modifier_name": "string",
      "price_delta": "number"
    }
  ],
  "unit_price": "number（最终单价，含规格和加料）",
  "original_price": "number（原价，用于展示划线价）",
  "product_image_url": "string",
  "production_profile": {
    "station_type": "string",
    "estimated_duration": "number"
  },
  "snapshot_at": "ISO8601"
}
```

### 6.4 快照服务接口

```
GetProductSnapshot(
  product_id: string,
  variant_id: string,
  modifier_ids: string[],
  channel_type: string,
  store_id: string
) -> ProductSnapshot
```

### 6.5 快照生成规则

1. 验证商品当前可售（调用可售规则引擎）
2. 计算最终价格（base_price + variant_delta + modifier_deltas）
3. 提取生产画像（用于后续履约域生成生产计划）
4. 冻结快照时间戳
5. 返回不可变快照对象

## 7. 品牌商品库与门店商品库

### 7.1 设计决策

**为什么需要两层商品库？** 连锁品牌要求所有门店销售统一的标准商品（保证品牌一致性），同时允许门店根据本地情况做有限度的调整（价格、图片等）。品牌商品库与门店商品库的继承机制平衡了"标准化"与"本地化"的矛盾。

### 7.2 连锁品牌场景

```
品牌商品库（Brand Product Library）
    +-- 品牌标准商品（ownership_scope=BRAND）
    |     +-- 统一定价（可锁定）
    |     +-- 统一规格（可锁定）
    |     +-- 统一生产画像
    |
    +-- 门店继承与覆盖
          +-- 门店可覆盖价格（如果品牌未锁定）
          +-- 门店可覆盖图片
          +-- 门店可添加本地商品（ownership_scope=STORE）
          +-- 门店不可修改品牌锁定字段
```

### 7.3 商品继承关系 `product_inheritance`

```
inheritance_id          继承关系ID
brand_product_id        品牌商品ID
store_product_id        门店商品ID（继承后的副本）
store_id                门店ID
override_fields         覆盖字段列表（JSON）
  field_name            字段名
  original_value        原始值
  override_value        覆盖值
  override_at           覆盖时间
  override_by           覆盖人
locked_fields           锁定字段列表（品牌禁止门店修改）
sync_status             同步状态（SYNCED|OUT_OF_SYNC）
last_sync_at            最后同步时间
```

**设计决策：** `locked_fields` 由品牌方设置，门店无法覆盖这些字段。例如品牌可以锁定"商品名称"和"基础价格"，确保全国统一定价和命名，但开放"图片"让门店上传本地实拍图。

### 7.4 品牌商品变更同步

当品牌修改标准商品时：

```
品牌商品变更
    |
    v
生成 BrandProductUpdated 事件
    |
    v
门店商品同步任务
    |
    v
对每个继承该商品的门店：
  - 检查门店是否覆盖了变更字段
  - 如果未覆盖，自动同步
  - 如果已覆盖，保留门店覆盖值
  - 如果字段被锁定，强制同步
    |
    v
更新门店商品
    |
    v
触发 MenuProductUpdated 事件（如果商品在菜单中）
```

**同步策略设计理由：** "未覆盖自动同步、已覆盖保留、锁定强制同步"三级策略，既保证品牌标准变更能自动传播到门店，又尊重门店的本地化调整，同时通过锁定机制保留品牌方的最终控制权。

## 8. 核心领域服务

### 8.1 MenuPublishService（菜单发布服务）

**职责：** 协调菜单发布流程，确保发布前的完整性检查。

**核心方法：**

```
PublishMenu(menu_id: string, operator_id: string) -> Result
  1. 验证菜单状态（必须是APPROVED）
  2. 检查菜单完整性（至少一个分区和商品）
  3. 验证所有商品均为ACTIVE状态
  4. 检查是否与现有菜单冲突
  5. 停用旧菜单（如果有）
  6. 激活新菜单
  7. 发布 MenuPublished 事件
  8. 触发渠道同步任务
```

**设计决策：** 菜单发布是领域服务而非聚合方法，因为它需要跨多个聚合协调（检查商品状态、停用旧菜单、触发渠道同步），单个聚合无法完成这些跨聚合操作。

### 8.2 ProductAvailabilityService（商品可售服务）

**职责：** 统一的商品可售判断入口。

**核心方法：**

```
CheckAvailability(
  product_id: string,
  store_id: string,
  channel_type: string,
  check_time: DateTime
) -> AvailabilityResult {
  is_available: boolean,
  unavailable_reason: string,
  available_from: DateTime,
  estimated_wait_minutes: number
}
```

**设计决策：** 返回值包含 `unavailable_reason` 和 `available_from`，不仅告诉调用方"不可售"，还告诉"为什么不可售"和"什么时候可售"。这样前端可以展示"该商品仅午餐时段供应（11:00起）"，而非简单的"暂时不可售"。

### 8.3 ChannelProductSyncService（渠道商品同步服务）

**职责：** 管理商品到各渠道的同步任务。

**核心方法：**

```
SyncProductToChannel(
  product_id: string,
  store_id: string,
  channel_type: string
) -> SyncResult

BatchSyncMenuToChannel(
  menu_id: string,
  channel_type: string
) -> BatchSyncResult
```

## 9. 关键领域事件

### 9.1 MenuPublished（菜单已发布）

**触发时机：** 菜单审核通过并正式发布生效

**消费方：**
- 渠道集成域（触发各渠道商品同步）——菜单变更后外部平台展示的商品列表需要同步更新
- 运营治理域（记录审计日志）——菜单发布是重要经营操作，需要留痕

### 9.2 ProductAvailabilityChanged（商品可售状态变更）

**触发时机：** 商品上架/下架/售罄/补货

**消费方：**
- 渠道集成域（实时同步各渠道库存状态）——避免外部平台展示已售罄商品导致无效订单

### 9.3 StoreOperatingStatusChanged（门店营业状态变更）

**触发时机：** 门店开店/暂停/打烊

**消费方：**
- 渠道集成域（同步各渠道店铺营业状态）——门店关闭后外部平台应同步关店
- 交易域（停止接受新订单）——门店暂停/关闭时不应再接受新订单

### 9.4 ProductSoldOut（商品已沽清）

**触发时机：** 库存扣减后 remaining_quantity <= sold_out_threshold

**消费方：**
- 渠道集成域（外卖平台下架）
- 设备管理（菜单刷新命令到POS/KDS）

### 9.5 ProductStockRestored（商品库存已恢复）

**触发时机：** 补货或库存重置后商品恢复可售

**消费方：**
- 渠道集成域（外卖平台上架）
- 设备管理（菜单刷新）

### 9.6 PriceChanged（商品价格已变更）

**触发时机：** 商品 base_price 或菜单 override_price 变更

**消费方：**
- 渠道集成域（同步外卖平台价格）
- 设备管理（菜单刷新）

### 9.7 BrandProductUpdated（品牌商品已更新）

**触发时机：** 品牌修改标准商品

**消费方：**
- 本域内部（门店商品同步任务）——触发所有继承该商品的门店执行同步

### 9.8 MenuProductUpdated（菜单商品已更新）

**触发时机：** 门店商品同步后，商品在菜单中

**消费方：**
- 渠道集成域（触发渠道商品重新同步）

## 10. 与其他域的交互

### 10.1 向交易域提供服务

**接口：** `GetProductSnapshot`

**调用时机：** 交易域创建订单时

**返回内容：** 商品快照（价格、名称、规格、生产画像）

**约定：** 快照一旦生成，商品域后续变更不影响已生成的快照。这是跨域数据隔离的核心保障。

### 10.2 向渠道域提供服务

**接口：** `GetChannelProductProjection`

**调用时机：** 渠道域需要向外部平台同步商品时

**返回内容：** 符合目标渠道格式的商品投影数据

### 10.3 向履约域提供服务

**接口：** `GetProductionProfile`

**调用时机：** 履约域生成生产计划时

**返回内容：** 商品的生产画像（制作岗位、时长、复杂度）

### 10.4 接收交易域反馈

**事件：** `OrderItemSold`

**用途：** 更新商品的每日限量已售数量、实时库存扣减

**设计决策：** 库存扣减由事件驱动而非同步调用，因为库存扣减不应阻塞订单创建流程。但这引入了短暂的数据不一致窗口（订单已创建但库存尚未扣减），通过沽清阈值（`sold_out_threshold`）提前预留缓冲来降低超卖风险。

## 11. 数据一致性保障

### 11.1 菜单版本一致性

- 菜单一旦发布，不可直接修改
- 修改需求通过创建新版本实现
- 新版本继承旧版本内容，修改后重新审核发布
- 旧版本保留完整历史，用于审计和回溯

**设计理由：** 如果允许直接修改已发布的菜单，可能导致正在使用该菜单下单的顾客看到不一致的信息。版本机制保证切换是原子的——要么用旧菜单，要么用新菜单，不存在中间状态。

### 11.2 商品价格一致性

- 商品 base_price 变更不影响已生成的订单快照
- 菜单中的 override_price 优先级高于商品 base_price
- 渠道投影中的价格以生成投影时的价格为准
- 价格变更触发渠道重新同步

**价格优先级链：** override_price（菜单级） > base_price（商品级）。下单时取当前菜单的 override_price，如果没有则取商品的 base_price。

### 11.3 库存一致性

- 库存扣减采用乐观锁机制
- 售罄状态实时同步到各渠道
- 补货后自动恢复可售状态
- 库存不足时拒绝下单，不允许超卖

**设计理由：** 乐观锁而非悲观锁，因为库存扣减的冲突概率低（同一商品同一时刻被多个订单扣减的概率不高），乐观锁在低冲突场景下性能更好。高并发场景下可通过 `sold_out_threshold` 提前沽清来进一步降低冲突。

## 12. 设计原则总结

| 原则 | 具体表现 | 解决的问题 |
|---|---|---|
| 快照不可变 | 订单商品快照一旦生成，不受后续商品变更影响 | 历史订单数据准确性 |
| 投影隔离 | 内部主数据与外部渠道格式通过投影机制隔离 | 外部平台格式变化不污染核心模型 |
