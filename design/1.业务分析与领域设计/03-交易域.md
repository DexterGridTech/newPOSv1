# 交易域

## 1. 域定位

交易域是平台的**订单真相源**，维护统一、稳定、可审计的订单事实。

本域的设计质量直接影响四个关键指标：

| 关键指标 | 本域保障机制 | 设计决策理由 |
|---|---|---|
| 业务准确性 | 完整的状态机 + 不变式 | 状态机定义合法路径，不变式保证数据自洽，二者配合杜绝非法状态和数据不一致 |
| 可审计性 | 状态线（StatusLine）记录每次状态转换 | 餐饮行业涉及资金流转，每次操作必须有操作人、时间、原因的完整记录 |
| 可扩展性 | 渠道/场景/履约三维度统一建模 | 一个订单模型覆盖所有渠道和场景，不因渠道不同而分叉，避免维护N套订单系统 |
| 一致性 | 领域服务 + 事件机制 | 跨域数据通过事件最终一致，域内数据通过领域服务和不变式保证强一致 |

---

## 2. 域职责边界

### 2.1 核心职责

- 接收并创建统一订单（来自各渠道）
- 管理订单完整生命周期状态
- 处理订单支付确认
- 管理订单变更（加菜/退菜/取消）
- 协调退款流程
- 维护订单操作审计日志
- 提供订单查询服务

### 2.2 不负责的事项

| 事项 | 归属域 | 边界划分理由 |
|---|---|---|
| 生产执行 | 履约域 | 制作和交付是独立的领域行为，有自己的岗位、工序、并行任务模型 |
| 直接对接外部渠道API | 渠道域 | 防腐层职责，外部API格式变化不应渗入订单核心模型 |
| 结算清分 | 结算域 | 清分涉及多方分账逻辑（平台/品牌/门店/骑手），是独立的领域问题 |
| 支付通道 | 支付网关 | 支付通道管理（微信/支付宝/现金）是基础设施层关注点，不是业务域 |

## 3. 核心聚合根设计

### 3.1 订单聚合 `SaleOrder`

**聚合根：** `sale_order`

**设计决策：** SaleOrder 作为聚合根，包含 SaleOrderItem（订单行）、SaleOrderStatusLine（状态线）、SaleOrderPayment（支付记录）、SaleOrderCoupon（优惠券使用记录）作为内部实体。所有这些实体共享订单的一致性边界——修改订单行需要重算金额，状态转换需要同步记录状态线，支付和优惠券状态需要和订单状态联动。把它们放在同一聚合内保证这些操作的原子性。

#### 3.1.1 基础信息字段

```
order_id                订单唯一标识（UUID）
order_no                订单业务单号（对外展示）
store_id                门店ID
tenant_id               租户ID
brand_id                品牌ID（可选）
entity_id               经营主体ID（结算用）
```

**字段设计说明：**

- `order_id`（UUID）用于系统内部唯一标识，`order_no`（业务单号）用于对外展示和沟通（如顾客取餐时报单号）。分离的原因是业务单号需要可读性（如 "MCD-20260404-0001"），而UUID保证全局唯一且无序列冲突。
- `entity_id`（经营主体ID）：结算域需要知道这笔订单的收入归属哪个法律实体，因为一个品牌可能有多个经营主体（不同区域用不同公司）。

#### 3.1.2 渠道与场景三维度

```
channel_type            渠道类型（MINI_PROGRAM|H5|POS_DIRECT|KIOSK|MEITUAN|ELEME|DOUYIN）
order_scene             业务场景（DINE_IN|TAKEAWAY_SELF|DELIVERY_ONESELF|DELIVERY_PLATFORM）
fulfillment_type        履约方式（TABLE_SERVICE|SELF_PICKUP|THIRD_PARTY_DELIVERY|PLATFORM_DELIVERY|STORE_DELIVERY）
external_order_no       外部平台原始单号（三方外卖必填）
```

**设计决策——为什么用三维度而非单一字段？**

单一字段（如 `order_type = MEITUAN_DELIVERY`）会导致枚举值爆炸（美团堂食、美团外卖、饿了么堂食、饿了么外卖、小程序堂食、小程序自取……）。三维度正交建模：`channel_type` 表示订单从哪来，`order_scene` 表示顾客怎么用餐，`fulfillment_type` 表示怎么交付。三者独立组合，N个渠道 x M个场景 x K个履约方式 = N*M*K 种组合，而非 N*M*K 个枚举值。

- **MINI_PROGRAM**：自有小程序渠道
- **H5**：桌台扫码 H5（微信/支付宝内嵌浏览器），堂食专用
- **POS_DIRECT**：POS直录单，门店收银员操作
- **KIOSK**：自助点餐机
- **MEITUAN / ELEME / DOUYIN**：三方外卖平台渠道

- **DINE_IN**：堂食
- **TAKEAWAY_SELF**：到店自取
- **DELIVERY_ONESELF**：商家自配送
- **DELIVERY_PLATFORM**：平台配送

- **TABLE_SERVICE**：桌台服务（服务员送餐到桌）
- **SELF_PICKUP**：顾客自取（凭取餐码）
- **THIRD_PARTY_DELIVERY**：第三方配送
- **PLATFORM_DELIVERY**：平台配送

#### 3.1.3 顾客信息

```
customer_id             顾客ID（可选，匿名订单为null）
customer_name           顾客姓名
customer_phone          顾客手机号（加密存储）
customer_phone_suffix   手机号后4位（明文，用于叫号）
```

**设计决策：** `customer_id` 可选，因为POS现场点单的顾客不一定会注册会员。`customer_phone` 加密存储保护隐私，但 `customer_phone_suffix` 明文保留后4位用于叫号（"尾号3456的顾客请取餐"），避免每次叫号都需要解密。

#### 3.1.4 关联信息

```
booking_id              关联预订单ID（可选，从预订转单时填入）
session_id              关联桌台会话ID（堂食开台后填入）
```

#### 3.1.5 金额信息

```
total_amount            订单总金额（分）
discount_amount         优惠总金额（分）
extra_charge_amount     附加费总金额（分，服务费/包间费等）
extra_charge_items      附加费明细（JSON）
  charge_id             附加费规则ID
  charge_name           名称
  charge_type           类型
  calc_basis            计算基数描述
  charge_amount         金额（分）
payable_amount          应付金额（分）= total_amount - discount_amount + extra_charge_amount
paid_amount             实付金额（分）
refund_amount           已退款金额（分）
```

**金额计算公式：** `payable_amount = total_amount - discount_amount + extra_charge_amount`

**设计决策：**
- 所有金额使用**分**为单位（整数），避免浮点运算误差。
- `extra_charge_items` 存储附加费明细而非只存总额，因为结算时需要知道每项附加费的类型和金额（服务费可能不参与平台抽成，包间费可能需要单独开票）。
- `paid_amount` 和 `payable_amount` 分离：正常情况下二者相等，但部分退款后 `refund_amount > 0`，需要这些字段配合才能准确反映订单的资金状态。

#### 3.1.6 优惠明细

```
platform_coupon_amount  平台优惠券金额
channel_coupon_amount   渠道优惠券金额
member_points_amount    会员积分抵扣金额
store_promotion_amount  门店活动优惠金额
```

**设计决策：** 优惠拆分为四个来源，因为结算清分时需要知道谁承担优惠成本——平台优惠券由平台承担，渠道优惠券由渠道承担（如美团券），会员积分由会员系统承担，门店活动由门店承担。

#### 3.1.7 服务上下文（JSON）

```
service_context         服务上下文
  table_id              桌台ID（堂食）
  table_no              桌台编号
  table_name            桌台名称
  dine_in_count         就餐人数
  pickup_code           取餐码（自取）
  delivery_address      配送地址（外卖）
    province            省
    city                市
    district            区
    street              街道
    detail              详细地址
    latitude            纬度
    longitude           经度
    contact_name        收货人
    contact_phone       收货电话
  delivery_time_type    配送时间类型（ASAP|SCHEDULED）
  scheduled_time        预约时间（预约单）
  delivery_note         配送备注
```

**设计决策：** 服务上下文用 JSON 而非独立字段，因为不同场景需要的字段完全不同——堂食需要桌台信息，自取需要取餐码，外卖需要配送地址。用 JSON 避免在主表上放大量可空字段。同时，不同场景的不变式也不同（堂食必须有 table_id，外卖必须有 delivery_address），这些在订单创建时通过验证服务检查。

#### 3.1.8 订单状态字段

```
status                  订单状态（见状态机）
status_display          状态展示文本
cancel_reason_code      取消原因码
cancel_reason_text      取消原因文本
cancel_initiator        取消发起方（CUSTOMER|STORE|PLATFORM|SYSTEM_TIMEOUT）
```

**设计决策：** `cancel_initiator` 区分四种取消发起方，因为不同发起方的取消规则不同（顾客取消在部分状态需要门店同意，门店取消可能需要承担违约金，系统超时取消是自动行为），后续结算清分也需要根据取消发起方判断责任归属。

#### 3.1.9 时间信息

```
created_at              创建时间
paid_at                 支付时间
accepted_at             接单时间
production_started_at   开始生产时间
production_completed_at 生产完成时间
fulfilled_at            履约完成时间
completed_at            订单完成时间
cancelled_at            取消时间
```

**设计决策：** 每个关键节点都有独立时间字段而非只看状态线，因为这些时间字段需要高频查询（如"今天未接单超过5分钟的订单"），从状态线中聚合查询效率太低。状态线用于审计回溯，时间字段用于业务查询。

#### 3.1.10 操作人信息

```
created_by              创建人（POS录单时）
accepted_by             接单人
cancelled_by            取消操作人
```

#### 3.1.11 SLA时间

```
accept_deadline         接单截止时间
expected_ready_time     预计出餐时间
expected_delivery_time  预计送达时间
```

**设计决策：** SLA时间在订单创建/接单时计算并持久化，而非每次查询时动态计算。持久化的好处是可以基于这些字段建索引做超时告警查询，并且SLA一旦确定就不变（除非人工调整），动态计算可能因规则变更导致历史订单的SLA回溯性变化。

#### 3.1.12 其他字段

```
remark                  订单备注
tags                    订单标签（急单/VIP等）
version                 乐观锁版本号
```

### 3.2 实体：SaleOrderItem（订单行）

```
item_id               订单行ID
order_id              订单ID
line_number           行号
product_snapshot      商品快照（JSON，来自商品域）
  product_id          商品ID
  product_code        商品编码
  product_name        商品名称
  product_type        商品类型
  selected_variant    选中规格
  selected_modifiers  选中加料
  unit_price          单价
  product_image_url   商品图片
  production_profile  生产画像
quantity              数量
subtotal              小计金额
discount_amount       优惠金额
actual_amount         实付金额
item_status           行状态（NORMAL|REFUNDED|CANCELLED）
refund_quantity       已退数量
refund_amount         已退金额
remark                行备注（口味要求等）
```

**设计决策——快照机制：**

`product_snapshot` 冗余存储商品的完整信息（名称、价格、规格、加料、图片、生产画像），而非只存 `product_id` 再关联查询。原因：

1. 商品信息会变化（改名、调价、换图），但订单记录的必须是下单那一刻的商品真相
2. 履约域需要从订单行中获取生产画像（制作岗位、时长）来排产，不应再回查商品域
3. 订单查询时不需要跨域调用商品域，降低系统耦合度
4. 快照是不可变的，一旦订单创建就冻结，后续商品变更不影响

### 3.3 实体：SaleOrderStatusLine（状态线）

```
status_line_id        状态线ID
order_id              订单ID
from_status           原状态
to_status             目标状态
transition_type       转换类型（NORMAL|EXCEPTION|COMPENSATION）
operator_id           操作人
operator_type         操作人类型（CUSTOMER|STAFF|SYSTEM）
reason                原因
occurred_at           发生时间
context_data          上下文数据（JSON）
```

**设计决策——StatusLine 为什么是独立实体而非日志表？**

StatusLine 是订单聚合的核心组成部分，不是旁路日志。理由：

1. `transition_type` 区分 NORMAL / EXCEPTION / COMPENSATION，支撑补偿事务的回溯分析
2. `operator_type` 区分 CUSTOMER / STAFF / SYSTEM，结合 `operator_id` 可以追溯每次状态变更的责任人
3. 状态线和订单��同一聚合边界内，保证状态转换和状态线记录的原子性——不会出现"状态已变但状态线没记录"的情况
4. `context_data` 存储每次转换的上下文（如取消原因、退款金额），为审计和纠纷处理提供完整证据

### 3.4 实体：SaleOrderPayment（支付记录）

```
payment_id            支付ID
order_id              订单ID
payment_method        支付方式（WECHAT|ALIPAY|CASH|MEMBER_WALLET）
payment_channel       支付渠道（MINI_PROGRAM|POS|PLATFORM）
paid_amount           支付金额
transaction_no        支付流水号
third_party_trade_no  第三方交易号
payment_status        支付状态（PENDING|SUCCESS|FAILED|REFUNDED）
paid_at               支付时间
refund_amount         已退款金额
refund_at             退款时间
```

**设计决策：**

- `payment_method`（支付方式）和 `payment_channel`（支付渠道）分离：支付方式是"用什么付"（微信/支付宝/现金），支付渠道是"在哪付"（小程序内支付/POS刷卡/平台代收）。二者独立组合，例如"在小程序内用支付宝付"。
- `transaction_no`（内部流水号）和 `third_party_trade_no`（第三方交易号）分离：内部流水号用于系统内追踪，第三方交易号用于和微信/支付宝对账。

### 3.5 实体：SaleOrderCoupon（优惠券使用记录）

```
coupon_usage_id       使用记录ID
order_id              订单ID
coupon_source         优惠券来源（PLATFORM|MEITUAN|ELEME|DOUYIN|MEMBER）
coupon_id             优惠券ID
coupon_code           优惠券码
coupon_type           优惠券类型（DISCOUNT|CASH|ITEM）
discount_amount       抵扣金额
external_coupon_id    外部平台优惠券ID（三方券）
verification_status   核销状态（LOCKED|VERIFIED|CANCELLED）
locked_at             锁定时间
verified_at           核销时间
reverse_verification_status  反核销状态（NOT_REVERSED|REVERSING|REVERSED）
reverse_at            反核销时间
reverse_reason        反核销原因
```

**设计决策——优惠券的两阶段处理：**

优惠券使用分为"锁定"和"核销"两个阶段：

1. **锁定（LOCKED）**：创建订单时锁定优惠券，防止同一张券被多个订单使用。此时券还未真正消费。
2. **核销（VERIFIED）**：支付确认后核销优惠券，标记为已消费。
3. 如果订单取消，锁定的优惠券会被释放（CANCELLED）。

这种两阶段设计避免了"下单占券但未支付，导致其他订单无法使用该券"和"多个订单同时使用同一张券"两个问题。

**反核销设计：** `reverse_verification_status` 处理退款场景——订单退款后已核销的优惠券需要反核销，将券退还给顾客。反核销是异步操作（需要和外部平台交互），因此有独立的状态跟踪。

### 3.6 SaleOrder 状态机

#### 3.6.1 完整状态流转图

```
PENDING_PAYMENT（待支付）
    | [PaymentConfirmed 事件]
    v
PAID（已支付/待接单）
    | [AcceptOrder 命令]
    v
ACCEPTED（已接单）
    | [OrderInProduction 事件]（来自履约域）
    v
IN_PRODUCTION（生产中）
    | [ProductionCompleted 事件]（来自履约域）
    v
PRODUCTION_DONE（生产完成/待交付）
    | [FulfillmentCompleted 事件]（来自履约域）
    v
FULFILLED（已履约）
    | [CloseOrder 命令]
    v
COMPLETED（已完成）

异常路径：
PENDING_PAYMENT  -> CANCELLED（超时未支付）
PAID             -> REJECTED（门店拒单）
PAID             -> CANCELLED（顾客取消）
ACCEPTED         -> CANCELLED（门店取消或顾客取消）
```

**退款状态线（payment_refund_status，独立于订单主状态）：**

```
NONE（无退款，默认值）
    | [发起部分退款]
    v
PARTIAL_REFUNDING（部分退款处理中）
    | [退款成功]
    v
PARTIALLY_REFUNDED（已部分退款）
    | [继续退款至全额]
    v
FULL_REFUNDING（全额退款处理中）
    | [退款成功]
    v
FULLY_REFUNDED（已全额退款）

NONE / PARTIALLY_REFUNDED -> FULL_REFUNDING（发起全额退款）
任意 REFUNDING 状态 -> REFUND_FAILED（退款失败，可重试）
```

> 退款状态独立于订单主状态，部分退款不改变订单 status，全额退款完成后订单 status 变为 CANCELLED。发布 `RefundProcessed` 事件时携带 refund_type（FULL/PARTIAL）和最新 payment_refund_status，供结算域和渠道域消费。

#### 3.6.2 状态设计理由

**为什么将交易状态、支付状态、退款状态、履约状态分离到不同维度？**

SaleOrder 的 `status` 字段跟踪订单的业务生命周期，`payment_refund_status` 跟踪退款进度，SaleOrderPayment 的 `payment_status` 跟踪支付状态，履约状态通过接收履约域事件来更新。四者分离的原因：

- 退款可以发生在订单完成之后，甚至跨结算周期，不能用订单主状态表达退款中间态。
- 部分退款不应终止订单生命周期（订单仍在履约中），独立退款状态线可以精确表达"已部分退款但订单仍在进行"的状态。
- 订单状态和支付状态的生命周期不同：一个订单可以有多次支付尝试（支付失败后重试），订单还是同一个订单但支付记录有多条。
- 订单状态和履约状态的驱动方不同：订单状态由交易域自身驱动，履约状态由履约域通过事件推送。如果混在一起，交易域就需要了解履约域的内部细节。
- 分离后各状态可以独立演进，不会因为增加一个履约状态（如"骑手已取餐"）而修改订单状态机。

**为什么 PAID 和 ACCEPTED 分离？**

POS直录单可以跳过PAID直接进入ACCEPTED（因为是现场操作，无需异步支付确认），但外卖订单必须经过 PAID → ACCEPTED（需要门店决定是否接单）。分离这两个状态让模型能覆盖这两种场景。

**为什么 PRODUCTION_DONE 和 FULFILLED 分离？**

PRODUCTION_DONE 表示食物已做好，FULFILLED 表示已交付给顾客。对堂食桌台服务场景，二者之间有一个"送餐到桌"的过程；对外卖场景，二者之间有一个"骑手配送"的过程。分离这两个状态让系统能精确追踪这个交付过程。

---

### 3.7 SaleOrder 不变式

- 订单总金额 = 所有订单行小计之和
- 应付金额 = 订单总金额 - 优惠总金额 + 附加费总金额
- 实付金额 = 应付金额（支付完成后）
- 已退款金额 <= 实付金额
- 订单状态只能按状态机定义的路径转换
- POS_DIRECT 渠道的订单无需支付确认，直接进入PAID状态
- 三方外卖订单必须有 external_order_no
- 堂食订单必须有 table_id
- 外卖订单必须有 delivery_address

### 3.8 订单关系聚合 `SaleOrderRelation`

**用途：** 管理订单间的关系（原单-改单、原单-退单）

```
relation_id             关系ID
source_order_id         源订单ID
target_order_id         目标订单ID
relation_type           关系类型（MODIFICATION|REFUND|SPLIT|MERGE）
relation_reason         关系原因
created_at              创建时间
```

**设计决策：** 订单关系独立为聚合而非订单聚合的内部实体，因为关系涉及两个订单（source 和 target），不属于任何一个订单的一致性边界。如果放在某个订单聚合内，查询另一个订单的关系时需要跨聚合访问。

### 3.9 桌台会话聚合 `TableSession`（正餐堂食）

**用途：** 管理正餐堂食的就餐上下文，将开台与订单创建解耦。

#### 3.9.1 完整字段定义

```
session_id              会话ID（UUID）
store_id                门店ID
table_id                桌台ID
table_no                桌台编号
order_id                关联订单ID（首次点菜后填入，可空）
booking_id              关联预订单ID（从预订转单时填入，可空）

就餐信息：
party_size              就餐人数
opened_by               开台操作人ID
opened_at               开台时间

状态：
session_status          会话状态（OPEN|BILLING|CLOSED）
closed_at               结束时间
```

**设计决策——为什么 TableSession 独立为聚合？**

正餐堂食的流程是"先开台、再点菜、可多轮加菜、最后结账"。如果把开台和订单绑定在一起，开台时就必须创建订单，但此时顾客还没有点菜（订单没有商品行），这违反了"订单必须至少有一个商品行"的不变式。TableSession 将"就餐上下文"从"交易行为"中解耦：

1. 开台时创建 TableSession，此时没有订单
2. 顾客点菜时创建 SaleOrder 并关联 session_id
3. 加菜时向已有 SaleOrder 追加订单行
4. 结账时 TableSession 关闭

`order_id` 初始为空，首次点菜后填入。这个设计自然支持了"开台但未点菜"的合法状态。

> **正餐改单策略（P1-3 决策）：** 本系统采用**策略A**：`TableSession + 单SaleOrder + 增量OrderItem + 改单日志`。即同一桌台会话内的加菜通过 `AddOrderItemCommand` 追加到同一订单，不创建新订单。`SaleOrderRelation` 聚合用于管理**改单/退单**的原单-目标单关系（如整单改单、退单），不用于同桌多次加菜场景。
>
> 选择策略A的理由：简化结账流程（一张账单），降低部分退款复杂度，与改单日志配合可完整追溯加菜历史。若未来需要支持"同桌多人各自独立结账"场景，可升级为策略B（多SaleOrder + SaleOrderRelation），但当前阶段不引入该复杂度。

#### 3.9.2 正餐三步流程

```
第一步：开台（OpenTableCommand）
  -> 创建 TableSession（session_status=OPEN）
  -> 桌台状态变为 OCCUPIED，current_session_id 填入
  -> 此时无订单

第二步：点菜/加菜（CreateOrderCommand / AddOrderItemCommand）
  -> 首次点菜时创建 SaleOrder（关联session_id）
  -> 后续加菜通过 AddOrderItemCommand 追加到同一订单
  -> 支持多轮加菜

第三步：结账（PrintPreBillCommand -> PaymentCommand）
  -> 打印预结单（可选）
  -> 顾客确认后发起支付
  -> 支付完成 -> TableSession关闭 -> 桌台释放

挂账路径：
  -> HangBillCommand -> 记录挂账信息
  -> 桌台释放（不等待支付）
  -> 后续统一结算
```

### 5.2 支付确认步骤

```
1. 验证订单状态（必须是PENDING_PAYMENT）
    |
    v
2. 验证支付金额（必须等于应付金额）
    |
    v
3. 核销优惠券（调用会员系统/渠道域）
    |
    v
4. 更新订单状态为PAID
    |
    v
5. 记录支付信息
    |
    v
6. 记录状态线
    |
    v

    |
```

- 步骤8（接单决策）在支付确认的最后：支付成功后立即进入接单流程，减少顾客等待时间。

## 6. 接单流程

### 6.1 接单决策逻辑

```
if (channel_type == POS_DIRECT) {
    // POS直录单，无需接单，直接进入ACCEPTED
    autoAccept()
} else if (channel_type == MEITUAN || channel_type == ELEME) {
    // 三方外卖，必须在SLA时间内接单
    if (store.auto_accept_enabled) {
        autoAccept()
    } else {
        waitForManualAccept(deadline: channel_accept_deadline)
    }
} else if (channel_type == MINI_PROGRAM) {
    // 一方小程序
    if (store.auto_accept_enabled) {
        autoAccept()
    } else {
        waitForManualAccept(timeout: store.accept_timeout_seconds)
    }
}
```

**设计决策：**

- POS_DIRECT 跳过接单步骤：POS录单是收银员现场操作，录单行为本身就意味着门店已接受这笔订单。
- 三方外卖使用 `channel_accept_deadline`（渠道方的SLA截止时间）而非门店自定义超时：因为美团/饿了么有自己的接单SLA要求，超时会自动取消订单并影响门店评分。
- 小程序使用 `store.accept_timeout_seconds`（门店自定义超时）：自有渠道没有外部SLA约束，由门店自行决定接单超时策略。

### 6.2 接单命令 `AcceptOrderCommand`

```json
{

  "operator_id": "string（人工接单时）",
```

```
    |
    v

    |
    v

    |
    v

    |
    v

    |
    v

    v
```


### 7.2 退菜命令 `RemoveOrderItemCommand`

```json
  "refund_reason": "string",
```

### 7.3 订单变更约束

- 只有 ACCEPTED 和 IN_PRODUCTION 状态的订单可以加菜
- 只有未开始制作的订单行可以退菜
- 已开始制作的订单行退菜需要门店确认
- 加菜后需要重新计算订单金额
- 退菜后需要处理退款

**设计理由：**

- 限制加菜状态为 ACCEPTED 和 IN_PRODUCTION：PENDING_PAYMENT 状态的订单还未支付，加菜会改变应付金额导致支付流程复杂化；PRODUCTION_DONE 之后的状态食物已做好或已交付，加菜没有意义。
- "未开始制作的订单行可以退菜"：食材还没有被消耗，可以无条件退菜退款。"已开始制作的订单行退菜需要门店确认"：食材已部分消耗，门店需要判断是否接受退菜并承担损失。

## 8. 取消与退款流程

### 8.1 取消订单命令 `CancelOrderCommand`

```json
{
  "order_id": "string",
  "cancel_initiator": "CUSTOMER|STORE|PLATFORM|SYSTEM_TIMEOUT",
  "cancel_reason_code": "string",
  "cancel_reason_text": "string",
  "operator_id": "string"
}
```

### 8.2 取消规则

| 订单状态 | 顾客取消 | 门店取消 | 退款规则 | 设计理由 |
|---|---|---|---|---|
| PENDING_PAYMENT | 允许 | 不适用 | 无需退款 | 未支付，直接取消无成本 |
| PAID | 允许 | 允许（拒单） | 全额退款 | 已支付但未开始制作，无食材损失 |
| ACCEPTED | 允许（需门店同意） | 允许 | 全额退款 | 已接单但未制作，顾客取消需门店同意是因为门店可能已准备食材 |
| IN_PRODUCTION | 不允许 | 允许（特殊情况） | 部分退款或不退 | 食材已消耗，顾客不能单方面取消；门店可因特殊原因取消（如食材问题） |
| PRODUCTION_DONE | 不允许 | 不允许 | 不退款 | 食物已做好，双方都不能取消 |
| FULFILLED | 不允许 | 不允许 | 售后退款 | 已交付，走售后退款流程（如食品质量问题） |

### 8.3 退款命令 `ProcessRefundCommand`

```json
{
  "order_id": "string",
  "refund_type": "FULL|PARTIAL",
  "refund_amount": "number",
  "refund_items": [
    {
      "item_id": "string",
      "quantity": "number"
    }
  ],
  "refund_reason": "string",
  "operator_id": "string"
}
```

**设计决策：**

- `refund_type` 区分全额退款和部分退款：全额退款整单取消，部分退款只退指定商品行。
- `refund_items` 指定退款的具体商品行和数量：支持"3份水饺退1份"的细粒度退款。
- 退款金额由业务层计算后传入，而非自动计算：因为部分退款场景下，退款金额可能不等于商品原价（如使用了优惠券的订单，退款金额需要按比例分摊优惠）。

## 10. 关键领域事件

### 10.1 OrderCreated（订单已创建）

**消费方：**
- 结算域（登记订单）——结算域需要知道有新订单，为后续清分做准备
- 渠道集成域（回传接收确认）——三方外卖平台需要知道订单已被系统接收
- 运营治理域（审计日志）——订单创建是重要业务事件，需要留痕

### 10.2 PaymentConfirmed（支付已确认）

**消费方：**
- 履约域（准备生产）——支付完成后可以开始备料，减少接单后的生产等待时间
- 结算域（记录支付事实）——结算需要支付金额、支付方式等信息
- 渠道集成域（回传支付状态）——三方平台需要同步支付状态

### 10.3 OrderAccepted（门店已接单）

**消费方：**
- 履约域（生成生产计划）——接单后正式开始排产，分配制作岗位和工序
- 渠道集成域（回传接单状态）——三方平台需要知道门店已接单

### 10.4 OrderCompleted（订单已完成）

**消费方：**
- 结算域（生成结算事实）——订单完成后开始清分计算
- 渠道集成域（回传完成状态）——三方平台需要同步完成状态
- 外部集成域（ERP数据输出）——ERP需要完成的订单数据做收入确认
- 运营治理域（经营指标更新）——更新门店的营业额、客单价等经营指标

## 11. 与其他域的交互

### 11.1 调用商品域

**接口：** `GetProductSnapshot`

**时机：** 创建订单时

**用途：** 获取商品快照，冻结商品信息

**设计决策：** 这是交易域对商品域的唯一同步依赖。获取快照后，交易域不再需要访问商品域——订单行的所有商品信息都在快照中。这种"获取一次后自给自足"的模式最大程度降低了运行时耦合。

### 11.2 调用会员系统（通过外部集成域）

**接口：** `LockCoupon`, `RedeemCoupon`, `DeductPoints`

**时机：** 创建订单、支付确认、取消订单

**用途：** 优惠券和积分管理

**设计决策：** 通过外部集成域的防腐层访问会员系统，而非直接调用。因为会员系统可能是第三方系统（如商场统一会员系统），其API格式和调用方式可能变化，防腐层隔离这些变化。

### 11.3 调用渠道域

**接口：** `VerifyChannelCoupon`

**时机：** 创建订单时

**用途：** 验证外部平台优惠券有效性

**设计决策：** 外部平台优惠券（如美团券）的验证必须通过渠道域，因为验证逻辑涉及调用外部平台API，这些API的格式和协议不应暴露给交易域。

### 11.4 接收履约域事件

**事件：** `OrderInProduction`, `ProductionCompleted`, `FulfillmentCompleted`

**用途：** 更新订单状态

**设计决策：** 交易域通过事件被动接收履约域的状态推送，而非主动轮询。这保持了交易域和履约域的松耦合——交易域不需要知道履约域的内部实现细节（如工序如何拆分、岗位如何分配），只关心三个关键节点："开始制作了"、"制作完了"、"交付完了"。

## 12. 数据一致性保障

### 12.1 订单金额一致性

- 订单创建时计算金额，持久化后不可篡改
- 加菜/退菜触发金额重算，记录变更历史
- 退款金额不能超过实付金额
- 所有金额计算使用整数（分），避免浮点误差

**设计理由：** 金额不可篡改+变更记录，保证任何时候都可以追溯"订单金额为什么是这个数"。整数（分）运算而非浮点运算，是因为浮点数在做大量加减和比较时会产生精度问题（如 0.1 + 0.2 != 0.3），餐饮订单涉及大量金额计算，必须用整数避免这类问题。

### 12.2 状态转换一致性

- 状态转换必须符合状态机定义
- 每次状态转换记录状态线
- 并发状态转换使用乐观锁控制
- 异常状态转换记录异常原因

**设计理由：** 乐观锁而非悲观锁控制并发状态转换。餐饮订单的并发更新冲突概率低（通常一个订单在某一时刻只有一个操作者），乐观锁在低冲突场景下性能更好，且不会造成锁等待。乐观锁失败时抛出异常让调用方重试，比持有锁等待更适合分布式场景。

### 12.3 优惠券一致性

- 优惠券锁定与订单创建在同一事务
- 优惠券核销与支付确认在同一事务
- 订单取消时自动释放优惠券
- 外部平台优惠券通过渠道域防腐层处理

**设计理由：** 优惠券的锁定/核销与订单操作在同一事务内，保证原子性。如果分成两个事务，可能出现"订单创建成功但券未锁定"（券被别的订单用了）或"券已核销但订单创建失败"（券无法退还）的数据不一致。外部平台优惠券通过防腐层处理，因为三方平台的券核销API可能有延迟、重试、幂等性等复杂性，需要在防腐层统一处理。

## 13. 设计原则总结

| 原则 | 具体表现 | 解决的问题 |
|---|---|---|
| 统一订单模型 | 所有渠道的订单进入统一模型，不因渠道不同而分叉 | 避免维护N套订单系统，降低复杂度 |
| 状态机驱动 | 订单状态转换严格遵循状态机定义 | 杜绝非法状态转换，保证业务正确性 |
| 快照隔离 | 商品信息通过快照机制隔离后续变更 | 历史订单数据不被商品后续变更影响 |
| 事件驱动 | 订单状态变更通过事件通知其他域 | 松耦合，交易域不依赖下游域的实现细节 |
| 审计完整 | 每次操作记录完整的操作人、时间、原因 | 纠纷处理、合规审计、运营分析 |
