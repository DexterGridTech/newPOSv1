# 05-渠道集成域

## 1. 域定位与职责边界

渠道集成域是平台与外部渠道之间的**防腐层**，负责隔离外部渠道差异，代理商户完成渠道侧操作。它的核心价值在于：无论外部平台（美团、饿了么、抖音等）的 API 如何变化，核心业务域（交易域、商品域、履约域）都不受影响。

### 1.1 核心职责

- 管理渠道店铺绑定关系和ISV授权
- 接收外部平台订单推送，翻译为统一格式
- 代理商户向外部平台回传订单状态
- 同步商品信息到各渠道
- 同步门店营业状态到各渠道
- 管理渠道优惠券核销记录
- 处理渠道Token刷新和过期告警
- 记录所有渠道操作日志

### 1.2 不负责的事项

| 事项 | 归属域 | 为什么排除 |
|------|--------|-----------|
| 订单业务逻辑 | 交易域 | 渠道域只负责"翻译"和"传递"，不参与订单的业务决策（定价、优惠计算、接单拒单策略等） |
| 商品主数据 | 商品域 | 渠道域只消费商品的"渠道投影"，不管理商品的核心定义 |
| 门店主数据 | 租户与组织域 | 渠道域只管理门店在外部平台上的映射关系，不管理门店的基础信息 |
| 结算清分 | 结算域 | 渠道优惠券的金额最终由结算域处理，渠道域只负责核销记录 |

## 2. 核心聚合根设计

### 2.1 渠道店铺绑定聚合 `ChannelShopBinding`

**聚合根：** `channel_shop_binding`

**设计意图：** ChannelShopBinding 是渠道集成域的基础锚点。它建模了一个内部门店与一个外部平台店铺之间的绑定关系，包括 ISV 授权信息。所有渠道操作（接单、回传、同步）都以绑定关系为前提——没有有效的绑定，就无法与外部平台通信。

**完整字段定义：**

```
binding_id              绑定关系ID（UUID）
store_id                内部门店ID
channel_type            渠道类型（MEITUAN|ELEME|DOUYIN）
external_shop_id        外部平台店铺ID
external_shop_name      外部平台店铺名称
external_poi_code       外部平台POI编码

ISV授权信息：
app_key                 ISV应用Key
access_token            授权Token（加密存储）
refresh_token           刷新Token（加密存储）
token_type              Token类型（BEARER|OAUTH2）
token_expires_at        Token过期时间
token_scope             授权范围

绑定状态：
binding_status          绑定状态（见状态机）
binding_mode            绑定模式（ISV|DIRECT）
authorization_status    授权状态（PENDING|AUTHORIZED|EXPIRED|REVOKED）

配置信息：
sync_config             同步配置（JSON）
  auto_sync_menu        是否自动同步菜单
  auto_sync_stock       是否自动同步库存
  auto_accept_order     是否自动接单
  accept_timeout        接单超时时间（秒）
notification_config     通知配置（JSON）
  webhook_url           回调地址
  notification_events   订阅事件列表

时间信息：
effective_date          生效日期
expire_date             失效日期（可选）
bound_at                绑定时间
authorized_at           授权时间
last_sync_at            最后同步时间
last_token_refresh_at   最后Token刷新时间

操作人信息：
bound_by                绑定操作人
authorized_by           授权操作人

其他：
remark                  备注
version                 乐观锁版本号
```

**为什么 Token 要加密存储？** access_token 和 refresh_token 是访问外部平台 API 的凭证，泄露后攻击者可以冒充商户进行操作（如接单、拒单、修改商品）。加密存储确保即使数据库被攻破，Token 仍然安全。

**为什么有 `binding_mode`（ISV / DIRECT）两种模式？** ISV 模式下，平台作为 ISV 服务商，通过 OAuth 授权代理多个商户操作，适合SaaS平台场景。DIRECT 模式下，商户自己注册开放平台账号，直接提供密钥，适合大型连锁品牌。两种模式的 Token 获取和刷新流程不同，因此需要显式区分。

**ChannelShopBinding 状态机：**

```
PENDING_AUTHORIZATION（待授权）
    ↓ [AuthorizeBinding 命令]
AUTHORIZED（已授权）
    ↓ [ActivateBinding 命令]
ACTIVE（已激活）
    ↓ [SuspendBinding 命令]
SUSPENDED（已暂停）
    ↓ [ResumeBinding 命令]
ACTIVE
    ↓ [UnbindShop 命令]
UNBOUND（已解绑）

异常路径：
AUTHORIZED → TOKEN_EXPIRED（Token过期）
TOKEN_EXPIRED → AUTHORIZED（Token刷新成功）
ACTIVE → AUTHORIZATION_REVOKED（授权被撤销）
```

**为什么 AUTHORIZED 和 ACTIVE 是两个独立状态？** 授权成功后，商户可能还需要进行额外配置（如菜单同步、接单设置）才能正式上线。AUTHORIZED 表示"技术上已打通"，ACTIVE 表示"业务上已启用"。这个分离允许运营人员在激活前进行验证，防止未配置好就开始接单。

**不变式：**

- 同一渠道类型下，一个external_shop_id在同一时间段只能绑定一个store_id
- ACTIVE状态的绑定必须有有效的access_token
- Token过期前7天必须触发刷新告警
- 解绑前必须确认该绑定下没有处理中的订单

**为什么同一外部店铺只能绑定一个内部门店？** 外部平台推送订单时只携带 external_shop_id，如果同一个外部店铺绑定了多个内部门店，系统就无法确定订单应该分配给哪个门店。一对一绑定确保订单路由的确定性。

### 2.2 渠道订单聚合 `ChannelOrder`

**聚合根：** `channel_order`

**设计意图：** ChannelOrder 的核心设计原则是**完整保留原始数据**。外部平台推送的原始 JSON 完整存储在 `raw_order_data` 字段中，翻译后的统一格式存储在 `unified_order_payload` 中。这两份数据互不影响——翻译过程只读取原始数据并生成新格式，永远不修改原始数据。这样做的好处是：当翻译逻辑有 bug 或外部平台字段含义变更时，可以用原始数据重新翻译，不会丢失信息。

**完整字段定义：**

```
channel_order_id        渠道订单ID（UUID）
binding_id              渠道绑定ID
store_id                内部门店ID
channel_type            渠道类型
external_order_no       外部平台原始单号
external_order_status   外部平台订单状态码

订单原始数据：
raw_order_data          原始订单数据（JSON，完整保留）
received_at             接收时间
parsed_at               解析时间

翻译后的统一数据：
unified_order_payload   统一订单载荷（JSON）
  order_scene           业务场景
  fulfillment_type      履约方式
  customer_info         顾客信息
  items                 订单行列表
  total_amount          订单总金额
  discount_amount       优惠金额
  service_context       服务上下文
  channel_coupons       渠道优惠券列表

处理状态：
processing_status       处理状态（见状态机）
internal_order_id       内部订单ID（创建成功后）
error_code              错误码（处理失败时）
error_message           错误信息
retry_count             重试次数

渠道SLA：
channel_accept_deadline 渠道接单截止时间
channel_delivery_time   渠道要求送达时间

其他：
idempotency_key         幂等键（防重复）
version                 乐观锁版本号
```

**为什么需要 `idempotency_key`？** 外部平台可能因为网络超时等原因重复推送同一个订单。幂等键（通常等于 external_order_no）确保重复推送只处理一次，避免重复创建订单。

**ChannelOrder 状态机：**

```
RECEIVED（已接收）
    ↓ [ParseChannelOrder 命令]
PARSED（已解析）
    ↓ [CreateInternalOrder 命令]
ORDER_CREATED（内部订单已创建）
    ↓ [ConfirmReceived 命令]
CONFIRMED（已确认接收）

异常路径：
RECEIVED → PARSE_FAILED（解析失败）
PARSED → CREATE_FAILED（创建失败）
```

**为什么有 RECEIVED 和 PARSED 两个独立状态？** 接收和解析是异步解耦的：先快速接收原始数据并返回成功响应给外部平台（避免超时），然后异步进行解析和翻译。如果解析失败（如字段格式变更），原始数据已经保存，可以修复后重试。

### 2.3 渠道商品映射聚合 `ChannelProductMapping`

**聚合根：** `channel_product_mapping`

**设计意图：** 内部商品和外部平台商品不是一一对应的——名称、价格、图片、规格都可能不同。ChannelProductMapping 管理这种映射关系，支持自动映射（通过编码匹配）和手动映射（运营人员指定），并记录完整的同步历史用于排查问题。

**完整字段定义：**

```
mapping_id              映射ID
store_id                门店ID
product_id              内部商品ID
channel_type            渠道类型
binding_id              渠道绑定ID
external_product_id     外部平台商品ID
external_sku_id         外部平台SKU ID
external_product_code   外部平台商品编码

映射状态：
mapping_status          映射状态（PENDING|ACTIVE|FAILED|UNMAPPED）
sync_status             同步状态（NOT_SYNCED|SYNCING|SYNCED|SYNC_FAILED）
last_sync_at            最后同步时间
sync_error_code         同步失败错误码
sync_error_message      同步失败原因

字段映射配置：
field_mapping_config    字段映射配置（JSON）
  name_mapping          名称映射规则
    internal_name       内部名称
    external_name       外部名称
    name_suffix         名称后缀（可选）
  price_mapping         价格映射规则
    price_strategy      定价策略（SAME|MARKUP|CUSTOM）
    markup_rate         加价率（MARKUP时）
    custom_price        自定义价格（CUSTOM时）
  image_mapping         图片映射规则
    use_internal_image  是否使用内部图片
    external_image_url  外部图片URL（可选）
  modifier_mapping      加料映射规则
    modifier_groups     加料组映射列表

同步历史：
sync_history            同步历史记录（JSON数组）
  sync_at               同步时间
  sync_result           同步结果
  sync_details          同步详情

创建信息：
created_at              创建时间
created_by              创建人
mapping_type            映射类型（AUTO|MANUAL）
```

**为什么需要 `price_strategy`（SAME|MARKUP|CUSTOM）？** 不同渠道的定价策略可能不同：有些商户在外卖平台加价（覆盖平台佣金），有些保持同价，有些针对特定渠道设定独立价格。映射配置让这些策略可以灵活配置，而不是硬编码到同步逻辑中。

### 2.4 渠道操作日志聚合 `ChannelOperationLog`

**聚合根：** `channel_operation_log`

**设计意图：** 与外部平台的每一次交互都可能出错（网络超时、签名失败、格式变更等）。ChannelOperationLog 完整记录每次操作的请求、响应、耗时和结果，支持问题排查和审计。它是渠道集成域可追溯性的基础。

**完整字段定义：**

```
log_id                  日志ID
binding_id              渠道绑定ID
store_id                门店ID
channel_type            渠道类型
operation_type          操作类型（见枚举）
operation_direction     操作方向（INBOUND|OUTBOUND）

操作内容：
target_type             目标类型（ORDER|PRODUCT|SHOP|STOCK）
target_id               目标ID
operation_action        操作动作（CREATE|UPDATE|DELETE|QUERY|CALLBACK）
request_payload         请求载荷（JSON）
response_payload        响应载荷（JSON）

执行结果：
execution_status        执行状态（SUCCESS|FAILED|TIMEOUT）
http_status_code        HTTP状态码
error_code              错误码
error_message           错误信息
retry_count             重试次数

时间信息：
started_at              开始时间
completed_at            完成时间
duration_ms             耗时（毫秒）

其他：
correlation_id          关联ID（链路追踪）
idempotency_key         幂等键
```

**为什么区分 INBOUND 和 OUTBOUND？** INBOUND 是外部平台主动推送给我们的操作（如订单推送、配送回调），OUTBOUND 是我们主动调用外部平台的操作（如接单回传、商品同步）。方向区分便于运营人员快速定位问题——"是我们没发出去"还是"是平台推过来的数据有问题"。

**操作类型枚举：**

```
RECEIVE_ORDER           接收订单
CONFIRM_ORDER           确认接收订单
ACCEPT_ORDER            回传接单
REJECT_ORDER            回传拒单
UPDATE_ORDER_STATUS     更新订单状态
CANCEL_ORDER            回传取消
SYNC_PRODUCT            同步商品
UPDATE_STOCK            更新库存
UPDATE_SHOP_STATUS      更新店铺状态
REFRESH_TOKEN           刷新Token
QUERY_ORDER             查询订单
CALLBACK_DELIVERY       配送回调
```

## 3. 渠道订单接收与翻译

### 3.1 订单接收流程

```
外部平台推送订单
    ↓
1. 验证签名（防止伪造）
    ↓
2. 幂等检查（防止重复处理）
    ↓
3. 保存原始订单数据（完整保留）
    ↓
4. 创建ChannelOrder（状态：RECEIVED）
    ↓
5. 记录操作日志
    ↓
6. 返回接收成功响应
    ↓
7. 异步解析订单
```

**为什么先返回成功响应再异步解析？** 外部平台的推送接口通常有严格的超时限制（如美团要求3秒内响应）。如果同步完成解析、创建内部订单等全部流程，很容易超时导致平台重试。先快速保存原始数据并返回成功，再异步处理，既满足平台的超时要求，又确保数据不丢失。

### 3.2 订单解析服务 `ChannelOrderParserService`

**职责：** 将外部平台订单格式翻译为统一格式

**解析步骤：**

```
1. 根据channel_type选择对应的解析器
    ↓
2. 提取订单基本信息
    ↓
3. 翻译订单状态码
    ↓
4. 解析顾客信息
    ↓
5. 解析订单行（商品映射）
    ↓
6. 解析金额信息
    ↓
7. 解析服务上下文（配送地址等）
    ↓
8. 解析渠道优惠券
    ↓
9. 生成unified_order_payload
    ↓
10. 更新ChannelOrder状态为PARSED
```

### 3.3 美团订单解析示例

**美团原始订单字段 -> 统一字段映射：**

```
wm_order_id_view        → external_order_no
wm_poi_receive_detail_time → received_at
recipient_name          → customer_info.customer_name
recipient_phone         → customer_info.customer_phone
recipient_address       → service_context.delivery_address
wm_order_status         → 状态码翻译（见状态码映射表）
total                   → total_amount（元转分）
original_price          → 原价
poi_receive_detail      → items解析
  app_food_code         → 商品映射查询
  food_name             → product_name
  quantity              → quantity
  price                 → unit_price
  spec                  → selected_variant
  food_property         → selected_modifiers
caution                 → remark
```

**为什么 total_amount 要做"元转分"？** 外部平台的金额通常以元为单位（如 "25.50"），内部系统统一使用分为单位（如 2550）。分为单位避免浮点数精度问题，确保金额计算的精确性。

### 3.4 饿了么订单解析示例

**饿了么原始订单字段 -> 统一字段映射：**

```
id                      → external_order_no
created_at              → received_at
consignee               → customer_info.customer_name
phone                   → customer_info.customer_phone
deliver_address         → service_context.delivery_address
status_code             → 状态码翻译
total_price             → total_amount（元转分）
original_price          → 原价
groups                  → items解析
  item_id               → 商品映射查询
  name                  → product_name
  quantity              → quantity
  price                 → unit_price
  spec                  → selected_variant
  attributes            → selected_modifiers
description             → remark
```

**为什么美团和饿了么需要各自独立的解析器？** 两个平台的字段命名、数据结构、状态码含义完全不同。例如美团用 `recipient_name` 表示收件人，饿了么用 `consignee`；美团用 `spec` 表示规格，饿了么用 `spec` + `attributes`。独立的解析器确保每个平台的特异性被正确处理。

## 4. 渠道状态码翻译

### 4.1 美团订单状态码映射

```
美团状态码 → 内部动作

2（待接单）              → 创建订单，等待接单
4（已接单）              → （已由我们主动回传，无需处理）
6（已完成）              → （由履约域触发完成，无需处理）
8（已取消-用户）         → CancelOrderCommand(initiator=CUSTOMER)
9（已取消-商家）         → CancelOrderCommand(initiator=STORE)
10（已取消-平台）        → CancelOrderCommand(initiator=PLATFORM)
```

### 4.2 饿了么订单状态码映射

```
饿了么状态码 → 内部动作

1（待接单）              → 创建订单，等待接单
2（已接单）              → （已由我们主动回传，无需处理）
3（配送中）              → （配送状态，由履约域处理）
4（已完成）              → （由履约域触发完成，无需处理）
5（已取消-用户）         → CancelOrderCommand(initiator=CUSTOMER)
6（已取消-商家）         → CancelOrderCommand(initiator=STORE)
7（已取消-平台）         → CancelOrderCommand(initiator=PLATFORM)
```

**为什么有些状态码标注"无需处理"？** 因为这些状态是由我们的系统主动触发的。例如"已接单"状态是我们调用回传接单 API 后平台更新的，平台再推送过来只是确认通知，不需要重复处理。如果处理了反而会导致重复操作。

## 5. 渠道状态回传

### 5.1 状态回传服务 `ChannelStatusSyncService`

**职责：** 将内部订单状态变更回传给外部平台

**回传时机：**

| 内部事件 | 回传动作 | 目标渠道 |
|---|---|---|
| OrderCreated | 确认接收 | 美团/饿了么 |
| OrderAccepted | 回传接单 | 美团/饿了么 |
| OrderRejected | 回传拒单 | 美团/饿了么 |
| ProductionCompleted | 回传备餐完成 | 美团/饿了么 |
| OrderCompleted | 回传订单完成 | 美团/饿了么 |
| OrderCancelled | 回传取消 | 美团/饿了么 |

### 5.2 回传接单流程

```
接收 OrderAccepted 事件
    ↓
1. 查询渠道绑定信息
    ↓
2. 根据channel_type选择对应的适配器
    ↓
3. 构造回传请求（美团/饿了么格式）
    ↓
4. 调用外部平台API
    ↓
5. 记录操作日志
    ↓
if (成功) {
    更新回传状态
} else {
    记录失败原因
    加入重试队列
}
```

### 5.3 美团接单回传示例

**请求格式：**

```json
{
  "app_poi_code": "门店POI编码",
  "wm_order_id_view": "美团订单号",
  "confirm_type": 2,
  "expected_delivery_time": "预计送达时间戳"
}
```

**API：** `POST /waimai/order/confirm`

### 5.4 饿了么接单回传示例

**请求格式：**

```json
{
  "order_id": "饿了么订单号",
  "push_time": "推送时间戳"
}
```

**API：** `POST /order/confirm`

## 6. 渠道商品同步

### 6.1 商品同步服务 `ChannelProductSyncService`

**职责：** 将内部商品同步到各渠道

**同步触发时机：**

- 菜单发布（MenuPublished事件）
- 商品价格变更（ProductPriceChanged事件）
- 商品上下架（ProductAvailabilityChanged事件）
- 商品图片更新（ProductImageUpdated事件）
- 手动触发同步

### 6.2 商品同步流程

```
接收 MenuPublished 事件
    ↓
1. 查询门店的所有渠道绑定
    ↓
2. 对每个绑定的渠道：
    ↓
3. 查询商品映射关系
    ↓
4. 生成渠道商品投影
    ↓
5. 调用渠道API同步商品
    ↓
6. 更新映射状态和同步时间
    ↓
7. 记录操作日志
    ↓
if (失败) {
    创建异常台账
    加入重试队列
}
```

**为什么同步失败不影响内部商品主数据？** 渠道同步是"最终一致"的。内部商品变更是权威数据源，渠道同步是衍生操作。即使某个渠道同步失败，内部系统照常运行，失败的同步通过重试机制最终完成。如果因为渠道同步失败而回滚内部变更，会导致核心业务被外部系统绑架。

**上下架：** `POST /waimai/dish/status`

### 6.4 饿了么商品同步API

## 7. 渠道优惠券处理

### 7.1 渠道优惠券核销记录 `ChannelCouponRecord`

**设计意图：** 外部平台订单经常携带平台优惠券（如美团红包、饿了么折扣券），这些优惠券的核销和资金结算涉及平台方。ChannelCouponRecord 记录每张渠道优惠券的核销状态，为后续结算对账提供依据。

```
coupon_record_id        核销记录ID
channel_order_id        渠道订单ID
internal_order_id       内部订单ID
channel_type            渠道类型
coupon_source           优惠券来源（MEITUAN|ELEME|DOUYIN）
external_coupon_id      外部平台优惠券ID
coupon_code             优惠券码
coupon_type             优惠券类型（PLATFORM|MERCHANT|CHANNEL）
discount_amount         抵扣金额（分）
coupon_name             优惠券名称
coupon_description      优惠券描述
verification_status     核销状态（PENDING|VERIFIED|FAILED|CANCELLED）
verified_at             核销时间
verification_result     核销结果（JSON）
error_code              错误码（失败时）
error_message           错误信息
```

**为什么区分 PLATFORM / MERCHANT / CHANNEL 三种优惠券类型？** 不同类型的优惠券由不同方承担成本：PLATFORM 由平台方承担（美团/饿了么补贴），MERCHANT 由商户自己承担（商户在平台上设置的优惠），CHANNEL 由渠道合作方承担。区分类型是结算清分的基础。

### 7.2 优惠券核销流程

```
订单创建时
    ↓
1. 提取渠道订单中的优惠券信息
    ↓
2. 创建ChannelCouponRecord（状态：PENDING）
    ↓
3. 调用渠道API验证优惠券有效性
    ↓
if (有效) {
    更新状态为VERIFIED
    将优惠券信息传递给交易域
} else {
    更新状态为FAILED
    记录失败原因
    拒绝订单创建
}
```

### 7.3 多平台优惠券汇总

订单可能同时使用多个来源的优惠券：

```
订单优惠券列表：
  - 购物中心平台券（PLATFORM）
  - 美团平台券（MEITUAN）
  - 饿了么平台券（ELEME）
  - 抖音平台券（DOUYIN）
  - 会员积分抵扣（MEMBER）

优惠金额计算：
total_discount = platform_coupon + channel_coupon + member_points
payable_amount = total_amount - total_discount
```

## 8. Token管理与刷新

### 8.1 Token刷新服务 `ChannelTokenRefreshService`

**职责：** 管理渠道授权Token的刷新

**为什么需要自动刷新而不是等过期后再处理？** Token 过期后所有渠道操作都会失败（接单、回传、商品同步等），直接影响业务。自动刷新机制确保 Token 始终有效，7天预警给运营人员留出手动处理的时间窗口。

**刷新策略：**

```
定时任务（每小时执行）
    ↓
1. 查询所有ACTIVE状态的绑定
    ↓
2. 检查Token过期时间
    ↓
if (距离过期 < 7天) {
    触发Token刷新告警
    发送通知给门店管理员
}
    ↓
if (距离过期 < 1天) {
    自动调用刷新API
    更新access_token和refresh_token
    更新token_expires_at
    记录刷新日志
}
    ↓
if (刷新失败) {
    更新绑定状态为TOKEN_EXPIRED
    创建异常台账
    发送紧急通知
}
```

**为什么7天告警但1天才自动刷新？** 7天告警是给运营人员的预警，让他们有时间处理可能的授权问题（如平台侧调整、商户更改密码等）。1天时自动刷新是兜底机制，确保在无人干预的情况下也能自动续期。过早刷新会浪费 API 调用配额，过晚刷新风险太高。

### 8.2 美团Token刷新

**API：** `POST /oauth/token/refresh`

**请求：**

```json
{
  "app_key": "应用Key",
  "refresh_token": "刷新Token"
}
```

**响应：**

```json
{
  "access_token": "新的访问Token",
  "refresh_token": "新的刷新Token",
  "expires_in": 7776000
}
```

### 8.3 饿了么Token刷新

**API：** `POST /token/refresh`

**请求：**

```json
{
  "app_key": "应用Key",
  "refresh_token": "刷新Token"
```

## 12. 数据一致性保障

### 12.1 订单接收幂等性

- 使用external_order_no作为幂等键
- 重复推送的订单只处理一次
- 幂等检查在事务外完成，避免锁竞争

**为什么幂等检查在事务外完成？** 如果在事务内做幂等检查，高并发下（多个重复推送同时到达）会产生行锁竞争，降低吞吐量。事务外用缓存或唯一索引做快速判断，重复请求直接返回，不进入事务流程。

### 12.2 状态回传一致性

- 状态回传失败自动重试（最多5次）
- 重试采用指数退避策略
- 超过最大重试次数后转入异常台账
- 所有回传操作记录完整日志

**为什么用指数退避而非固定间隔重试？** 固定间隔重试在外部平台故障时会产生大量无效请求，加重对方负担（甚至触发限流）。指数退避（如 1s, 2s, 4s, 8s, 16s）让重试间隔逐渐拉大，给外部平台恢复的时间。

### 12.3 商品同步一致性

- 商品映射关系持久化存储
- 同步失败不影响内部商品主数据
- 定期全量对账，发现差异自动修复
- 同步操作支持手动重试

## 13. 核心设计原则总结

渠道集成域的设计质量直接影响系统稳定性、数据准确性、运营效率和异常处理能力。

| 原则 | 含义 | 为什么重要 |
|------|------|-----------|
| 完整保留原始数据 | 外部平台推送的原始数据完整保存，不丢弃任何字段 | 翻译逻辑有 bug 时可以用原始数据重跑，也为争议订单提供证据 |
| 翻译不修改 | 翻译过程只生成新格式，不修改原始数据 | 原始数据是"事实"，翻译结果是"解读"，两者必须独立保存 |
| 幂等处理 | 所有外部回调和推送都实现幂等处理 | 外部平台会因超时等原因重复推送，幂等确保业务不受影响 |
| 操作可追溯 | 所有渠道操作记录完整日志，支持审计和排查 | 渠道集成涉及外部系统，出问题时需要完整的请求/响应记录来定位 |
| 自动重试 | 失败操作自动重试，超过阈值转入人工处理 | 外部系统不稳定是常态，自动重试覆盖大部分瞬时故障 |
