# 10-预订域

## 1. 域定位与设计意图

预订域管理正餐预订场景的完整业务闭环，从顾客预订到到店开台。在购物中心餐饮场景中，正餐（尤其是包间和VIP桌台）的预订需求普遍存在——顾客希望提前预订确保有位，商家希望通过预订提前规划接待能力和备料。

预订域被设计为**独立的经营链路**，与桌台管理域和交易域通过事件解耦，原因如下：

1. **预订是交易的前置环节而非交易本身**：预订时尚未产生订单和消费，只有顾客到店后才会触发开台和创建交易订单。如果将预订逻辑嵌入交易域会使交易域变得臃肿且职责不清。
2. **桌台状态的所有权归租户与组织域**：预订域不直接修改桌台状态，而是通过事件通知桌台管理方——预订确认时桌台变为 RESERVED，到店时变为 OCCUPIED，取消时恢复 AVAILABLE。这保持了桌台状态管理的单一职责。
3. **预订有独立的生命周期**：预订单从待确认到已确认、已到店、已完成/已取消/已逾期，有自己的状态机和业务规则，适合作为独立域管理。

---

## 2. 域职责边界

### 2.1 核心职责

- 管理预订单完整生命周期（创建 → 确认 → 到店 → 完成/取消）
- 管理餐段配置（时段容量、提前预订天数、截止时间）
- 触发桌台预订状态变更（通过事件）
- 触发到店开台（通过事件通知交易域）
- 沉淀预订经营数据

### 2.2 不负责的事项

- 不管理桌台状态（租户与组织域职责）——预订域只通知桌台状态变更，不直接修改桌台数据
- 不管理订单和支付（交易域职责）——到店后的消费由交易域负责
- 不管理订金退款（由外部集成域的支付网关能力执行，交易域负责退款业务决策）——取消预订时如果需要退订金，预订域发布 BookingCancelled 事件，由交易域判断退款规则并调用外部集成域的支付网关完成退款

## 3. 核心聚合根设计

### 3.1 预订单聚合 `BookingOrder`

**聚合根：** `booking_order`

**设计意图：** BookingOrder 聚合封装了一次预订的全部信息。booking_type 区分 B 端（商家代客预订，通常来自 POS 或电话）和 C 端（顾客自助通过小程序预订），不同来源的预订在确认流程上可能不同。table_id 设计为可选字段，支持"先预订后分桌"模式——部分商家不在预订时指定桌台，而是等顾客到店时再根据实际情况分配。deposit（订金）机制用于减少爽约，deposit_refund_policy 定义取消时的退款规则。linked_order_id 在顾客到店开台后填入，建立预订与交易的关联。

**完整字段定义：**

```
booking_id              预订单ID（UUID）
booking_no              预订单号（对外展示）
store_id                门店ID
booking_type            预订类型（B_END|C_END）
source_channel          来源渠道（MINI_PROGRAM|PHONE|THIRD_PARTY|POS）

顾客信息：
customer_name           顾客姓名
customer_phone          顾客电话（加密存储）
customer_phone_suffix   手机号后4位
party_size              就餐人数

预订时间：
booking_date            预订日期
period_id               餐段ID
period_name             餐段名称
arrival_time            预计到店时间
expire_at               预订截止时间（超时自动取消）

桌台信息：
table_id                预订桌台ID（可选，到店再分配）
table_no                桌台编号
table_type              桌台类型（REGULAR|VIP|PRIVATE_ROOM）

订金信息：
deposit_required        是否需要订金
deposit_amount          订金金额（分）
deposit_paid            是否已付
deposit_payment_id      订金支付ID
deposit_refund_policy   退订金规则（FULL|PARTIAL|NO_REFUND）

预点餐：
pre_order_items         预点菜品列表（JSON）
  product_id / product_name / quantity / unit_price

状态：
status                  预订状态（PENDING|CONFIRMED|ARRIVED|COMPLETED|CANCELLED|OVERDUE）
cancel_reason_code      取消原因码
cancel_initiator        取消发起方（CUSTOMER|STORE|SYSTEM）

关联：
linked_order_id         关联交易订单ID（到店开台后生成）

时间戳：
created_at / confirmed_at / arrived_at / cancelled_at / completed_at
```

### 3.2 餐段配置值对象 `BookingPeriod`

**设计意图：** BookingPeriod 定义门店的可预订时段，每个餐段有独立的容量上限和预订规则。max_bookings 限制同一餐段的最大预订桌数，防止超额预订。advance_booking_days 控制可提前预订的天数范围（如只允许预订未来7天内的）。cutoff_minutes 定义截止接受预订的时间（如开餐前30分钟停止接受预订），给商家留出备料和准备时间。days_of_week 支持不同星期设置不同餐段（如周末增加下午茶时段）。

```
period_id               餐段ID
store_id                门店ID
period_name             餐段名称（早餐/午餐/下午茶/晚餐/夜宵）
start_time              开始时间（HH:mm）
end_time                结束时间（HH:mm）
days_of_week            适用星期（1-7）
max_bookings            最大预订桌数
advance_booking_days    可提前预订天数（如7天内）
cutoff_minutes          截止预订时间（开餐前N分钟停止接受预订）
enabled                 是否启用
```

## 4. 预订状态机

**设计意图：** 预订状态机覆盖了预订的完整生命周期，包括正常路径（待确认 → 已确认 → 已到店 → 已完成）和异常路径（取消、逾期）。OVERDUE 状态由系统定时任务在到店时间加缓冲时间后自动触发，标记未到店的预订。逾期后系统自动关闭释放桌台资源，避免桌台长时间被无效预订占用。

```
PENDING（待确认）
    ↓ [ConfirmBooking 命令]（商家确认）
CONFIRMED（已确认）
    ↓ [ConfirmArrival 命令]（顾客到店）
ARRIVED（已到店）→ 触发开台，生成交易订单
    ↓ [CompleteBooking 命令]（用餐完成）
COMPLETED（已完成）

异常路径：
PENDING → CANCELLED（顾客/商家取消）
CONFIRMED → CANCELLED（顾客/商家取消）
CONFIRMED → OVERDUE（系统定时任务：到店时间+缓冲后未到店）
OVERDUE → CANCELLED（系统自动关闭）
```

**不变式：**

这些不变式确保数据完整性和业务规则的正确执行：

- CONFIRMED 状态必须有 confirmed_at 时间——确认操作必须记录时间
- ARRIVED 状态必须已触发 TableSession 创建（session_id 不为空）——到店必须已经开台建立桌台会话；linked_order_id 在首次点菜提交后由交易域回填，到店时可为空
- deposit_required=true 时，deposit_payment_id 不能为空——需要订金的预订必须完成订金支付
- expire_at 必须晚于 booking_date+arrival_time——预订截止时间必须在预计到店时间之后才有意义

## 5. 核心领域服务

### 5.1 BookingAvailabilityService（预订可用性服务）

**设计意图：** 在顾客发起预订前，需要查询指定日期各餐段的剩余容量和可用桌台。该服务整合餐段配置和已有预订数据，计算出实时的可用性信息，支持顾客选择合适的时段和桌台。

```
QueryAvailablePeriods(
  store_id: string,
  date: Date
) → PeriodAvailability[] {
  period_id, period_name, start_time, end_time,
  remaining_capacity, available_tables[]
}
```

### 5.2 BookingLifecycleService（预订生命周期服务）

**设计意图：** 封装预订单的全部生命周期操作。CreateBooking 负责创建预订单并进行容量检查；ConfirmBooking 由商家操作确认预订；ConfirmArrival 在顾客到店时触发，同时发布事件通知交易域开台；CancelBooking 支持顾客、商家和系统三方发起取消，并记录取消原因和发起方以便分析爽约原因。

```
CreateBooking(command: CreateBookingCommand) → BookingOrder
ConfirmBooking(booking_id: string, operator_id: string) → Result
ConfirmArrival(booking_id: string, operator_id: string) → Result
CancelBooking(booking_id: string, reason: string, initiator: string) → Result
```

## 6. 关键域事件

### 6.1 BookingCreated（预订单已创建）

**触发时机：** 新预订单创建成功后发布。

**消费方：**
- 运营治理域（审计）——记录预订创建操作
- 外部集成域（通知顾客）——通过短信或微信通知顾客预订已提交

### 6.2 BookingConfirmed（预订单已确认）

**触发时机：** 商家确认预订后发布。

**消费方：**
- 外部集成域（发送确认通知给顾客）——通知顾客预订已被商家确认
- 租户与组织域（更新桌台状态为 RESERVED）——将预订桌台标记为已预留，防止被其他渠道使用

### 6.3 BookingArrived（顾客已到店）

**触发时机：** 确认顾客到店后发布。这是预订域与交易域的关键衔接点。

**消费方：**
- 交易域（触发开台，创建 TableSession）——交易域收到事件后自动创建桌台会话，关联 booking_id，开始正式的点餐和消费流程
- 租户与组织域（更新桌台状态为 OCCUPIED）——桌台从预留状态变为占用状态

### 6.4 BookingCancelled（预订单已取消）

**触发时机：** 预订被取消后发布，无论取消发起方是谁。

**消费方：**
- 租户与组织域（释放桌台预订）——清空桌台的 current_booking_id，恢复为 AVAILABLE 状态
- 外部集成域（通知顾客）——通知顾客预订已取消
- 交易域（退订金，如有）——如果顾客已支付订金，交易域根据 deposit_refund_policy 判断退款规则，调用外部集成域支付网关执行退款

### 6.5 BookingOverdue（预订单已逾期）

**触发时机：** 系统定时任务检测到已确认的预订在到店时间加缓冲后仍未到店时发布。

**消费方：**
- 外部集成域（通知顾客）——提醒顾客预订即将被取消
- 运营治理域（记录逾期统计）——沉淀逾期数据用于分析和改善预订策略

## 7. 与其他域的交互

### 7.1 向租户与组织域发送事件

**设计意图：** 预订域不直接修改桌台状态，而是通过事件通知租户与组织域来更新。这保持了桌台状态管理的单一职责——桌台状态的唯一所有者是租户与组织域。

- BookingConfirmed → 桌台状态变为 RESERVED，填入 current_booking_id
- BookingArrived → 桌台状态变为 OCCUPIED
- BookingCancelled → 桌台状态恢复 AVAILABLE，清空 current_booking_id

### 7.2 向交易域发送事件

**设计意图：** 预订到店是交易的触发点。交易域监听 BookingArrived 事件自动创建 TableSession，实现预订流程到消费流程的无缝衔接，无需前台手动操作开台。

- BookingArrived → 交易域创建 TableSession，关联 booking_id

### 7.3 查询租户与组织域

**设计意图：** 预订创建时需要检查桌台可用性和门店餐段配置，这些数据的所有权在租户与组织域，预订域通过查询接口获取。

- 创建预订时查询桌台可用性
- 查询门店餐段配置
