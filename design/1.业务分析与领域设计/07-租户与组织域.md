# 07-租户与组织域

## 1. 域定位

租户与组织域是平台的**组织结构真相源**（Organizational Source of Truth），负责维护 Platform（平台/集团）、Project（购物中心）、Tenant（商户）、Brand（品牌）、Store（门店）、Contract（合同）的组织关系和权限体系。

> **重要定位约束：** 餐饮中台是**业务执行系统**，不负责合同签署、商户入驻审核等前置业务。所有组织主数据（Platform/Tenant/Brand/Contract）均由外部系统（如招商管理系统）维护，通过数据同步接口推送到本平台。本域的职责是**接收、存储、查询**这些主数据，并为各业务域提供组织归属校验服务。

> **为什么需要独立的租户与组织域？** 购物中心餐饮平台是多租户架构，所有业务数据（订单、商品、结算）都必须归属于某个 Platform 和门店。如果组织关系散落在各业务域中，隔离保障和权限控制将无法统一。将组织层级抽象为独立域，是多租户平台的架构基石。

---

## 2. 域职责边界

### 2.1 核心职责

- 接收并存储从上游系统同步的组织主数据（Platform、Project、Tenant、Brand、Store、Contract）
- 维护组织层级关系和 Platform 隔离边界
- 维护经营主体信息（用于结算）
- 维护门店基础信息（地址、营业时间、联系方式）
- 管理员工账号和角色权限
- 管理门店设备绑定关系（POS、KDS、打印机）
- 提供组织架构查询服务

### 2.2 不负责的事项

| 事项 | 说明 |
|------|------|
| 商户入驻审核 | 由外部招商管理系统负责，本系统只接收同步结果 |
| 合同签署管理 | 由外部合同管理系统负责，本系统只同步合同状态和关键信息 |
| 证照资质管理 | 营业执照、食品许可证等资质由外部系统维护，本系统不存储原件 |
| 门店营业状态 | 归商品域，营业状态与菜单、库存强相关，属于经营维度而非组织维度 |
| 订单和交易 | 归交易域，租户域只提供"谁的店"，不管"店里卖了什么" |
| 结算清分 | 归结算域，分钱逻辑是财务维度，租户域只提供经营主体信息供引用 |
| 商品数据 | 归商品域，菜单和商品有独立的生命周期和变更频率 |

## 3. 组织层级模型

```
Platform（平台/集团）
    ├── Project（项目/购物中心）
    │        └── Store（门店）← 通过 Contract 与 Tenant/Brand 关联
    ├── Tenant（租户/商户）← 按统一社会信用代码识别，Platform 独立维护
    ├── Brand（品牌）← Platform 独立维护的品牌库，与 Tenant 无直接关联
    └── Contract（合同）← 连接 Project + Tenant + Brand + Store(unit_no)
```

**各层级说明：**

| 层级 | 业务含义 | 关键特性 |
|------|---------|---------|
| Platform | 集团/平台方（如万达集团、大悦城集团） | 顶层隔离边界，多 Platform 可共存于同一环境 |
| Project | 单个购物中心（如深圳万象城） | 归属某个 Platform |
| Tenant | 经营主体/法人（按营业执照统一代码区分） | 每个 Platform 独立维护自己的 Tenant 列表；同一商户在不同 Platform 是两条独立记录 |
| Brand | 餐饮品牌（如肯德基、奈雪） | 直接归属 Platform，与 Tenant 无直接关联；多个 Tenant 可通过不同 Contract 以同一 Brand 经营 |
| Store | 具体门店（如"肯德基大悦城店"，铺位 L0102） | 由 unit_no + Project 唯一确定物理位置；当前生效 Contract 决定关联的 Tenant 和 Brand |
| Contract | 经营/租赁合同 | 约定合同编号、铺位号、Tenant、Brand、Project、起止日期；门店可有多份历史合同 |

> **为什么 Platform 是隔离边界而非 Tenant？** 传统多租户系统以 Tenant 为隔离边界。但本平台面向多个购物中心集团（万达、大悦城）运营，同一个餐饮商户（如肯德基）可能同时是万达的租户和大悦城的租户。以 Platform 为最高隔离层，使各集团的商户数据完全独立，既满足集团数据主权需求，又支持同一商户跨平台经营的现实。

> **为什么通过 Contract 而非直接字段关联 Tenant/Brand？** 门店的运营主体会随合同变更而变化（如换品牌、换承租商户）。Contract 记录了每个时期的关联关系和有效期，历史合同不删除，保证经营事实可追溯。`Store.active_contract_id` 始终指向当前生效合同，其他域通过它快速获取当前 Tenant/Brand。

> **为什么 Tenant 按统一社会信用代码识别？** 统一社会信用代码是法人主体的唯一标识，与营业执照一一对应。以此作为 Tenant 的业务唯一键，在上游系统推送相同商户数据时可做幂等判断，避免重复创建。

## 4. 核心聚合根设计

### 4.1 平台聚合 `Platform`

**聚合根：** `platform`

> **设计意图：** Platform 是整个系统的顶层隔离单元，代表一个购物中心集团（如万达集团）。多个 Platform 可在同一套系统环境中并行运行，彼此数据完全隔离。Platform 主数据由系统管理员维护，不由上游同步。

**完整字段定义：**

```
platform_id             平台ID（UUID）
platform_code           平台编码（全局唯一，如"wanda"）
platform_name           平台名称（如"万达集团"）
platform_status         状态（ACTIVE|INACTIVE）
contact_person          平台联系人
contact_phone           联系电话
created_at              创建时间
updated_at              更新时间
```

---

### 4.2 租户聚合 `Tenant`

**聚合根：** `tenant`

> **设计意图：** Tenant 代表一个经营主体（法人），按营业执照统一社会信用代码识别。每个 Platform 独立维护自己的 Tenant 列表——即使同一个商户（如肯德基）同时是万达和大悦城的租户，在两个 Platform 下也是两条独立的 Tenant 记录，互不干扰。Tenant 数据由外部招商管理系统维护并同步进来，本系统**不发起任何入驻或审核流程**。

**完整字段定义：**

```
tenant_id               租户ID（UUID）
platform_id             归属平台ID（Platform 隔离键）
tenant_code             租户编码（platform 内唯一）
tenant_name             租户名称

企业信息：
company_name            企业名称
unified_social_credit_code 统一社会信用代码（platform 内唯一业务键）
legal_representative    法定代表人
contact_person          联系人
contact_phone           联系电话
contact_email           联系邮箱

租户类型：
tenant_type             租户类型（SINGLE_STORE|CHAIN_BRAND）
business_model          经营模式（SELF_OPERATED|FRANCHISED|MIXED）

账户状态：
account_status          账户状态（ACTIVE|SUSPENDED|INACTIVE）
status_reason           状态原因（上游同步时携带）

同步信息：
external_tenant_id      上游系统中的商户ID
synced_at               最近同步时间
created_at              创建时间
updated_at              更新时间
version                 乐观锁版本号
```

> **为什么删除入驻/审核字段？** 入驻审核是前置业务，由外部招商管理系统负责。本系统收到同步数据时，商户已通过审核。在本系统中重建审核流程是职责越界，会形成两套冲突的数据源。

> **为什么 account_status 简化为三种？** ACTIVE（正常经营）、SUSPENDED（临时停业）、INACTIVE（停止合作）三种状态已覆盖执行系统需要感知的全部场景。复杂的入驻流程状态（DRAFT/PENDING_REVIEW/APPROVED）不属于执行系统的职责范围。

> **为什么 Tenant 不通过 Project 归属，而直接归属 Platform？** 商户可能同时在同一集团下的多个购物中心经营（如肯德基进入万达的多个商场），但商户主体只有一个。Tenant 归属 Platform（集团），通过 Contract 与具体的 Project（购物中心）关联，避免为同一商户重复创建多条 Tenant 记录。

### 4.3 品牌聚合 `Brand`

**聚合根：** `brand`

> **设计意图：** Brand 是 Platform 下独立维护的品牌库，与 Tenant **无直接关联**。多个 Tenant 可以通过不同 Contract 以同一个 Brand 经营不同的 Store。同一餐饮品牌在不同 Platform 下是独立的 Brand 记录。Brand 数据由上游系统同步，本系统不创建或修改品牌主数据。

**完整字段定义：**

```
brand_id                品牌ID
platform_id             归属平台ID（Platform 隔离键，Brand 直接归属于 Platform）
brand_code              品牌编码（platform 内唯一）
brand_name              品牌名称
brand_name_en           品牌英文名称
brand_logo_url          品牌Logo URL
brand_description       品牌描述

品牌定位：
brand_category          品牌类别（咖啡/茶饮/烘焙/正餐等）
positioning             品牌定位（高端/中端/大众）

标准化配置：
standard_menu_enabled   是否启用标准菜单
standard_pricing_locked 是否锁定标准定价
standard_operation_sop  标准作业流程（JSON）

ERP集成：
erp_system_type         ERP系统类型
erp_integration_enabled 是否启用ERP集成
erp_api_endpoint        ERP API端点
erp_auth_config         ERP认证配置（加密存储）

同步信息：
external_brand_id       上游系统中的品牌ID
synced_at               最近同步时间

状态：
brand_status            品牌状态（ACTIVE|INACTIVE）
created_at              创建时间
```

### 4.4 门店聚合 `Store`

**聚合根：** `store`

> **设计意图：** Store 对应购物中心内一个物理铺位在某时期内的经营状态。`unit_no`（铺位号）+ `project_id` 唯一确定物理位置。当前经营的 Tenant 和 Brand 由 `active_contract_id` 指向的生效 Contract 决定。Store 本身只存储快照引用，不直接持有 Tenant/Brand 归属。

**完整字段定义：**

```
store_id                门店ID（UUID）
platform_id             归属平台ID（冗余存储，便于查询隔离）
project_id              归属购物中心项目ID
store_code              门店编码（platform 内唯一）
store_name              门店名称
unit_no                 铺位号（在 project 内唯一，如"L0102"）

当前合同快照（随 active_contract 更新）：
active_contract_id      当前生效合同ID（为空表示铺位暂无经营主体）
tenant_id               当前租户ID（冗余自 active_contract）
brand_id                当前品牌ID（冗余自 active_contract，可为null）
entity_id               当前经营主体ID

门店类型：
store_type              门店类型（DIRECT|FRANCHISE|JOINT_VENTURE）
business_format         业态（堂食/外卖/自取/综合）

地址信息：
address_detail          详细地址（含楼层铺位描述）
latitude                纬度
longitude               经度
poi_id                  POI ID（地图标识）

联系信息：
store_phone             门店电话
store_manager           店长姓名
manager_phone           店长电话

营业信息：
opening_date            开业日期
business_area           营业面积（平方米）
seat_count              座位数

设施信息：
has_dine_in             是否支持堂食
has_takeaway            是否支持外卖
has_parking             是否有停车位
wifi_available          是否有WiFi
facilities              其他设施（JSON）

门店状态：
store_status            门店状态（OPERATING|SUSPENDED|CLOSED|VACANT）
status_reason           状态原因
status_changed_at       状态变更时间

同步信息：
external_store_id       上游系统中的门店ID
synced_at               最近同步时间
created_at              创建时间
version                 乐观锁版本号
```

> **为什么 tenant_id/brand_id 在 Store 上冗余存储？** Store 是高频查询对象，冗余存储避免每次都 JOIN Contract 表，但这两个字段的权威值是 Contract，随 active_contract_id 变更而更新。

> **为什么新增 VACANT 状态？** 铺位在两个合同之间存在空档期，这段时间没有经营主体，VACANT 反映这个现实。

### 4.5 合同聚合 `Contract`

**聚合根：** `contract`

> **设计意图：** Contract 是连接 Store（铺位）与 Tenant（经营主体）、Brand（品牌）、Project（购物中心）的核心关系实体，记录某商户在某铺位经营某品牌的完整约定和有效期。合同数据由外部合同管理系统维护并同步。一个门店可有多份历史合同，`store.active_contract_id` 指向当前生效的那份。

**完整字段定义：**

```
contract_id             合同ID（UUID）
platform_id             归属平台ID
project_id              归属购物中心项目ID
store_id                归属门店ID
tenant_id               经营主体/租户ID
brand_id                品牌ID（单店模式可为null）
unit_no                 铺位号（冗余自 store，便于查询）

合同信息：
contract_no             合同编号（platform 内唯一）
contract_type           合同类型（OPERATING|LEASE）
start_date              合同开始日期
end_date                合同结束日期

合同状态：
contract_status         合同状态（PENDING|ACTIVE|EXPIRED|TERMINATED）

同步信息：
external_contract_no    上游系统合同号
synced_at               最近同步时间
created_at              创建时间
updated_at              更新时间
```

**状态机：**

```
PENDING（待生效）→ ACTIVE（生效中）→ EXPIRED（已到期）
PENDING/ACTIVE  → TERMINATED（提前终止，由上游同步驱动）
```

> **为什么合同状态变更由定时任务驱动？** 合同生效/到期是时间事件，不需要人工干预。定时任务每日检查，将到期合同标记 EXPIRED 并触发 ContractExpired 事件，通知商品域停用菜单、结算域关闭账期。

### 4.4 桌台聚合 `Table`

**聚合根：** `table`

> **设计意图：** Table 是堂食场景的核心资源。顾客到店用餐必须分配桌台，扫码点单需要桌台二维码，翻台率是门店经营效率的关键指标。Table 聚合管理桌台的物理属性和实时状态。

**完整字段定义：**

```
table_id                桌台ID（UUID）
store_id                归属门店ID
table_no                桌台编号（门店内唯一）
table_name              桌台名称（可选）

桌台属性：
area                    区域（大厅/包间/露台等）
capacity                容纳人数
table_type              桌台类型（REGULAR|VIP|OUTDOOR|PRIVATE_ROOM）
qr_code_url             桌台二维码URL（扫码点单）

桌台状态：
status                  桌台状态（AVAILABLE|OCCUPIED|RESERVED|CLEANING|DISABLED）
current_session_id      当前桌台会话ID（堂食占用的主锚点，关联table_sessions.session_id）
current_booking_id      当前预订单ID（已预订时，关联booking_orders.booking_id）
current_order_id        当前订单ID（查询投影字段，仅用于快速跳转，不作为占用判断依据）
current_customer_count  当前就餐人数

占用信息：
occupied_at             占用开始时间
occupied_by             占用操作人
estimated_duration      预计用餐时长（分钟）

其他：
sort_order              排序
is_active               是否启用
created_at              创建时间
updated_at              更新时间
```

> **为什么桌台状态需要五种？** 每种状态代表桌台的一个真实物理状态：AVAILABLE（空桌可用）、OCCUPIED（有人在吃）、RESERVED（已被预订）、CLEANING（刚吃完在收拾）、DISABLED（暂停使用，如设备维修）。遗漏任何一种都会导致桌台分配逻辑出错。

> **为什么 current_booking_id 关联的是预订域的 booking_id？** 桌台的预订详情（预订时间、预订人信息、截止时间等）由预订域的 BookingOrder 聚合管理，桌台只需要知道"这张桌子被谁预订了"（通过 booking_id 引用），而不需要复制预订信息。这是跨域引用而非数据冗余的设计选择。

> **current_session_id 是堂食占用的唯一主锚点（P1-5）：** 判断桌台是否被占用，以 `current_session_id` 是否有值为准，而不是 `current_order_id`。原因：正餐场景"先开台后点菜"，开台时 session 已存在但 order 尚未创建；同桌多轮加菜时 order 内容持续变化，session 始终稳定。`current_order_id` 仅作为查询投影字段保留，方便前端快速跳转到当前订单，不参与占用判断逻辑。

**状态机：**

```
AVAILABLE（空闲可用）
    ↓ [OccupyTable 命令]
OCCUPIED（已占用）
    ↓ [ReleaseTable 命令]
CLEANING（清理中）
    ↓ [FinishCleaning 命令]
AVAILABLE

AVAILABLE
    ↓ [ReserveTable 命令]（预订域BookingConfirmed事件触发）
RESERVED（已预订，current_booking_id已填入）
    ↓ [OccupyTable 命令]（预订域BookingArrived事件触发，顾客到店开台）
OCCUPIED
    ↓ [CancelReservation 命令]（预订域BookingCancelled事件触发）
AVAILABLE

AVAILABLE/OCCUPIED/RESERVED
    ↓ [DisableTable 命令]
DISABLED（已停用）
    ↓ [EnableTable 命令]
AVAILABLE
```

> **为什么预订相关状态转换由预订域事件驱动？** 桌台域不管理预订业务逻辑（如预订时间冲突检测、超时取消策略）。当预订域确认了一个预订（BookingConfirmed），桌台只需响应性地将状态改为 RESERVED。这保持了职责边界的清晰。

**不变式：**

- 桌台编号在门店内必须唯一
- 容纳人数必须大于0
- OCCUPIED 状态必须关联 current_session_id
- RESERVED 状态必须关联 current_booking_id（预订单ID由预订域管理）
- 预订相关字段（预订时间、截止时间、顾客信息）已迁移至预订域 BookingOrder 聚合

**业务规则：**

- **扫码点单**：顾客扫描桌台二维码自动关联桌台信息
- **占用检查**：开台时检查桌台是否可用（AVAILABLE 或 RESERVED）
- **自动释放**：订单完成后自动释放桌台（状态改为 CLEANING）
- **预订联动**：桌台 RESERVED/AVAILABLE 状态由预订域事件（BookingConfirmed/BookingCancelled）驱动，桌台本身不存储预订详情
- **翻台率统计**：记录占用时长用于翻台率分析

> **为什么自动释放后进入 CLEANING 而非直接 AVAILABLE？** 现实中桌台在客人离开后需要清洁，不能立即安排下一桌。如果直接变为 AVAILABLE，可能导致新客人被带到未清理的桌位。CLEANING 状态反映了物理世界的真实约束。

### 4.5 经营主体聚合 `BusinessEntity`

**聚合根：** `business_entity`

> **设计意图：** BusinessEntity 是结算和开票的法律主体。一个租户（如某餐饮集团）可能在不同区域注册了不同的公司实体来经营门店。结算域需要知道"这笔钱该打给哪个公司的哪个银行账户"，这就是 BusinessEntity 的价值。

**完整字段定义：**

```
entity_id               经营主体ID
entity_name             主体名称
entity_type             主体类型（INDIVIDUAL|COMPANY|PARTNERSHIP）
tenant_id               归属租户ID

企业信息：
company_name            公司名称
unified_social_credit_code 统一社会信用代码
legal_representative    法定代表人
registered_address      注册地址

银行账户：
bank_name               开户银行
bank_account_name       账户名称
bank_account_no         银行账号（加密存储）
bank_branch             开户支行

税务信息：
tax_registration_no     税务登记号
taxpayer_type           纳税人类型（GENERAL|SMALL_SCALE）
tax_rate                税率

结算配置：
settlement_cycle        结算周期（DAILY|WEEKLY|MONTHLY）
settlement_day          结算日（每月几号）
auto_settlement_enabled 是否自动结算

状态：
entity_status           主体状态（ACTIVE|INACTIVE）
created_at              创建时间
```

> **为什么 bank_account_no 需要加密存储？** 银行账号是敏感金融信息，属于 PCI DSS 合规范畴。明文存储银行账号一旦发生数据泄露，将直接导致商户资金安全风险。加密存储是安全底线。

> **为什么结算配置放在 BusinessEntity 而非 Tenant？** 因为结算是按经营主体进行的，不是按租户。同一租户的不同经营主体可能有不同的结算周期（如上海子公司日结，北京子公司月结）。结算配置跟着法律实体走，不跟着商业品牌走。

> **为什么 entity_type 区分三种？** INDIVIDUAL（个体工商户）、COMPANY（公司法人）、PARTNERSHIP（合伙企业）在税务处理、发票类型、结算流程上有本质区别。例如个体工商户通常只能开普通发票，而一般纳税人公司可以开增值税专用发票。

---

### 4.6 员工聚合 `Staff`

**聚合根：** `staff`

> **设计意图：** Staff 管理平台上所有操作人员的身份认证和权限控制。从店长到收银员到厨师，每个人能看什么数据、能做什么操作，都由 Staff 聚合的角色权限决定。

**完整字段定义：**

```
staff_id                员工ID
staff_code              员工编码
staff_name              员工姓名
tenant_id               归属租户ID
store_id                归属门店ID（可选，总部员工为null）

账号信息：
username                登录用户名
password_hash           密码哈希
phone                   手机号
email                   邮箱

员工信息：
id_card_no              身份证号（加密存储）
gender                  性别
birth_date              出生日期
hire_date               入职日期
employment_type         雇佣类型（FULL_TIME|PART_TIME|INTERN）

职位信息：
position                职位（店长/收银员/厨师等）
department              部门
role_ids                角色ID列表

账号状态：
account_status          账号状态（ACTIVE|LOCKED|RESIGNED）
last_login_at           最后登录时间
password_expire_at      密码过期时间

其他：
created_at              创建时间
created_by              创建人
```

> **为什么 store_id 是可选的？** 连锁品牌的总部员工（如运营总监、财务经理）不归属于任何门店，他们需要跨门店查看数据。store_id 为 null 表示"总部级别"，配合角色权限可以控制其可见的门店范围。

> **为什么 id_card_no 加密存储？** 身份证号是个人隐私信息，受《个人信息保护法》保护。明文存储违反法律要求，且一旦泄露可能被用于身份冒用。

> **为什么 account_status 区分 ACTIVE/LOCKED/RESIGNED？** ACTIVE 是正常使用，LOCKED 是临时锁定（如密码错误次数过多、安全事件处理中），RESIGNED 是永久停用（离职）。LOCKED 可以恢复为 ACTIVE，RESIGNED 不可以。这三种状态覆盖了员工生命周期的所有阶段。

### 4.7 设备聚合 `Device`

**聚合根：** `device`

> **设计意图：** Device 管理门店中所有需要与平台通信的硬件设备。餐饮门店是"重设备"场景：POS 机收银、KDS 屏幕显示订单、打印机出小票、自助点餐机下单。Device 聚合统一管理设备的注册、绑定、状态监控和命令下发。

**完整字段定义：**

```
device_id               设备ID
device_code             设备编码
device_name             设备名称
device_type             设备类型（POS|KDS|PRINTER|SELF_SERVICE_KIOSK）
store_id                归属门店ID

设备信息：
manufacturer            制造商
model                   型号
serial_number           序列号
mac_address             MAC地址
ip_address              IP地址

绑定信息：
binding_status          绑定状态（UNBOUND|BOUND|OFFLINE）
bound_at                绑定时间
station_id              工作站ID（KDS专用）
station_type            工作站类型（HOT_DRINK|COLD_DRINK等）

配置信息：
device_config           设备配置（JSON）
  print_template        打印模板
  display_layout        显示布局
  network_config        网络配置

设备会话：
current_session_id      当前会话ID
session_token           会话令牌（设备登录后获取）
logged_in_staff_id      当前登录员工ID
logged_in_at            登录时间

心跳：
last_heartbeat_at       最后心跳时间
heartbeat_interval_sec  心跳间隔（秒，默认30）
offline_threshold_sec   离线判定阈值（秒，默认90）

能力：
capability_version      设备能力版本
supported_commands      支持的命令列表（JSON，如["PRINT","DISPLAY_ORDER","CALL_PICKUP"]）
app_version             应用版本号
os_version              操作系统版本

状态信息：
device_status           设备状态（ONLINE|OFFLINE|FAULT）
fault_reason            故障原因

其他：
created_at              创建时间
```

> **为什么需要心跳机制？** 设备在门店的物理环境中运行，网络不稳定是常态（如WiFi信号弱、断电重启）。心跳机制通过定期"报平安"来检测设备是否在线。当超过 offline_threshold_sec（默认90秒，即3个心跳周期）未收到心跳时，判定设备离线并触发告警。

> **为什么 station_id 和 station_type 是 KDS 专用？** KDS（Kitchen Display System）通常按工作站分类：热饮站、冷饮站、出餐口等。当订单到达时，需要根据菜品类型将制作指令分发到正确的工作站屏幕上。station_id 建立了 KDS 设备与履约域工作站的绑定关系。

> **为什么需要 supported_commands？** 不同型号的设备支持的功能不同。老款打印机可能只支持 PRINT 命令，新款 KDS 可能还支持 DISPLAY_ORDER 和 CALL_PICKUP（叫号取餐）。在下发命令前需要检查设备是否支持该命令，避免发送无法执行的指令。

**实体：DeviceCommand（设备命令）**

```
command_id              命令ID（UUID）
device_id               目标设备ID
command_type            命令类型（PRINT|DISPLAY_ORDER|CALL_PICKUP|REFRESH_MENU|LOGOUT）
command_payload         命令内容（JSON）
status                  状态（PENDING|SENT|ACKNOWLEDGED|FAILED|TIMEOUT）
sent_at                 发送时间
ack_at                  回执时间
retry_count             重试次数（最多3次）
idempotency_key         幂等键
created_at              创建时间
```

> **为什么需要 idempotency_key？** 在网络不稳定的环境中，命令可能被重复发送（如重试时上一次的命令其实已经到达设备）。幂等键确保设备收到重复命令时只执行一次，避免出现"打了两张小票"或"叫了两次号"的问题。

> **为什么 status 有五种？** 命令从创建到完成经历明确的生命周期：PENDING（等待发送）-> SENT（已推送到设备）-> ACKNOWLEDGED（设备确认已执行）。如果设备执行失败返回 FAILED，如果超时未回执返回 TIMEOUT。每种状态都对应不同的处理逻辑。

**实体：OfflineOperationLog（离线操作日志）**

```
log_id                  日志ID（UUID）
device_id               设备ID
store_id                门店ID
operation_type          操作类型（OPEN_TABLE|ADD_ITEM|PAYMENT等）
operation_payload       操作内容（JSON）
local_timestamp         设备本地时间
server_timestamp        服务端接收时间
sync_status             同步状态（PENDING|SYNCED|CONFLICT|FAILED）
conflict_reason         冲突原因（如"桌台已被占用"）
created_at              创建时间
```

> **为什么需要离线操作日志？** 餐饮门店不能因为网络断开就停止营业。POS 设备需要支持离线模式：网络断开时在本地缓存操作，网络恢复后批量同步到服务端。OfflineOperationLog 记录了每一条离线操作及其同步结果，用于处理同步冲突（如离线期间两个设备都开了同一张桌）。

> **为什么需要 local_timestamp 和 server_timestamp 两个时间？** 离线操作发生在设备本地，local_timestamp 记录真实操作时间。同步到服务端后的 server_timestamp 可能晚很多。两个时间的差值反映了离线时长，也用于在冲突解决时判断操作的先后顺序。

**设备命令下发流程：**

```
平台下发命令
    ↓
DeviceCommand 创建（status=PENDING）
    ↓
通过 WebSocket/长连接 推送到设备
    ↓
设备执行并回执（status=ACKNOWLEDGED）
    ↓
超时未回执（>30s）→ 重试（最多3次）→ 告警
```

> **为什么用 WebSocket 而非 HTTP 轮询？** 设备需要实时接收命令（如新订单到达时立即打印小票），HTTP 轮询的延迟（通常数秒）在餐饮高峰期不可接受。WebSocket 建立持久连接，服务端可以主动推送，延迟在毫秒级。

> **为什么超时阈值是30秒、最多重试3次？** 30秒足以覆盖绝大多数网络抖动场景。如果3次重试（总计约2分钟）仍未成功，大概率是设备故障或网络断开，此时应该触发告警通知门店人员处理，而非无限重试浪费资源。

## 5. 核心领域服务

### 5.1 MasterDataSyncService（主数据同步服务）

> 接收外部系统推送的组织主数据，执行幂等写入。

```
SyncPlatform(data: PlatformSyncData) → Platform

SyncTenant(data: TenantSyncData) → Tenant
  // 以 platform_id + unified_social_credit_code 做幂等键

SyncBrand(data: BrandSyncData) → Brand
  // 以 platform_id + external_brand_id 做幂等键

SyncContract(data: ContractSyncData) → Contract
  // 合同状态变更时，触发门店 active_contract_id 更新

SyncStore(data: StoreSyncData) → Store
  // 以 platform_id + project_id + unit_no 做幂等键
```

### 5.2 OrganizationHierarchyService（组织层级服务）

```
GetPlatformHierarchy(
  platform_id: string

GetStoresByTenant(

) → Store[]

ValidateStoreAccess(
  store_id: string,
  platform_id: string,
  tenant_id: string  // 可选，用于校验租户归属
) → boolean

GetActiveContract(store_id: string) → Contract
```

> **为什么 ValidateStoreAccess 同时校验 platform_id 和 tenant_id？** platform_id 防止跨平台数据越权，tenant_id（可选）防止跨租户操作。每一个涉及门店的 API 调用都需经过此校验。

### 5.3 StaffAuthorizationService（员工授权服务）

```
AuthenticateStaff(
  username: string,
  password: string
) → AuthResult

GetStaffPermissions(
  staff_id: string
) → Permission[]

ValidatePermission(
  staff_id: string,
  resource: string,
  action: string
) → boolean
```

> **为什么权限校验需要 resource + action 两个维度？** 单纯的"有没有权限"不够精确。例如店长可以"查看"所有订单（resource=order, action=read），但不能"退款"大额订单（resource=order, action=refund, amount>500）。resource + action 的组合支持细粒度的 RBAC。

### 5.4 DeviceManagementService（设备管理服务）

```
RegisterDevice(
  device_info: DeviceInfo
) → Device

BindDeviceToStore(
  device_id: string,
  store_id: string,
  station_id: string
) → Result

UpdateDeviceStatus(
  device_id: string,
  status: string
) → Result

GetStoreDevices(
  store_id: string,
  device_type: string
) → Device[]

SendDeviceCommand(
  device_id: string,
  command_type: string,
  payload: JSON,
  idempotency_key: string
) → DeviceCommand

ProcessOfflineOperations(
  device_id: string,
  operations: OfflineOperationLog[]
) → SyncResult[]
```

> **为什么 SendDeviceCommand 需要 idempotency_key 参数？** 调用方（如履约域派发打印任务）在网络超时后可能重试调用。idempotency_key 确保同一个业务动作（如"订单001的小票打印"）不会因重试而产生多条命令。这个幂等性保证在设备侧和服务侧双重生效。

> **为什么 ProcessOfflineOperations 返回 SyncResult 数组？** 设备恢复在线后会批量提交多条离线操作，每条操作的同步结果可能不同（有的成功、有的冲突、有的失败）。返回数组让设备端可以逐条处理同步结果。

## 6. 关键域事件

### 6.1 ContractActivated（合同已生效）

**触发时机：** 合同 start_date 到达，定时任务将合同状态改为 ACTIVE，同时更新 store.active_contract_id

**消费方：**
- 商品域（初始化/激活门店菜单）
- 结算域（创建结算账期配置）
- 运营治理域（创建商户运营账户）

> **为什么用 ContractActivated 而非 TenantActivated 触发下游初始化？** 门店的菜单和结算配置与具体的经营合同绑定，而非与商户主体绑定。同一商户的不同门店（不同合同）有独立的菜单和结算账期，以合同为触发点更精确。

### 6.2 ContractExpired（合同已到期）

**触发时机：** 合同 end_date 到达，定时任务将合同状态改为 EXPIRED，并清空 store.active_contract_id

**消费方：**
- 商品域（停用该门店菜单）
- 结算域（关闭结算账期）

### 6.3 ContractTerminated（合同提前终止）

**触发时机：** 上游系统推送终止事件，同步到本系统

**消费方：** 与 ContractExpired 相同

### 6.4 StoreSynced（门店信息已同步）

**触发时机：** 上游系统推送门店信息更新（地址、联系方式等变更）

**消费方：**
- TDP（更新推送给终端的门店配置）

### 6.5 DeviceBound（设备已绑定）

**消费方：**
- 履约域（更新工作站配置）

### 6.6 DeviceCommandAcknowledged（设备命令已回执）

**消费方：**
- 运营治理域（记录执行结果）

### 6.7 DeviceOffline（设备已离线）

**消费方：**
- 运营治理域（告警通知）

## 7. 与其他域的交互

### 7.1 向商品域提供服务

**接口：** `GetStoreInfo`

**时机：** 商品域需要门店信息时

**返回内容：** 门店基础信息

> **交互方式：** 同步查询。商品域在加载门店菜单、校验营业状态时需要实时的门店信息。

### 7.2 向结算域提供服务

**接口：** `GetBusinessEntityInfo`

**时机：** 结算域生成账单时

**返回内容：** 经营主体和银行账户信息

> **为什么结算域不直接存储银行账户？** 银行账户是敏感信息，应该由单一域（租户域）统一管理和加密存储，其他域通过接口查询。这减少了敏感数据的扩散面，降低了安全风险。

### 7.3 向履约域提供服务

**接口：** `GetStoreDevices`

**时机：** 履约域派发工作单时

**返回内容：** 门店设备列表

> **为什么履约域需要设备列表？** 履约域在分派制作任务时需要知道门店有哪些 KDS 设备、分布在哪些工作站，才能将订单中的菜品分派到正确的制作位。

## 8. 数据一致性保障

### 8.1 组织层级一致性

- Platform 是所有实体的顶层隔离键，不同 Platform 的数据完全隔离
- Contract 是 Store/Tenant/Brand 关联关系的权威来源
- store.tenant_id / store.brand_id 是从 active_contract 冗余的快照，必须与 Contract 保持一致
- 合同状态变更时，原子性更新 store.active_contract_id 及快照字段

### 8.2 员工权限一致性

- 员工只能访问其归属 platform_id + tenant_id 范围内的数据
- 角色权限变更实时生效
- 员工离职后账号自动锁定

### 8.3 设备绑定一致性

- 设备只能绑定到一个门店
- 设备解绑后才能重新绑定
- 设备离线超过阈值自动告警

### 8.4 主数据同步幂等性

- 所有 Sync 接口必须幂等：以业务唯一键（如 platform_id + unified_social_credit_code）判断是创建还是更新
- 同步数据以上游系统为准，本系统不应修改同步来的主数据字段

## 9. 核心设计原则总结

| 原则 | 内容 | 为什么 |
|------|------|--------|
| Platform 隔离 | platform_id 是最顶层隔离键，不同集团数据完全独立 | 支持多集团共存，满足各集团数据主权需求 |
| 主数据只读 | 本系统不创建/修改 Platform/Tenant/Brand/Contract 主数据 | 餐饮中台是执行系统，主数据由招商管理系统负责 |
| Contract 驱动 | 门店与租户/品牌的关联通过 Contract 管理，支持历史追溯 | 商户/品牌变更有历史记录，经营事实可追溯 |
| 层级清晰 | Platform→(Project/Tenant/Brand)，Store 通过 Contract 关联 Tenant+Brand，层级关系明确 | 清晰的组织结构支撑权限分配、数据汇总、报表生成 |
| 权限最小化 | 员工只能访问必要的 platform + tenant 范围内数据 | 降低数据泄露风险，满足合规要求 |
| 设备唯一绑定 | 设备与门店一对一绑定 | 消除设备配置歧义，确保命令正确路由 |
