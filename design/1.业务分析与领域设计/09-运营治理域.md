# 09-运营治理域

## 1. 域定位与设计意图

运营治理域是平台的**治理中枢**，负责异常处理、补偿机制、审计日志和运营监控。在一个涉及多个外部系统、多个业务域的复杂平台中，异常是常态而非例外——订单支付超时、配送派单失败、ERP同步错误、渠道推送失败等问题持续发生。如果没有统一的治理机制，这些异常将散落在各个业务域中，缺乏全局视图和标准化的处理流程。

运营治理域的核心设计意图是：

1. **异常可追溯**：所有异常记录完整保留，处理过程有迹可查。异常台账提供全局视图，运营人员可以从一个入口看到平台所有异常。
2. **补偿可重试**：补偿任务支持自动重试和人工介入，确保业务数据最终一致。补偿机制是分布式系统中实现最终一致性的关键手段。
3. **日志不可变**：审计日志只能追加不能修改，满足合规和审计要求，支持事后追责和问题复盘。
4. **告警有抑制**：避免告警风暴，在外部系统大规模故障时不会产生数千条重复告警淹没运维人员。
5. **指标可聚合**：支持多维度（时间、门店、渠道）指标分析，提供经营决策数据支撑。

---

## 2. 域职责边界

### 2.1 核心职责

- 管理异常台账和补偿任务
- 记录完整的审计日志
- 监控系统健康状态和业务指标
- 管理人工介入工作流
- 提供运营报表和数据分析
- 管理告警规则和通知

### 2.2 不负责的事项

- 不管理业务主流程（各核心域职责）——运营治理域只观察和治理，不参与正常的业务流转
- 不直接修改业务数据（通过补偿命令）——本域发现问题后通过向各业务域发送补偿命令来修正，而不是直接操作业务域的数据，保持域间边界清晰
- 不管理用户权限（租户域职责）

## 3. 核心聚合根设计

### 3.1 异常台账聚合 `ExceptionRecord`

**聚合根：** `exception_record`

**设计意图：** 异常台账是运营治理的核心数据结构，将平台各处发生的异常统一收集、分类、跟踪直至解决。异常记录包含丰富的上下文（业务信息、影响评估、处理过程），使得运营人员无需跨系统查看日志即可理解异常全貌。乐观锁版本号防止并发处理冲突。SLA 截止时间驱动处理时效管理。

**完整字段定义：**

```

```

**状态机：**

异常处理遵循标准的工单流转模型：检测 → 分配 → 处理 → 解决。支持升级机制处理超出一线运维能力的复杂问题，支持直接关闭无需处理的误报。

```
DETECTED（已检测）
    ↓ [AssignException 命令]
ASSIGNED（已分配）
    ↓ [StartHandling 命令]
IN_PROGRESS（处理中）
    ↓ [ResolveException 命令]
RESOLVED（已解决）

异常路径：
IN_PROGRESS → ESCALATED（已升级）
ESCALATED → IN_PROGRESS（重新处理）
任意状态 → CLOSED（已关闭，无需处理）
```

### 3.2 补偿任务聚合 `CompensationTask`

**聚合根：** `compensation_task`

**设计意图：** 在分布式系统中，业务操作可能部分成功部分失败（如支付成功但通知失败），需要补偿机制来修正不一致状态。CompensationTask 将补偿操作标准化为可跟踪、可重试、可审批的任务，避免补偿操作散落在代码各处难以管理。审批流程确保涉及金额的补偿（如退款、发券）经过授权。

**完整字段定义：**

```
compensation_task_id    补偿任务ID
exception_record_id     关联异常记录ID
task_type               任务类型（见枚举）
task_category           任务分类（AUTOMATIC|MANUAL）

业务上下文：
business_type           业务类型
business_id             业务ID
business_no             业务单号
store_id                门店ID
tenant_id               租户ID

补偿方案：
compensation_action     补偿动作（见枚举）
compensation_target     补偿目标（CUSTOMER|MERCHANT|PLATFORM）
compensation_amount     补偿金额（分）
compensation_reason     补偿原因
compensation_plan       补偿方案（JSON）

执行配置：
execution_mode          执行模式（IMMEDIATE|SCHEDULED|MANUAL）
scheduled_at            计划执行时间
max_retry_count         最大重试次数
retry_interval          重试间隔（秒）

执行状态：
task_status             任务状态（见状态机）
execution_count         执行次数
last_execution_at       最后执行时间
last_execution_result   最后执行结果
error_message           错误信息

审批流程：
approval_required       是否需要审批（YES|NO）
approval_status         审批状态（PENDING|APPROVED|REJECTED）
submitted_for_approval_at 提交审批时间
approved_at             审批时间
approved_by             审批人
approval_comment        审批意见

完成信息：
completed_at            完成时间
completed_by            完成人
completion_note         完成备注

其他：
created_at              创建时间
created_by              创建人
version                 乐观锁版本号
```

**补偿动作枚举：**

补偿动作按业务域分类，覆盖了从技术重试到客户安抚的完整补偿场景。AUTOMATIC 类型的补偿（如重试派单、刷新 Token）由系统自动执行；涉及金额变动的补偿（如退款、发券）通常需要人工审批。

```
订单补偿：
RETRY_ORDER_CREATION        重试订单创建
RETRY_PAYMENT_CONFIRMATION  重试支付确认
RETRY_ORDER_ACCEPTANCE      重试订单接单
CANCEL_ORDER_WITH_REFUND    取消订单并退款
ADJUST_ORDER_AMOUNT         调整订单金额

渠道补偿：
RETRY_CHANNEL_PUSH          重试渠道推送
RETRY_STATUS_SYNC           重试状态同步
REFRESH_CHANNEL_TOKEN       刷新渠道Token
MANUAL_CHANNEL_OPERATION    人工渠道操作

结算补偿：
ADJUST_SETTLEMENT_AMOUNT    调整结算金额
REGENERATE_SETTLEMENT_BILL  重新生成结算账单
MANUAL_RECONCILIATION       人工对账

配送补偿：
RETRY_DELIVERY_DISPATCH     重试配送派单
SWITCH_DELIVERY_PLATFORM    切换配送平台
CANCEL_DELIVERY_ORDER       取消配送订单

客户补偿：
ISSUE_COUPON                发放优惠券
REFUND_TO_CUSTOMER          退款给客户
COMPENSATE_POINTS           补偿积分
```

**状态机：**

补偿任务的状态机重点设计了失败和重试路径。在分布式环境中，补偿操作本身也可能失败（如目标系统不可用），因此需要支持多次重试。重试次数耗尽后转入人工介入状态，避免无限重试消耗系统资源。

```
PENDING（待执行）
    ↓ [ExecuteCompensation 命令]
EXECUTING（执行中）
    ↓ [执行成功]
COMPLETED（已完成）

异常路径：
EXECUTING → FAILED（执行失败）
FAILED → RETRYING（重试中）
RETRYING → COMPLETED（重试成功）
RETRYING → FAILED（重试失败）
FAILED → MANUAL_INTERVENTION（需人工介入）
```

### 3.3 审计日志聚合 `AuditLog`

**聚合根：** `audit_log`

**设计意图：** 审计日志是平台合规能力的基石。不同于业务日志，审计日志强调不可变性——一旦写入不可修改或删除（在保留期限内），确保操作记录的法律效力。变更内容（before_data/after_data）完整记录数据变更前后的快照，支持事后审计和回滚分析。保留天数根据合规要求不同而不同。

**日志保留策略：**

保留期限根据数据敏感程度和合规要求确定。涉及资金流转的日志（订单、支付、结算）保留5年以满足财税审计要求；员工操作日志保留3年以满足劳动法规要求；商品操作等非敏感日志保留期较短以节省存储成本。

```
日志类型          保留期限
订单操作日志      5年
支付操作日志      5年
结算操作日志      5年
员工操作日志      3年
商品操作日志      1年
访问日志          6个月
系统日志          3个月
```

### 3.5 告警规则聚合 `AlertRule`

**聚合根：** `alert_rule`

**设计意图：** 告警规则将指标异常检测自动化。支持三种规则类型：阈值型（THRESHOLD，指标超过固定值时告警）、趋势型（TREND，指标同比/环比变化异常时告警）和异常型（ANOMALY，基于统计模型检测异常波动）。抑制配置（suppression）是告警系统的关键设计——当外部系统大规模故障时，同一规则可能每分钟都被触发，如果不抑制将产生告警风暴淹没运维人员。

**完整字段定义：**

```
alert_rule_id           告警规则ID
rule_name               规则名称
rule_type               规则类型（THRESHOLD|TREND|ANOMALY）
rule_category           规则分类（BUSINESS|TECHNICAL|QUALITY）

监控对象：
monitor_target          监控目标（ORDER|PAYMENT|DELIVERY|SYSTEM）
metric_name             指标名称
aggregation_method      聚合方法（SUM|AVG|MAX|MIN|COUNT）
time_window             时间窗口（分钟）

触发条件：

alert_message_template  告警消息模板
notification_channels   通知渠道（EMAIL|SMS|WECHAT|WEBHOOK）
notification_recipients 通知接收人列表

抑制配置：

suppression_duration    抑制时长（分钟）

规则状态：

last_triggered_at       最后触发时间

其他：
created_at              创建时间

```

## 4. 接入治理聚合设计（P1-2）

**设计意图：** 当中台开放 API 给第三方 ISV 或商户自有系统调用时，需要一套完整的接入治理机制——客户端注册审核、访问授权管理、回调事件推送。接入治理确保 API 调用的安全性（认证授权）、可控性（权限范围）和可观测性（回调投递日志）。

### 4.1 应用客户端聚合 `AppClient`

**聚合根：** `app_client`

**设计意图：** 每个接入中台 API 的第三方应用需要先注册为客户端并经过审核。requested_scopes 与 approved_scopes 分离，使得审核人员可以在审批时缩减请求方的权限范围。

```
client_id               客户端ID（UUID）
client_name             客户端名称
client_type             类型（ISV|MERCHANT_SELF）
client_secret           客户端密钥（加密存储）
contact_email           联系邮箱
status                  状态（PENDING_REVIEW|ACTIVE|SUSPENDED|REVOKED）
requested_scopes        申请的权限范围（JSON数组）
approved_scopes         已批准的权限范围（JSON数组）
created_at / updated_at
```

### 4.2 授权记录聚合 `AuthorizationGrant`

**设计意图：** 授权记录管理客户端对特定门店的访问权限。一个客户端可能被授权访问多个门店，每个授权关系独立管理。Token 机制支持过期和刷新，撤销授权时可以立即使 Token 失效。

```
grant_id                授权记录ID
client_id FK            客户端ID
store_id FK             授权门店ID
granted_scopes          授权范围（JSON数组）
access_token            访问令牌（加密）
token_expires_at        令牌过期时间
refresh_token           刷新令牌
granted_at              授权时间
revoked_at              撤销时间（可空）
```

### 4.3 回调订阅聚合 `CallbackSubscription`

**设计意图：** 第三方系统需要实时接收中台的业务事件（如订单状态变更），回调订阅机制允许客户端按需订阅感兴趣的事件类型和门店范围，中台在事件发生时主动推送到客户端指定的回调地址。

```
subscription_id         订阅ID
client_id FK            客户端ID
event_types             订阅事件类型列表（JSON数组）
callback_url            回调地址
store_ids               订阅门店范围（JSON数组，空=全部）
status                  状态（ACTIVE|PAUSED|REVOKED）
created_at
```

### 4.4 回调投递日志 `CallbackDeliveryLog`

**设计意图：** 回调投递可能因为客户端系统故障而失败，需要记录每次投递的结果并支持重试。投递日志使运营人员可以查看投递失败的详细原因，也支持手动重放失败的回调。

```
log_id                  日志ID
subscription_id FK      订阅ID
event_id                事件ID
event_type              事件类型
payload                 投递内容（JSON）
status                  状态（SUCCESS|FAILED|PENDING）
http_status_code        HTTP响应码
retry_count             重试次数
next_retry_at           下次重试时间
created_at / delivered_at
```

**投递重试策略：**

采用指数退避策略，避免在客户端系统故障时频繁重试加重对方负担。3次重试后标记失败，支持问题排查后手动重放。

```
首次失败 → 1分钟后重试
第2次失败 → 5分钟后重试
第3次失败 → 30分钟后重试
第3次仍失败 → 标记FAILED，支持手动重放
```

## 5. 异常处理流程

### 5.1 异常检测与记录

**设计意图：** 异常检测是治理流程的起点。系统自动从各业务域的异常事件中创建异常记录，评估严重程度和影响范围，对于 CRITICAL 和 HIGH 级别的异常立即通知并自动分配，确保关键问题不被遗漏。

```
系统检测到异常
    ↓
1. 创建异常记录
    ↓
2. 评估严重程度
    ↓
3. 提取业务上下文
    ↓
4. 记录异常详情
    ↓
5. 计算影响范围
    ↓
6. 发布 ExceptionDetected 事件
    ↓
7. 触发告警通知
    ↓
if (severity == CRITICAL || severity == HIGH) {
    立即通知相关人员
    自动分配给值班人员
}
```

### 5.2 异常处理流程

**设计意图：** 异常处理遵循标准的工单处理流程，每一步都有记录。处理人可以记录调查过程和处理备注，确保知识沉淀。如果异常涉及数据不一致，处理人确定方案后创建补偿任务来自动执行修正操作。

```
异常记录已创建
    ↓
1. 分配给处理人
    ↓
2. 处理人开始调查
    ↓
3. 记录处理备注
    ↓
4. 确定解决方案
    ↓
5. 执行解决方案
    ↓
if (需要补偿) {
    创建补偿任务
    执行补偿操作
}
    ↓
6. 验证解决结果
    ↓
7. 更新异常状态为RESOLVED
    ↓
8. 记录解决过程
    ↓
9. 发布 ExceptionResolved 事件
```

### 5.3 补偿任务执行

**设计意图：** 补偿任务支持三种执行模式：立即执行（适用于自动化补偿如重试推送）、计划执行（适用于需要等待特定时间窗口的操作如批量对账）、手动执行（适用于需要人工确认的复杂操作）。审批机制确保涉及资金的补偿操作经过授权。重试耗尽后转入人工介入，避免自动化补偿在特殊场景下造成更大问题。

```
创建补偿任务
    ↓
if (approval_required) {
    提交审批
    等待审批结果
    if (rejected) {
        关闭任务
        return
    }
}
    ↓
if (execution_mode == IMMEDIATE) {
    立即执行
} else if (execution_mode == SCHEDULED) {
    等待计划时间
} else {
    等待人工触发
}
    ↓
执行补偿动作
    ↓
if (成功) {
    更新状态为COMPLETED
    记录完成信息
    发布 CompensationCompleted 事件
} else {
    if (retry_count < max_retry_count) {
        加入重试队列
    } else {
        更新状态为MANUAL_INTERVENTION
        通知人工处理
    }
}
```

## 6. 审计日志管理

### 6.1 审计日志记录规则

**设计意图：** 审计日志的记录范围基于合规要求和业务风险确定。所有涉及资金流转、数据变更、权限操作的行为必须记录，确保事后可追溯。

**必须记录的操作：**

- 所有订单状态变更
- 所有支付和退款操作
- 所有商品和菜单变更
- 所有员工账号操作
- 所有权限变更
- 所有结算和对账操作
- 所有敏感数据访问

### 6.2 审计日志查询

**设计意图：** 提供多维度的查询能力，支持按操作人查询（"这个员工做了什么"）、按资源查询（"这条记录被谁改过"）、按业务查询（"这个订单经历了什么"）。GenerateAuditReport 支持导出审计报告用于外部审计。

## 7. 运营监控与告警

### 7.1 指标采集

**设计意图：** 指标采集通过定时任务驱动，不同时间维度的指标采集频率不同——实时指标每分钟采集，小时指标每小时聚合，日指标每天汇总。采集后发布事件触发告警规则评估。

```
定时任务触发（每分钟/每小时/每天）
    ↓
1. 查询各域的业务数据
    ↓
2. 计算各项指标
    ↓
3. 聚合到不同维度
    ↓
4. 持久化指标数据
    ↓
5. 发布 MetricUpdated 事件
```

### 7.2 告警触发

**设计意图：** 告警触发将指标值与告警规则进行匹配。抑制机制是关键设计——在抑制期内同一规则的重复告警被跳过，避免告警风暴。抑制计时在首次告警后启动，抑制期结束后如果问题仍然存在则再次告警。

```
指标更新
    ↓
1. 查询相关的告警规则
    ↓
2. 评估触发条件
    ↓
if (满足触发条件) {
    if (在抑制期内) {
        跳过本次告警
    } else {
        创建告警记录
        发送告警通知
        记录触发时间
        启动抑制计时
    }
}
```

### 7.3 告警通知

**设计意图：** 告警通知支持多渠道送达，确保关键告警不被遗漏。不同告警级别可配置不同的通知渠道——CRITICAL 级别同时发送短信和企业微信，MEDIUM 级别只发送邮件。

```
告警触发
    ↓
1. 根据告警级别选择通知渠道
    ↓
2. 渲染告警消息
    ↓
3. 发送通知：
    - 邮件通知
    - 短信通知
    - 企业微信通知
    - Webhook通知
    ↓
4. 记录通知结果
```

## 9. 关键域事件

### 9.1 ExceptionDetected（异常已检测）

**触发时机：** 系统检测到异常并创建异常记录后发布。

**消费方：**
- 运营治理域内部（触发告警）——异常检测事件驱动告警规则评估，对严重异常立即通知

### 9.2 CompensationCompleted（补偿已完成）

**触发时机：** 补偿任务执行成功后发布。

**消费方：**
- 相关业务域（更新业务状态）——补偿完成后通知对应业务域更新状态，如重试派单成功后通知履约域

### 9.3 CriticalAlertTriggered（严重告警已触发）

**触发时机：** CRITICAL 级别的告警规则被触发时发布。

**消费方：**
- 运营治理域内部（发送通知）——驱动多渠道通知发送

## 10. 与其他域的交互

### 10.1 接收所有域的异常事件

**事件：** 各域的异常事件

**处理：** 创建异常记录，触发处理流程。运营治理域是所有异常事件的中央收集点，将分散在各域的异常汇聚为统一的治理视图。

### 10.2 向各域发送补偿命令

**命令：** 各种补偿命令

**时机：** 执行补偿任务时

**用途：** 修正业务数据或重试失败操作。运营治理域通过命令而非直接修改来驱动业务域执行补偿，保持域间解耦。

### 10.3 记录所有域的审计日志

**接口：** `RecordAuditLog`

**时机：** 各域执行关键操作时

**用途：** 记录完整的操作审计轨迹。各域通过统一接口上报审计日志，运营治理域负责存储和查询。

## 11. 数据一致性保障

### 11.1 异常记录一致性

- 异常记录一旦创建不可删除
- 异常处理过程完整记录
- 补偿任务与异常记录强关联
- 所有状态变更记录时间戳

### 11.2 审计日志一致性

- 审计日志只能追加不能修改
- 审计日志必须包含完整上下文
- 敏感数据脱敏后记录
- 日志保留期限严格执行

### 11.3 指标数据一致性

- 指标计算结果可重现
- 指标聚合逻辑统一
- 指标数据定期归档
- 历史指标不可修改

## 12. 核心设计原则总结

运营治理域是平台的治理中枢，其设计质量直接影响：

- **异常处理效率**：通过异常台账和补偿机制快速响应问题
- **系统可审计性**：通过完整的审计日志支持合规和追溯
- **运营可见性**：通过监控指标和告警及时发现问题
- **治理自动化**：通过自动化补偿减少人工介入
