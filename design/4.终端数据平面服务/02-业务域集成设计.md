## 1. 集成模式

### 1.1 两种集成模式

**V1（直接写入模式）：**
- 业务服务在处理完业务逻辑后，直接调用 TDP 写入 API
- 简单，耦合度略高，适合快速落地

**V2（Adapter 模式）：**
- 独立的 Projection Adapter 订阅 RocketMQ 领域事件，将其转化为 TDP 写入
- 业务服务无感，解耦彻底，适合稳定后演进

**本设计推荐 V1 起步，V2 目标演进。**

### 1.2 Projection Adapter 架构

```
┌─────────────────────────────────────┐
│         RocketMQ 事件总线            │
│  order-events / fulfillment-events  │
│  product-events / booking-events    │
└──────────────┬──────────────────────┘
               │ 订阅
        ┌──────▼──────┐
        │  Projection │
        │   Adapter   │
        │   Service   │
        └──────┬──────┘
               │ HTTP Write API
        ┌──────▼──────┐
        │     TDP     │
        │  L1 Service │
        └─────────────┘
```

**Projection Adapter 职责：**
1. 订阅 RocketMQ 领域事件
2. 将领域事件转化为终端可消费的 Projection Payload
3. 确定目标 Scope（orgScopeId）
4. 调用 TDP 写入 API
5. 处理写入失败重试

**Adapter 不负责：**
1. 业务逻辑处理
2. 数据权威存储
3. 终端连接管理

## 2. 商品与门店经营域集成

### 2.1 领域事件到 TDP 写入映射

| 领域事件 | 触发的 TDP 写入 |
|---------|--------------|
| `MenuPublished` | `PUT menu.catalog` @ `store` or `brand` scope |
| `ProductAvailabilityChanged` | `PUT menu.availability/{productId}` @ `store` scope |
| `StoreHoursUpdated` | `PUT store.config/hours` @ `store` scope |
| `StoreSettingsChanged` | `PUT store.config/{key}` @ `store` scope |

### 2.2 MenuPublished 处理示例

**领域事件 Payload：**
```json
{
  "eventId": "evt_menu_001",
  "eventType": "MenuPublished",
  "aggregateId": "menu_v20240405",
  "occurredAt": "2026-04-05T10:00:00Z",
  "payload": {
    "storeId": "s-001",
    "menuId": "menu_v20240405",
    "menuVersion": "v20240405",
    "effectiveDate": "2026-04-05",
    "categories": [...],
    "products": [...]
  }
}
```

**Adapter 转化逻辑：**
```java
@RocketMQMessageListener(
    topic = "product-events",
    consumerGroup = "tdp-adapter-product-group",
    selectorExpression = "MenuPublished"
)
public class MenuPublishedAdapter implements RocketMQListener<MenuPublishedEvent> {

    @Autowired
    private TdpWriteClient tdpClient;

    @Override
    public void onMessage(MenuPublishedEvent event) {
        // 1. 构造终端可消费的 Payload
        MenuCatalogPayload payload = buildMenuCatalog(event);

        // 2. 判断 Payload 大小
        if (payload.size() > 64 * 1024) {
            // 大菜单：上传到 CDN，只推送版本指针
            String downloadUrl = uploadToCdn(payload);
            payload = MenuCatalogPayload.builder()
                .version(event.getMenuVersion())
                .downloadUrl(downloadUrl)
                .checksum(calculateChecksum(payload))
                .build();
        }

        // 3. 写入 TDP
        ProjectionWriteRequest request = ProjectionWriteRequest.builder()
            .tenantId(event.getTenantId())
            .topic("menu.catalog")
            .itemKey("menu-version:" + event.getMenuVersion())
            .scopeType("store")
            .scopeId("store:" + event.getStoreId())
            .payload(payload)
            .schemaVersion("1.0")
            .sourceSystem("product-service")
            .occurredAt(event.getOccurredAt())
            .idempotencyKey(event.getEventId())
            .build();

        tdpClient.writeProjection(request);
    }
}
```

## 3. 交易域集成

### 3.1 领域事件到 TDP 写入映射

| 领域事件 | 触发的 TDP 写入 |
|---------|--------------|
| `OrderPaid` | `PUT order.pending/{orderId}` @ `store` scope（status=PAID） |
| `OrderAccepted` | `PUT order.pending/{orderId}` @ `store` scope（status=ACCEPTED） |
| `OrderStatusChanged` | `PUT order.pending/{orderId}` @ `store` scope（更新 status） |
| `OrderCompleted` | `DELETE order.pending/{orderId}` @ `store` scope（tombstone） |
| `OrderCancelled` | `DELETE order.pending/{orderId}` @ `store` scope（tombstone） |
| `OrderUrged` | `PUT order.urge/{orderId}:{timestamp}` @ `store` scope（ephemeral） |
| `RefundStatusChanged` | `PUT order.pending/{orderId}` @ `store` scope（更新 refundStatus） |

### 3.2 OrderPaid 处理示例

**领域事件 Payload：**
```json
{
  "eventId": "evt_order_001",
  "eventType": "OrderPaid",
  "aggregateId": "order_20240405001",
  "occurredAt": "2026-04-05T10:30:00Z",
  "payload": {
    "orderId": "order_20240405001",
    "orderNo": "#20240405001",
    "storeId": "s-001",
    "scene": "DINE_IN",
    "items": [...],
    "totalAmount": 5800,
    "payableAmount": 5800,
    "status": "PAID"
  }
}
```

**Adapter 转化逻辑：**
```java
@RocketMQMessageListener(
    topic = "order-events",
    consumerGroup = "tdp-adapter-order-group",
    selectorExpression = "OrderPaid"
)
public class OrderPaidAdapter implements RocketMQListener<OrderPaidEvent> {

    @Autowired
    private TdpWriteClient tdpClient;

    @Override
    public void onMessage(OrderPaidEvent event) {
        // 构造 POS 需要的订单摘要
        OrderPendingPayload payload = OrderPendingPayload.builder()
            .orderId(event.getOrderId())
            .orderNo(event.getOrderNo())
            .scene(event.getScene())
            .items(simplifyItems(event.getItems()))  // 简化 items，只保留 POS 需要的字段
            .amount(event.getPayableAmount())
            .status("PAID")
            .createdAt(event.getOccurredAt())
            .build();

        ProjectionWriteRequest request = ProjectionWriteRequest.builder()
            .tenantId(event.getTenantId())
            .topic("order.pending")
            .itemKey("order:" + event.getOrderId())
            .scopeType("store")
            .scopeId("store:" + event.getStoreId())
            .payload(payload)
            .schemaVersion("1.0")
            .sourceSystem("trading-service")
            .occurredAt(event.getOccurredAt())
            .idempotencyKey(event.getEventId())
            .build();

        tdpClient.writeProjection(request);
    }
}
```

### 3.3 OrderCompleted / OrderCancelled 处理

订单完成或取消后，应从 POS 的待处理列表中移除：

```java
ProjectionDeleteRequest request = ProjectionDeleteRequest.builder()
    .tenantId(event.getTenantId())
    .topic("order.pending")
    .itemKey("order:" + event.getOrderId())
    .scopeType("store")
    .scopeId("store:" + event.getStoreId())
    .idempotencyKey(event.getEventId())
    .build();

tdpClient.deleteProjection(request);
```

TDP 会将该 Projection 标记为 `tombstone=true`，并推送 `operation=delete` 消息给终端。

### 3.4 OrderUrged 处理（耐久事件）

催单是典型的"不能只看最终态"的事件，每次催单都是独立动作：

```java
ProjectionWriteRequest request = ProjectionWriteRequest.builder()
    .tenantId(event.getTenantId())
    .topic("order.urge")
    .itemKey("urge:" + event.getOrderId() + ":" + System.currentTimeMillis())
    .scopeType("store")
    .scopeId("store:" + event.getStoreId())
    .payload(Map.of(
        "orderId", event.getOrderId(),
        "orderNo", event.getOrderNo(),
        "urgeCount", event.getUrgeCount()
    ))
    .schemaVersion("1.0")
    .sourceSystem("trading-service")
    .occurredAt(event.getOccurredAt())
    .idempotencyKey(event.getEventId())
    .expiresAt(event.getOccurredAt().plusMinutes(30))  // 30 分钟后过期
    .build();

tdpClient.writeProjection(request);
```

## 4. 履约与生产域集成

### 4.1 领域事件到 TDP 写入映射

| 领域事件 | 触发的 TDP 写入 |
|---------|--------------|
| `WorkUnitCreated` | `PUT workunit.active/{workUnitId}` @ `store` scope |
| `WorkUnitStatusChanged` | `PUT workunit.active/{workUnitId}` @ `store` scope（更新 status） |

### 4.2 WorkUnitCreated 处理

**领域事件 Payload：**
```json
{

  "eventType": "WorkUnitCreated",

  "occurredAt": "2026-04-05T10:35:00Z",

    "workUnitId": "wu_001",

    "orderNo": "#20240405001",
    "storeId": "s-001",
    "stationId": "hot-dish",
    "items": [...],
    "status": "PENDING",
    "priority": 1
  }
}
```

**TDP 写入：**
```java
ProjectionWriteRequest request = ProjectionWriteRequest.builder()
    .tenantId(event.getTenantId())
    .topic("workunit.active")
    .itemKey("workunit:" + event.getWorkUnitId())
    .scopeType("store")
    .scopeId("store:" + event.getStoreId())
    .payload(event.getPayload())
    .schemaVersion("1.0")
    .sourceSystem("fulfillment-service")
    .occurredAt(event.getOccurredAt())
    .idempotencyKey(event.getEventId())

```

**终端侧过滤：**
KDS 终端订阅 `workunit.active`，但只处理 `stationId` 匹配自身的工单。这个过滤逻辑在终端的 Topic Handler 中完成，而非在 TDP 层面。

### 4.3 WorkUnitRecalled 处理（双写）

工单撤销需要双写：
1. 将 `workunit.active` 标记为 tombstone（从活跃列表移除）
2. 写入 `workunit.recalled` 耐久事件（触发终端撤销提示动作）

```java
// 1. 删除活跃工单
ProjectionDeleteRequest deleteReq = ProjectionDeleteRequest.builder()
    .tenantId(event.getTenantId())
    .topic("workunit.active")
    .itemKey("workunit:" + event.getWorkUnitId())
    .scopeType("store")
    .scopeId("store:" + event.getStoreId())
    .idempotencyKey(event.getEventId() + "-delete")
    .build();

tdpClient.deleteProjection(deleteReq);

// 2. 写入撤销事件
ProjectionWriteRequest recallReq = ProjectionWriteRequest.builder()
    .tenantId(event.getTenantId())
    .topic("workunit.recalled")
    .itemKey("recall:" + event.getWorkUnitId() + ":" + System.currentTimeMillis())
    .scopeType("store")
    .scopeId("store:" + event.getStoreId())
    .payload(Map.of(
        "workUnitId", event.getWorkUnitId(),
        "orderId", event.getOrderId(),
        "reason", event.getReason()
    ))
    .schemaVersion("1.0")
    .sourceSystem("fulfillment-service")
    .occurredAt(event.getOccurredAt())
    .idempotencyKey(event.getEventId() + "-recall")
    .expiresAt(event.getOccurredAt().plusMinutes(30))
    .build();

tdpClient.writeProjection(recallReq);
```

## 9. 关键业务场景链路

### 9.1 新订单到达（顾客小程序 → POS 接单）

```
1. 顾客小程序提交订单 + 支付
2. 交易域完成支付确认，发布 OrderPaid 领域事件
3. Projection Adapter 订阅到 OrderPaid
4. Adapter 构造订单摘要 Payload，调用 TDP 写入 API：
   PUT /v1/projections/items
   { topic: "order.pending", itemKey: "order:xxx", scopeType: "store", scopeId: "s-001", payload: {...} }
5. TDP L1 分配 Revision，写入 Projection Store，记录 Change Log
6. TDP 向该 store 的在线终端推送：
   { type: "PROJECTION_CHANGED", topic: "order.pending", itemKey: "order:xxx", operation: "upsert", revision: 10042, payload: {...} }
7. POS 收到推送，Topic Handler Registry 匹配 "order.pending" 处理器
8. Handler 从本地 Projection Store 更新该订单记录
9. Business Command Dispatcher 派发 newOrderReceived 命令
10. POS 订单列表 UI 更新，提示音响起
```

### 9.2 菜单发布（后台 → 所有终端刷新）

```
1. 店长在商户后台提交菜单发布
2. 商品域处理完成，发布 MenuPublished 领域事件
3. Adapter 构造菜单版本 Payload（若超 64KB 则只写版本指针）
4. TDP 写入 menu.catalog @ brand scope（连锁）或 store scope（单店）
5. TDP 向该 scope 下所有在线终端推送
6. 离线终端重连后，在 WS 握手时 lastCursor 落后于 highWatermark
7. 终端发起增量追平（HTTP），获取 menu.catalog 变更
8. 如果是版本指针，终端通过 downloadUrl 拉取完整菜单数据
9. Handler 触发 invalidateMenuCache 命令
```

### 9.3 工单分发（交易域 → KDS）

```
1. 门店 POS 接单（OrderAccepted）
2. 履约域订阅 OrderAccepted，创建履约订单和工单
3. 发布 WorkUnitCreated 领域事件（含 stationId）
4. Adapter 写入 TDP：workunit.active @ store scope
5. TDP 推送到该 store 下订阅 workunit.active 的 KDS 终端
6. KDS 的 workunit.active Handler 接收，只处理 stationId 匹配自身的工单
7. KDS 展示新工单，厨师确认开始制作
```

### 9.4 终端离线重连追平

```
1. KDS 网络中断 15 分钟
2. 期间后台有 6 个工单创建、3 个工单完成
3. KDS 重连，WS 握手携带 lastCursor=10020
4. 服务端返回 sessionReady + highWatermark=10047 + syncMode=incremental
5. KDS 发起 POST /v1/sync/changes { cursor: 10020, topics: ["workunit.active"] }
6. 服务端返回 9 条 ProjectionChangeSet（6 创建 + 3 完成）
7. KDS 按 Revision 顺序应用，tombstone 的工单从活跃列表移除
8. 同步完成，进入实时推送态
```

## 10. 写入幂等保证

### 10.1 幂等键规范

TDP 写入必须携带 `idempotencyKey`，推荐格式：

```
{eventId}          # 来自领域事件的 event_id，天然幂等
{eventId}-{seq}    # 一个事件产生多条写入时追加序号
```

### 10.2 幂等实现

TDP 在 Projection Store 中对 `(tenantId, topic, itemKey, scopeType, scopeId, idempotencyKey)` 建立唯一索引。

重复写入时：
- 如果 `idempotencyKey` 相同且 Payload 相同 → 返回成功，不分配新 Revision
- 如果 `idempotencyKey` 相同但 Payload 不同 → 返回冲突错误

### 10.3 Adapter 重试策略

Adapter 写入失败时，应按以下策略重试：

```java
@Retryable(
    value = {TdpWriteException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 1000, multiplier = 2)
)
public void writeToTdp(ProjectionWriteRequest request) {
    tdpClient.writeProjection(request);
}
```

如果 3 次重试仍失败，应：
1. 记录到异常台账
2. 触发告警
3. 由运营治理域定期重试
