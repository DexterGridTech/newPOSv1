# 终端侧集成指南

> 版本：1.0 | 状态：正式稿 | 日期：2026-04-05

---

## 1. 终端侧定位

### 1.1 终端在 TDP 体系中的角色

终端在终端数据平面（TDP）体系中的定位不是"某个业务服务的被动推送客户端"，而是"终端数据平面的标准消费节点"。

**终端内核负责：**
1. 消费后端数据平面定义的统一协议
2. 维护本地可恢复的保留态投影
3. 将投影解释成业务命令
4. 屏蔽终端业务模块对传输协议、同步机制、连接切换的感知

**终端业务模块负责：**
1. 接收稳定的业务命令或稳定业务态
2. 处理自身业务运行逻辑与 UI 逻辑
3. 不直接依赖 WS 消息格式、同步游标、连接迁移协议

---

## 2. 终端侧总体架构

### 2.1 五层架构

```
┌─────────────────────────────────────────────┐
│  Terminal Session Layer                     │
│  连接 / 会话 / 迁移 / 降级                   │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  Sync Engine                                │
│  增量追平 / 全量对账 / 推送接入              │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  Local Projection Store                     │
│  Topic/Item/Revision 持久化                 │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  Topic Handler Registry                     │
│  解释 Topic 并生成业务命令                  │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  Business Command Dispatcher                │
│  分发到各业务模块                            │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│  POS / KDS / PDA 业务模块                   │
└─────────────────────────────────────────────┘
```

**层次说明：**
1. **Terminal Session Layer**：连接、会话、迁移、降级
2. **Sync Engine**：增量追平、全量对账、推送接入
3. **Local Projection Store**：本地统一投影存储
4. **Topic Handler Registry**：统一 Topic 解释与投影转命令
5. **Business Command Dispatcher**：将解释结果投递给业务模块

---

## 3. 核心设计原则

### 3.1 先落统一数据层，再转业务态

终端收到后端变更后，第一步不是直接更新订单、配置、任务等业务状态，而是先写入本地投影层。

**价值：**
1. 可持久化
2. 可回放
3. 可重建
4. 可审计
5. 可断点恢复

### 3.2 业务模块不直接消费传输协议

业务模块不应直接知道：
1. WebSocket 消息结构
2. Topic Revision
3. Cursor
4. L2 降级与迁移细节

这些都应由统一终端内核屏蔽。

### 3.3 传输入口统一，状态更新统一

无论是：
1. WebSocket 推送
2. HTTP 增量追平
3. HTTP 全量对账

最终都要产出统一的 `ProjectionChangeSet`，写入同一个本地 Projection Store。

### 3.4 事实层与运行层分离

终端本地状态应分成：
1. **Projection State**：收到的终端事实层
2. **Business Runtime State**：业务模块计算后的运行层

运行层可以被重新构建，事实层必须可恢复。

### 3.5 统一内核，差异化装配

POS、KDS、PDA 不应各自维护一套不同连接和同步内核，而应共享统一终端数据平面内核。差异体现在：
1. Topic 白名单不同
2. 能力声明不同
3. Topic Handler 装配不同
4. 业务模块装配不同

## 4. 核心对象模型

| 对象 | 含义 | 说明 |
|------|------|------|
| `SessionState` | 会话状态 | 终端当前连接的节点、连接态、迁移态 |
| `SyncState` | 同步状态 | cursor、highWatermark、lastSyncAt、digest 等 |
| `ProjectionRecord` | 投影记录 | 本地持久化的一条 Topic/Item 当前状态 |
| `ProjectionChangeSet` | 统一变更集 | 来自 WS 或 HTTP 的标准变更集 |
| `TopicHandler` | Topic 处理器 | 负责将投影转换成业务命令 |
| `BusinessCommand` | 业务命令 | 投递到业务模块的标准命令 |
| `TerminalCapability` | 终端能力声明 | 终端类型、角色、支持的 Topic 集合 |
| `ConnectionRoute` | 候选接入点 | 当前可连接的 L2/L1 地址列表 |

## 5. 终端本地状态分层

### 5.1 Session State（会话状态）

用于描述会话与连接状态，建议包含：

```typescript
interface SessionState {
  connectionState: 'disconnected' | 'connecting' | 'connected' | 'reconnecting' | 'switching';
  sessionMode: 'healthy' | 'degraded' | 'rehome-recommended' | 'rehome-required' | 'isolated-readonly';
  connectedNodeId: string;
  connectedRegion: string;
  connectedEndpoint: string;
  connectedAt: number;
  lastHeartbeatAt: number;
  lastDisconnectReason?: string;
  lastMigrationReason?: string;
}
```

**说明：**
- 默认不做长期持久化，或只保留少量诊断信息

---

### 5.2 Sync State（同步状态）

用于描述终端同步进度，建议持久化：

```typescript
interface SyncState {
  lastCursor: number;
  highWatermark: number;
  lastSyncAt: number;
  lastReconcileAt: number;
  perTopicDigest: Record<string, string>;
  lastAppliedRevision: number;
  lastReceivedRevision: number;
}
```

---

### 5.3 Projection State（投影状态）

这是终端最核心的持久化层，必须默认持久化：

```typescript
interface ProjectionRecord {
  topic: string;
  itemKey: string;
  scopeType: string;
  scopeId: string;
  payload: any;
  revision: number;
  updatedAt: number;
  tombstone: boolean;
  schemaVersion: string;
  sourceNodeId?: string;
}
```

**存储结构建议：**
```typescript
// 按 Topic 分组存储
interface ProjectionStore {
  [topic: string]: {
    [itemKey: string]: ProjectionRecord;
  };
}
```

---

### 5.4 Business Runtime State（业务运行态）

由各业务模块维护，例如：
1. 订单展示态
2. 任务执行态
3. UI 界面态
4. 本地临时交互态

这层可以是持久化的，也可以是临时态，但不应承担"终端数据事实层"的职责。

### 6.4 多地址接入策略

**优先级顺序：**
1. 优先连接本地 L2
2. 本地 L2 不可达时切换其他 L2
3. 所有 L2 不可达时回退 L1
4. 接收到服务端下发的重定向地址时，动态更新地址优先级

**实现示例：**
```typescript
class ConnectionManager {
  private endpoints: string[] = [
    'wss://l2-beijing-01.example.com/tdp/ws/connect',
    'wss://l2-beijing-02.example.com/tdp/ws/connect',
    'wss://l1-central.example.com/tdp/ws/connect'
  ];

  async connect(): Promise<WebSocket> {
    for (const endpoint of this.endpoints) {
      try {
        const ws = await this.tryConnect(endpoint);
        return ws;
      } catch (error) {
        console.warn(`Failed to connect to ${endpoint}, trying next...`);
      }
    }
    throw new Error('All endpoints failed');
  }
}
```

## 7. 终端迁移与降级逻辑

### 7.1 系统控制消息

#### EDGE_DEGRADED（边缘节点降级）

```typescript
interface EdgeDegradedMessage {
  type: 'EDGE_DEGRADED';
  data: {
    reason: 'upstream_disconnected' | 'high_load' | 'maintenance_mode';
    issuedAt: string;
    nodeState: 'grace' | 'degraded';
    gracePeriodSeconds: number;
    alternativeEndpoints: string[];
  };
}
```

**终端处理：**
1. 更新 `sessionMode = 'degraded'`
2. 标识当前连接不再保证实时新鲜度
3. 暂不主动中断当前连接
4. 记录候选接入点

---

#### SESSION_REHOME_REQUIRED（会话迁移要求）

```typescript
interface SessionRehomeRequiredMessage {
  type: 'SESSION_REHOME_REQUIRED';
  data: {
    reason: 'node_draining' | 'load_balancing';
    deadline: string;
    alternativeEndpoints: string[];
  };
}
```

**终端处理：**
1. 更新 `sessionMode = 'rehome-required'`
2. 必须在 `deadline` 前完成迁移
3. 旧连接不再允许无限期保留

---

### 7.2 平滑切主流程

**推荐流程：**
1. 当前旧连接保持可用
2. 新连接后台建立
3. 新连接完成握手
4. 新连接完成追平
5. 终端把新连接标记为 active session
6. 旧连接主动关闭

**实现示例：**
```typescript
class SessionMigrationManager {
  async migrateSession(newEndpoint: string): Promise<void> {
    // 1. 建立新连接
    const newWs = await this.connectToEndpoint(newEndpoint);

    // 2. 握手
    await this.handshake(newWs);

    // 3. 增量追平
    await this.syncEngine.incrementalSync(newWs);

    // 4. 切换主连接
    this.activeSession = newWs;

    // 5. 关闭旧连接
    this.oldSession?.close();
  }
}
```

## 8. 同步与追平层

### 8.1 终端启动同步流程

```
1. 加载本地持久化 Projection State
2. 加载本地持久化 Sync State
3. 基于本地 Projection 重建关键业务运行态
4. 建立 WS 会话
5. 根据 lastCursor 执行增量追平
6. 若追平失败或游标过旧，则执行全量对账
7. 应用追平结果
8. 进入实时推送态
```

---

### 8.2 增量追平

**请求：**
```typescript
interface IncrementalSyncRequest {
  cursor: number;
  topics: string[];
  limit: number;
}
```

**响应：**
```typescript
interface IncrementalSyncResponse {
  changes: ProjectionChange[];
  nextCursor: number;
  hasMore: boolean;
  highWatermark: number;
}

interface ProjectionChange {
  revision: number;
  topic: string;
  itemKey: string;
  operation: 'upsert' | 'delete';
  scopeType: string;
  scopeId: string;
  payload?: any;
  occurredAt: string;
}
```

**实现示例：**
```typescript
class SyncEngine {
  async incrementalSync(): Promise<void> {
    const syncState = await this.loadSyncState();
    let cursor = syncState.lastCursor;
    let hasMore = true;

    while (hasMore) {
      const response = await this.httpClient.post('/v1/sync/changes', {
        cursor,
        topics: this.subscribedTopics,
        limit: 100
      });

      // 应用变更
      await this.applyChanges(response.changes);

      cursor = response.nextCursor;
      hasMore = response.hasMore;
    }

    // 更新同步状态
    await this.updateSyncState({
      lastCursor: cursor,
      highWatermark: response.highWatermark,
      lastSyncAt: Date.now()
    });
  }
}
```

---

### 8.3 全量对账

**触发条件：**
1. 本地 `cursor` 无效
2. 服务端返回变更日志已裁剪
3. 本地投影疑似损坏
4. Topic Digest 校验失败

**请求：**
```typescript
interface ReconcileRequest {
  topics: string[];
  localDigest: Record<string, string>;
}
```

**响应：**
```typescript
interface ReconcileResponse {
  fullSnapshot: ProjectionRecord[];
  highWatermark: number;
}
```

**实现示例：**
```typescript
class SyncEngine {
  async fullReconcile(): Promise<void> {
    const response = await this.httpClient.post('/v1/sync/reconcile', {
      topics: this.subscribedTopics,
      localDigest: await this.calculateDigest()
    });

    // 清空本地对应 Topic 的数据
    await this.projectionStore.clearTopics(this.subscribedTopics);

    // 重新写入
    await this.projectionStore.batchUpsert(response.fullSnapshot);

    // 更新同步状态
    await this.updateSyncState({
      lastCursor: response.highWatermark,
      highWatermark: response.highWatermark,
      lastReconcileAt: Date.now()
    });
  }
}
```

### 8.4 实时推送处理

**推送消息：**
```typescript
interface ProjectionChangedMessage {
  type: 'PROJECTION_CHANGED';
  eventId: string;
  timestamp: number;
  data: {
    topic: string;
    itemKey: string;
    operation: 'upsert' | 'delete';
    scopeType: string;
    scopeId: string;
    revision: number;
    payload?: any;
    occurredAt: string;
    sourceSystem: string;
  };
}
```

**处理流程：**
```typescript
class MessageHandler {
  async handleProjectionChanged(message: ProjectionChangedMessage): Promise<void> {
    // 1. 转换为 ProjectionChangeSet
    const changeSet = this.convertToChangeSet(message);

    // 2. 写入 Projection Store
    await this.projectionStore.apply(changeSet);

    // 3. 触发 Topic Handler
    await this.topicHandlerRegistry.handle(changeSet.topic);

    // 4. 更新同步状态
    await this.updateSyncState({
      lastAppliedRevision: message.data.revision
    });
  }
}
```

## 9. Local Projection Store

### 9.1 存储接口

```typescript
  clearTopics(topics: string[]): Promise<void>;

  // 工具
  calculateDigest(topic: string): Promise<string>;
}
```

---

### 9.2 Scope 解析规则

当同一 Topic/Item 在多个 Scope 上都有 Projection 时，终端按以下规则确定最终生效值：

**优先级规则：**
1. `TerminalScope > ProfileScope > OrgScope`
2. 同类型中：`更深层级（叶子）> 更浅层级（根）`
3. 同节点内：`Revision 新者 > 旧者`

**实现示例：**
```typescript
class ProjectionResolver {
  resolve(topic: string, itemKey: string): any {
    const records = this.projectionStore.getAll(topic)
      .filter(r => r.itemKey === itemKey && !r.tombstone);

    // 按优先级排序
    records.sort((a, b) => {
      // 1. Scope 类型优先级
      const scopePriority = this.getScopePriority(a.scopeType) - this.getScopePriority(b.scopeType);
      if (scopePriority !== 0) return scopePriority;

      // 2. 同类型中深度优先
      const depthPriority = this.getScopeDepth(b.scopeId) - this.getScopeDepth(a.scopeId);
      if (depthPriority !== 0) return depthPriority;

      // 3. Revision 新者优先
      return b.revision - a.revision;
    });

    return records[0]?.payload;
  }

  private getScopePriority(scopeType: string): number {
    const priority = { terminal: 3, profile: 2, org: 1 };
    return priority[scopeType] || 0;
  }
}
```

## 10. Topic Handler Registry

### 10.1 Handler 接口

```typescript
interface TopicHandler {
  topic: string;
  handle(projectionStore: ProjectionStore): Promise<BusinessCommand[]>;
}
```

---

### 10.2 Handler 注册

```typescript
class TopicHandlerRegistry {
  private handlers = new Map<string, TopicHandler>();

  register(handler: TopicHandler): void {
    this.handlers.set(handler.topic, handler);
  }

  async handle(topic: string): Promise<void> {
    const handler = this.handlers.get(topic);
    if (!handler) return;

    const commands = await handler.handle(this.projectionStore);
    await this.commandDispatcher.dispatch(commands);
  }
}
```

---

### 10.3 Handler 示例

#### menu.catalog Handler

```typescript
class MenuCatalogHandler implements TopicHandler {
  topic = 'menu.catalog';

  async handle(projectionStore: ProjectionStore): Promise<BusinessCommand[]> {
    const records = await projectionStore.getAll(this.topic);
    const latestMenu = records[0]; // 假设只有一条

    if (!latestMenu) return [];

    // 如果是版本指针，需要下载完整菜单
    if (latestMenu.payload.downloadUrl) {
      const fullMenu = await this.downloadMenu(latestMenu.payload.downloadUrl);
      return [{ type: 'invalidateMenuCache', data: fullMenu }];
    }

    return [{ type: 'invalidateMenuCache', data: latestMenu.payload }];
  }
}
```

#### order.pending Handler

```typescript
class OrderPendingHandler implements TopicHandler {
  topic = 'order.pending';

  async handle(projectionStore: ProjectionStore): Promise<BusinessCommand[]> {
    const records = await projectionStore.getAll(this.topic);
    const activeOrders = records.filter(r => !r.tombstone);

    return [{ type: 'updateOrderList', data: activeOrders.map(r => r.payload) }];
  }
}
```

## 12. 异常与边界场景

### 12.1 本地有快照但无法连网

终端可基于本地 Projection 启动为有限可用模式，但必须向上层明确标识当前不是实时态。

```typescript
if (!this.isConnected && this.hasLocalSnapshot) {
  this.sessionMode = 'isolated-readonly';
  this.showOfflineIndicator();
}
```

---

### 12.2 收到旧 Revision

应直接忽略，不得回滚本地 Projection。

```typescript
if (incomingRevision <= currentRevision) {
  console.warn('Ignoring old revision', incomingRevision);
  return;
}
```

---

### 12.3 Topic Handler 失败

Projection 应先成功落盘，再记录 Handler 失败。不能因为某个 Topic Handler 失败而丢失底层事实层数据。

```typescript
try {
  await this.projectionStore.upsert(record);
} catch (error) {
  console.error('Failed to persist projection', error);
  throw error;
}

try {
  await this.topicHandlerRegistry.handle(record.topic);
} catch (error) {
  console.error('Handler failed', error);
  // 记录失败，但不影响 Projection 持久化
}
```

---

### 12.4 业务模块状态损坏

可通过重新执行 Topic Handler，从 Projection Store 重建业务状态，而不依赖重新拉取所有业务服务数据。

```typescript
async rebuildBusinessState(): Promise<void> {
  const topics = this.subscribedTopics;
  for (const topic of topics) {
    await this.topicHandlerRegistry.handle(topic);
  }
}
```
