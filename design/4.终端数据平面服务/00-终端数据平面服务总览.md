# 终端数据平面服务总览

> 版本：1.0 | 状态：正式稿 | 日期：2026-04-05

---

## 1. 服务定位与战略价值

### 1.1 为什么需要终端数据平面服务

购物中心餐饮中台面临的终端数据分发挑战：

**挑战一：终端类型多样，数据需求各异**
- POS 需要订单列表、支付状态、打印任务
- KDS 需要工单队列、催单提醒、沽清状态
- PDA 需要桌台状态、预订列表、上菜确认
- 顾客小程序需要菜单、订单跟踪、优惠券
- 叫号屏需要叫号信息、广告轮播

**挑战二：业务系统众多，各自推送复杂度高**
- 交易域、履约域、商品域、预订域、渠道域各自产生需要推送的数据
- 如果每个业务系统各自维护终端连接、各自定义推送协议，会导致：
  - 后台复杂度失控（每个系统都要处理连接管理、断线重连、幂等去重）
  - 终端复杂度失控（要对接多个 WebSocket、多套协议、多套数据模型）
  - 职责边界混乱（业务系统不该承担基础设施责任）

**挑战三：全国分布，网络质量差异大**
- 购物中心遍布全国，不同地区网络质量不稳定
- 必须支持多节点、多地域接入和自动切换
- 离线恢复困难，容易错过关键状态

**挑战四：现有推送体系的局限**
- 当前 WebSocket 推送是门店级简单推送，缺乏：
  - 统一的 Revision 机制（只有 timestamp）
  - 可靠的断线补偿（`/events/missed` 不够健壮）
  - 边缘接入能力（无 L2 节点）
  - 平滑迁移机制（只能硬断重连）

### 1.2 终端数据平面服务的定位

**终端数据平面服务（Terminal Data Plane，简称 TDP）** 是购物中心餐饮中台的**终端被动数据分发基础设施**，处于业务系统与终端之间，承担如下职责：

```
业务系统（交易/履约/商品/预订/渠道）
        │ 产出领域事件
        ▼
  Projection Adapter
        │ 转化为终端可消费的投影数据
        ▼
终端数据平面服务（TDP）
        │ 统一存储、版本管理、分发推送
        ▼
终端（POS/KDS/PDA/KIOSK/小程序/叫号屏）
```

**核心职责：**
1. 接收业务系统提交的终端可见数据变更
2. 把变更整理为终端投影数据并持久化
3. 根据终端所属组织、机型、实例等作用域决定哪些终端应感知该变更
4. 向在线终端推送变更
5. 为离线终端提供重连后的追平能力
6. 在中心层与边缘层之间同步投影数据

**明确不做的事情：**
1. 不执行终端主动业务请求（终端主动操作仍通过 HTTP 调用具体业务服务）
2. 不替代业务系统的权威数据库
3. 不充当统一业务查询网关
4. 不为所有业务系统做复杂的领域翻译或聚合编排

### 1.3 战略价值

**对平台运营方：**
- 统一终端数据分发基础设施，降低整体系统复杂度
- 支持全国多购物中心、多云、多地域的边缘接入
- 提供可观测、可追溯的终端数据推送链路
- 为未来新业务域、新终端类型提供标准化接入方式

**对业务系统：**
- 无需关心终端连接管理、断线重连、幂等去重等基础设施问题
- 只需关注"哪些数据需要推送给终端"，而非"如何推送"
- 通过 Projection Adapter 解耦，业务系统不直接依赖 TDP

**对终端：**
- 统一的连接入口、统一的协议、统一的数据模型
- 可靠的断线补偿机制（Cursor 增量追平 + 全量对账）
- 就近接入（L2 边缘节点）+ 自动降级与迁移
- 本地 Projection Store 支持离线可用

**对研发团队：**
- 清晰的职责边界：业务系统产出数据，TDP 负责分发
- 标准化的 Topic 治理体系，避免消息类型失控
- 可复用的终端内核（Session / Sync / Projection / Handler 分层）

## 2. 核心设计理念

### 2.1 主动与被动分离

- **主动业务动作**：终端调用具体业务系统 HTTP 接口并获得应答（如下单、支付、接单）
- **被动状态感知**：后台把变更交给 TDP 统一分发（如新订单到达、菜单更新、工单推送）

这个分工边界是 TDP 设计的基石。

### 2.2 分发的是数据，不只是通知

当后台数据发生变化且终端需要立刻知道时，业务系统应提交**终端可直接消费的数据**，而不是只发"请你去查一下"的空通知。

这样终端处理链路更短，也更利于离线补偿。

### 2.3 保留态优先

系统保留当前有效数据投影（Projection Store）。终端在线时立即收到推送，离线时下次连接可继续对齐最新状态。

保留态是 TDP 的核心，不依赖终端在线才能成立。

### 2.4 中心权威，边缘分发

- **L1 中心层**：负责权威写入、版本生成、投影权威存储
- **L2 边缘层**：负责就近接入、边缘推送、断线容忍、区域缓存

不允许业务系统同时向多个层级直接写入同一类业务数据，避免多主冲突。

### 2.5 业务系统产出终端就绪数据

TDP 不应成长为"大型业务翻译中心"。业务系统或其旁路 Projection Adapter 应负责把数据整理成终端可消费结构。

TDP 只负责校验、存储、分发、同步。

### 2.6 协议先行，Topic 治理先行

要让多个业务系统长期稳定接入，必须把 Topic、Schema、作用域、权限、保留策略、大小限制、版本兼容规则前置治理。

不能任由接入方随意写 JSON。

## 3. 总体架构

### 3.1 服务在中台中的位置

```
┌─────────────────────────────────────────────────────────┐
│                    餐饮中台业务层                         │
│  交易域   履约域   商品域   渠道域   预订域   组织域     │
│     │       │       │       │       │       │            │
│     └───────┴───────┴───────┴───────┴───────┘            │
│                       │（领域事件 via RocketMQ）          │
│              ┌────────▼────────┐                          │
│              │  Projection     │                          │
│              │  Adapter Layer  │  ← 事件转投影的桥接层   │
│              └────────┬────────┘                          │
└───────────────────────┼─────────────────────────────────┘
                        │（HTTP Write API）
          ┌─────────────▼──────────────┐
          │   L1 终端数据平面服务       │
          │  权威写入 + 投影存储 +      │
          │  版本分配 + 联邦网关        │
          └─────────────┬──────────────┘
                        │（Federation Sync）
           ┌────────────┴────────────┐
    ┌──────▼──────┐          ┌───────▼──────┐
    │  L2 北京商场 │          │  L2 上海商场  │
    │  边缘节点   │          │  边缘节点    │
    └──────┬──────┘          └───────┬──────┘
           │ WebSocket                │ WebSocket
    ┌──────┴──────────┐       ┌───────┴────────────┐
    │ POS KDS PDA ... │       │ POS KDS PDA ...     │
    └─────────────────┘       └────────────────────┘
```

### 3.2 四层架构

**Ingress Layer（接入层）**
- 接收业务系统写入，支持直调 API 和事件订阅接入
- 校验 Topic 合约、权限、幂等键

**Projection Layer（投影层）**
- 校验 Topic Schema、生成投影、分配 Revision、写入保留态
- 追加 Change Log

**Distribution Layer（分发层）**
- 终端会话管理、在线推送、离线追平
- Session Presence Store

**Federation Layer（联邦层）**
- L1 到 L2 的投影同步、按需副本、边缘缓存与容灾切换

### 3.3 核心对象模型

| 对象 | 含义 | 与 POC 对应 |
|------|------|------------|
| `OrgScope` | 组织作用域树节点（平台/商场/租户/品牌/门店） | entity unit |
| `ProfileScope` | 终端画像树节点（POS/KDS/PDA/型号/角色） | model unit |
| `TerminalScope` | 具体终端实例 | terminal unit |
| `Topic` | 注册在册的数据分发主题 | group |
| `Item` | Topic 内最小数据单元 | path |
| `Projection` | 某 Scope 下某 Topic/Item 的当前有效值 | unit_data |
| `Revision` | 全局单调递增版本号（L1 分配） | updatedAt 的升级 |
| `Cursor` | 终端或 L2 已消费到的版本位置 | — |
| `Session` | 终端当前连接会话 | — |
| `Command` | 点对点命令型消息（不进 Projection Store） | command |

## 4. 与现有推送体系的关系

### 4.1 现有设计回顾

餐饮中台现有的 WebSocket 推送（`3.前台服务设计/14-统一消息通知与实时同步.md`）是一个**门店级简单推送**体系：

- 连接 URL 绑定 `storeId + deviceType`
- `INIT_SNAPSHOT` 只在连接时推送一次当前状态
- 事件类型与业务逻辑强耦合（24+ 消息类型）
- 断线恢复靠 `/api/v1/events/missed?since=timestamp`，无 Revision 机制
- 无 L2 边缘节点，无联邦架构

### 4.2 TDP 的定位

TDP 的引入**不是推翻现有设计，而是在下方建立更可靠的基础**：

| 现有设计 | TDP 对应能力 | 关系 |
|--------|------------|------|
| INIT_SNAPSHOT | Projection Store + 增量追平 | TDP 替代，语义增强 |
| eventId 去重 Set | Revision + 幂等写入 | TDP 替代，机制更健壮 |
| `events/missed?since=timestamp` | Cursor 增量追平 + 全量对账 | TDP 替代，更可靠 |
| 各业务域直接 push | Projection Adapter 写入 | TDP 封装 |
| 简单指数退避重连 | Session 状态机 + 平滑迁移 | TDP 增强 |
| 无 L2 / 无联邦 | L1 + L2 联邦架构 | TDP 新增 |

### 4.3 迁移路径

**Phase 1：TDP 基础服务上线（不影响现有推送）**
- 部署 TDP L1 服务，创建 Topic Registry
- 开发 Projection Adapter，订阅 RocketMQ 领域事件
- Adapter 开始写入 TDP，但不推送终端（只写 Projection Store）
- 验证数据写入正确性

**Phase 2：灰度切换单一终端类型**
- 选择 KDS 作为首批迁移终端（数量少、逻辑简单）
- KDS 终端升级，连接 TDP WebSocket 端点
- 关闭 KDS 的旧 WebSocket 连接
- 观察 1 周，验证稳定性

**Phase 3：逐步迁移其他终端**
- POS → PDA → KIOSK → 打包台 → 叫号屏
- 每类终端迁移间隔 2 周
- 旧 WebSocket 服务保留 3 个月后下线

**Phase 4：L2 边缘节点部署**
- 在 3-5 个商场部署 L2 节点
- 配置 L1 → L2 联邦同步
- 终端连接地址切换到 L2
- 验证降级与迁移机制

## 5. 关键设计决策

### D1：Topic 语义分三类

```
retained_state   保留态数据（菜单、配置、订单列表、桌台状态）
durable_event    耐久事件（不能只看最终态，需要逐条处理的场景）
ephemeral_cmd    点对点短生命周期命令（打印任务、远程控制）
```

现有 24 种消息类型按此分类后，绝大多数是 `retained_state` 或 `ephemeral_cmd`，真正需要 `durable_event` 的极少（主要是催单、工单撤销）。

### D2：Scope 解析规则与 POC 完全兼容

优先级：`TerminalScope > ProfileScope > OrgScope`
同类型中：`更深层级（叶子）> 更浅层级（根）`

这与 POC 中的 `TERMINAL > MODEL > ENTITY`、深路径覆盖浅路径保持一致，只是正式化命名。

### D3：Projection Adapter 是业务域与 TDP 之间的防腐层

业务域产出领域事件，Projection Adapter 订阅 RocketMQ 并将其转化为 TDP Projection 写入，业务域本身不直接依赖 TDP。

这与餐饮中台"渠道差异在边界消化"的设计哲学一致。

### D4：终端连接身份从 `storeId + deviceType` 升级为三维 Scope

当前连接 URL 只携带 `storeId + deviceType`，这是 Scope 最小化的简化。

TDP 中，终端连接时提交完整 Scope 三元组（`orgScopeId + profileScopeId + terminalId`），服务端以此确定哪些 Projection 是该终端可见的。

### D5：L2 部署单位是购物中心项目，非门店

一个购物中心有多个门店，部署一个 L2 边缘节点服务所有门店，而非每门店单独部署。

这是合理的资源密度。

### D6：大 Payload 采用引用地址模式

`menu.catalog` 等大 Payload（> 64KB）只推送版本指针 `{version, downloadUrl}`，终端通过 HTTP 拉取完整数据。

避免 WebSocket 传输大对象造成的性能问题。

## 6. 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| **开发语言** | Java 17 | 与中台主体一致 |
| **应用框架** | Spring Boot 3.x | 与中台主体一致 |
| **WebSocket** | Spring WebSocket + STOMP | 成熟方案 |
| **消息队列** | RocketMQ | 与中台事件总线一致 |
| **Projection Store** | PostgreSQL | 关系型存储，支持复杂查询 |
| **Change Log** | PostgreSQL (时序表) + 定期归档 | 或 ClickHouse（可选） |
| **Session Store** | Redis | 内存存储，快速查询 |
| **Cursor/Revision** | Redis + PostgreSQL 双写 | Redis 快速读，PG 持久化 |
| **服务注册** | Nacos | 与中台一致 |
| **配置中心** | Nacos | 与中台一致 |
| **监控** | Prometheus + Grafana | 与中台一致 |
| **链路追踪** | SkyWalking | 与中台一致 |

## 7. 规模估算

### 7.1 单购物中心规模

| 指标 | 估算值 |
|------|--------|
| 门店数 | 50-200 家 |
| 每店终端数 | POS 2-5台、KDS 1-3台、PDA 2-10台、KIOSK 0-2台 |
| 总终端数 | 300-1000 台 |
| 高峰并发连接 | 500-1500（含顾客小程序） |
| 每秒写入 QPS | 10-50（正常）、100-200（高峰） |
| Projection Store 大小 | 10GB-50GB（单商场） |
| Change Log 保留 7 天 | 50GB-200GB |

### 7.2 全国规模

| 规模 | L1 配置 | L2 配置 | PostgreSQL | Redis |
|------|---------|---------|-----------|-------|
| 单商场 | 4C8G × 1 | — | 4C8G 100GB | 2C4G |
| 10 商场 | 8C16G × 2 | 4C8G × 2/商场 | 8C16G 500GB | 4C8G Cluster |
| 100 商场 | 16C32G × 3 | 4C8G × 2/商场 | 16C32G 2TB | 8C16G Cluster |
