## 1. 部署架构

### 1.1 单商场部署架构

```
┌─────────────────────────────────────────────────────────────┐
│                         负载均衡 (Nginx)                      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    TCP 服务集群 (2 节点)                      │
│  ├─ tcp-service-1 (4C8G)                                    │
│  └─ tcp-service-2 (4C8G)                                    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│              PostgreSQL + Citus (4C8G 100GB)                │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                   Redis Cluster (2C4G)                       │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                   RocketMQ (4C8G)                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                   MinIO / OSS (对象存储)                      │
└─────────────────────────────────────────────────────────────┘
```

```yaml
          periodSeconds: 10

            path: /actuator/health
```

## 5. 监控指标

### 5.1 业务指标

```java
@Component
public class TcpMetrics {

    private final MeterRegistry meterRegistry;

    public TcpMetrics(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
    }

    /**
     * 激活成功率
     */
    public void recordActivation(boolean success) {
        Counter.builder("tcp.activation")
            .tag("result", success ? "success" : "failure")
            .register(meterRegistry)
            .increment();
    }

    /**
     * 配置发布成功率
     */
    public void recordConfigRelease(boolean success) {
        Counter.builder("tcp.config.release")
            .tag("result", success ? "success" : "failure")
            .register(meterRegistry)
            .increment();
    }

    /**
     * 热更新成功率
     */
    public void recordUpgradeRelease(boolean success) {
        Counter.builder("tcp.upgrade.release")
            .tag("result", success ? "success" : "failure")
            .register(meterRegistry)
            .increment();
    }

    /**
     * 远程控制成功率
     */
    public void recordControlTask(String command, boolean success) {
        Counter.builder("tcp.control.task")
            .tag("command", command)
            .tag("result", success ? "success" : "failure")
            .register(meterRegistry)
            .increment();
    }

    /**
     * 终端在线率
     */
    public void recordTerminalOnlineRate(double rate) {
        Gauge.builder("tcp.terminal.online.rate", () -> rate)
            .register(meterRegistry);
    }

    /**
     * 任务执行平均时长
     */
    public void recordTaskExecutionTime(String taskType, long duration) {
        Timer.builder("tcp.task.execution.time")
            .tag("type", taskType)
            .register(meterRegistry)
            .record(duration, TimeUnit.MILLISECONDS);
    }
}
```

## 6. 告警规则

### 6.1 Prometheus 告警规则

```yaml
groups:
- name: tcp-alerts
  rules:
  # P0 告警（紧急）
  - alert: TcpServiceDown
    expr: up{job="tcp-service"} == 0
    for: 1m
    labels:
      severity: critical
    annotations:
      summary: "TCP 服务不可用"
      description: "TCP 服务已停止响应"

  - alert: ActivationSuccessRateLow
    expr: rate(tcp_activation_total{result="success"}[5m]) / rate(tcp_activation_total[5m]) < 0.95
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "激活成功率低于 95%"
      description: "当前激活成功率：{{ $value }}"

  # P1 告警（高）
  - alert: ApiResponseTimeSlow
    expr: histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m])) > 1
    for: 5m
    labels:
      severity: high
    annotations:
      summary: "API 响应时间 P99 超过 1 秒"
      description: "当前 P99 响应时间：{{ $value }}s"

  - alert: TerminalOnlineRateLow
    expr: tcp_terminal_online_rate < 0.8
    for: 10m
    labels:
      severity: high
    annotations:
      summary: "终端在线率低于 80%"
      description: "当前在线率：{{ $value }}"

  # P2 告警（中）
  - alert: JvmMemoryHigh
    expr: jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.8
    for: 10m
    labels:
      severity: medium
    annotations:
      summary: "JVM 堆内存使用率超过 80%"
      description: "当前使用率：{{ $value }}"

  - alert: DatabaseSlowQuery
    expr: rate(postgresql_slow_queries_total[1m]) > 100
    for: 5m
    labels:
      severity: medium
    annotations:
      summary: "数据库慢查询超过 100 次/分钟"
      description: "当前慢查询数：{{ $value }}"
```

## 8. 备份与恢复

### 8.1 数据库备份策略

```bash
#!/bin/bash

# 备份脚本
BACKUP_DIR="/backup/tcp"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="tcp_db_${DATE}.sql.gz"

# 执行备份
pg_dump -h localhost -U tcp_user tcp_db | gzip > ${BACKUP_DIR}/${BACKUP_FILE}

# 保留最近 30 天的备份
find ${BACKUP_DIR} -name "tcp_db_*.sql.gz" -mtime +30 -delete

# 上传到对象存储
aws s3 cp ${BACKUP_DIR}/${BACKUP_FILE} s3://tcp-backup/
```

**备份策略：**

| 备份类型 | 频率 | 保留时间 | 存储位置 |
|---------|------|---------|---------|
| 全量备份 | 每天凌晨3点 | 30天 | 本地 + 对象存储 |
| 增量备份 | 每小时一次 | 7天 | 本地 |
| Binlog | 实时 | 7天 | 本地 |

**备份内容：**
- 终端实例数据（terminal_instances）
- 激活码数据（activation_codes）
- 任务发布数据（task_releases, terminal_task_instances）
- 配置版本数据（config_versions）
- 应用版本数据（app_versions）
- 审计日志数据（audit_logs）

### 8.2 数据库恢复

```bash
#!/bin/bash

# 恢复脚本
BACKUP_FILE=$1

if [ -z "$BACKUP_FILE" ]; then
    echo "用法: $0 <backup_file>"
    exit 1
fi

# 从对象存储下载
aws s3 cp s3://tcp-backup/${BACKUP_FILE} /tmp/

# 执行恢复
gunzip -c /tmp/${BACKUP_FILE} | psql -h localhost -U tcp_user tcp_db

echo "数据库恢复完成"
```

**恢复流程：**

1. **停止应用服务**
   ```bash
   kubectl scale deployment tcp-service --replicas=0
   ```

2. **备份当前数据库**（防止恢复失败）
   ```bash
   pg_dump -h localhost -U tcp_user tcp_db > /tmp/tcp_db_before_restore.sql
   ```

3. **执行恢复**
   ```bash
   ./restore.sh tcp_db_20260406_030000.sql.gz
   ```

4. **验证数据完整性**
   ```sql
   -- 检查关键表记录数
   SELECT 'terminal_instances' AS table_name, COUNT(*) FROM terminal_instances
   UNION ALL
   SELECT 'task_releases', COUNT(*) FROM task_releases
   UNION ALL
   SELECT 'audit_logs', COUNT(*) FROM audit_logs;
   ```

5. **重启应用服务**
   ```bash
   kubectl scale deployment tcp-service --replicas=3
   ```

6. **验证服务可用性**
   ```bash
   curl http://tcp-service:8092/actuator/health
   ```

### 8.3 关键操作软删除

为支持数据恢复，关键实体使用软删除：

```java
/**
 * 软删除基类
 */
@MappedSuperclass
public abstract class SoftDeletableEntity {

    @Column(name = "deleted")
    private Boolean deleted = false;

    @Column(name = "deleted_at")
    private LocalDateTime deletedAt;

    @Column(name = "deleted_by")
    private String deletedBy;

    /**
     * 标记删除
     */
    public void markDeleted(String deletedBy) {
        this.deleted = true;
        this.deletedAt = LocalDateTime.now();
        this.deletedBy = deletedBy;
    }

    /**
     * 恢复删除
     */
    public void undelete() {
        this.deleted = false;
        this.deletedAt = null;
        this.deletedBy = null;
    }

    /**
     * 是否已删除
     */
    public boolean isDeleted() {
        return deleted != null && deleted;
    }
}
```

**软删除实体：**
- TerminalInstance（终端实例）
- ActivationCode（激活码）
- TaskRelease（任务发布）
- ConfigVersion（配置版本）
- AppVersion（应用版本）

**查询过滤：**

```java
@Repository
public interface TerminalInstanceRepository extends JpaRepository<TerminalInstance, String> {

    /**
     * 查询未删除的终端
     */
```

### 8.4 增量备份与 Binlog

**配置 PostgreSQL WAL 归档：**

```ini
# postgresql.conf
wal_level = replica
archive_mode = on
archive_command = 'cp %p /backup/tcp/wal/%f'
archive_timeout = 3600  # 每小时归档一次
```

**基于 WAL 的增量恢复：**

```bash
#!/bin/bash

# 增量恢复脚本
BASE_BACKUP=$1
WAL_DIR="/backup/tcp/wal"

# 1. 恢复基础备份
gunzip -c ${BASE_BACKUP} | psql -h localhost -U tcp_user tcp_db

# 2. 应用 WAL 日志
pg_waldump ${WAL_DIR}/* | psql -h localhost -U tcp_user tcp_db

echo "增量恢复完成"
```

### 8.5 备份验证

**定期验证备份可用性：**

```bash
#!/bin/bash

# 备份验证脚本
BACKUP_FILE=$1
TEST_DB="tcp_db_test"

# 1. 创建测试数据库
createdb -h localhost -U tcp_user ${TEST_DB}

# 2. 恢复到测试数据库
gunzip -c ${BACKUP_FILE} | psql -h localhost -U tcp_user ${TEST_DB}

# 3. 验证数据完整性
psql -h localhost -U tcp_user ${TEST_DB} -c "
    SELECT
        'terminal_instances' AS table_name,
        COUNT(*) AS record_count
    FROM terminal_instances
    UNION ALL
    SELECT 'task_releases', COUNT(*) FROM task_releases;
"

# 4. 删除测试数据库
dropdb -h localhost -U tcp_user ${TEST_DB}

echo "备份验证完成"
```

**验证频率**：每周一次

### 8.6 灾难恢复计划

**RTO（恢复时间目标）**：2小时
**RPO（恢复点目标）**：1小时

**灾难恢复步骤：**

1. **评估损失**
   - 确认数据丢失范围
   - 确认最近可用备份时间点

2. **准备恢复环境**
   - 准备新的数据库实例
   - 准备新的应用服务实例

3. **恢复数据**
   - 恢复最近的全量备份
   - 应用增量备份和 WAL 日志
   - 验证数据完整性

4. **恢复服务**
   - 更新应用配置（数据库连接）
   - 启动应用服务
   - 验证服务可用性

5. **数据对账**
   - 对比恢复后数据与预期
   - 检查关键业务数据
   - 通知相关方

6. **切换流量**
   - 更新 DNS 或负载均衡配置
   - 逐步切换流量到新环境
   - 监控服务稳定性

```bash
#!/bin/bash

# 健康检查脚本
HEALTH_URL="http://localhost:8092/actuator/health"
```
