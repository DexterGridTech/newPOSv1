## P0-1: 统一任务模型重构 ✅

### 修改的文件

1. **新增文件**：`10-统一任务模型设计.md`
2. **修改文件**：`01-领域模型设计.md`

### 核心变更

#### 1. 创建统一任务模型

**新增聚合根：**
- `TaskRelease`：统一的任务发布聚合根
- `TerminalTaskInstance`：统一的终端任务实例

**移除的独立聚合根：**
- ~~`ConfigRelease` + `ConfigTaskInstance`~~
- ~~`UpgradeRelease` + `UpgradeTaskInstance`~~
- ~~`ControlTask` + `TaskExecutionRecord`~~

#### 2. 统一状态机

```
PENDING → DISPATCHED → DELIVERED → EXECUTING → SUCCESS/FAILED
```

**状态说明：**
- `PENDING`：任务已创建，等待投递到 TDP
- `DISPATCHED`：任务已投递到 TDP，等待送达终端
- `DELIVERED`：任务已送达终端，等待执行
- `EXECUTING`：终端正在执行任务
- `SUCCESS`：任务执行成功
- `FAILED`：任务执行失败

#### 3. 差异化 Payload

**TaskPayload 接口：**
- `ConfigPublishPayload`：配置发布载荷
- `AppUpgradePayload`：应用升级载荷
- `RemoteControlPayload`：远程控制载荷

**TaskResult 接口：**
- `ConfigPublishResult`：配置发布结果
- `AppUpgradeResult`：应用升级结果
- `RemoteControlResult`：远程控制结果

#### 4. 任务类型枚举

```java
public enum TaskType {
    CONFIG_PUBLISH,   // 配置发布
    APP_UPGRADE,      // 应用升级
    REMOTE_CONTROL    // 远程控制
}
```

### 优势

1. **代码复用**：三种任务类型共享相同的聚合根和状态机
2. **逻辑一致**：统一的状态流转和异常处理
3. **易于扩展**：新增任务类型只需添加新的 Payload 和 Result
4. **协议简化**：TDP 只需一个投递接口

## P0-2: TCP-TDP协同边界明确化 ✅

### 修改的文件

1. **修改文件**：`08-与TDP协同设计.md`
2. **修改文件**：`01-领域模型设计.md`

### 核心变更

#### 1. 职责边界明确

**TCP（控制面）：**
- 管理终端身份、归属、能力
- **内部完成目标圈选**，产出明确的 `terminalId → taskInstance` 集合
- 追踪任务执行结果（lifecycleStatus 权威）

**TDP（数据面）：**
- 管理终端在线连接（presenceStatus 权威）
- 可靠投递任务到终端
- 同步终端在线状态到 TCP

#### 2. 统一投递接口

**旧设计（分裂）：**
```
POST /api/v1/tdp/tasks/config
POST /api/v1/tdp/tasks/upgrade
POST /api/v1/tdp/tasks/control
```

**新设计（统一）：**
```
POST /internal/control-plane/tasks/dispatch
```

**关键改进：**
- TCP 提交的是**已解算后的终端级投递列表**
- 不再使用 `orgScopeId + profileScopeId + terminalIds` 这种复合表达
- TDP 只负责投递，不负责解析目标范围

#### 3. 三类状态分层

**LifecycleStatus（生命周期状态）：**
- 权威来源：TCP 控制面
- 变更方式：通过 TCP API 操作
- 用途：控制终端的业务生命周期

**PresenceStatus（在线状态）：**
- 权威来源：TDP 数据面
- 变更方式：TDP 通过 WebHook 同步给 TCP
- 用途：反映终端与 TDP 的连接状态

**HealthStatus（健康状态）：**
- 权威来源：TCP 计算
- 变更方式：TCP 根据多种指标综合计算
- 用途：运维监控和告警

#### 4. Delivery vs Execution 分层

**Delivery 状态（TDP 负责）：**
- `PENDING → DISPATCHED → DELIVERED`
- TDP 同步送达状态到 TCP

**Execution 状态（终端执行）：**
- `DELIVERED → EXECUTING → SUCCESS/FAILED`
- 终端回报执行结果到 TCP

#### 5. 结果回报以 Token 为准

终端回报结果时：
- 使用 `instanceId` 标识任务实例
- 终端身份从 JWT token 中提取 `terminalId`
- **以 token 为准**，不信任请求体中的 `terminalId`

### 协议示例

**任务投递（TCP → TDP）：**
```json
POST /internal/control-plane/tasks/dispatch

{
  "releaseId": "release-001",
  "taskType": "CONFIG_PUBLISH",
  "instances": [
    {
      "instanceId": "inst-001",
      "terminalId": "pos-001",
      "payload": { ... }
    }
  ]
}
```

**送达状态同步（TDP → TCP）：**
```json
POST /api/v1/tcp/tasks/delivery-status

{
  "instanceId": "inst-001",
  "deliveryStatus": "DELIVERED",
  "deliveredAt": "2026-04-06T10:00:30Z"
}
```

**结果回报（终端 → TCP）：**
```json
POST /api/v1/tcp/tasks/results
Authorization: Bearer {terminal_token}

{
  "instanceId": "inst-001",
  "status": "SUCCESS",
  "result": { ... }
}
```
