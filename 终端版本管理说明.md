# 终端版本管理说明

## 1. 适用范围

本说明适用于当前终端工程中的版本管理与发布操作，覆盖以下层面：

- `1-kernel`
- `2-ui`
- `3-adapter`
- `4-assembly`

重点覆盖：

- Android 终端整合包
- Electron 终端整合包
- 安装包发布
- JS Bundle / OTA 发布
- 本地版本维护
- 后续 CI 集成方式

本说明只讲怎么做，不讲历史背景。

---

## 2. 当前版本体系

当前统一采用以下版本字段：

### 2.1 Assembly Version

用途：安装包主版本。

位置：

- `4-assembly/android/mixc-retail-rn84v2/release.manifest.json`
- `4-assembly/electron/mixc-retail-v1/release.manifest.json`

字段：

- `assemblyVersion`

示例：

```json
"assemblyVersion": "1.1.0"
```

---

### 2.2 Build Number

用途：构建号，给机器识别，要求递增。

位置：

- `release.manifest.json` 中的 `buildNumber`
- Android 同步到 `android/app/build.gradle` 的 `versionCode`

示例：

```json
"buildNumber": 1000002
```

---

### 2.3 Bundle Version

用途：JS Bundle / OTA 版本。

位置：

- `release.manifest.json` 中的 `bundleVersion`

示例：

```json
"bundleVersion": "1.1.0+ota.2"
```

---

### 2.4 Runtime Version

用途：标识运行时兼容范围，用于区分不同 assembly / 平台的 OTA 兼容关系。

位置：

- `release.manifest.json` 中的 `runtimeVersion`

示例：

```json
"runtimeVersion": "android-mixc-retail-rn84v2@1.1"
```

---

### 2.5 Package Version

用途：包自身版本，供模块内部版本展示和开发追踪使用。

来源：

- 每个包自己的 `package.json.version`

同步方式：

- 通过生成文件 `src/generated/packageVersion.ts`
- 模块 `src/index.ts` 中统一使用 `packageVersion`

---

## 3. 版本真相源

当前版本真相源为各 assembly 自己的：

## 5. 日常开发怎么做

### 5.1 只开发功能，不发版

如果只是日常开发，不需要改 `release.manifest.json`。

正常做：

- 修改代码
- 跑 `type-check`
- 跑本地 dev

如果新加了 kernel/ui 包，或修改了包版本号后需要同步源码版本文件，可以执行：

```bash
node scripts/release/generate-package-version-info.cjs
```

作用：

- 为 `1-kernel`、`2-ui` 下符合规范的包生成 `src/generated/packageVersion.ts`

---

### 5.2 修改某个包的 `package.json.version` 后

如果你手工改了某个 kernel/ui 包的 `package.json.version`，需要重新生成版本文件：

```bash
node scripts/release/generate-package-version-info.cjs
```

然后执行相关包的 `type-check`。

## 6. Android 安装包发布怎么做

### 6.1 一次性完整流程

推荐直接用完整流程脚本：

```bash
node scripts/release/release-assembly-android-full.cjs --app assembly-android-mixc-retail-rn84v2 --version 1.2.0 --channel production
```

这个脚本会自动完成：

1. 更新 `assemblyVersion`
2. 递增 `buildNumber`
3. 刷新 `targetPackages`
4. 写入 git 信息
5. 校验 manifest
6. 同步 assembly `package.json.version`
7. 同步 Android `versionCode / versionName`
8. 生成 `src/generated/releaseInfo.ts`
9. 执行 Android assembly `type-check`
10. 执行现有 Android release 构建
11. 写回构建产物信息到 manifest

---

### 6.2 如果想拆开执行

#### 第一步：设置主版本

```bash
node scripts/release/bump-assembly-version.cjs --app assembly-android-mixc-retail-rn84v2 --version 1.2.0
```

#### 第二步：递增构建号

```bash
node scripts/release/bump-build-number.cjs --app assembly-android-mixc-retail-rn84v2
```

#### 第三步：刷新 manifest 快照

```bash
node scripts/release/prepare-release-manifest.cjs --app assembly-android-mixc-retail-rn84v2 --channel production
```

#### 第四步：注入 git 信息

```bash
node scripts/release/inject-git-metadata.cjs --app assembly-android-mixc-retail-rn84v2
```

#### 第五步：校验 manifest

```bash
node scripts/release/validate-release-manifest.cjs --app assembly-android-mixc-retail-rn84v2
```

#### 第六步：同步版本到 package 与 Gradle

```bash
node scripts/release/sync-assembly-version.cjs --app assembly-android-mixc-retail-rn84v2
node scripts/release/sync-android-version.cjs --app assembly-android-mixc-retail-rn84v2
node scripts/release/generate-assembly-release-info.cjs --app assembly-android-mixc-retail-rn84v2
```

#### 第七步：构建 release 包

```bash
corepack yarn assembly:android-mixc-retail-rn84v2:build:release
```

#### 第八步：归档产物与 release note

```bash
node scripts/release/finalize-release-manifest.cjs --app assembly-android-mixc-retail-rn84v2
node scripts/release/generate-release-note.cjs --app assembly-android-mixc-retail-rn84v2
```

## 10. 修改版本号时应该改哪里

### 10.1 改终端安装包版本

不要手改：

- `android/app/build.gradle`
- `assembly package.json version`
- `src/index.ts`

正确做法：

- 用 `bump-assembly-version.cjs`
- 再执行对应同步脚本或 full-flow

---

### 10.2 改终端 JS Bundle 版本

不要手改 `bundleVersion` 字符串。

正确做法：

- 用 `bump-bundle-version.cjs`
- 或直接用 `release-bundle-full.cjs`

---

### 10.3 改构建号

不要手改 Android `versionCode`。

正确做法：

- 用 `bump-build-number.cjs`
- 再执行 `sync-android-version.cjs`

---

### 10.4 改包内部模块版本

不要手改源码里的 `version: '0.0.1'`。

当前规范：

- kernel/ui 包统一使用 `src/generated/packageVersion.ts`
- assembly 包统一使用 `src/generated/releaseInfo.ts`

如果 `package.json.version` 有变化，执行：

```bash
node scripts/release/generate-package-version-info.cjs
```

---

## 11. 新建一个包时，如何纳入当前版本管理体系

### 11.1 对 `1-kernel` / `2-ui` 新包

如果你新建一个 kernel/ui 包，要满足以下结构：

- 必须有 `package.json`
- 必须有 `src/index.ts`

然后执行：

```bash
node scripts/release/generate-package-version-info.cjs
```

脚本会自动生成：

- `src/generated/packageVersion.ts`

你需要保证 `src/index.ts` 中模块版本写法为：

```ts
import {packageVersion} from './generated/packageVersion'

export const xxxModule: AppModule = {
  name: moduleName,
  version: packageVersion,
}
```

不要再写：

```ts
version: '0.0.1'
```

---

### 11.2 对 `4-assembly` 新包

如果你新建一个 assembly，需要额外做这些事：

#### 1）新增 `release.manifest.json`

至少包含：

- `appId`
- `platform`
- `product`
- `assemblyVersion`
- `buildNumber`
- `bundleVersion`
- `runtimeVersion`
- `channel`
- `minSupportedAppVersion`
- `targetPackages`
- `git`
- `updatedAt`

#### 2）在 `src/index.ts` 中使用 releaseInfo

写法：

```ts
import {releaseInfo} from './generated/releaseInfo'

export const xxxAssemblyModule: AppModule = {
  name: moduleName,
  version: releaseInfo.assemblyVersion,
}
```

#### 3）新增 `src/generated/releaseInfo.ts`

初始可以手工放一个占位文件，后续通过：

```bash
node scripts/release/generate-assembly-release-info.cjs --app <appId>
```

自动生成。

#### 4）把新 assembly 注册进脚本映射

当前以下脚本内部有 appId 映射：

- `scripts/release/shared.cjs`
- `scripts/release/generate-assembly-release-info.cjs`
- `scripts/release/finalize-release-manifest.cjs`
- `scripts/release/release-bundle-full.cjs`
- 以及后续任何按 appId 分流的脚本

新建 assembly 后，必须补这些映射。

#### 5）为新 assembly 增加根脚本（可选但推荐）

如果这个 assembly 进入正式发布链，建议在根 `package.json` 补对应构建命令。

---

### 11.3 对 `3-adapter` 新包

当前 adapter 主要通过 `package.json.version` 参与版本快照。

如果新 adapter 被某个 assembly 引用，需要做两件事：

1. 确保它有自己的 `package.json.version`
2. 把它加入该 assembly 对应的 `targetPackages` 收集规则

当前收集规则在：

- `scripts/release/shared.cjs`

新 adapter 加入某个 assembly 后，需要同步更新这里的包列表。

## 14. CI 集成建议

### 14.1 CI 最少要接哪些脚本

后续接 CI 时，建议直接复用当前本地脚本，不要重新造一套 CI 内逻辑。

最少接入：

- `validate-release-manifest.cjs`
- `inject-git-metadata.cjs`
- `generate-assembly-release-info.cjs`
- `generate-package-version-info.cjs`
- `release-assembly-android-full.cjs`
- `release-assembly-electron-full.cjs`
- `release-bundle-full.cjs`
- `finalize-release-manifest.cjs`
- `generate-release-note.cjs`

---

### 14.2 推荐的 CI Job 划分

#### Job 1：版本校验

执行：

- manifest validate
- package version generate
- assembly releaseInfo generate
- type-check

#### Job 2：安装包构建

Android：

- `release-assembly-android-full.cjs`

Electron：

- `release-assembly-electron-full.cjs`

#### Job 3：OTA 构建

- `release-bundle-full.cjs`

#### Job 4：发布记录归档

- `finalize-release-manifest.cjs`
- `generate-release-note.cjs`
- 上传 manifest / release note / 产物

---

### 14.3 CI 中哪些步骤自动，哪些步骤人工

建议：

#### 自动

- validate
- inject git
- generate version info
- type-check
- build
- finalize manifest
- generate release note

#### 人工审批

- 生产环境 Android 安装包发布
- 生产环境 Electron 安装包发布
- 生产环境 OTA 放量

---

## 15. 当前未完成项

当前体系已经能用，但还未覆盖这些内容：

- Electron OTA 构建与上传闭环
- targetPackages 的自动依赖图分析
- 更多 package 的版本生成覆盖（如部分 adapter）
- 发布审批平台接入
- 灰度与回滚自动化接入

后续如需扩展，优先沿用当前 `release.manifest.json + scripts/release/*` 体系继续加，不要另起版本系统。

---

## 16. 当前操作原则

后续版本管理统一遵循下面几条：

1. 不直接手改散落在源码中的版本字符串
2. 不直接手改 Android `versionCode / versionName`
3. 不直接手改 assembly `package.json.version` 作为发布主入口
4. 所有发布先改 `release.manifest.json`，再通过脚本同步
5. 所有发布尽量优先走 full-flow 脚本
6. 所有新包必须尽快接入生成版本文件机制
7. 所有 CI 后续只复用现有本地脚本，不重复实现发布逻辑
