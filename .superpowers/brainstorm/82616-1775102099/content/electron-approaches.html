Electron V1 Architecture Options

Target already fixed: dual-window, dual-runtime, strict bridge, full 7 adapters, near-production delivery

A. Monolithic Main Host
Main Process
Window lifecycle + all adapter services + IPC router
Primary Renderer
Secondary Renderer

Option A

Everything terminates in one main-process host. Fastest to start, but adapter boundaries are weaker and host complexity grows quickly.

B. Host Service Container
Main Process Shell
Window Orchestrator
IPC Bridge
Adapter Services
Local Server / Connector Hub
Primary Renderer
Secondary Renderer

Option B

Recommended. Main process stays thin as shell + coordinator; adapter capabilities live in explicit service modules with typed IPC contracts.

C. Utility Process Heavy
Main Process Shell
Utility Process 1
Utility Process 2
IPC mesh + renderers

Option C

Most isolated, best long-term for hardware-heavy adapters, but the coordination and packaging cost is too high for V1.

Recommendation

Why B

  • Matches Android assembly responsibilities without copying Android process model mechanically.
  • Keeps strict bridge discipline and typed capability boundaries.
  • Leaves room for later moving single services into utility processes if hardware pressure appears.

Trade-off

  • Requires explicit IPC schemas, preload contracts, and lifecycle ownership from day one.
