# 小白手册

> 适用对象：第一次接触 `Mock Terminal Platform`，完全不了解 TCP / TDP / 终端联调体系的同学。  
> 目标：用最容易理解的话，帮你搞清楚这个系统里“谁是谁”“各自干什么”“什么时候该用谁”，以及你下一步怎么开始开发。

---

## 1. 先用一句人话理解这个系统

你可以把整套系统理解成：

- **TCP（Terminal Control Plane，终端控制平面）**：负责“管理终端、下达治理动作、记录终端状态”的后台指挥中心
- **TDP（Terminal Data Plane，终端数据平面）**：负责“把终端应该感知到的数据，稳定地送到终端”的数据配送网络

如果把终端系统比作“外卖体系”：

- **TCP** 像“运营后台 + 调度中心”
  - 决定哪些门店、哪些终端需要做什么
  - 比如：激活终端、绑定门店、下发配置、指定升级版本、发远控任务
- **TDP** 像“配送网络 + 快递站”
  - 负责把这些数据或状态送到终端
  - 比如：把“你应该升级到 2.4.0”“你现在要用 config-v8”“这个终端有个待执行任务”送到设备上

**一句话总结：**
- **TCP 决策和治理**
- **TDP 存储和分发**

---

## 2. 为什么要拆成 TCP 和 TDP 两个东西？

因为现实里终端系统有两种完全不同的需求：

### 2.1 终端主动发起业务动作
比如：
- 收银员点击“提交订单”
- 终端点击“查询库存”
- 设备点击“上报升级结果”
- 终端激活时发 HTTP 请求

这类动作是：
- 终端主动发起
- 有明确请求和响应
- 更像正常的业务接口调用

这类通常走：
- **HTTP 接口**
- 对接真实业务系统，或者对接 TCP

### 2.2 终端被动感知后台变化
比如：
- 总部决定某一批终端升级到 `2.4.0`
- 某门店的配置变更了
- 某终端需要看到新的任务
- 某终端断网恢复后，需要追平之前漏掉的数据

这类动作是：
- 后台发生变化，终端要“被动知道”
- 更像“后台下发”“状态同步”“数据投递”

这类通常走：
- **TDP**

所以要拆开：
- **TCP** 更像“管控平台”
- **TDP** 更像“数据下发基础设施”

---

## 3. 一张最简单的关系图

```text
业务系统 / TCP
   │
   │ 产生任务、配置、升级目标、终端治理动作
   ▼
TDP
   │
   │ 以 Topic / Projection / ChangeLog 的形式分发给终端
   ▼
终端
   │
   ├─ 主动业务请求：HTTP 调业务系统 / TCP
   └─ 被动数据接收：通过 TDP 感知后台变化
```

再说得更生活化一点：

- **TCP** 说：“这 100 台终端今晚升级到新版本。”
- **TDP** 负责把“升级目标”这个信息真正送到这 100 台终端。
- **终端** 收到后，择机升级。
- **终端升级完成** 后，再通过 HTTP 把结果告诉 **TCP**。
- **TCP** 再把“当前版本是多少、是否成功”记录下来。

这就是现实里非常典型的一条链路。

## 4. TCP 和 TDP 分别负责什么？

## 4.1 TCP 负责什么？

TCP 主要负责“治理、控制、记录”。

你可以理解为它关心的是：

- 这台终端是谁？
- 它属于哪个门店？
- 它当前健康吗？在线吗？
- 它现在应该用哪个配置？
- 它应该升级到哪个版本？
- 我要不要对它下发任务？
- 任务执行结果是什么？

### TCP 典型职责

- 终端注册与主数据维护
- 终端激活、绑定、换绑、停用、退役
- 终端能力管理
- 版本热更新目标管理
- 配置版本管理
- 远程控制任务管理
- 任务发布单、任务实例、结果回收
- 终端资产台账与审计记录

### 什么时候应该想到 TCP？

当你脑子里想的是下面这些问题时，大概率就是 TCP：

- “这台终端归谁管？”
- “这批终端应该升级到哪个版本？”
- “我要给这些终端发一批配置任务。”
- “这台终端是否已经激活？”
- “我想看某台终端最近执行过哪些任务。”
- “我要知道这台终端当前安装了哪个包、哪个配置版本。”

---

## 4.2 TDP 负责什么？

TDP 主要负责“存储终端应该看到的数据，并把它可靠送出去”。

你可以理解为它关心的是：

- 某类数据叫什么名字？
- 这类数据对哪些范围生效？
- 某个范围当前的最新数据是什么？
- 这个数据历史改过几次？
- 终端现在在线吗？能不能把这个数据推给它？
- 如果终端刚才不在线，之后怎么补给它？

### TDP 典型职责

- Topic 定义与治理
- Scope 范围定义
- Projection 当前快照维护
- Change Log 历史变更记录
- Session 会话连接与心跳
- 终端断线重连后的数据追平
- 中心到边缘的数据同步（正式系统会有）

### 什么时候应该想到 TDP？

当你脑子里想的是下面这些问题时，大概率就是 TDP：

- “终端应该被动收到哪些数据？”
- “同一类数据，门店 A 和门店 B 是否不同？”
- “某终端离线几分钟后回来，怎么补齐这期间漏掉的数据？”
- “当前终端看到的配置快照到底是什么？”
- “最近一次下发的数据历史长什么样？”
- “这个任务信息到底有没有送到终端？”

## 5. 系统中的核心实体和名词，分别是什么意思？

这一章最重要。你只要把下面这些词看懂，整个系统就基本入门了。

---

## 5.1 Sandbox（沙箱）

### 它是什么
一个“独立的测试环境容器”。

### 它的作用
把一组终端、任务、Topic、故障规则、审计日志放在一个相对独立的环境里，避免互相干扰。

### 什么时候用
- 你要做开发联调时
- 你要造一批测试数据时
- 你想做故障注入但不影响别的同学时

### 真实场景例子
- 你正在开发“热更新能力”，可以在 `sandbox-default` 里造一批测试终端，不影响别人。
- 测试同学想做“配置下发失败”的演练，也可以在独立沙箱里做。

---

## 5.2 Terminal / TerminalInstance（终端实例）

### 它是什么
一台真正被系统管理的终端设备实例。

### 它的作用
是整个系统最核心的管理对象。

### 你可以把它理解成
“门店里那台真正的 POS / PDA / KDS 设备，在系统里的数字身份”。

### 它通常包含什么信息
- `terminalId`：终端 ID
- `storeId`：所属门店
- `profileId`：终端画像
- `templateId`：终端模板
- `lifecycleStatus`：生命周期状态
- `presenceStatus`：在线状态
- `healthStatus`：健康状态
- `currentAppVersion`：当前应用版本
- `currentConfigVersion`：当前配置版本

### 什么时候用
只要你是在讨论“某一台设备”，你几乎都在和 Terminal 打交道。

### 真实场景例子
- 门店新增了一台收银机，先生成激活码，等设备真正激活后再创建 Terminal。
- 某台终端坏了，后台把它标记为告警或退役。
- 某台终端升级失败，需要查看它的版本和任务历史。

---

## 5.3 终端机型（TerminalProfile）

### 它是什么
终端能力类型的描述。

### 它的作用
回答“这类终端通常具备什么能力”。

### 你可以把它理解成
终端的“机型角色说明书”。

### 什么时候用
当你需要区分：
- 收银 POS
- 厨显 KDS
- 手持 PDA
- 自助终端 KIOSK

不同终端能做的事情不同，所需配置也不同，就需要终端机型。

### 真实场景例子
- POS 支持支付、打印、小票补打
- KDS 支持工单展示、叫菜提醒
- PDA 支持上菜确认、桌台巡检

---

## 5.4 终端模板（TerminalTemplate）

### 它是什么
终端默认配置的预设模板。

### 它的作用
避免每次新建终端都从零开始配置。

### 你可以把它理解成
“装机预设包”或“默认参数包”。

### 什么时候用
- 一批新终端要按同样的规则初始化
- 同一类门店希望终端默认行为一致

### 真实场景例子
- 华南门店 POS 默认启用双屏模式
- 某业务线的终端默认开启某些功能开关

---

## 5.5 ActivationCode（激活码）

### 它是什么
让终端完成首次激活和绑定的凭证。

### 它的作用
把“真实设备”绑定进系统。

### 什么时候用
- 新设备第一次接入系统
- 门店装机
- 设备重置后重新激活

### 真实场景例子
运维同学拿到一台新设备，在后台先生成激活码；设备启动时输入激活码，系统就知道“这台设备属于谁、应该使用什么上下文”。

---

## 5.6 TaskRelease（任务发布单）

### 它是什么
一次“治理动作”的总任务单。

### 它的作用
描述“我要对哪些终端做什么事情”。

### 常见任务类型
- 配置下发
- 热更新升级
- 远程控制
- 某种系统维护动作

### 什么时候用
当你是从后台角度去发起一个批量动作时，用的就是 TaskRelease。

### 真实场景例子
- 总部要把 300 台终端配置切到 `config-v8`
- 指定一批灰度终端升级到 `2.4.0`
- 对一组故障终端下发远程诊断任务

---

## 5.7 TaskInstance（任务实例）

### 它是什么
TaskRelease 拆分到单台终端后的具体执行项。

### 它的作用
回答：“这张总任务单，落到每台设备上之后，现在各自执行得怎么样？”

### 什么时候用
- 你想看某台终端是否已收到任务
- 你想看一批终端里哪些成功、哪些失败
- 你想排查具体某一台设备的执行问题

### 真实场景例子
一张升级发布单发给 100 台终端：
- 总体上看，这是 1 个 TaskRelease
- 落到每台设备上，就是 100 个 TaskInstance

---

## 5.8 Session（会话）

### 它是什么
终端和 TDP 之间的一次连接会话。

### 它的作用
表示“这台终端当前是否连上了数据面”。

### 它通常关心什么
- 终端 ID
- 客户端版本
- 协议版本
- 连接状态
- 最近心跳时间

### 什么时候用
- 想知道终端是否在线
- 想知道终端是否还能收到 TDP 数据
- 排查连接问题

### 真实场景例子
终端启动后先和 TDP 建立 Session，之后定期发心跳，TDP 才知道它还活着。

---

## 5.9 Topic（主题）

### 它是什么
TDP 中“一类数据”的名字。

### 它的作用
告诉系统：“这是一种什么类型的数据。”

### 你可以把它理解成
“数据频道”或者“数据分类名”。

### 当前采用的设计
**先注册 Topic，再允许 Payload 自由扩展。**

也就是说：
- Topic 名字要先确定
- 但具体 JSON 字段可以灵活扩展

### 什么时候用
当你有一种新的“终端需要被动感知的数据”时，就应该考虑是否要定义一个 Topic。

### 真实场景例子
- `tcp.task.release`：终端任务发布数据
- `terminal.config.state`：终端配置状态
- `terminal.runtime.config`：终端运行时配置

### 小白容易犯的错
不要把 Topic 理解成“接口名”。

它更像是：
- 某类投递数据的统一分类名
- 后续多个终端范围都可以基于这个 Topic 有自己的快照

---

## 5.10 Schema（数据结构约定）

### 它是什么
描述某个 Topic 的 Payload 大概长什么样。

### 它的作用
帮助前后端、终端、测试同学对齐数据结构。

### 什么时候用
- 你要定义一个新 Topic
- 你要约定这个 Topic 的 JSON 长什么样
- 你要避免字段乱飞

### 真实场景例子
比如 `terminal.runtime.config` 的 payload 可能长这样：

```json
{
  "desiredVersion": "2.4.0",
  "rollout": "gray"
}
```

### 小白理解重点
当前系统不是死板的强约束模式，而是：
- **先有 Topic 治理**
- **再允许 Payload 灵活扩展**

所以 Schema 更像“推荐结构”和“沟通契约”，而不是完全僵硬的数据库表。

---

## 5.11 Scope（作用范围）

### 它是什么
一条数据到底对“谁”生效。

### 它的作用
回答：“这个 Topic 的这份数据，是给单台终端、某家门店，还是整个租户用的？”

### 当前常见 Scope 类型
- `TERMINAL`：某台终端
- `STORE`：某个门店
- `TENANT`：某个租户
- `GLOBAL`：全局

### 什么时候用
只要你在讨论“数据生效范围”，你就在用 Scope。

### 真实场景例子
- 某台测试机单独启用一个灰度参数：`TERMINAL`
- 某门店统一切换打印配置：`STORE`
- 某品牌统一调整策略：`TENANT`
- 全部终端统一启用公告：`GLOBAL`

---

## 5.12 Projection（投影快照）

### 它是什么
某个 Topic 在某个 ScopeKey 下的“当前最新值”。

### 它的作用
回答：“此时此刻，这个范围下，终端应该看到的最新数据是什么？”

### 这是 TDP 里最核心的对象之一
你可以把它理解成：
- Topic 是“数据分类”
- Scope 是“对谁生效”
- Projection 是“当前最终结果”

### 什么时候用
- 想看终端当前应该拿到什么配置
- 想看某门店当前应该收到什么数据
- 想查任务投递后的最新状态

### 真实场景例子
Topic：`tcp.task.release`  
ScopeType：`TERMINAL`  
ScopeKey：`T-1001`  
Projection Payload：

```json
{
  "releaseId": "release-001",
  "taskType": "CONFIG_PUBLISH",
  "payload": {
    "configVersion": "config-v8"
  },
  "deliveryStatus": "PENDING"
}
```

这就表示：
> 当前终端 `T-1001` 应该感知到的最新任务数据，就是这份内容。

---

## 5.13 ChangeLog（变更日志）

### 它是什么
Projection 的历史变更记录。

### 它的作用
回答：“这份数据之前怎么变过？”

### 什么时候用
- 排查到底是谁改了数据
- 排查某次任务是不是重复投递了
- 排查终端看到的数据为什么和后台预期不一致

### 真实场景例子
- revision 1：配置是 `config-v7`
- revision 2：配置改成 `config-v8`
- revision 3：回滚到 `config-v7`

这些历史都可以通过 ChangeLog 看出来。

---

## 5.14 AuditLog（审计日志）

### 它是什么
后台操作和重要系统动作的记录。

### 它的作用
回答：“谁在什么时候做了什么事？”

### 什么时候用
- 排查误操作
- 查导入导出记录
- 查故障模拟是谁触发的
- 查谁创建了任务发布单

### 真实场景例子
- 某同学手动创建了 Topic
- 某测试同学导入了 Fault 模板
- 某人模拟了故障命中

---

## 5.15 SceneTemplate（场景模板）

### 它是什么
一套预设的联调脚本模板。

### 它的作用
帮助你快速造场景，而不是每次手工点很多按钮。

### 什么时候用
- 你要快速搭一组“像真的一样”的测试场景
- 你要反复复现同一类联调问题
- 你要给测试同学一个一键复现入口

### 真实场景例子
- 门店批量上线
- 灰度升级演练
- 配置回滚演练

---

## 5.16 FaultRule（故障规则）

### 它是什么
用来故意制造异常的规则。

### 它的作用
帮助开发、测试模拟真实世界里的各种问题。

### 典型故障动作
- 延迟
- 超时
- 失败
- 伪造返回结果

### 什么时候用
- 测试超时重试是否正常
- 测试终端断线恢复逻辑
- 测试任务失败后的回滚或重试

### 真实场景例子
- 把配置任务的投递故意延迟 3 秒
- 把升级任务故意模拟成超时
- 把某类任务故意回一个假的成功结果

---

## 5.17 TemplateLibrary（模板库）

### 它是什么
平台内置的一批可复用模板。

### 它的作用
让你少从零开始。

### 常见类型
- Topic 模板
- Fault 模板

### 什么时候用
- 你是新手，不知道 Topic 怎么写
- 你想快速创建一条故障规则
- 你想先跑通流程，再决定如何定制

## 6. 几个最常见的真实场景，分别怎么落在 TCP / TDP 上？

## 6.1 场景一：新终端装机

### 发生了什么
门店来了一台新 POS，要纳入系统时，通常先生成激活码，设备完成激活后才真正创建 Terminal。

### 一般流程
1. 后台在 TCP 里创建终端或准备激活码
2. 设备启动后输入激活码
3. 终端通过 HTTP 调用激活接口
4. TCP 完成终端绑定与状态更新
5. 终端建立 TDP Session
6. TDP 开始把这台终端该看到的 Projection 推给它

### 这里谁负责什么
- **TCP**：终端身份、激活、绑定
- **TDP**：会话建立、数据投递

---

## 6.2 场景二：配置下发

### 发生了什么
总部要给某批终端下发新的配置版本。

### 一般流程
1. TCP 创建一个 `TaskRelease`
2. 拆成多个 `TaskInstance`
3. TDP 为各终端写入新的 `Projection`
4. 终端连接 TDP 后感知到配置变化
5. 终端执行后再通过 HTTP 把结果回报给 TCP
6. TCP 更新任务执行状态和当前配置版本

### 这里谁负责什么
- **TCP**：决定谁该用哪个配置、记录最终结果
- **TDP**：把配置目标真正送给终端

---

## 6.3 场景三：热更新

### 发生了什么
你要灰度发布一个新版本，比如 `2.4.0`。

### 一般流程
1. TCP 指定：某一批终端的目标版本是 `2.4.0`
2. TCP 创建升级发布单
3. TDP 把升级目标作为数据分发给这些终端
4. 终端择机下载 / 安装 / 重启
5. 终端通过 HTTP 把升级结果告诉 TCP
6. TCP 记录当前版本和升级结果

### 这里谁负责什么
- **TCP**：决定升级策略、记录版本结果
- **TDP**：下发升级目标

### 小白重点理解
热更新不是“终端主动去问 TCP 今天要不要更新”。
更推荐是：
- TCP 做治理
- TDP 做下发
- 终端做执行
- 终端再回报 TCP

---

## 6.4 场景四：远程控制

### 发生了什么
运维想让某台终端执行一个远控动作，比如采集日志或触发诊断。

### 一般流程
1. TCP 创建远程控制任务
2. TDP 把任务下发给目标终端
3. 终端执行远控动作
4. 终端通过 HTTP 回报结果到 TCP
5. TCP 记录成功 / 失败 / 输出内容

### 这里谁负责什么
- **TCP**：下任务、收结果、做审计
- **TDP**：把任务送达终端

---

## 6.5 场景五：终端断网恢复

### 发生了什么
终端刚才断网了，期间错过了一批数据。

### 一般流程
1. 终端重新和 TDP 建立 Session
2. TDP 根据当前 Projection 和 ChangeLog 帮它追平
3. 终端恢复到最新状态

### 这里谁负责什么
- **TCP**：通常不处理连接细节
- **TDP**：负责数据补偿和状态追平

### 小白重点理解
这正是 TDP 的核心价值之一：
**终端不在线时，数据也不会丢成一团乱麻。**

---

## 7. 新手最容易混淆的几个点

## 7.1 Topic 和 Task 是一回事吗？
不是。

- **Task** 是“要做的事”
- **Topic** 是“某类数据的名字”

任务可以通过某个 Topic 下发，但 Topic 本身不是任务。

---

## 7.2 Projection 和 ChangeLog 是一回事吗？
不是。

- **Projection** 是“当前最新结果”
- **ChangeLog** 是“历史变化过程”

一个看现在，一个看历史。

---

## 7.3 终端在线状态和终端健康状态是一回事吗？
不是。

- **在线状态（presenceStatus）**：现在连没连上
- **健康状态（healthStatus）**：这台设备整体状态好不好

### 例子
- 一台终端可以在线，但是一直报错，所以健康状态差
- 一台终端可以离线，但设备本身没问题，只是暂时没联网

---

## 7.4 终端主动业务请求也走 TDP 吗？
通常不走。

比如：
- 下单
- 查询商品
- 查询库存
- 提交支付

这些一般走业务系统 HTTP 接口。

TDP 更适合：
- 被动接收状态
- 配置同步
- 任务下发
- 版本目标同步

---

## 11. 最后一段：你应该怎么记住 TCP 和 TDP？

如果你只想记一句话，请记这个：

- **TCP 管“终端应该做什么、当前是什么状态”**
- **TDP 管“终端应该被动看到什么数据，以及这些数据怎么稳定送到终端”**

或者更白话一点：

- **TCP 是指挥部**
- **TDP 是配送网**
- **终端是执行者**

只要你记住这三者关系，后面再看具体实体，就不会太乱。
