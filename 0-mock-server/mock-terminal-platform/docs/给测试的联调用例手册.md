# 给测试的联调用例手册

> 适用对象：测试同学、联调同学、实施验收同学。  
> 目标：告诉你如何围绕 Mock Terminal Platform 设计和执行联调测试，而不是只会“点几个按钮看看有没有报错”。

---

## 1. 这份手册解决什么问题

这份手册主要回答：

- 测试这个平台时，应该测什么
- 联调时，应该从什么角度设计用例
- 哪些场景属于 TCP
- 哪些场景属于 TDP
- 正常链路怎么测
- 异常链路怎么测
- 测试结果应该去哪里看

如果《后台使用手册》解决的是“怎么用后台”，那这份手册解决的是：

> “怎么把后台用成一套真正的联调测试工具。”

---

## 2. 测试同学必须先建立的 3 个认知

## 2.1 第一，测的不是页面，而是链路

这个后台虽然有界面，但真正要测的是链路：

- TCP 动作有没有真的引起状态变化
- TDP 有没有真的写出 Projection
- ChangeLog 有没有记录
- 审计日志有没有沉淀
- Mock 故障是否真的生效

所以不要只测：
- 按钮能不能点
- 页面会不会报错

更要测：
- 点完之后对象有没有联动变化
- 上下游是否一致
- 结果是否可观察

---

## 2.2 第二，正常链路和异常链路都必须测

只测正常流程是不够的。

这套平台的价值很大一部分来自：
- 可造场景
- 可造异常
- 可回看链路

因此测试时一定要覆盖：

- 正常创建
- 正常投递
- 正常回报
- 延迟
- 超时
- 失败
- 断线
- 恢复

---

## 2.3 第三，每个测试动作都要有“观察点”

测试一个动作时，至少要知道：

- 当前页应该发生什么变化
- 关联页应该发生什么变化
- 审计日志里应该出现什么记录
- 导出数据里是否应该体现

如果没有观察点，测试就会变成“感觉好像成功了”。

---

## 3. 测试视角下，后台 5 个页面分别负责什么

- **总览**：看全局数字和审计，适合做“结果回看”
- **TCP 控制面**：发治理动作，适合做“动作入口”
- **TDP 数据面**：看数据投递结果，适合做“链路验证”
- **场景引擎**：快速造测试环境，适合做“前置准备”
- **故障注入**：制造异常，适合做“异常用例”

测试时不要只停留在一个页面里。  
一般都是：

1. 在一个页面发起动作
2. 到另一个页面确认结果
3. 最后到总览或审计看落日志

---

## 4. 联调测试的 5 大类用例

建议你把联调用例分成这 5 类：

### 第一类：环境准备类
比如：
- 批量造终端
- 场景模板运行
- Topic 模板套用
- 故障模板导入

### 第二类：正常主链路类
比如：
- 激活终端
- 创建任务发布单
- TDP 建立 Session
- Projection 注入
- 结果回报

### 第三类：观察验证类
比如：
- 审计日志正确
- Projection 正确
- ChangeLog 正确
- trace 正确

### 第四类：异常联调类
比如：
- 故障规则命中
- 延迟
- 超时
- 伪结果
- 终端断开 Session

### 第五类：数据管理类
比如：
- 导入预检
- 导入模板
- 导出 JSON
- 下载导出文件

这样拆用例，会非常清楚。

---

## 5. 推荐的测试执行顺序

建议测试同学不要随机点后台。  
推荐按下面顺序执行：

### 第一步：准备环境
- 运行场景模板
- 确认终端、任务、Topic 基础数据存在

### 第二步：跑正常链路
- 终端激活
- 发布配置任务
- 建 Session
- 看 Projection / ChangeLog

### 第三步：跑异常链路
- 新增故障规则
- 模拟故障命中
- 看命中次数、状态变化、日志

### 第四步：做回归验证
- 审计日志
- 导出预览
- 关键对象详情

这样做的好处是：
- 不容易漏步骤
- 更像真实联调节奏
- 方便复用为标准测试剧本

## 6. 正常链路用例怎么设计

下面给你一套推荐模板。

---

## 6.1 用例模板：终端激活成功

### 用例名称
终端激活成功

### 前置条件
- 系统已启动
- 后台可正常打开
- 已生成至少 1 个激活码

### 操作步骤
1. 进入 `TCP 控制面`
2. 查看激活码列表
3. 在激活区输入可用激活码
4. 点击“执行终端激活”

### 预期结果
- 页面提示激活成功
- 终端列表出现新终端或终端状态更新
- 审计日志出现激活相关记录
- 导出数据中能看到对应终端对象

### 观察点
- TCP 页面终端列表
- 总览页统计数字
- 审计日志

---

## 6.2 用例模板：发布配置任务成功

### 用例名称
发布配置任务成功

### 前置条件
- 系统里已有终端
- 平台数据已刷新

### 操作步骤
1. 进入 `TCP 控制面`
2. 点击“发布配置任务”
3. 查看任务发布和任务实例
4. 切换到 `TDP 数据面`
5. 查看 Projection / ChangeLog

### 预期结果
- 新增 1 条任务发布单
- 新增若干任务实例
- TDP 中新增或更新 Projection
- TDP 中新增 ChangeLog
- 审计日志出现 `CREATE_TASK_RELEASE`

### 观察点
- TCP 任务发布列表
- TCP 任务实例列表
- TDP Projection 列表
- TDP ChangeLog 列表
- 审计日志

---

## 6.3 用例模板：Session 建立与心跳成功

### 用例名称
TDP Session 建立与心跳成功

### 前置条件
- 系统中存在有效终端

### 操作步骤
1. 进入 `TDP 数据面`
2. 输入终端 ID
3. 点击“建立 Session”
4. 查看 Session 列表
5. 点击“发送心跳”

### 预期结果
- Session 列表出现对应终端会话
- 状态为 `CONNECTED`
- 最近心跳时间更新
- 审计日志出现连接记录

### 观察点
- Session 列表
- 最近心跳字段
- 审计日志

---

## 6.4 用例模板：Topic 创建与 Projection 注入成功

### 用例名称
Topic 创建与 Projection 注入成功

### 前置条件
- TDP 页面可正常访问

### 操作步骤
1. 进入 `TDP 数据面`
2. 填写 Topic Key / 名称 / Scope
3. 点击“注册 Topic”
4. 填写 Projection Payload
5. 点击“注入 Projection”

### 预期结果
- Topic 列表出现新 Topic
- Projection 列表出现对应记录
- ChangeLog 记录对应变更
- 审计日志出现创建 Topic 记录

### 观察点
- Topic 列表
- Projection 列表
- ChangeLog 列表
- 审计日志

## 10. 测试记录建议怎么写

推荐测试同学每条联调用例都记录：

- 用例名称
- 前置条件
- 操作步骤
- 预期结果
- 实际结果
- 是否通过
- 关联截图
- 关联对象 ID
- 关联审计日志时间点

这样问题回溯会非常高效。

---

## 11. 建议测试同学重点关注的“高价值观察点”

每次测试时，优先关注：

### 1）对象数量变化
比如终端数、任务数、Topic 数

### 2）关键对象状态变化
比如：
- 终端状态
- 任务实例状态
- Session 状态

### 3）TDP 快照变化
Projection 是否新增、revision 是否变化

### 4）历史变化记录
ChangeLog 是否新增

### 5）审计日志
有没有把这次动作记录下来

## 13. 最后给测试同学的一句话建议

测试这套平台时，最重要的不是“按钮能不能点”，而是：

- 动作有没有真的发生
- 关联对象有没有真的变化
- 上下游有没有真的打通
- 问题有没有被真的观察到

你如果总能带着这 4 个问题去测，测试价值就会非常高。
