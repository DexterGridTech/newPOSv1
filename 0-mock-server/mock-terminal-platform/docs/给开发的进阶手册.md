# 给开发的进阶手册

> 适用对象：已经看过《小白手册》和《后台使用手册》，准备真正开始开发 Mock Terminal Platform 的前端、后端、终端接入、联调工具能力的同学。

---

## 1. 这份手册解决什么问题

如果《小白手册》解决的是“这套系统是什么”，那这份手册解决的是：

- 你作为开发，应该先理解什么
- 你应该从哪一层切入
- 什么改动属于 TCP，什么属于 TDP
- Mock 平台的开发边界是什么
- 哪些能力应该先做，哪些能力不要一开始就做得太重
- 你改完之后应该怎么自测和联调

这份手册不是 API 手册，也不是代码注释。  
它更像是一份“入场作战指南”。

---

## 2. 开发前，你必须先建立的 4 个认知

## 2.1 第一，Mock 平台不是临时玩具

虽然它叫 Mock，但它的目标不是“随便造几个假数据看看”。

它应该具备：
- **和真实 TCP / TDP 一致的对象模型**
- **和真实链路尽量一致的接口形态**
- **足够高的自由度，便于测试和联调**
- **足够强的观察能力，便于排查问题**

所以开发时不要把它当成“只有几条假接口”的 demo。

---

## 2.2 第二，Mock-only 能力是增强，不是污染主链路

在 Mock 平台里，会有一些真实生产系统不会开放的能力，比如：

- 强制修改终端状态
- 直接注入 Projection
- 直接写伪结果
- 模拟故障命中
- 快速批量造终端

这些能力是为了联调效率而存在的。  
但是开发时一定要记住：

- 标准链路仍然应该保留
- Mock 能力要通过专用接口和专用页面进入
- 不要把 Mock 逻辑写到正常链路中间，导致边界混乱

---

## 2.3 第三，TCP 和 TDP 不是两个“页面模块”，而是两个职责系统

开发时很容易犯一个错误：

> 只按页面分代码，不按职责分模型。

这会导致：
- 数据结构混乱
- 接口命名混乱
- 页面能跑，但长期不可维护

你应该始终从职责边界来思考：

### 属于 TCP 的问题
- 终端是谁
- 终端当前是什么状态
- 要发布什么任务
- 热更新目标是什么
- 配置目标是什么
- 结果回收如何记录

### 属于 TDP 的问题
- Topic 怎么定义
- Session 怎么通过真实 WebSocket 建立
- Projection 当前值是什么
- ChangeLog 怎么追踪
- Scope 怎么生效

页面只是展示层，职责边界才是根。

---

## 2.4 第四，开发的核心目标不是“多做功能”，而是“打通链路”

在这个系统里，最重要的不是单个页面做得多炫，而是链路能跑通。

你应该优先保证下面这些链路顺畅：

1. **TCP 创建任务 -> TDP 产生 Projection / ChangeLog**
2. **终端 Session 建立 -> TDP 可观察连接状态**
3. **终端结果回报 -> TCP 更新 TaskInstance**
4. **场景模板运行 -> 多对象联动变化**
5. **故障规则触发 -> 状态与日志可观察**

只要这些链路顺，Mock 平台就有价值。  
如果功能很多，但链路断裂，平台就会变成“看起来很丰富，实际上不好用”。

---

## 3. 你应该如何理解当前实现结构

当前 Mock Terminal Platform 可以简单理解为三层：

- **后台 Web**：管理员可视化界面
- **Server**：对象管理、接口编排、Mock 扩展能力
- **SQLite**：本地持久化与种子数据

它虽然不是正式生产服务，但在设计思路上已经尽量贴近真实系统：

- 对象模型明确
- TCP / TDP 分区明确
- 场景与故障作为扩展能力独立存在
- 审计与导出作为观察和交接能力存在

开发时建议始终按下面 4 个维度思考：

1. **对象模型是否清晰**
2. **接口边界是否清晰**
3. **页面动作是否真的对应后端动作**
4. **动作结果是否能被观察到**

### 当前代码实现里还需要特别记住的边界

- `基础资料` 已经是独立主入口，不再只是 TCP 页面里的附属能力
- `项目`、`租户`、`品牌` 是参考数据，`门店` 直接承载业务结果
- 当前系统不再维护 `品牌授权` 实体，也不再校验门店创建时的授权关系
- `终端机型 / 终端模板` 是当前后台里的标准术语，不再建议继续用 `Profile / Template` 作为用户侧主称呼
- `激活码` 是 12 位纯数字，只代表待激活凭证，不等于终端实例
- TDP 现在的主接入口已经不是纯 HTTP mock，而是 `WebSocket + HTTP 补偿` 的组合：
  - WebSocket：`/api/v1/tdp/ws/connect?terminalId=...&token=...`
  - HTTP：`/api/v1/tdp/terminals/:terminalId/snapshot`、`/api/v1/tdp/terminals/:terminalId/changes`
  - 旧的 `sessions/connect|heartbeat|disconnect` 仍保留，但更偏管理端兼容与调试用途

### 当前 WebSocket TDP 已实现到哪一层

当前已经落地：

- 严格鉴权：`terminalId + token` 必须匹配且未过期
- `HANDSHAKE -> SESSION_READY`
- `FULL_SNAPSHOT`
- `CHANGESET`
- `PING / PONG`
- Projection 写入后的服务端主动推送 `PROJECTION_CHANGED`
- 短时间连续变更会合并成 `PROJECTION_BATCH`
- 客户端 `ACK`
- Session 持久化记录 `lastDeliveredRevision / lastAckedRevision / lastAppliedRevision`
- Session 列表返回 `highWatermark / ackLag / applyLag`
- `tcp.task.release` 在客户端 ACK 后，会把对应 `task_instance.deliveryStatus` 推进到 `ACKED`
- 管理端可以主动向在线 session 发送 `EDGE_DEGRADED / SESSION_REHOME_REQUIRED`
- `REMOTE_CONTROL` 已改为 command 通道，不再走 projection/change log，而是直接推送 `COMMAND_DELIVERED`
- `print.command` 也可通过 command 通道下发，后台可在 `Command Outbox` 里查看投递和 ACK 状态

当前还没有完全落地：

- 全局单调 revision
- 更完整的 command catalog / TTL / replay 治理

## 5. 前端开发应该怎么切入

## 5.1 前端开发的核心目标

不是单纯把页面画出来，而是要做到：

- 让用户知道自己当前在操作什么对象
- 让用户知道操作完成了没有
- 让用户知道下一步应该去哪看结果
- 让用户能快速从问题定位到对象和链路

也就是说，前端在这个系统里本质上是一个“联调控制台”和“问题观察台”。

---

## 5.2 前端优先做什么，不要一上来做什么

### 先做
- 列表可筛选
- 对象详情易读
- 状态标签明显
- 重要动作有反馈
- 关键对象之间可跳转或联动观察
- 复杂 JSON 有良好展示和编辑体验

### 不要一上来做
- 很重的动画
- 很复杂的拖拽布局
- 过早抽象出特别庞大的组件系统
- 很多看起来高级但实际不帮助联调的装饰 UI

在这个后台里，“信息清楚、定位快、链路直观”比“花哨”更重要。

---

## 5.3 前端最适合的首批改进方向

### 方向一：可观测性增强
比如：
- 列表筛选更丰富
- 关键状态加彩色 Tag
- 详情里加字段解释
- 表格里增加常用快捷查看入口

### 方向二：联动路径更顺
比如：
- 从 TaskInstance 直接跳到对应 Projection
- 从 Projection 直接看到来源发布单
- 从审计日志直接跳到目标对象详情

### 方向三：JSON 体验更专业
比如：
- JSON 编辑区更大
- 格式化按钮
- 校验报错更友好
- 示例模板更直观

---

## 5.4 前端自测的最小清单

每次改完页面，至少自测下面这些：

- 按钮点击后有没有明确反馈
- 数据刷新后列表是否同步
- 没数据时空态是否正常
- 长 JSON 是否还能看清楚
- 操作后审计日志里是否能看到对应记录
- 页面之间联动是否符合预期

---

## 6. 后端开发应该怎么切入

## 6.1 后端开发的核心目标

后端在这个平台里的职责，不只是“返回点 JSON”。

而是要负责：
- 对象状态正确
- 链路动作正确
- Mock 增强能力边界正确
- 数据可观察
- 测试和联调可复现

如果后端只做“能返回数据”，但对象状态关系不严谨，后续联调会越来越乱。

---

## 6.2 后端最重要的 5 类对象关系

开发时你应该重点关注：

### 1）Terminal
核心主体对象，很多链路最终都会落到某台终端上。

### 2）TaskRelease / TaskInstance
一张发布单如何拆成多台终端上的实例，是理解 TCP 的关键。

### 3）Session
终端是否在线、是否活跃，是 TDP 观察的重要基础。

### 4）Topic / Projection / ChangeLog
这是 TDP 的核心三件套。

### 5）SceneTemplate / FaultRule
这是 Mock 扩展能力的核心对象。

只要你把这 5 类关系理顺，整个系统基本就不会乱。

---

## 6.3 后端开发优先级建议

### P1：先保证主链路正确
- 创建任务发布单
- 拆成任务实例
- 委托到 TDP
- 写 Projection
- 记录 ChangeLog
- 回收结果

### P2：再补查询与观察能力
- 列表查询
- 分页
- 筛选
- 详情
- trace

### P3：再补 Mock 增强能力
- 批量造数
- 场景模板执行
- 故障规则命中模拟
- 导入导出

### P4：最后再考虑复杂抽象
- Scene DSL 执行器
- 更复杂的故障 DSL
- 多维度数据回放

---

## 6.4 后端开发的常见陷阱

### 陷阱 1：为了方便，把所有逻辑都塞到一个接口里

这样短期快，长期一定乱。  
应该尽量保持：
- TCP 逻辑归 TCP
- TDP 逻辑归 TDP
- Mock 扩展逻辑独立

### 陷阱 2：只改数据库，不记录审计

这个平台是强联调平台，很多动作都需要可追溯。  
所以重要动作尽量都应该有审计。

### 陷阱 3：直接改最终状态，不留下过程痕迹

比如你只把 Projection 改了，但没有 ChangeLog。  
这样问题排查会很困难。

### 陷阱 4：Mock 能力和标准能力混写

比如：
- 正常任务下发接口里偷偷带上模拟故障逻辑
- 正常投递逻辑里塞“测试专用快捷路径”

这样很容易把系统边界弄脏。

---

## 6.5 后端自测最小清单

每次改完后端，至少自己验证：

- 对象状态是否真的变化
- 关联对象是否同步变化
- 审计日志是否存在
- 导出数据里是否看得到新对象
- 前端页面是否能读到对应变化
- 错误输入时是否返回明确错误信息

---

## 7. 终端 / 客户端开发应该怎么切入

## 7.1 终端开发最重要的认知

终端开发必须先把“主动”和“被动”分清楚：

### 主动
终端自己发请求，比如：
- 激活
- 查询
- 提交结果
- 请求业务数据

通常走 HTTP。

### 被动
后台有变化，终端应该感知，比如：
- 配置变化
- 升级目标变化
- 某条任务下发

通常通过 TDP。

很多终端接入混乱，都是因为这两类链路没分清楚。

---

## 7.2 终端开发最适合的切入点

### 第一步：先模拟一台终端最小接入

最小闭环建议是：
- 激活
- 建 Session
- 查看当前 Projection
- 回报结果

### 第二步：再接入一种任务

比如：
- 配置下发
- 升级任务

### 第三步：再补异常处理

比如：
- 超时
- 重连
- 断线恢复

终端接入最怕一开始想把所有东西都接全。  
建议从最小闭环开始。

---

## 8. 你应该如何规划下一步开发

这一章是最重要的“路线图”。

---

## 8.1 如果你的目标是“让平台更能看”

你应该做：
- 更好的筛选
- 更好的状态展示
- 更好的 JSON 展示
- 更完整的详情页
- 更直接的跳转链路

这是前端增强路线。

---

## 8.2 如果你的目标是“让平台更能用”

你应该做：
- 更多真实任务类型
- 更真实的场景模板
- 更真实的故障动作
- 更完整的 trace
- 更完整的导入导出对象

这是后端和联调增强路线。

---

## 8.3 如果你的目标是“让平台更接近真实生产链路”

你应该做：
- 更严格的对象状态流转
- 更接近真实系统的任务模型
- 更接近真实系统的协议字段
- 更接近真实系统的终端回报模型
- 更完整的热更新 / 配置 / 远控专属对象

这是体系化深化路线。

---

## 9. 推荐的开发任务拆解方式

不要按“页面”拆任务，建议按“能力链路”拆。

### 好的拆法
- 补齐 TaskInstance trace
- 给 Projection 增加筛选与详情
- 补一条热更新任务类型的完整闭环
- 实现场景运行记录
- 实现故障命中历史查看

### 不好的拆法
- 改一下这个页面
- 再加个按钮
- 顺便补个字段

前者有明确链路目标，后者容易越做越碎。

---

## 10. 开发完成后，你应该怎么验收自己做的东西

推荐使用“三层验收法”：

### 第一层：看当前页面
动作有没有成功反馈？数据有没有变化？

### 第二层：看关联页面
比如：
- TCP 动作后，要去看 TDP
- 场景动作后，要去看总览 / TCP / TDP
- 故障动作后，要去看审计和状态变化

### 第三层：看审计和导出
- 审计里有没有记录
- 导出里能不能看到对象变化

如果只看第一层，很多隐藏问题会漏掉。

---

## 11. 建议你现在就去做的 5 个进阶练习

### 练习 1：跑通一条 TCP -> TDP 链路
发布配置任务，然后去 TDP 看 Projection / ChangeLog。

### 练习 2：给 Projection 增加一个更好的详情查看体验
比如更清楚的字段解释或高亮。

### 练习 3：给任务实例 trace 增加更多上下文
比如来源发布单、目标终端、对应 Projection。

### 练习 4：补一条新的场景模板
比如“配置回滚演练”。

### 练习 5：补一条新的故障动作
比如“伪造成功但延迟上报”。

这 5 个练习做完，你就基本不是“接手者”，而是“真正理解这个平台的人”了。

---

## 12. 最后给开发同学的一句话建议

在这个项目里，最重要的不是“多做几个页面”，而是：

- 把对象关系做对
- 把链路做通
- 把观察能力做好
- 把 Mock 边界守住

只要你一直按这个方向推进，平台会越来越专业，而不会越来越乱。
