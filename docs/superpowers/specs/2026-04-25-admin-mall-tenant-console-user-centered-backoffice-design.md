# Admin Mall Tenant Console 用户视角后台设计

## 1. 本次重启的原因

上一版 `/customer` 的最大问题不是某个按钮位置错了，而是设计出发点错了：它把后台做成了“数据对象浏览器”，而不是业务用户每天用来找资料、确认关系、完成操作、追溯原因的管理后台。

这些误解必须明确记录，避免下一版继续犯同样的错。

1. 把实体表当成产品页面。页面只是把 `platform`、`project`、`store`、`role`、`menu` 等对象摆出来，却没有回答用户正在做什么。
2. 把通用组件套到所有实体。沙箱被显示“所属集团”，项目被显示“时区”，角色只能选一个权限，都是因为没有按实体语义设计。
3. 把技术字段直接给业务用户看。“natural scope”“payload data”“timezone”“platform-scoped permission”这类词可以存在于调试层，但不能出现在主工作流。
4. 详情被做成附属面板，而不是明确的查看动作。用户要求点击行进入详情弹窗，创建和修改也应是明确弹窗或独立编辑页。
5. 新建和修改表单只覆盖了极少字段。它们不能支撑真实实体属性，也不能表达 N 对 N 关系，例如角色与权限、菜单与商品、用户与角色。
6. 列表列太少，筛选太弱。业务用户无法回答“某集团下有哪些角色”“某项目下有哪些门店”“某菜单包含哪些商品”这类最常见问题。
7. 默认上下文没有解释。系统自动选第一个集团、品牌、门店，会让用户困惑“为什么是它”。上下文必须可见、可切换、可解释。
8. 多个列表并排显示增加了阅读负担。后台工作台更应该纵向堆叠，每张表保留足够宽度展示关键列。
9. 状态修改被误认为普通编辑。合同生效、菜单发布、角色停用、门店打烊都不是直接改字段，而是业务动作。
10. 详情省略了业务关键字段，却把原始 JSON 当补救。正确做法是详情默认展示业务字段，原始数据只作为隐藏的技术辅助。

下一版设计从用户旅途出发：用户先有问题，然后进入合适页面，通过筛选、关系导航、详情和动作完成任务。实体只是支撑任务的载体。

## 2. 设计目标

这个后台面向商业综合体、餐饮品牌、门店和权限管理的真实运营场景。它应当让用户快速回答四类问题：

1. 我现在在哪个业务范围内工作：沙箱、集团、项目、门店、品牌分别是谁。
2. 我想看的对象在哪里：某集团下所有角色、某项目下所有门店、某菜单下所有菜品。
3. 我能对这个对象做什么：新增、编辑、发布、停用、续签、授权、撤销、回滚。
4. 这个对象为什么是现在这样：合同决定门店归属，菜单版本决定可售商品，角色绑定决定用户权限，生命周期动作决定状态。

设计原则：

1. 场景优先。页面标题、列表列、筛选器、详情字段和按钮都从用户任务倒推。
2. 关系显性。所有关键关系都用可点击标签、关系图、关联列表表达，不能靠用户猜 ID。
3. 主流程业务化。新建、修改、发布、授权、续签等都用业务语言和业务动作。
4. 技术信息后置。原始数据、主题、修订号、投影、TDP 发布状态放在“技术信息”或“调试”区。
5. 列表为工作表，不是摘要卡。列表要承载比较、筛选、批量操作和快速判断。
6. 修改受约束。可编辑字段、不可编辑字段、只能通过业务动作变化的字段必须明确。
7. N 对 N 是一等交互。权限、角色、菜单商品、门店发布范围、商品规格/加料不能用单选糊弄。

## 3. 用户角色

### 3.1 平台运维管理员

关注系统是否可用、沙箱是否正确、集团平台是否已接入、投影和同步是否正常。

常见任务：

1. 创建和切换联调沙箱。
2. 创建集团平台并配置 ISV 授权。
3. 查看某个沙箱内已经准备了哪些业务资料。
4. 查看投影是否发布到终端平台。
5. 排查某个实体为什么没有到终端。

### 3.2 平台运营管理员

关注集团初始化、基础能力开通、跨集团模板和系统角色。

常见任务：

1. 查看所有集团平台。
2. 为集团开通功能点。
3. 查看系统角色和系统权限。
4. 审计高风险授权。

### 3.3 集团管理员

关注集团下的项目、品牌、租户、角色、权限、门店和经营关系。

常见任务：

1. 查看某集团下所有项目、门店、租户、品牌。
2. 创建项目、大区、品牌、租户。
3. 查看某集团下所有角色和权限。
4. 创建自定义角色并选择多个权限。
5. 查看一个用户在集团下到底拥有哪些权限。

### 3.4 购物中心管理员

关注具体项目下的门店、铺位、合同和入驻关系。

常见任务：

1. 查看某项目下全部门店和铺位状态。
2. 开设门店。
3. 入驻租户、选择品牌、创建经营合同。
4. 处理合同生效、到期、续签、提前终止。
5. 查看某门店当前由谁经营、合同何时到期。

### 3.5 品牌运营

关注品牌商品、规格、加料组、品牌菜单和门店发布。

常见任务：

1. 创建商品。
2. 配置规格、加料组、套餐结构和制作环节。
3. 创建品牌标准菜单。
4. 查看某份菜单包含哪些菜品。
5. 发布菜单到一批门店。
6. 查看菜单版本差异和回滚。

### 3.6 门店管理员

关注本门店营业状态、桌台、工作站、库存、价格、可售规则、员工授权。

常见任务：

1. 开店、打烊、暂停接单。
2. 管理桌台和工作站。
3. 查看本店当前菜单、沽清商品、库存和价格规则。
4. 创建员工并授予门店范围角色。
5. 查询某员工现在能做什么。

### 3.7 审计员和财务人员

关注授权记录、合同历史、法人主体、结算主体和高风险操作。

常见任务：

1. 查看某合同历史和附件。
2. 查看某角色的权限清单和使用者。
3. 查看某用户授权变更历史。
4. 导出法人主体和合同清单。

## 4. 信息架构

### 4.1 全局壳

全局壳只提供上下文，不塞业务指标。

左侧导航：

1. 工作台
2. 环境与集团
3. 项目与门店
4. 租户与合同
5. 品牌与商品
6. 菜单中心
7. 门店经营
8. 员工与权限
9. 审计与同步

顶部上下文条：

1. 当前沙箱：仅平台运维和联调场景显示。
2. 当前集团：大多数业务页面必选。
3. 当前项目：门店、合同、设施相关页面显示。
4. 当前品牌：商品、品牌菜单页面显示。
5. 当前门店：门店菜单、经营配置、员工授权页面显示。

上下文条规则：

1. 任何自动默认值都必须有说明，例如“默认选择最近访问的集团”或“默认选择该集团下第一个活跃项目”。
2. 用户可以随时切换上下文。
3. 切换上下文后，页面筛选和列表立即刷新。
4. 上下文不等于筛选器。上下文决定工作范围，筛选器用于在范围内找对象。

### 4.2 工作台

工作台只放跨模块待办和风险，不放每个页面都重复的统计。

内容：

1. 当前工作范围卡：显示用户当前沙箱、集团、项目、门店、品牌，并提供切换。
2. 待处理事项：合同即将到期、菜单待审核、Token 即将过期、权限审批、投影失败。
3. 快捷入口：新建项目、新建门店、创建合同、新建商品、创建角色、发布菜单。
4. 最近操作：用户最近查看和修改的对象。

不放：

1. 每个页面重复的全局实体数。
2. 技术型 outbox 详情。
3. 无法操作的装饰性指标。

## 5. 通用交互规范

### 5.1 列表页

列表页是主工作面，必须支持用户完成查找、比较、进入详情和批量动作。

固定结构：

1. 页面标题：用业务语言描述当前范围，例如“深圳湾万象城门店”。
2. 范围说明：告诉用户现在看的是哪个集团、项目、品牌或门店。
3. 筛选区：常用筛选默认展开，高级筛选可折叠。
4. 列表工具条：新建、导出、批量操作、列设置、保存视图。
5. 数据表：每行点击打开详情弹窗或详情页。
6. 分页和结果数：显示当前筛选命中数量。

列表列设计规则：

1. 第一列永远是业务名称，不是 ID。
2. 第二列通常是编码或位置，便于运营核对。
3. 中间列放用户作决策最关心的关系和状态。
4. 最后一列放状态和可执行动作。
5. ID、修订号、TDP topic 不在默认列中，放技术列设置里。

### 5.2 详情弹窗或详情页

轻量对象用详情弹窗，复杂对象用详情页。

详情通用结构：

1. 标题区：名称、状态、所属范围、关键关系。
2. 动作区：编辑、生命周期动作、复制、导出、查看历史。
3. 概览：业务字段，不省略关键字段。
4. 关系：上游和下游对象，例如门店关联项目、合同、租户、品牌、菜单、员工。
5. 业务明细：菜单分区和菜品、角色权限、商品规格和加料、合同条款。
6. 历史：创建、修改、发布、授权、撤销、回滚等事件。
7. 技术信息：原始数据、修订号、投影状态，默认收起。

关闭按钮规则：

1. 弹窗右上角只保留一个“关闭”按钮。
2. 底部只放主业务动作，不重复关闭。
3. 有未保存变更时，关闭前提示“放弃修改”。

### 5.3 新建

新建不是把所有字段堆进一个表单，而是按照业务步骤组织。

简单实体：使用弹窗。

1. 权限
2. 桌台
3. 工作站
4. 法人主体
5. 沙箱

复杂实体：使用向导或独立编辑页。

1. 集团平台：基础信息、联系人、ISV 授权、功能开通。
2. 项目：基础信息、大区地址、营业配置、渠道默认配置。
3. 门店：铺位信息、经营场景、联系人、设施初始配置。
4. 合同：选择门店、租户、品牌、法人主体、合同周期和费用。
5. 商品：基础信息、规格、加料、套餐、制作环节。
6. 菜单：基础信息、分区、菜品、发布范围、审核。
7. 角色：基础信息、权限选择、适用范围、风险校验。
8. 用户授权：选择用户、角色、范围、生效时间、审批。

### 5.4 修改

修改必须区分三类字段：

1. 可直接编辑：名称、描述、联系人、地址、展示排序、可见说明。
2. 创建后不可改：编码、系统 ID、合同编号、菜单历史版本、商品编码、门店铺位号。
3. 只能通过业务动作变化：状态、合同生效、合同终止、菜单发布、角色停用、用户授权撤销、商品上架/下架。

编辑入口：

1. 详情弹窗的“编辑”进入编辑弹窗。
2. 复杂对象的“编辑”进入详情页的编辑模式。
3. 业务动作单独成按钮，不混在字段编辑里。

### 5.5 N 对 N 关系选择器

任何 N 对 N 都不能用单选。

通用组件：

1. 左侧候选列表：支持搜索、分组、筛选。
2. 右侧已选择列表：显示数量、可排序、可移除。
3. 关系属性：每个关系可配置额外字段，例如菜单商品展示价、角色授权范围、生效时间。
4. 冲突提示：重复选择、权限冲突、不可用商品、已下架商品即时提示。

应用场景：

1. 角色选择多个权限。
2. 用户绑定多个角色。
3. 角色授权多个门店或大区范围。
4. 菜单分区包含多个商品。
5. 商品包含多个规格、多个加料组选项、多个制作环节。
6. 菜单发布到多个门店。

## 6. 关键用户旅途

### 6.1 从零准备一个集团联调环境

目标用户：平台运维管理员。

旅途：

1. 进入“环境与集团”。
2. 创建沙箱，填写用途、负责人、说明。
3. 在沙箱内创建集团平台，填写集团名称、联系人、联系电话、ISV 授权状态。
4. 创建项目，选择大区、城市、地址和商业体类型。
5. 开设门店，填写铺位、楼层、面积、经营场景。
6. 入驻租户和品牌。
7. 创建法人主体。
8. 创建经营合同，合同生效后门店自动显示当前租户和品牌。
9. 创建商品和菜单。
10. 创建员工、角色和授权。
11. 在“审计与同步”确认投影发布成功。

页面必须让用户看到当前进度，例如“项目已建、门店未签约、菜单未发布、员工未授权”。

### 6.2 查看某集团下所有角色

目标用户：集团管理员或审计员。

入口：

1. 全局上下文选择集团。
2. 进入“员工与权限”。
3. 默认打开“角色”标签。

筛选：

1. 集团：默认当前集团，可切换。
2. 角色类型：系统角色、自定义角色。
3. 状态：启用、停用、废弃。
4. 权限包含：输入权限名称或编码，反查包含该权限的角色。
5. 使用范围：平台、项目、门店、大区、标签。
6. 使用情况：未被使用、已授权用户、存在高风险权限。

列表列：

1. 角色名称。
2. 角色编码。
3. 类型：系统/自定义。
4. 权限数量。
5. 权限摘要：按模块展示前几个，例如“商品、库存、退款”。
6. 已授权用户数。
7. 授权范围数。
8. 高风险权限提示。
9. 状态。
10. 操作：查看、复制为自定义角色、停用。

详情：

1. 基础信息。
2. 权限清单，按模块分组。
3. 使用者列表：用户、门店/大区范围、生效时间、失效时间。
4. 角色对比入口：可选 2 到 3 个角色比较权限差异。
5. 审计历史。

新建自定义角色：

1. 选择创建方式：从零开始、基于系统角色。
2. 填写角色名称、编码、描述。
3. 使用权限多选矩阵，按模块选择多个权限。
4. 系统显示已选择权限数量和风险等级。
5. 保存后状态为启用。

可修改：

1. 自定义角色名称。
2. 描述。
3. 权限集合。

不可修改：

1. 系统角色不可编辑。
2. 角色编码创建后不可改。
3. 角色类型不可改。
4. 已有用户使用的角色不能删除，只能停用或复制新角色替换。

### 6.3 查看某菜单包含哪些菜品

目标用户：品牌运营、门店管理员、审计员。

入口：

1. 从“菜单中心”进入。
2. 选择品牌菜单或门店菜单。
3. 列表点击菜单名称进入详情。

筛选：

1. 菜单类型：品牌标准菜单、门店有效菜单。
2. 品牌。
3. 门店。
4. 审核状态。
5. 生命周期状态。
6. 生效时间。
7. 是否包含某商品。

列表列：

1. 菜单名称。
2. 类型。
3. 品牌/门店。
4. 分区数。
5. 商品数。
6. 审核状态。
7. 当前版本。
8. 生效时间。
9. 最近发布人/发布时间。
10. 操作：查看内容、复制、提交审核、发布、版本历史。

菜单详情：

1. 概览：菜单名称、品牌/门店、渠道、生效时间、版本、状态。
2. 菜品结构：左侧分区树，右侧该分区菜品表。
3. 菜品表列：菜品名称、商品类型、基础价、菜单展示价、规格数、加料组数、是否推荐、是否可售、来源。
4. 搜索“菜品名称/商品编码/品类”。
5. 支持在详情内切换“按分区看”和“平铺全部菜品”。
6. 支持导出菜单菜品清单。
7. 支持对比两个版本的新增、删除、价格变化和排序变化。
8. 技术信息收起显示原始结构。

新增品牌菜单：

1. 填写菜单名称、适用渠道、生效时间。
2. 添加分区。
3. 每个分区通过多选商品选择器添加商品。
4. 对每个菜单商品可设置推荐、展示名、展示价、排序。
5. 保存草稿。
6. 提交审核。

新增门店菜单：

1. 选择来源：继承品牌菜单、复制门店历史菜单、完全自定义。
2. 选择继承方式：完全继承、允许本店调整、完全自定义。

2. 未发布版本的生效时间。

不可修改：

1. 已发布菜单不能直接改内容，只能复制成新版本。
2. 历史版本不能改，只能查看、对比、回滚为新版本。
3. 下架商品不能新增进待发布菜单。

### 6.5 为新员工授权

## 9. 关键交互细节

### 9.1 搜索和筛选

每张核心列表至少支持：

1. 关键词搜索。
2. 状态筛选。
3. 所属范围筛选。
4. 关系筛选。
5. 保存当前筛选为视图。

示例：

1. 角色列表支持“包含权限”。
2. 菜单列表支持“包含商品”。
3. 门店列表支持“合同即将到期”。
4. 商品列表支持“被菜单引用”。
5. 用户列表支持“拥有角色”。

### 9.2 关系跳转

详情内所有关系都应可点击。

示例：

1. 门店详情点击合同，打开合同详情。
2. 合同详情点击品牌，打开品牌详情。
3. 菜单详情点击商品，打开商品详情。
4. 角色详情点击权限，打开权限详情。
5. 用户详情点击角色绑定，打开绑定详情。

### 9.3 原始数据入口

业务页面默认不展示原始数据。

入口形式：

1. 详情弹窗底部“技术信息”折叠区。
2. 审计与同步页的“原始数据查看器”。
3. 开发模式下可以显示“查看原始数据”按钮。

原始数据展示内容：

1. 聚合原始数据。
2. 修订号。
3. 最近事件。
4. 投影主题和发布状态。

### 9.4 空状态

空状态要告诉用户下一步，而不是只说“暂无数据”。

示例：

1. 项目下没有门店：显示“先开设门店，再通过合同绑定租户和品牌”。
2. 菜单没有商品：显示“先添加分区，再从商品库选择商品”。
3. 角色没有权限：显示“至少选择一个权限才能保存角色”。
4. 门店没有合同：显示“门店处于空置状态，创建合同后才能经营”。

### 9.5 风险提示

高风险动作必须解释影响。

示例：

1. 停用角色：提示影响多少用户。
2. 终止合同：提示门店将失去当前租户和品牌，菜单可能失效。
3. 发布菜单：提示覆盖哪些门店当前菜单。
4. 停用商品：提示影响哪些菜单。
5. 撤销授权：提示用户将失去哪些权限。

## 11. 下一版原型验收清单

下一版 UI 不是“能看见数据”就算完成，至少要通过这些场景验收。

1. 用户能在 10 秒内查看某集团下所有角色，并按权限反查角色。
2. 用户能打开任意菜单，直接看到分区和菜品，不需要看原始数据。
3. 用户能查看某项目下所有门店，列表列能表达铺位、租户、品牌、合同、菜单、营业状态。
4. 用户能创建角色并一次选择多个权限。
5. 用户能创建菜单并一次选择多个商品。
6. 用户能查看员工的最终有效权限，并看到权限来源。
7. 用户能理解当前上下文为什么是这个集团/项目/门店。
8. 业务页面不出现 `payload`、`natural scope`、`timezone`、`platform-scoped` 这类技术语言。
9. 沙箱详情不显示所属集团，也不显示没有业务意义的资料数。
10. 已发布菜单、已生效合同、系统角色等不可直接编辑的对象必须有清晰约束说明。
11. 所有多列表页面纵向堆叠，保证表格宽度足够。
12. 每个详情页都默认展示业务关键字段，原始数据只在技术信息中展开。

## 12. 评审

### 12.1 产品方向评审

结论：通过。文档已经从数据模型页面转向用户任务页面，覆盖了“查集团角色”“查菜单菜品”“查项目门店”这些高频场景，也明确了每个实体的列表列、筛选、详情、新建、修改和不可改边界。

关键改进：

1. 页面不再按后端对象机械堆叠，而是按用户任务组织。
2. 关键关系被放到列表和详情的一等位置。
3. N 对 N 关系有明确交互，不再用单选替代。
4. 状态变化被收敛为业务动作，不再直接改状态字段。
5. 技术信息被隔离到调试区。

### 12.2 交互完整性评审

结论：基本通过，但后续原型要重点验证列表密度和复杂编辑器。

需要在原型中验证：

1. 菜单编辑器的分区树和菜品表是否足够顺手。
2. 权限多选矩阵是否能支撑大量权限。
3. 项目门店列表列较多时，是否需要列冻结和列设置。
4. 合同详情和门店详情之间的关系跳转是否清晰。
5. 编辑弹窗和独立编辑页的边界是否自然。

### 12.3 领域完整性评审

结论：通过第一版设计评审。组织、合同、商品菜单、门店经营、IAM 的核心关系都已覆盖。

仍需补充的二期内容：

1. 外部 IdP、用户组、资源标签、SoD、高风险审批的完整页面。
2. 渠道集成后的渠道店铺、渠道菜单、渠道商品映射。
3. 结算、对账、预订、履约等非本轮后台范围。
4. 终端设备与 TDP 订阅的完整运维页面。

### 12.4 可实施性评审

结论：可以作为下一版 `/customer` 重建 PRD，但实现应分阶段。

推荐阶段：

1. 第一阶段：环境、集团、项目、门店、合同、角色、权限、商品、菜单的核心列表/详情/新建。
2. 第二阶段：复杂编辑器，包含角色权限矩阵、菜单分区菜品编辑器、商品规格加料编辑器。
3. 第三阶段：生命周期动作、审计历史、版本对比、权限测试。
4. 第四阶段：投影诊断、终端同步、外部 IdP 和高风险审批。

### 12.5 风险

1. 如果继续从已有 mock API 能力倒推 UI，仍然会退化成数据编辑器。
2. 如果列表列配置不足，用户还是无法完成“找对象”和“比较对象”。
3. 如果详情默认字段继续省略，用户会被迫看原始数据。
4. 如果复杂关系不做专门组件，角色、菜单、商品都会继续粗糙。
5. 如果生命周期动作和字段编辑混在一起，业务真相会被破坏。

### 12.6 最终评审意见

这份文档可以作为新一版用户视角后台设计基线。下一步不应继续修补旧代码，而应先按本文档切出第一阶段原型范围，并为三个验收场景优先做高保真交互：

1. 集团角色和权限管理。
2. 菜单内容查看与编辑。
3. 项目门店和合同关系查看。
