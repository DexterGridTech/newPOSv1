# 01-架构与边界设计

> 版本：2.0 | 状态：详细设计稿 | 日期：2026-04-06

---

## 文档说明

本文档详细阐述 `terminal-data` 包的架构定位、边界划分、与旧包的关系、内部子域划分、与 1-kernel 框架的对齐方式、依赖策略、分层设计以及外部模块的使用方式。

## 1. 总体定位

### 1.1 核心定位

`terminal-data` 的定位是：

> **面向终端侧的 TCP/TDP 客户端通用数据内核**

它位于 `1-kernel` 的 core 层，既不是业务模块，也不是 UI 模块，而是终端运行时最底层的数据与同步内核之一。

### 1.2 在系统中的位置

```
┌─────────────────────────────────────────────────────────┐
│                    4-assembly 整合层                      │
│              (mixc-retail-v1, 其他项目)                   │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                    3-adapter 适配层                       │
│         (RN 适配、Electron 适配、设备能力注入)             │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                     2-ui UI 层                           │
│              (React Native 组件、页面)                    │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                  1-kernel 业务逻辑层                      │
│  ┌───────────────────────────────────────────────────┐  │
│  │          1.1-cores 核心包                          │  │
│  │  ┌─────────────────────────────────────────────┐  │  │
│  │  │  terminal-data (本包)                        │  │  │
│  │  │  - TCP/TDP 客户端能力                        │  │  │
│  │  │  - 终端身份与绑定                            │  │  │
│  │  │  - 数据同步与投递                            │  │  │
│  │  └─────────────────────────────────────────────┘  │  │
│  │  ┌─────────────────────────────────────────────┐  │  │
│  │  │  base, interconnection, task, navigation... │  │  │
│  │  └─────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 1.3 核心价值

1. **统一的终端数据内核**
   - 为所有终端形态（RN/Electron/Web）提供统一的数据层
   - 封装 TCP/TDP 协议细节
   - 提供一致的状态管理接口

2. **与服务端架构对齐**
   - 客户端概念与服务端设计文档一致
   - 降低前后端沟通成本
   - 便于理解和维护

3. **跨平台能力**
   - 通过 adapter 抽象平台差异
   - 核心逻辑可在多平台复用
   - 降低多端开发成本

4. **可扩展性**
   - 清晰的模块边界
   - 灵活的扩展点
   - 便于后续功能增强

## 2. 与旧 terminal 包的关系

### 2.1 旧包的本质定位

旧 `1-kernel/1.1-cores/terminal` 包本质上是：

1. **终端身份模块**
   - 设备信息采集
   - 终端激活与反激活
   - token 管理

2. **kernel HTTP + WS 协议客户端**
   - kernel API 封装
   - kernel WebSocket 长连接
   - 心跳与消息接收

3. **unitData 本地状态机**
   - unitData 组数据同步
   - 本地 unitData 缓存
   - 数据变更通知

4. **老组织树语义容器**
   - operatingEntity 管理
   - terminal / model / entity 树状关系
   - rootPath 优先级覆盖

### 2.2 新包替代范围

新 `terminal-data` **首期替代**旧包中的以下核心职责：

#### 替代内容

1. **终端身份与激活状态**
   - ✅ 设备信息采集 → `DeviceInfoAdapter`
   - ✅ 终端激活流程 → `ActivationCoordinator`
   - ✅ 激活状态管理 → `terminalIdentity` slice

2. **Token 与认证态**
   - ✅ Token 存储 → `CredentialRepository`
   - ✅ Token 刷新 → `refreshTerminalCredential` command
   - ✅ 认证态管理 → `terminalIdentity` slice

3. **TDP 会话与连接**
   - ✅ Session 连接 → `TdpSessionClient`
   - ✅ 心跳保活 → `heartbeatTdpSession` command
   - ✅ 断线重连 → `SessionCoordinator`

4. **终端本地数据快照与投递同步**
   - ✅ Projection 同步 → `ProjectionSyncCoordinator`
   - ✅ Change 增量推送 → `TdpSyncActor`
   - ✅ 本地缓存 → `ProjectionRepository`

5. **TCP 任务接收与结果回报**
   - ✅ 任务接收 → `tdpInbox` slice
   - ✅ 结果上报 → `reportTaskResult` command
   - ✅ 状态上报 → `TcpControlClient`

#### 替代方式

| 旧包功能 | 新包实现 | 说明 |
|---------|---------|------|
| 设备激活 | `activateTerminal` command | 使用 TCP 激活接口 |
| token 管理 | `terminalIdentity` slice | 显式的 credential 状态 |
| kernel WS | `TdpSessionClient` | 使用 TDP WebSocket 协议 |
| unitData 同步 | `tdpProjection` slice | 使用 Projection/Change 模型 |
| 远程消息 | `tdpInbox` slice | 使用 Delivery Envelope 模型 |
| 状态上报 | `reportTaskResult` command | 使用 TCP Report 接口 |

### 2.3 暂不直接替代范围

以下内容**不在本次新内核首期**中作为核心模型落入：

#### 不替代内容

1. **旧 unitData 的 group 模型**
   - 原因：与新 TDP 的 Topic/Projection 模型不一致
   - 方案：可提供兼容层，但不作为核心模型

2. **operatingEntity / model 的历史树覆盖语义**
   - 原因：与新的显式实体模型不一致
   - 方案：使用 `TerminalBindingContext` 替代

3. **老 kernel WS 特定消息协议**
   - 原因：已被 TDP WebSocket 协议替代
   - 方案：通过 adapter 适配旧协议（如需要）

4. **项目级远程命令语义**
   - 原因：应由上层业务模块处理
   - 方案：新包只提供通用的任务投递机制

#### 迁移策略

对于暂不替代的内容，提供以下迁移路径：

1. **兼容层模块**
   - 创建独立的 `@impos2/kernel-compat-terminal-legacy` 包
   - 提供 unitData 到 Projection 的转换
   - 提供 operatingEntity 到 BindingContext 的映射

2. **逐步迁移**
   - 新功能使用新包
   - 老功能保持旧包
   - 逐步替换旧包调用

3. **文档支持**
   - 提供详细的迁移指南
   - 提供代码示例
   - 提供常见问题解答

### 2.4 新旧包对比

| 维度 | 旧 terminal 包 | 新 terminal-data 包 |
|------|---------------|-------------------|
| 核心模型 | Unit / unitData | TerminalIdentity / ProjectionRecord |
| 组织关系 | 树状覆盖（rootPath） | 显式引用（TenantRef/BrandRef/StoreRef） |
| 协议 | kernel HTTP + WS | TCP HTTP + TDP WebSocket |
| 数据同步 | unitData group | Topic / Projection / Change |
| 任务投递 | 远程消息 | Task Delivery Envelope |
| 状态管理 | 混合在一起 | 分域管理（identity/tcp/tdp/runtime） |
| 类型安全 | 弱类型（any） | 强类型（TypeScript） |
| 平台依赖 | 直接依赖 RN | 通过 adapter 抽象 |
| 扩展性 | 难以扩展 | 清晰的扩展点 |

## 3. 新内核包含什么

新 `terminal-data` 包含以下 5 个核心子域：

### 3.1 Identity 子域

**职责**：
- 设备信息采集后形成终端身份上下文
- 终端激活态管理
- Token / Credential 管理
- 绑定上下文管理

**核心实体**：
- `TerminalIdentity`：终端身份
- `TerminalCredential`：终端凭证
- `TerminalBindingContext`：终端绑定上下文
- `TenantRef` / `BrandRef` / `StoreRef` / `TerminalProfileRef`：组织引用

**核心状态**：
- `terminalIdentity` slice：身份与凭证
- `terminalBinding` slice：绑定上下文

**核心命令**：
- `activateTerminal`：激活终端
- `refreshTerminalCredential`：刷新凭证
- `resetTerminalIdentity`：重置身份

**核心 Actor**：
- `IdentityActor`：处理激活、刷新、重置

### 3.2 TCP Client 子域

**职责**：
- 与 TCP 服务的 HTTP 通信
- 激活 / Token Refresh / Bootstrap / 结果上报
- 配置、版本、任务结果等控制面交互

**核心实体**：
- `TcpTaskDeliveryEnvelope`：任务投递信封
- `TerminalReportPayload`：终端上报载荷
- `TerminalRuntimeSnapshot`：终端运行时快照

**核心状态**：
- `tcpClient` slice：TCP 客户端状态

**核心命令**：
- `fetchTcpBootstrap`：获取引导数据
- `reportTaskResult`：上报任务结果
- `reportConfigApplied`：上报配置应用
- `reportVersionApplied`：上报版本应用
- `reportLifecycleState`：上报生命周期状态

**核心 Actor**：
- `TcpControlActor`：处理 TCP 控制面交互

### 3.3 TDP Client 子域

**职责**：
- Session Connect / Heartbeat / Disconnect
- 连接重建
- Snapshot / Change Catch-up
- Topic / Projection 消费与本地缓存

**核心实体**：
- `TdpSessionDescriptor`：TDP 会话描述符
- `TopicDefinition`：Topic 定义
- `ProjectionRecord`：Projection 记录
- `ChangeRecord`：Change 记录

**核心状态**：
- `tdpSession` slice：TDP 会话状态
- `tdpProjection` slice：Projection 缓存
- `tdpInbox` slice：投递收件箱

**核心命令**：
- `connectTdpSession`：连接 TDP 会话
- `heartbeatTdpSession`：心跳保活
- `disconnectTdpSession`：断开连接
- `reconnectTdpSession`：重新连接
- `syncProjectionSnapshot`：同步 Projection 快照
- `catchUpProjectionChanges`：追赶 Change 增量

**核心 Actor**：
- `TdpSessionActor`：处理会话管理
- `TdpSyncActor`：处理数据同步

### 3.4 Runtime State 子域

**职责**：
- 本地终端运行时数据
- 终端目标版本、目标配置、任务收件箱
- 最近同步时间、最近错误、最近会话结果

**核心实体**：
- `DesiredConfigState`：期望配置状态
- `DesiredVersionState`：期望版本状态
- `TerminalCapabilityState`：终端能力状态
- `DeliveryInboxItem`：投递收件箱项
- `SyncCheckpoint`：同步检查点

**核心状态**：
- `terminalRuntime` slice：运行时状态

**核心命令**：
- `updateDesiredConfig`：更新期望配置
- `updateDesiredVersion`：更新期望版本
- `consumeDeliveryEnvelope`：消费投递信封
- `markDeliveryConsumed`：标记投递已消费

**核心 Actor**：
- `RuntimeActor`：处理运行时状态更新

### 3.5 Derived View 子域

**职责**：
- 将显式实体引用构造成客户端派生树
- 构造 Scope 解析辅助结构
- 提供 UI / Selector 更友好的组织视图

**核心实体**：
- `DerivedTreeNode`：派生树节点
- `DerivedTreeNodeType`：派生树节点类型

**核心状态**：
- `terminalDerivedTree` slice：派生树视图

**核心命令**：
- `rebuildDerivedTree`：重建派生树
- `refreshScopeView`：刷新 Scope 视图

**核心 Actor**：
- `DerivedTreeActor`：处理派生树构建

## 4. 新内核不包含什么

为了保持边界清晰，新内核明确**不包含**以下内容：

### 4.1 不包含业务域解释器

**说明**：
- 不包含菜品域解释
- 不包含订单域解释
- 不包含履约域解释
- 不包含支付域解释

**原因**：
- 这些应由更上层业务模块处理
- 内核应该保持通用性
- 避免内核与业务耦合

**示例**：
```typescript
// ❌ 不应该在内核中
interface MenuProjection {
  menuId: string
  dishName: string
  price: number
  category: string
  // ... 餐饮业务特有字段
}

// ✅ 应该在业务模块中
// 内核只提供通用的 ProjectionRecord
interface ProjectionRecord {
  projectionId: string
  topicKey: string
  scopeType: TdpScopeType
  scopeKey: string
  payload: Record<string, unknown>  // 通用 payload
}

// 业务模块自行解释
const menuProjection = projection.payload as MenuProjection
```

### 4.2 不包含项目定制终端动作实现

**说明**：
- 不包含某项目专属打印参数处理
- 不包含某项目专属远控命令执行器
- 不包含某项目专属设备能力封装

**原因**：
- 项目定制逻辑应在整合层实现
- 内核应该保持项目无关
- 便于跨项目复用

**示例**：
```typescript
// ❌ 不应该在内核中
async function executePrintReceipt(params: PrintReceiptParams) {
  // 项目特定的打印逻辑
}

// ✅ 应该在整合层中
// 内核只提供通用的任务投递机制
interface TcpTaskDeliveryEnvelope {
  taskType: string
  payload: Record<string, unknown>
}

// 整合层实现具体的任务处理器
const taskHandlers = {
  'PRINT_RECEIPT': handlePrintReceipt,
  'REBOOT_DEVICE': handleRebootDevice,
  // ...
}
```

### 4.3 不包含服务端权威组织树

**说明**：
- 客户端可以派生树，但服务端不需要为了这个内核反向改造成树权威模型
- 树是客户端视图层的辅助结构，不是领域模型

**原因**：
- 树模型会导致服务端领域模型复杂化
- 树模型会导致 TDP Scope 解析复杂化
- 树模型会重新引入 rootPath 优先级体系

**设计原则**：
- **服务端权威模型**：显式实体引用（Tenant/Brand/Store/Terminal）
- **客户端派生树**：基于显式实体构建的视图结构

### 4.4 不包含 UI 层逻辑

**说明**：
- 不包含页面组件
- 不包含导航逻辑
- 不包含表单验证
- 不包含样式定义

**原因**：
- 该包是 `1-kernel` 的 core 包，只负责数据层
- UI 层应该通过 hooks 和 selectors 使用内核
- 保持内核的纯粹性

**使用方式**：
```typescript
// ✅ UI 层通过 hooks 使用内核
function TerminalStatusScreen() {
  const identity = useTerminalIdentity()
  const sessionStatus = useTdpSessionStatus()

  return (
    <View>
      <Text>Terminal ID: {identity.terminalId}</Text>
      <Text>Session Status: {sessionStatus}</Text>
    </View>
  )
}
```

## 5. 为什么采用"显式实体 + 派生树"的混合模式

这是本设计的核心边界选择。

### 5.1 显式实体的好处

**显式实体模型**：
```typescript
interface TenantRef {
  tenantId: string
  code?: string
  name: string
}

interface BrandRef {
  brandId: string
  tenantId: string
  code?: string
  name: string
}

interface StoreRef {
  storeId: string
  tenantId: string
  brandId: string
  code?: string
  name: string
}

interface TerminalBindingContext {
  terminalId: string
  tenant: TenantRef
  brand: BrandRef
  store: StoreRef
  profile: TerminalProfileRef
}
```

**好处**：

1. **语义清楚**
   - 一眼就能看出是 Tenant、Brand 还是 Store
   - 不需要猜测 `Unit` 的实际类型
   - IDE 可以提供准确的类型提示

2. **接口清楚**
   - 每个实体都有明确的字段定义
   - 不会出现"某个字段在某些情况下才有"的问题
   - 便于编写类型安全的代码

3. **与 TCP/TDP 服务端设计一致**
   - 服务端使用显式实体模型
   - 客户端与服务端概念对齐
   - 降低沟通成本

4. **不容易把组织关系和能力关系混淆**
   - 组织关系：Tenant → Brand → Store
   - 能力关系：Terminal → Profile → Capabilities
   - 两者分开建模，不会混淆

### 5.2 派生树的好处

**派生树模型**：
```typescript
enum DerivedTreeNodeType {
  ROOT = 'ROOT',
  TENANT = 'TENANT',
  BRAND = 'BRAND',
  STORE = 'STORE',
  TERMINAL = 'TERMINAL',
  PROFILE = 'PROFILE',
}

interface DerivedTreeNode {
  nodeId: string
  nodeType: DerivedTreeNodeType
  refId: string
  label: string
  parentNodeId?: string
  childrenNodeIds: string[]
  metadata?: Record<string, unknown>
}
```

**好处**：

1. **兼容旧终端包用户的心智**
   - 旧包用户习惯树形结构
   - 提供派生树可以降低迁移成本
   - 不需要完全改变使用习惯

2. **给 UI 提供树形浏览能力**
   - UI 层可以直接渲染树形结构
   - 不需要在 UI 层重新构建树
   - 提升 UI 开发效率

3. **给 Scope 解析提供更直观视图**
   - Scope 解析需要知道组织层级关系
   - 树形结构可以快速查找父子关系
   - 提升 Scope 解析性能

4. **方便做按组织节点的选择与过滤**
   - 可以快速找到某个 Brand 下的所有 Store
   - 可以快速找到某个 Store 下的所有 Terminal
   - 提升查询效率

### 5.3 为什么不让树成为权威模型

**如果树成为权威模型，会导致**：

1. **服务端领域模型复杂化**
   - 需要维护树的父子关系
   - 需要处理树的增删改查
   - 需要处理树的一致性问题

2. **TDP Scope 解析复杂化**
   - Scope 需要基于树路径解析
   - 需要处理树路径的变更
   - 需要处理树路径的冲突

3. **Profile 等非组织节点被强行塞进树**
   - Profile 不是组织节点，但需要挂在树上
   - 导致树的语义不清晰
   - 导致树的结构复杂化

4. **重新引入旧 rootPath 优先级体系**
   - 需要处理树路径的优先级
   - 需要处理树路径的覆盖
   - 重新引入旧包的复杂性

**因此**：
- **服务端权威模型保持显式实体**
- **客户端内部允许派生树视图**
- 这是最优平衡

### 5.4 派生树的构建时机

**构建时机**：
1. 终端激活成功后
2. 绑定上下文变更后
3. 显式调用 `rebuildDerivedTree` 命令后

**构建逻辑**：
```typescript
function buildDerivedTree(context: TerminalBindingContext): DerivedTreeNode[] {
  const nodes: DerivedTreeNode[] = []

  // 创建 ROOT 节点
  nodes.push({
    nodeId: 'root',
    nodeType: DerivedTreeNodeType.ROOT,
    refId: 'root',
    label: 'Root',
    childrenNodeIds: [context.tenant.tenantId]
  })

  // 创建 TENANT 节点
  nodes.push({
    nodeId: context.tenant.tenantId,
    nodeType: DerivedTreeNodeType.TENANT,
    refId: context.tenant.tenantId,
    label: context.tenant.name,
    parentNodeId: 'root',
    childrenNodeIds: [context.brand.brandId]
  })

  // 创建 BRAND 节点
  nodes.push({
    nodeId: context.brand.brandId,
    nodeType: DerivedTreeNodeType.BRAND,
    refId: context.brand.brandId,
    label: context.brand.name,
    parentNodeId: context.tenant.tenantId,
    childrenNodeIds: [context.store.storeId]
  })

  // 创建 STORE 节点
  nodes.push({
    nodeId: context.store.storeId,
    nodeType: DerivedTreeNodeType.STORE,
    refId: context.store.storeId,
    label: context.store.name,
    parentNodeId: context.brand.brandId,
    childrenNodeIds: [context.terminalId]
  })

  // 创建 TERMINAL 节点
  nodes.push({
    nodeId: context.terminalId,
    nodeType: DerivedTreeNodeType.TERMINAL,
    refId: context.terminalId,
    label: `Terminal ${context.terminalId}`,
    parentNodeId: context.store.storeId,
    childrenNodeIds: []
  })

  return nodes
}
```

## 6. 与 1-kernel 框架的对齐方式

根据旧 `terminal` 与其他 core 包的实现方式，新包必须遵循如下结构：

### 6.1 AppModule 结构

新包将以 `AppModule` 形式导出：

```typescript
import { AppModule } from '@impos2/kernel-core-base'

export const terminalDataModule: AppModule = {
  name: 'kernel.core.terminal-data',
  version: '0.0.1',
  slices: terminalDataSlices,
  middlewares: terminalDataMiddlewares,
  epics: terminalDataEpics,
  commands: terminalDataCommands,
  actors: terminalDataActors,
  errorMessages: terminalDataErrorMessages,
  parameters: terminalDataParameters,
  dependencies: [
    kernelCoreBaseModule,
    kernelCoreInterconnectionModule
  ],
  modulePreSetup: terminalDataModulePreSetup,
}
```

**关键字段说明**：

1. **name**：模块名称，遵循 `kernel.core.xxx` 命名规范
2. **version**：模块版本，遵循语义化版本
3. **slices**：Redux slices 数组，定义所有状态
4. **middlewares**：Redux middlewares 数组，处理副作用
5. **epics**：RxJS epics 数组，处理异步流
6. **commands**：命令定义对象，暴露所有动作
7. **actors**：Actor 数组，处理命令执行
8. **errorMessages**：错误消息定义对象
9. **parameters**：参数定义对象
10. **dependencies**：依赖的其他模块
11. **modulePreSetup**：模块预初始化函数

### 6.2 slices 负责状态权威

**原则**：
- 所有终端客户端状态必须以 slice 为权威来源
- 不能散落在 foundation 单例中
- 不能散落在全局变量中

**示例**：
```typescript
// ✅ 正确：状态在 slice 中
const terminalIdentitySlice = createSlice({
  name: 'terminalIdentity',
  initialState: {
    terminalId: null,
    deviceId: null,
    activationStatus: TerminalActivationStatus.UNACTIVATED,
    credential: null,
  },
  reducers: {
    setTerminalId: (state, action) => {
      state.terminalId = action.payload
    },
    setActivationStatus: (state, action) => {
      state.activationStatus = action.payload
    },
    setCredential: (state, action) => {
      state.credential = action.payload
    },
  },
})

// ❌ 错误：状态在单例中
class TerminalIdentityManager {
  private terminalId: string | null = null
  private credential: TerminalCredential | null = null
  // ...
}
```

### 6.3 commands 负责动作抽象

**原则**：
- 所有关键动作应使用 `createModuleCommands` 统一建模
- 避免上层直接调用散落函数
- 提供统一的命令接口

**示例**：
```typescript
export const terminalDataCommands = createModuleCommands({
  // Identity 命令
  activateTerminal: createCommand<{ deviceId: string }>(),
  refreshTerminalCredential: createCommand<void>(),
  resetTerminalIdentity: createCommand<void>(),

  // TCP 命令
  fetchTcpBootstrap: createCommand<void>(),
  reportTaskResult: createCommand<{ releaseId: string; result: any }>(),

  // TDP 命令
  connectTdpSession: createCommand<void>(),
  heartbeatTdpSession: createCommand<void>(),
  disconnectTdpSession: createCommand<void>(),

  // Sync 命令
  syncProjectionSnapshot: createCommand<{ topicKey: string }>(),
  catchUpProjectionChanges: createCommand<{ topicKey: string; fromRevision: number }>(),
})
```

### 6.4 actors 负责副作用执行

**原则**：
- 所有副作用应由 actor 承担编排
- 包括：HTTP 调用、Session 建立、Projection 同步、上报结果、本地持久化协作

**示例**：
```typescript
export const activateTerminalActor = createActor({
  command: terminalDataCommands.activateTerminal,
  execute: async ({ payload, dispatch, getState }) => {
    const { deviceId } = payload

    // 1. 调用 TCP 激活接口
    const response = await tcpControlClient.activate({
      deviceId,
      deviceFingerprint: await deviceInfoAdapter.getDeviceFingerprint(),
    })

    // 2. 更新状态
    dispatch(terminalIdentitySlice.actions.setTerminalId(response.terminalId))
    dispatch(terminalIdentitySlice.actions.setActivationStatus(
      TerminalActivationStatus.ACTIVATED
    ))
    dispatch(terminalIdentitySlice.actions.setCredential({
      accessToken: response.accessToken,
      refreshToken: response.refreshToken,
      expiresAt: response.expiresAt,
      tokenType: 'Bearer',
      updatedAt: Date.now(),
    }))

    // 3. 保存到本地存储
    await credentialRepository.save(response)

    // 4. 触发后续流程
    dispatch(terminalDataCommands.fetchTcpBootstrap())
    dispatch(terminalDataCommands.connectTdpSession())
  },
})
```

### 6.5 foundations 负责底层抽象

**原则**：
- 底层组件应归入 `foundations`
- 包括：transport adapter、storage adapter、tree builder、scope resolver、sync coordinator

**分类**：

1. **adapters**：平台适配器
   - `DeviceInfoAdapter`
   - `StorageAdapter`
   - `TcpTransportAdapter`
   - `TdpTransportAdapter`
   - `NetworkStateAdapter`
   - `AppLifecycleAdapter`
   - `LoggerAdapter`
   - `ClockAdapter`

2. **clients**：协议客户端
   - `TcpControlClient`
   - `TdpSessionClient`
   - `TdpSyncClient`
   - `TerminalReportClient`

3. **repositories**：本地存储
   - `CredentialRepository`
   - `ProjectionRepository`
   - `ChangeLogRepository`
   - `InboxRepository`
   - `BindingContextRepository`

4. **coordinators**：流程编排
   - `ActivationCoordinator`
   - `SessionCoordinator`
   - `ProjectionSyncCoordinator`
   - `ReportCoordinator`
   - `DerivedTreeCoordinator`

5. **builders / resolvers**：辅助工具
   - `DerivedTreeBuilder`
   - `ScopeResolver`
   - `BootstrapAssembler`
   - `ProjectionMerger`

### 6.6 模块初始化流程

```typescript
export const terminalDataModulePreSetup = async (context: ModulePreSetupContext) => {
  const { dispatch, getState, adapters } = context

  // 1. 注册 adapters
  registerDeviceInfoAdapter(adapters.deviceInfo)
  registerStorageAdapter(adapters.storage)
  registerTcpTransportAdapter(adapters.tcpTransport)
  registerTdpTransportAdapter(adapters.tdpTransport)

  // 2. 初始化 coordinators
  await sessionCoordinator.initialize()
  await projectionSyncCoordinator.initialize()

  // 3. 加载本地缓存
  const cachedIdentity = await credentialRepository.load()
  if (cachedIdentity) {
    dispatch(terminalIdentitySlice.actions.setTerminalId(cachedIdentity.terminalId))
    dispatch(terminalIdentitySlice.actions.setCredential(cachedIdentity.credential))
  }

  // 4. 注册 TCP token interceptor
  tcpTransportAdapter.registerRequestInterceptor(async (config) => {
    const state = getState()
    const credential = selectTerminalCredential(state)
    if (credential?.accessToken) {
      config.headers.Authorization = `Bearer ${credential.accessToken}`
    }
    return config
  })

  // 5. 注册生命周期监听器
  appLifecycleAdapter.onAppActive(() => {
    dispatch(terminalDataCommands.reconnectTdpSession())
  })

  appLifecycleAdapter.onAppBackground(() => {
    dispatch(terminalDataCommands.disconnectTdpSession())
  })
}
```

## 7. 依赖建议

### 7.1 必选依赖

**@impos2/kernel-core-base**：

- 提供 `createCommand` / `createActor` 工具
- 提供基础类型定义

### 7.2 可选依赖

**@impos2/kernel-core-task**：
- 如果需要与任务系统集成
- 可以将 TDP 投递的任务转换为 task 系统的任务

**@impos2/kernel-core-navigation**：
- 如果需要与导航系统集成
- 可以根据投递的消息触发导航

### 7.3 依赖原则

**建议原则**：
- 新 `terminal-data` 不主动依赖 task / navigation 作为核心职责
- 若未来某项目需要把任务执行器或导航逻辑接入，应由上层模块扩展

## 10. 边界总结

### 10.1 应在内核内的

**核心职责**：
- ✅ 激活
- ✅ Token
- ✅ Binding Context
- ✅ TDP Session
- ✅ Projection Cache
- ✅ Delivery Inbox
- ✅ Config/Version Target State
- ✅ Derived Tree View

**原因**：
- 这些是终端客户端的核心能力
- 与 TCP/TDP 服务端设计直接对应
- 所有终端形态都需要这些能力

### 10.2 不应在内核内的

**排除内容**：
- ❌ 旧 unitData 覆盖语义
- ❌ 项目业务域对象树
- ❌ 具体页面流程
- ❌ 具体业务远控动作解释

**原因**：
- 这些是项目特定或业务特定的逻辑
- 应由上层模块处理
- 保持内核的通用性和纯粹性

### 10.3 边界判断原则

**判断方法**：
1. 是否与 TCP/TDP 服务端设计直接对应？
   - 是 → 应在内核内
   - 否 → 不应在内核内

2. 是否所有终端形态都需要？
   - 是 → 应在内核内
   - 否 → 不应在内核内

3. 是否项目特定或业务特定？
   - 是 → 不应在内核内
   - 否 → 应在内核内

4. 是否涉及 UI 层逻辑？
   - 是 → 不应在内核内
   - 否 → 可能在内核内

## 11. 关键结论

### 11.1 架构定位

`terminal-data` 是面向终端侧的 TCP/TDP 客户端通用数据内核，位于 `1-kernel` 的 core 层，为所有终端形态提供统一的数据层能力。

### 11.2 核心价值

1. **统一的领域语言**：与服务端设计保持一致
2. **清晰的职责边界**：5 个子域分工明确
3. **强大的类型安全**：TypeScript 完整类型定义
4. **灵活的扩展能力**：通过 adapter 和 middleware 扩展

### 11.3 设计原则

1. **显式实体优先**：不使用泛化的 Unit 类型
2. **客户端派生树**：树是视图层辅助结构
3. **控制面与数据面解耦**：TCP 和 TDP 分别建模
4. **强状态可观察**：每类状态都有明确的 slice
5. **框架风格继承**：遵循 1-kernel 的模块框架
6. **适配层注入**：通过 adapter 抽象平台差异

### 11.4 与旧包的关系

- **替代范围**：身份、激活、Token、TDP 会话、数据同步、任务投递
- **不替代范围**：unitData 模型、operatingEntity 树、项目特定逻辑
- **迁移策略**：提供兼容层，逐步迁移
