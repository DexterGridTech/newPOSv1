# 05-兼容策略与迁移建议

> 版本：2.0 | 状态：详细设计稿 | 日期：2026-04-06

---

## 文档说明

本文档详细阐述 `terminal-data` 包的兼容策略和迁移建议，包括与旧 `terminal` 包的关系、迁移路径、风险控制、扩展包规划等内容。

---

## 1. 设计结论

### 1.1 核心原则

新 `terminal-data` **不是**旧 `terminal` 的逐文件平移，而是一次**概念重建**。

**关键差异**：

| 维度 | 旧 terminal 包 | 新 terminal-data 包 |
|------|---------------|-------------------|
| **核心概念** | Unit 树、unitData、group | TCP/TDP、Projection、Scope |
| **数据模型** | 树状覆盖模型 | 显式实体引用模型 |
| **状态管理** | 散落的 state | 统一的 Redux slices |
| **协议抽象** | kernel WS（混合协议） | TCP（控制）+ TDP（数据）分离 |
| **依赖关系** | 与业务模块耦合 | 纯数据内核，业务解耦 |

### 1.2 迁移策略

因此迁移策略必须遵循：

1. **先建立新内核**：完整实现 TCP/TDP 客户端能力
2. **再逐步迁移旧职责**：通过适配层和 Facade 模式
3. **不把旧设计原样搬进来**：避免 unitData / Unit 树污染新内核

## 2. 旧 `terminal` 包能力归位分析

### 2.1 终端激活与 Token 管理

**旧实现**：
```typescript
// 旧 terminal 包
export const activateDevice = (activationCode: string) => {
  // 混合了激活、token、设备信息等逻辑
}
```

**归位建议**：✅ **归入新 `terminal-data`**

**新实现**：
```typescript
// 新 terminal-data 包
export const terminalDataCommands = createModuleCommands({
  activateTerminal: createCommand<{ activationCode: string }>(),
  refreshTerminalCredential: createCommand<void>(),
})
```

**迁移路径**：
1. 保留旧 `activateDevice` 作为 Facade
2. 内部调用新 `terminalDataCommands.activateTerminal`
3. 逐步迁移调用方到新 API

### 2.2 Kernel API Token Request Interceptor

**旧实现**：
```typescript
// 旧 terminal 包
kernelAPI.interceptors.request.use((config) => {
  const token = getDeviceToken()
  config.headers.Authorization = `Bearer ${token}`
  return config
})
```

**归位建议**：✅ **归入新 `terminal-data.modulePreSetup`**

**新实现**：
```typescript
// 新 terminal-data 包
export const terminalDataModulePreSetup = async (context) => {
  const { adapters } = context

  // 注册 TCP Token 拦截器
  adapters.tcpTransport.registerRequestInterceptor(async (config) => {
    const state = getState()
    const credential = selectTerminalCredential(state)
    if (credential?.accessToken) {
      config.headers = config.headers || {}
      config.headers.Authorization = `Bearer ${credential.accessToken}`
    }
    return config
  })
}
```

**迁移路径**：
1. 新项目直接使用 `modulePreSetup`
2. 旧项目保留旧拦截器，逐步迁移

### 2.3 Kernel WS 连接

**旧实现**：
```typescript
// 旧 terminal 包
export const connectKernelWS = () => {
  // 混合了控制和数据通道
  const ws = new WebSocket('ws://kernel-server/ws')
  // ...
}
```

**归位建议**：❌ **不直接迁移**

**重构建议**：应重构为三层架构

```typescript
// 1. TdpTransportAdapter（平台抽象）
export interface ITdpTransportAdapter {
  connect(url: string): Promise<void>
  disconnect(): Promise<void>
  send(message: TdpMessage): Promise<void>
  onMessage(handler: (message: TdpMessage) => void): void
}

// 2. TdpSessionClient（协议封装）
export class TdpSessionClient {
  async connect(params: { terminalId: string; accessToken: string }): Promise<TdpSessionDescriptor>
  async disconnect(): Promise<void>
  async sendHeartbeat(): Promise<void>
}

// 3. SessionCoordinator（流程编排）
export class SessionCoordinator {
  async reconnect(params: { terminalId: string; accessToken: string }): Promise<void>
  startHeartbeat(): void
  stopHeartbeat(): void
}
```

**迁移路径**：
1. 新项目使用新三层架构
2. 旧项目保留 `connectKernelWS`，内部桥接到新架构
3. 逐步迁移调用方

### 2.4 unitData 同步

**旧实现**：
```typescript
// 旧 terminal 包
export const syncUnitData = (group: string) => {
  // 基于 group 和 unit path 的同步逻辑
}
```

**归位建议**：❌ **不作为核心迁移**

**原因**：
1. `unitData` 是旧的树状覆盖模型，与新 TDP 的 Projection 模型不兼容
2. 强行兼容会污染新内核的设计清晰度
3. 会导致新旧语义冲突

**兼容方案**：如确有兼容需求，应设计独立扩展包

```typescript
// 独立扩展包：@next/kernel-core-terminal-data-legacy
export class UnitDataAdapter {
  // 将 TDP Projection 转换为旧 unitData 格式
  convertProjectionToUnitData(projection: ProjectionRecord): UnitData {
    // 转换逻辑
  }

  // 将旧 unitData 查询转换为 TDP Projection 查询
  queryUnitData(group: string, unitPath: string): UnitData {
    // 查询逻辑
  }
}
```

### 2.5 operatingEntity / model / terminal 树

**旧实现**：
```typescript
// 旧 terminal 包
export interface Unit {
  id: string
  type: 'TENANT' | 'BRAND' | 'STORE' | 'TERMINAL'
  parentId?: string
  children: Unit[]
  data: Record<string, unknown>
}
```

**归位建议**：❌ **不迁移为权威模型**

**替代方案**：使用客户端派生树

```typescript
// 新 terminal-data 包
export interface DerivedTreeNode {
  nodeId: string
  nodeType: DerivedTreeNodeType
  refId: string  // 指向实际实体的 ID
  label: string
  parentNodeId?: string
  childrenNodeIds: string[]
  metadata?: Record<string, unknown>
}
```

**关键差异**：
- **旧 Unit 树**：权威模型，数据存储在树节点中
- **新派生树**：视图模型，数据存储在显式实体中，树只是引用关系

### 2.6 sendDeviceState / remoteCommandConfirm

**旧实现**：
```typescript
// 旧 terminal 包
export const sendDeviceState = (state: DeviceState) => {
  // 上报设备状态
}

export const remoteCommandConfirm = (commandId: string, result: string) => {
  // 确认远程命令执行结果
}
```

**归位建议**：⚠️ **需要拆分**

**通用部分**：归入新 `terminal-data`

```typescript
// 新 terminal-data 包
export const terminalDataCommands = createModuleCommands({
  reportLifecycleState: createCommand<{
    state: 'ACTIVE' | 'BACKGROUND' | 'TERMINATED'
  }>(),

  reportTaskResult: createCommand<{
    releaseId: string
    result: 'SUCCESS' | 'FAILURE'
    message?: string
  }>(),
})
```

**项目级部分**：不内置到内核

```typescript
// 上层业务模块
export const remoteControlCommands = createModuleCommands({
  confirmRemoteCommand: createCommand<{
    commandId: string
    result: string
  }>(),
})
```

## 3. 为什么不建议直接兼容旧 `unitData`

### 3.1 新旧语义冲突

**旧 unitData 的中心概念**：
- **group**：数据分组（如 systemParameters、errorMessages）
- **unit path**：树状路径（如 `/tenant/brand/store`）
- **覆盖优先级**：子节点覆盖父节点

**新 TDP 的中心概念**：
- **Topic**：数据主题（如 `product.catalog`、`member.profile`）
- **Scope**：数据范围（如 `STORE:store123`）
- **Projection**：数据快照
- **ChangeLog**：增量变更

**冲突点**：

| 场景 | 旧 unitData | 新 TDP |
|------|------------|--------|
| 数据定位 | group + unit path | topic + scope |
| 数据更新 | 覆盖合并 | Projection 替换或 Change 应用 |
| 数据查询 | 树遍历 + 优先级计算 | 直接查询 Projection |
| 数据订阅 | 订阅 group | 订阅 Topic |

**示例对比**：

```typescript
// 旧方式：查询门店的系统参数
const params = getUnitData('systemParameters', '/tenant123/brand456/store789')
// 需要遍历树，计算覆盖优先级

// 新方式：查询门店的系统参数
const projection = getProjection('system.parameters', 'STORE', 'store789')
// 直接获取，无需遍历
```

### 3.2 客户端会重新陷入"树优先"思维

**问题**：如果新内核继续以 `unitData` 为中心，会导致：

1. **削弱 TCP/TDP 设计边界**
   - TCP 和 TDP 的职责分离会变得模糊
   - 控制平面和数据平面的边界会混淆

2. **削弱显式实体设计优势**
   - 显式的 `TenantRef`、`BrandRef`、`StoreRef` 会被忽略
   - 重新回到隐式的树路径引用

3. **削弱终端侧数据模型清晰度**
   - 状态管理会变得复杂（树 + Projection 双重模型）
   - 数据流向会变得难以追踪

### 3.3 迁移成本会被无限拖延

**问题**：如果新内核里还继续以 `unitData` 为中心，就会导致：

1. **大量兼容代码常驻**
   - 转换逻辑、适配逻辑会永久存在
   - 代码复杂度持续增加

2. **永远无法真正退掉老设计**
   - 新旧代码并存，维护成本翻倍
   - 技术债务持续累积

3. **新功能开发受限**
   - 必须同时考虑新旧两套模型
   - 创新空间被压缩

## 4. 推荐迁移阶段

### 阶段一：建立新内核，不接旧 unitData

**目标**：完整实现 TCP/TDP 客户端能力

**交付物**：
- ✅ `@next/kernel-core-terminal-data` 包
- ✅ 完整的 Redux slices（8 个）
- ✅ 完整的 Commands 和 Actors
- ✅ 完整的 Foundation 层（Adapters、Clients、Repositories、Coordinators）
- ✅ 完整的 Hooks 层

**验收标准**：
- [ ] 可以完成终端激活
- [ ] 可以连接 TDP 会话
- [ ] 可以同步 Projection
- [ ] 可以接收任务投递
- [ ] 可以上报执行结果

**时间估算**：2-3 周

### 阶段二：让新项目直接接入新内核

**目标**：验证新内核的完整性和可用性

**实施方案**：
1. 选择一个新项目（如新门店类型、新业务场景）
2. 直接使用新 `terminal-data` 包
3. 不依赖旧 `terminal` 包
4. 收集反馈，优化新内核

**验收标准**：
- [ ] 新项目可以正常激活
- [ ] 新项目可以正常接收配置
- [ ] 新项目可以正常接收版本更新
- [ ] 新项目可以正常上报状态

**时间估算**：1-2 周

### 阶段三：对老项目做适配层

**目标**：让老项目可以逐步迁移到新内核

**实施方案**：

#### 3.1 Facade 模式

```typescript
// @next/kernel-core-terminal（旧包，保留作为 Facade）
import { terminalDataCommands } from '@next/kernel-core-terminal-data'

// 保留旧 API，内部调用新 API
export const activateDevice = (activationCode: string) => {
  return dispatch(terminalDataCommands.activateTerminal({ activationCode }))
}

export const connectKernelWS = () => {
  return dispatch(terminalDataCommands.connectTdpSession())
}
```

#### 3.2 Mapper 模式

```typescript
// @next/kernel-core-terminal-data-legacy
export class UnitDataMapper {
  // 将 Projection 映射为 unitData
  mapProjectionToUnitData(projection: ProjectionRecord): UnitData {
    return {
      group: projection.topicKey,
      unitPath: this.scopeToUnitPath(projection.scopeType, projection.scopeKey),
      data: projection.payload,
    }
  }

  // 将 Scope 映射为 unitPath
  private scopeToUnitPath(scopeType: TdpScopeType, scopeKey: string): string {
    // 从 bindingContext 构建 unitPath
    const context = selectBindingContext(getState())
    if (scopeType === 'STORE') {
      return `/${context.tenant.tenantId}/${context.brand.brandId}/${scopeKey}`
    }
    // ...
  }
}
```

#### 3.3 Legacy Adapter 模式

```typescript
// @next/kernel-core-terminal-data-legacy
export class LegacyTerminalAdapter {
  // 提供旧 terminal 包的完整接口
  activateDevice(activationCode: string): Promise<void>
  connectKernelWS(): Promise<void>
  syncUnitData(group: string): Promise<void>
  getUnitData(group: string, unitPath: string): UnitData
  sendDeviceState(state: DeviceState): Promise<void>
}
```

**验收标准**：
- [ ] 老项目可以通过 Facade 使用新内核
- [ ] 老项目的 unitData 查询可以正常工作
- [ ] 老项目的设备状态上报可以正常工作

**时间估算**：2-3 周

### 阶段四：逐步迁出旧 terminal 剩余职责

**目标**：彻底清理旧 `terminal` 包，拆分为多个专用包

**拆分方案**：

```
旧 @next/kernel-core-terminal
```

|-------|--------|
| 激活与 Token | terminal-data |
| TCP 控制平面 | terminal-data |
| TDP 数据平面 | terminal-data |
| unitData 同步 | terminal-data-legacy |
| Unit 树管理 | terminal-data-legacy |
| 热更新执行 | terminal-control |
| 配置治理 | terminal-control |
| 远控治理 | terminal-control |
| 设备信息采集 | terminal-runtime |
| 本地执行器 | terminal-runtime |

**验收标准**：
- [ ] 旧 `terminal` 包已废弃
- [ ] 所有职责已迁移到新包
- [ ] 所有项目已迁移到新包

**时间估算**：4-6 周

## 5. 客户端派生树的兼容价值

### 5.1 对开发者心智的兼容

**问题**：旧 terminal 包用户仍然习惯：
- 从上到下看层级（Tenant → Brand → Store → Terminal）
- 用树来理解归属关系
- 用树路径来定位数据

**解决方案**：提供派生树视图

```typescript
// 派生树提供熟悉的树状视图
const nodes = useDerivedTree()

// 开发者可以像以前一样遍历树
const rootNode = nodes.find(n => n.nodeType === 'ROOT')
const tenantNode = nodes.find(n => n.nodeId === rootNode.childrenNodeIds[0])
const brandNode = nodes.find(n => n.nodeId === tenantNode.childrenNodeIds[0])
// ...
```

### 5.2 对 UI 的兼容

**问题**：很多管理界面、调试界面依然更适合：
- 展示 Tenant → Brand → Store → Terminal 层级
- 做树状选择器
- 做组织范围过滤

**解决方案**：派生树可以直接用于 UI 渲染

```typescript
// UI 组件示例
function OrganizationTreeView() {
  const nodes = useDerivedTree()

  return (
    <TreeView>
      {renderTree(nodes, 'root')}
    </TreeView>
  )
}

function renderTree(nodes: DerivedTreeNode[], parentId: string) {
  const children = nodes.filter(n => n.parentNodeId === parentId)

  return children.map(node => (
    <TreeNode key={node.nodeId} label={node.label}>
      {renderTree(nodes, node.nodeId)}
    </TreeNode>
  ))
}
```

### 5.3 对 Scope 解析辅助的兼容

**问题**：虽然 Scope 权威判断仍依赖显式实体，但树能帮助客户端更直观地理解范围关系。

**解决方案**：派生树可以辅助 Scope 匹配判断

```typescript
// 判断某个 Scope 是否包含当前终端
function isScopeContainsCurrentTerminal(
  scopeType: TdpScopeType,
  scopeKey: string
): boolean {
  const nodes = useDerivedTree()
  const bindingContext = useTerminalBindingContext()

  // 使用派生树辅助判断
  const scopeNode = nodes.find(n =>
    n.nodeType === scopeType && n.refId === scopeKey
  )

  const terminalNode = nodes.find(n =>
    n.nodeType === 'TERMINAL' && n.refId === bindingContext.terminalId
  )

  // 检查 terminalNode 是否是 scopeNode 的后代
  return isDescendant(nodes, terminalNode, scopeNode)
}
```

### 5.4 派生树的关键原则

**必须遵守的原则**：

1. **派生树不是权威模型**
   - 数据不存储在树节点中
   - 树节点只是引用关系的视图

2. **派生树不参与 Scope 权威判断**
   - Scope 匹配仍基于显式实体引用
   - 派生树只是辅助工具

3. **派生树不持久化**
   - 每次启动或绑定变化时重建
   - 不保存到本地存储

4. **派生树不引入 rootPath**
   - 不使用旧的 `rootPath` 覆盖体系
   - 不引入树路径优先级计算

## 6. 风险与控制点

### 风险一：为了兼容，把老设计整包搬回新内核

**风险描述**：
- 为了快速兼容，直接把旧 `terminal` 包的代码复制到新包
- 导致新内核变得臃肿、复杂、难以维护

**控制策略**：

1. **明确新包职责边界**
   ```typescript
   // ✅ 新包应该包含的
   - TCP 控制平面客户端
   - TDP 数据平面客户端
   - 终端身份与凭证管理
   - Projection 同步与管理
   - 任务投递与上报

   // ❌ 新包不应该包含的
   - unitData 树状模型
   - 业务域特定逻辑
   - UI 组件
   - 导航逻辑
   ```

2. **代码审查检查清单**
   - [ ] 是否引入了 `unitData` 相关代码？
   - [ ] 是否引入了 `Unit` 树权威模型？
   - [ ] 是否引入了业务域特定逻辑？
   - [ ] 是否引入了 UI 组件？

3. **依赖检查**
   ```json
   // package.json
   {
     "dependencies": {
       // ✅ 允许的依赖
       "@reduxjs/toolkit": "^2.0.0",
       "redux-observable": "^3.0.0",

       // ❌ 不允许的依赖
       "@next/kernel-core-navigation": "❌",
       "@next/kernel-core-task": "❌",
       "@next/ui-*": "❌"
     }
   }
   ```

### 风险二：派生树失控，重新变成权威模型

**风险描述**：
- 派生树逐渐承载越来越多的数据
- 派生树开始参与 Scope 权威判断
- 派生树重新变成旧 Unit 树

**控制策略**：

1. **类型定义限制**
   ```typescript
   // ✅ 正确的派生树节点定义
   export interface DerivedTreeNode {
     nodeId: string
     nodeType: DerivedTreeNodeType
     refId: string  // 只存储引用，不存储数据
     label: string
     parentNodeId?: string
     childrenNodeIds: string[]
     metadata?: Record<string, unknown>  // 只存储少量元数据
   }

   // ❌ 错误的派生树节点定义
   export interface DerivedTreeNode {
     nodeId: string
     nodeType: DerivedTreeNodeType
     data: Record<string, unknown>  // ❌ 不应该存储业务数据
     children: DerivedTreeNode[]  // ❌ 不应该嵌套存储
   }
   ```

2. **文档明确声明**
   ```markdown
   # 派生树使用指南

   ## ✅ 正确用法
   - 用于 UI 展示
   - 用于辅助 Scope 理解
   - 用于组织关系可视化

   ## ❌ 错误用法
   - 不用于数据存储
   - 不用于 Scope 权威判断
   - 不用于业务逻辑计算
   ```

3. **代码审查检查清单**
   - [ ] 派生树节点是否只存储引用？
   - [ ] 派生树是否参与了 Scope 判断？
   - [ ] 派生树是否被持久化了？

### 风险三：新包又依赖太多上层包

**风险描述**：
- 为了实现某些功能，引入了上层业务包的依赖
- 导致新包变得难以复用、难以测试

**控制策略**：

1. **依赖方向检查**
   ```
   ✅ 正确的依赖方向
   上层业务包 → terminal-data → kernel-core-base

   ❌ 错误的依赖方向
   terminal-data → 上层业务包
   ```

2. **依赖注入原则**
   ```typescript
   // ✅ 正确：通过 Adapter 抽象依赖
   export interface IDeviceInfoAdapter {
     getDeviceId(): Promise<string>
   }

   // 上层提供实现
   class RNDeviceInfoAdapter implements IDeviceInfoAdapter {
     async getDeviceId(): Promise<string> {
       return DeviceInfo.getUniqueId()
     }
   }

   // ❌ 错误：直接依赖上层包
   import DeviceInfo from 'react-native-device-info'
   ```

3. **package.json 依赖审查**
   - 定期检查 `dependencies` 和 `peerDependencies`
   - 确保没有引入业务包依赖

### 风险四：新旧状态同时存在导致混乱

**风险描述**：
- 迁移期间，新旧两套状态并存
- 开发者不知道应该使用哪个状态
- 数据不一致导致 bug

**控制策略**：

1. **明确状态权威**
   ```typescript
   // 迁移期间的状态权威声明

   // ✅ 新项目：只使用新状态
   const identity = useSelector(state => state.terminalIdentity.identity)

   // ⚠️ 老项目迁移中：明确使用新状态，旧状态只读
   const identity = useSelector(state => state.terminalIdentity.identity)
   const legacyDevice = useSelector(state => state.terminal.device)  // 只读，不修改

   // ❌ 禁止：同时修改新旧状态
   dispatch(terminalIdentitySlice.actions.setIdentity(identity))
   dispatch(terminalSlice.actions.setDevice(device))  // ❌
   ```

2. **迁移检查清单**
   - [ ] 是否明确了哪个状态是权威？
   - [ ] 是否禁止了同时修改新旧状态？
   - [ ] 是否有数据同步机制（如果需要）？

3. **过渡期数据同步**
   ```typescript
   // 如果必须同时维护新旧状态，使用单向同步

   // ✅ 新状态 → 旧状态（单向同步）
   useEffect(() => {
     const identity = selectTerminalIdentity(state)
     if (identity) {
       // 同步到旧状态（只读）
       dispatch(legacyTerminalSlice.actions.syncFromNewIdentity(identity))
     }
   }, [identity])

   // ❌ 禁止：旧状态 → 新状态（会导致循环依赖）
   ```

## 7. 命名与接口兼容建议

### 7.1 命名规范对比

新包应全面采用 TCP/TDP 语义命名，废弃旧的 `kernel`/`unit` 命名体系。

**WS 连接相关**：

| 旧命名 | 新命名 | 说明 |
|--------|--------|------|
| `connectKernelWS` | `connectTdpSession` | 明确是 TDP 会话 |
| `kernelWSConnected` | `tdpSessionConnected` | 明确是 TDP 状态 |
| `kernelWSDisconnected` | `tdpSessionDisconnected` | 明确是 TDP 状态 |
| `kernelWSError` | `tdpSessionError` | 明确是 TDP 错误 |
| `kernelWSReconnecting` | `tdpSessionReconnecting` | 明确是 TDP 重连 |

**状态 Slice 相关**：

| 旧命名 | 新命名 | 说明 |
|--------|--------|------|
| `unitData_systemParameters` | `tdpProjection['system.parameters']` | 基于 Topic 的 Projection |
| `unitData_errorMessages` | `tdpProjection['system.errors']` | 基于 Topic 的 Projection |
| `terminal.device` | `terminalIdentity.identity` | 显式身份实体 |
| `terminal.token` | `terminalIdentity.credential` | 显式凭证实体 |
| `terminal.storeInfo` | `terminalBinding.bindingContext.store` | 显式 Store 引用 |

**API 调用相关**：

| 旧命名 | 新命名 | 说明 |
|--------|--------|------|
| `activateDevice` | `activateTerminal` | 语义更准确 |
| `refreshDeviceToken` | `refreshTerminalCredential` | 语义更准确 |
| `sendDeviceState` | `reportLifecycleState` | 语义更准确 |
| `remoteCommandConfirm` | `reportTaskResult` | 语义更准确 |
| `fetchStoreConfig` | `syncProjectionSnapshot` | 基于 TDP Topic |

### 7.2 接口命名原则

新包的接口命名应遵循以下原则：

1. **围绕 TCP/TDP 语义**：不使用 `kernel`、`device`、`unit` 等旧语义
2. **使用显式实体名称**：`Terminal`、`Tenant`、`Brand`、`Store`、`Profile`
3. **使用 Topic/Scope/Projection 体系**：不使用 `group`/`unitPath`/`覆盖` 体系
4. **动词语义清晰**：`activate`（激活）、`connect`（连接）、`sync`（同步）、`report`（上报）

### 7.3 迁移命名 Checklist

在代码审查时，检查以下命名是否还存在于新包中：

```bash
# 应该被替换的旧命名
grep -r "kernelWS\|unitData\|rootPath\|operatingEntity\|storeUnit" src/

# 如果有结果，需要替换为新命名
```

### 8.2 `@next/kernel-core-terminal-data-legacy`

**职责**：承载旧 terminal / unitData 的兼容层

**包含能力**：

- 旧 terminal API Facade

```typescript
// @next/kernel-core-terminal-data-legacy
export class UnitDataLegacyAdapter {
  /**
   * 从 TDP Projection 获取旧格式的 unitData
   */
  getUnitData(group: string, unitPath: string): UnitData | null {
    // 将旧 group 映射到 topicKey
    const topicKey = this.mapGroupToTopicKey(group)

    // 从 Redux state 获取 Projection
    const projection = selectProjection(topicKey)(getState())
    if (!projection) return null

    // 将 Projection payload 转换为旧格式
    return this.convertToUnitData(projection.payload, unitPath)
  }

  /**
   * 将 group 名称映射到 TDP topicKey
   */
  private mapGroupToTopicKey(group: string): string {
    const mapping: Record<string, string> = {
      systemParameters: 'system.parameters',
      errorMessages: 'system.errors',
      storeInfo: 'store.info',
      // ...
    }
    return mapping[group] ?? group
  }

  /**
   * 将 Projection payload 转换为 unitData 格式
   */
  private convertToUnitData(
    payload: Record<string, unknown>,
    unitPath: string
  ): UnitData {
    // 根据 unitPath 提取对应层级的数据
    const pathParts = unitPath.split('/').filter(Boolean)
    let current = payload

    for (const part of pathParts) {
      if (current && typeof current === 'object' && part in current) {
        current = current[part] as Record<string, unknown>
      }
    }

    return current as UnitData
  }
}

// Facade：提供旧 terminal 包的 API
export const legacyTerminalFacade = {
  activateDevice: (activationCode: string) =>
    dispatch(terminalDataCommands.activateTerminal({ activationCode })),

  connectKernelWS: () =>
    dispatch(terminalDataCommands.connectTdpSession()),

  getUnitData: (group: string, unitPath: string) =>
    unitDataLegacyAdapter.getUnitData(group, unitPath),
}
```

## 9. 最终迁移路径总结

### 9.1 分阶段迁移路线图

```
阶段一（当前）
  ├─ 新建 terminal-data 包（已完成设计）
  ├─ 实现 Identity、Binding、Runtime、TCP、TDP 能力
  └─ 验收：新功能可直接使用新包

阶段二（近期）
  ├─ 在新项目中全面接入 terminal-data
  ├─ 收集反馈，打磨 API
  └─ 验收：新项目稳定运行

阶段三（中期）
  ├─ 为老项目构建 Legacy Adapter
  ├─ 老项目通过 Facade 使用新包
  └─ 验收：老项目可以使用新包能力

阶段四（长期）
  ├─ 逐步将老项目调用方迁移到新 API
  ├─ 拆出 terminal-control、terminal-runtime
  └─ 验收：所有项目迁移完成，旧包废弃
```

### 9.2 不同角色的迁移建议

**新项目开发者**：
- 直接使用 `@next/kernel-core-terminal-data`
- 不依赖旧 `terminal` 包
- 参考本文档和 04-关键流程设计

**老项目维护者**：
- 保留旧 `terminal` 包暂时不动
- 引入 `terminal-data-legacy` 包
- 逐步替换关键路径（激活、同步、上报）
- 最后替换 UnitData 查询

**基础设施开发者**：
- 实现对应平台的 Adapter（RN、Electron、Web）
- 确保 Adapter 接口完整实现
- 提供平台特定的测试和调试工具

### 9.3 迁移完成标准

当满足以下条件时，认为迁移完成：

- [ ] 所有新项目使用 `terminal-data`
- [ ] 所有老项目已迁移完毕
- [ ] 旧 `terminal` 包已废弃（标记 deprecated）
- [ ] `unitData` 查询已通过 Legacy Adapter 迁移
- [ ] 所有命名已更新为新语义
- [ ] 文档已更新

## 10. 一句话总结

新 `terminal-data` 的迁移策略，不是"把旧 terminal 包复制一份"，而是：

> **用 TCP/TDP 的新生产体系重建终端客户端内核；旧 terminal 的历史能力按职责拆散、通过适配层兼容、分阶段迁移退出。**

核心要点：
1. **新内核不承载兼容负担**：`terminal-data` 只关注 TCP/TDP 核心能力
2. **兼容由独立包承载**：`terminal-data-legacy` 专门处理历史兼容
3. **迁移分阶段进行**：不强求一次性完成，允许新旧并存过渡期
4. **派生树桥接心智**：保留树状视图，帮助开发者平滑过渡
