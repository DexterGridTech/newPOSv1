## 1. 设计背景

### 1.1 旧 terminal 包的历史定位

当前老工程中的 `_old_/1-kernel/1.1-cores/terminal` 包，是在早期项目中逐步演化形成的终端侧核心模块。它承担了以下职责：

#### 核心职责
1. **终端身份管理**
   - 设备信息采集与终端身份生成
   - 终端激活与反激活流程
   - 终端认证态（token）管理

2. **组织关系管理**
   - operatingEntity 设置与切换
   - terminal / model / entity 的树状关系
   - rootPath 优先级覆盖机制

3. **协议客户端**
   - kernel HTTP API 封装
   - kernel WebSocket 长连接管理
   - 心跳与远程消息接收

4. **数据同步**
   - unitData 组数据同步
   - 本地 unitData 状态机
   - 数据变更通知与消费

5. **设备状态上报**
   - 远程命令确认
   - 设备健康状态上报
   - 版本与配置状态上报

#### 实现特点
- 以 `Unit` 树作为核心抽象
- 以 `unitData` 作为数据载体
- 以 `rootPath` 作为覆盖优先级机制
- 以 `kernelWS` 作为通信通道

### 1.2 旧包存在的核心问题

虽然旧 `terminal` 包在历史项目中可以工作，但随着系统架构演进，其问题逐渐暴露：

#### 问题一：职责边界模糊

**现象**：
- 同一个包内同时包含：终端身份、组织树、数据分发、连接管理、状态上报、业务单元覆盖
- 无法清晰回答"这个包的核心职责是什么"
- 难以判断某个功能应该放在 terminal 包内还是外部

**影响**：
- 代码维护困难，修改一处可能影响多个不相关功能
- 新人学习成本高，需要理解整个包才能修改局部
- 测试困难，单元测试需要 mock 大量依赖

**根本原因**：
- 缺乏明确的领域划分
- 历史演进中不断叠加功能
- 没有与服务端架构保持一致

#### 问题二：抽象模型语义过于泛化

**现象**：
- `Unit` 可以表示 tenant、brand、store、terminal、profile、model 等任何对象
- `unitData` 可以承载任何业务数据
- `rootPath` 可以表示任何层级关系

**影响**：
- 阅读代码时很难直接判断当前对象的真实业务语义
- 容易把组织层级、终端能力、配置覆盖、实体归属混在一起
- IDE 无法提供有效的类型提示和自动补全
- 重构困难，不敢轻易修改类型定义

**根本原因**：
- 追求"万能树"的通用性，牺牲了类型安全
- 缺乏显式的领域实体建模
- 与新 TCP/TDP 的显式模型不一致

#### 问题三：协议绑定过重

**现象**：
- 直接围绕 old kernel HTTP API 实现
- 直接围绕 old kernel WebSocket 协议实现
- 直接围绕 old unitData group 机制实现

**影响**：
- 无法复用到其他协议体系
- 难以适配新的 TCP/TDP 服务
- 迁移成本高，需要重写大量代码

**根本原因**：
- 缺乏协议抽象层
- 业务逻辑与协议实现耦合
- 没有考虑多协议支持

#### 问题四：难以支持新体系演进

**现象**：
现在新的生产设计已经明确拆分为：
- **TCP（Terminal Control Plane）**：终端控制平面，负责激活、配置、升级、任务、监控
- **TDP（Terminal Data Plane）**：终端数据平面，负责数据投递、订阅、同步、变更通知

如果客户端内核还继续维持老 `terminal` 的思维方式，就会出现：
- 服务端一套概念（TCP/TDP）
- 客户端另一套概念（Unit/unitData）
- 中间需要大量翻译层
- 最终难以形成真正通用的内核

**影响**：
- 前后端概念不一致，沟通成本高
- 需要维护大量适配代码
- 新功能开发效率低
- 难以利用服务端的新能力

**根本原因**：
- 客户端架构没有跟随服务端演进
- 缺乏统一的领域语言
- 历史包袱过重，不敢重构

### 1.3 新体系的架构演进

#### 服务端架构演进

根据 `design/4.终端数据平面服务` 和 `design/5.终端控制平面服务` 的设计：

**TCP（Terminal Control Plane）**：
- 终端激活与绑定
- 配置管理与下发
- 热更新管理与下发
- 任务投递与结果收集
- 终端监控与健康检查
- 远程控制与指令下发

**TDP（Terminal Data Plane）**：
- WebSocket 会话管理
- Topic 定义与管理
- Projection 投递与同步
- Change 增量推送
- Cursor 管理与断点续传
- Scope 解析与权限控制

#### 客户端需要对齐的概念

为了与服务端保持一致，客户端内核需要明确支持：

**TCP 客户端能力**：
- 激活流程（Activation）
- Token 刷新（Token Refresh）
- Bootstrap 引导（获取初始配置）
- 配置拉取（Config Pull）
- 升级拉取（Upgrade Pull）
- 任务接收（Task Delivery）
- 结果上报（Result Report）
- 状态上报（Status Report）

**TDP 客户端能力**：
- Session 连接（Connect）
- 心跳保活（Heartbeat）
- Topic 订阅（Subscribe）
- Projection 接收（Receive Projection）
- Change 接收（Receive Change）
- Ack 确认（Acknowledge）
- Replay 重放（Replay from Cursor）
- 断线重连（Reconnect）

### 1.4 为什么需要新包

基于以上分析，我们需要一个全新的终端客户端数据内核，原因如下：

#### 原因一：对齐服务端架构
- 服务端已经明确拆分为 TCP 和 TDP
- 客户端必须跟随演进，否则会产生巨大的概念鸿沟
- 统一的领域语言可以大幅降低沟通成本

#### 原因二：提升代码质量
- 显式的类型定义可以提供更好的 IDE 支持
- 清晰的职责边界可以降低维护成本
- 模块化的设计可以提升测试覆盖率

#### 原因三：支持多终端形态
- React Native 裸工程终端
- Electron 桌面终端
- Web 调试终端
- 未来的其他终端形态

#### 原因四：为老包迁移留出空间
- 不是简单的重命名，而是重新设计
- 可以逐步迁移，不需要一次性替换
- 为历史项目提供兼容层

## 2. 新包设计目标

### 2.1 包命名与定位

**npm 包名**：`@next/kernel-core-terminal-data`
**模块名**：`kernel.core.terminal-data`
**目录位置**：`_old_/1-kernel/1.1-cores/terminal-data`

**核心定位**：
> 面向终端侧的 TCP/TDP 客户端通用数据内核

它不是简单的"把旧 terminal 包重命名"，而是要成为新的终端侧基础内核。

### 2.2 核心目标

#### 目标一：与 TCP/TDP 生产设计保持一致

**要求**：
- 新包的概念体系必须与服务端设计文档保持一致
- 实体命名、状态定义、协议流程必须对齐
- 开发者可以直接对照服务端文档理解客户端实现

**验证标准**：
- 服务端的 `Terminal` 实体，客户端有对应的 `TerminalIdentity`
- 服务端的 `Topic` 定义，客户端有对应的 `TopicDefinition`
- 服务端的 `Projection` 投递，客户端有对应的 `ProjectionRecord`
- 服务端的 `Task` 投递，客户端有对应的 `TcpTaskDeliveryEnvelope`

#### 目标二：成为终端侧通用数据内核

**要求**：
- 不绑定特定的终端形态（RN/Electron/Web）
- 不绑定特定的业务域（餐饮/零售/其他）
- 不绑定特定的项目（mixc-retail/其他）

**验证标准**：
- 可以在 React Native 裸工程中使用
- 可以在 Electron 应用中使用
- 可以在 Web 调试工具中使用
- 可以在未来的其他终端形态中使用

#### 目标三：延续 1-kernel 的模块框架

**要求**：
- 必须遵守现有 `1-kernel` 的开发框架
- 使用 `AppModule` 作为模块导出
- 使用 `slices` 管理状态
- 使用 `commands` 定义动作
- 使用 `actors` 处理副作用
- 使用 `foundations` 抽象底层能力

**验证标准**：
- 导出的模块符合 `AppModule` 接口
- 所有状态都在 `slices` 中定义
- 所有动作都通过 `commands` 暴露
- 所有副作用都在 `actors` 中处理
- 所有底层能力都通过 `foundations` 抽象

#### 目标四：支持显式实体 + 客户端派生树

**要求**：
- 服务端保持显式实体权威模型（tenant/brand/store/profile/terminal）
- 客户端允许基于这些实体构建派生树视图
- 派生树仅作为客户端视图与辅助计算结构，不替代服务端权威模型

**验证标准**：
- 核心状态使用显式实体引用（`TenantRef`/`BrandRef`/`StoreRef`）
- 提供派生树构建能力（`DerivedTreeNode`）
- 派生树不影响服务端模型
- 派生树可以根据需要重建

#### 目标五：为老 terminal 的后续拆迁留出空间

**要求**：
- 本次设计不把老 terminal 包的全部内容直接吞进去
- 先建立通用 TCP/TDP 客户端内核
- 再把老 terminal 的剩余内容迁移到更合适的位置

**验证标准**：
- 新包职责清晰，不包含业务域逻辑
- 提供兼容层接口，方便老代码迁移
- 老包可以逐步废弃，不需要一次性替换

### 2.3 设计原则

#### 原则一：显式实体优先

**说明**：
- `tenant`、`brand`、`store`、`profile`、`terminal` 为显式引用实体
- 不使用泛化的 `Unit` 类型
- 每个实体都有明确的类型定义

**示例**：
```typescript
// ❌ 旧方式：泛化类型
interface Unit {
  id: string
  type: string  // 可能是 'tenant' | 'brand' | 'store' | ...
  data: any
}

// ✅ 新方式：显式类型
interface TenantRef {
  tenantId: string
  code?: string
  name: string
}

interface BrandRef {
  brandId: string
  tenantId: string
  code?: string
  name: string
}
```

#### 原则二：客户端派生树

**说明**：
- 允许在客户端内部构建通用树
- 但不要求服务端转为树权威模型
- 树是视图层的辅助结构，不是领域模型

**示例**：
```typescript
// 服务端权威模型：显式引用
interface TerminalBindingContext {
  terminalId: string
  tenant: TenantRef
  brand: BrandRef
  store: StoreRef
  profile: TerminalProfileRef
}

// 客户端派生树：视图结构
interface DerivedTreeNode {
  nodeId: string
  nodeType: 'TENANT' | 'BRAND' | 'STORE' | 'TERMINAL'
  refId: string
  label: string
  parentNodeId?: string
  childrenNodeIds: string[]
}
```

#### 原则三：控制面与数据面解耦

**说明**：
- TCP 与 TDP 在客户端中分别建模与同步
- TCP 负责控制（激活、配置、升级、任务）
- TDP 负责数据（投递、订阅、同步、变更）

**示例**：
```typescript
// TCP 状态
interface TcpClientState {
  lastBootstrapAt?: number
  lastReportAt?: number
  lastError?: string
}

// TDP 状态
interface TdpSessionState {
  sessionId?: string
  status: TdpSessionStatus
  connectedAt?: number
  lastHeartbeatAt?: number
}
```

#### 原则四：强状态可观察

**说明**：
- 每类状态都有明确的 slice
- 每类状态都有对应的 selector
- 每类状态都有对应的 actor
- 每类状态都有对应的 command

**示例**：
```typescript
// Slice
const terminalIdentitySlice = createSlice({
  name: 'terminalIdentity',
  initialState: { ... },
  reducers: { ... }
})

// Selector
const selectTerminalId = (state: RootState) =>
  state.terminalIdentity.terminalId

// Command
const activateTerminal = createCommand({
  type: 'terminal/activate',
  payload: { deviceId: string }
})

// Actor
const activateTerminalActor = createActor({
  command: activateTerminal,
  execute: async (payload) => { ... }
})
```

#### 原则五：框架风格继承 1-kernel

**说明**：
- 保持 `AppModule + slices + commands + actors + supports + foundations` 结构
- 遵循现有的模块开发规范
- 与其他 core 包保持一致的代码风格

#### 原则六：适配层注入

**说明**：
- 设备、存储、网络、连接、时钟、日志全部通过 foundation/adapter 抽象接入
- 核心逻辑不直接依赖 React Native 或 Electron 的 API
- 通过依赖注入实现跨平台

**示例**：
```typescript
// Foundation 抽象
interface IDeviceInfoProvider {
  getDeviceId(): Promise<string>
  getDeviceFingerprint(): Promise<string>
}

interface IStorageAdapter {
  get(key: string): Promise<string | null>
  set(key: string, value: string): Promise<void>
}

// 适配层注入
const rnDeviceInfoProvider: IDeviceInfoProvider = {
  getDeviceId: async () => DeviceInfo.getUniqueId(),
  getDeviceFingerprint: async () => { ... }
}
```

## 3. 非目标

为了保证边界清晰，本次新内核明确**不承担**以下职责：

### 3.1 不以旧 unitData 为核心数据模型

**说明**：
- 可以提供迁移适配层，但不能让 `unitData` 再次成为内核中心
- 新内核的核心是 `ProjectionRecord` 和 `ChangeRecord`
- `unitData` 可以作为兼容层存在，但不是核心模型

**原因**：
- `unitData` 的语义与新 TDP 不一致
- `unitData` 的树覆盖机制过于复杂
- 保留 `unitData` 会阻碍架构演进

### 3.2 不以旧 Unit 树作为权威领域模型

**说明**：
- 客户端可以派生树，但服务端和内核的正式领域模型必须是显式实体引用
- 树是视图层的辅助结构，不是领域模型

**原因**：
- 树模型会导致服务端领域模型复杂化
- 树模型会导致 TDP scope 解析复杂化
- 树模型会重新引入 `rootPath` 优先级体系

### 3.3 不承担项目级业务域数据解释

**说明**：
- 不包含某个项目特有的菜单数据解析
- 不包含某个项目特有的远控命令解释
- 不包含某个项目特有的 operatingEntity 扩展逻辑

**原因**：
- 这些应该交由更上层模块处理
- 内核应该保持通用性
- 避免内核与业务耦合

**示例**：
```typescript
// ❌ 不应该在内核中
interface MenuItemData {
  menuId: string
  dishName: string
  price: number
  // ... 餐饮业务特有字段
}

// ✅ 应该在业务模块中
// 内核只提供通用的 ProjectionRecord
interface ProjectionRecord {
  projectionId: string
  topicKey: string
  payload: Record<string, unknown>  // 通用 payload
}
```

### 3.4 不承担 UI 层逻辑

**说明**：
- 该包是 `1-kernel` 的 core 包，只负责状态、协议、命令、actor、foundation
- 不负责页面与组件
- UI 层应该通过 hooks 和 selectors 使用内核

**原因**：
- 保持内核的纯粹性
- 便于跨平台复用
- 便于单元测试

## 4. 约束条件

### 4.1 必须遵守 Monorepo 约束

**约束**：
- 使用 yarn workspace 管理依赖
- 版本统一遵循根目录 package 约定
- `dev` 与 `src` 必须分离
- 依赖导出时不得暴露 `dev`

**验证**：
```json
// package.json
{
  "name": "@next/kernel-core-terminal-data",
  "main": "src/index.ts",
  "files": ["src"],
  "scripts": {
    "dev": "...",
    "build": "..."
  }
}
```

### 4.2 必须使用 TypeScript 强类型

**约束**：
- 实体、状态、接口、scope、topic 类型必须显式建模
- 能用类型表示的，不用字符串拼凑
- 所有导出的接口都必须有完整的类型定义

**示例**：
```typescript
// ❌ 不推荐：字符串拼凑
const scopeKey = `${tenantId}:${brandId}:${storeId}`

// ✅ 推荐：类型化
interface ScopeKey {
  scopeType: TdpScopeType
  tenantId?: string
  brandId?: string
  storeId?: string
  terminalId?: string
}

function buildScopeKey(scope: ScopeKey): string {
  // ...
}
```

### 4.3 必须强调抽象与复用

**约束**：
- 连接、存储、日志、网络、设备能力必须抽象
- React Native / Electron 不应在核心逻辑中写死
- 通过 foundation adapter 实现跨平台

**示例**：
```typescript
// ❌ 不推荐：直接依赖平台 API
import DeviceInfo from 'react-native-device-info'

async function getDeviceId() {
  return await DeviceInfo.getUniqueId()
}

// ✅ 推荐：通过 adapter 抽象
interface IDeviceInfoProvider {
  getDeviceId(): Promise<string>
}

// 在 RN 适配层实现
const rnDeviceInfoProvider: IDeviceInfoProvider = {
  getDeviceId: async () => DeviceInfo.getUniqueId()
}

// 在 Electron 适配层实现
const electronDeviceInfoProvider: IDeviceInfoProvider = {
  getDeviceId: async () => { /* Electron 实现 */ }
}
```

### 4.4 必须面向 TCP/TDP 双体系

**约束**：
- TCP 负责治理与结果收集
- TDP 负责数据投递与同步
- 新内核必须同时支持两侧客户端能力

**验证**：
- 有独立的 TCP 客户端模块
- 有独立的 TDP 客户端模块
- 两者可以独立工作，也可以协同工作

## 5. 设计成功标准

新 `terminal-data` 设计完成后，应达到以下标准：

### 5.1 概念一致性

**标准**：
阅读新包接口时，开发者能直接对照服务端设计文档，理解以下概念：
- `TerminalIdentity`：终端身份
- `TerminalBindingContext`：终端绑定上下文
- `TopicDefinition`：TDP Topic 定义
- `ProjectionRecord`：TDP Projection 记录
- `TcpTaskDeliveryEnvelope`：TCP 任务投递信封
- `TdpSessionState`：TDP 会话状态

**验证方法**：
- 新人可以在 30 分钟内理解核心概念
- 开发者可以直接对照服务端文档编写客户端代码
- 不需要额外的"翻译文档"

### 5.2 结构清晰性

**标准**：
开发者能快速知道：
- 哪些是 TCP 状态（`tcpClient` slice）
- 哪些是 TDP 状态（`tdpSession`、`tdpProjection` slice）
- 哪些是运行时状态（`terminalRuntime` slice）
- 哪些是派生视图状态（`terminalDerivedTree` slice）

**验证方法**：
- 目录结构清晰，一眼能看出模块职责
- 文件命名规范，能直接反映内容
- 代码注释完整，关键概念有说明

### 5.3 可替换性

**标准**：
未来项目可逐步用新包替代旧 `terminal` 包的核心职责。

**验证方法**：
- 提供迁移指南文档
- 提供兼容层接口
- 老代码可以逐步迁移，不需要一次性替换

### 5.4 可扩展性

**标准**：
后续可以基于新内核继续扩展：
- 热更新客户端模块
- 配置管理客户端模块
- 远控客户端模块
- 兼容老 `unitData` 的迁移适配模块

**验证方法**：
- 核心接口稳定，扩展不需要修改核心代码
- 提供扩展点（hooks、middlewares、adapters）
- 扩展模块可以独立开发和测试

### 5.5 性能指标

**标准**：
- 激活流程：< 3s（正常网络）
- TDP 连接建立：< 2s（正常网络）
- Projection 同步：< 1s（100 条记录）
- 状态更新响应：< 100ms（UI 刷新）
- 内存占用：< 50MB（正常运行）

**验证方法**：
- 提供性能测试用例
- 在真实设备上测试
- 监控关键指标

### 5.6 测试覆盖率

**标准**：
- 单元测试覆盖率：> 80%
- 集成测试覆盖率：> 60%
- 关键流程必须有 E2E 测试

**验证方法**：
- 使用 Jest 进行单元测试
- 使用 Mock Server 进行集成测试
- 使用真实环境进行 E2E 测试

## 6. 关键结论

### 6.1 为什么需要新包

1. **对齐服务端架构**：服务端已经明确拆分为 TCP 和 TDP，客户端必须跟随演进
2. **提升代码质量**：显式类型、清晰职责、模块化设计
3. **支持多终端形态**：RN、Electron、Web、未来的其他形态
4. **为老包迁移留出空间**：逐步迁移，不需要一次性替换

### 6.2 新包的核心价值

1. **统一的领域语言**：前后端使用相同的概念和术语
2. **清晰的职责边界**：每个模块的职责明确，易于维护
3. **强大的类型安全**：TypeScript 提供完整的类型检查
4. **灵活的扩展能力**：通过 adapter 和 middleware 实现扩展

### 6.3 设计的核心取舍

1. **显式实体 vs 泛化树**：选择显式实体，提供派生树作为视图
2. **类型安全 vs 灵活性**：选择类型安全，通过泛型提供灵活性
3. **通用性 vs 便利性**：选择通用性，通过 helper 提供便利性
4. **纯粹性 vs 功能完整**：选择纯粹性，通过扩展模块提供完整功能
