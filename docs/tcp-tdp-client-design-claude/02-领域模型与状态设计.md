## 1. 领域模型总览

### 1.1 模型分组

新 `terminal-data` 的领域模型分为 6 个核心组：

```
┌─────────────────────────────────────────────────────────┐
│                    领域模型体系                          │
│                                                          │
│  ┌────────────────┐  ┌────────────────┐                │
│  │  Identity      │  │  Binding /     │                │
│  │  实体          │  │  Organization  │                │
│  │                │  │  引用实体      │                │
│  └────────────────┘  └────────────────┘                │
│                                                          │
│  ┌────────────────┐  ┌────────────────┐                │
│  │  TCP 控制面    │  │  TDP 数据面    │                │
│  │  实体          │  │  实体          │                │
│  └────────────────┘  └────────────────┘                │
│                                                          │
│  ┌────────────────┐  ┌────────────────┐                │
│  │  Runtime       │  │  DerivedTree   │                │
│  │  运行时实体    │  │  派生视图实体  │                │
│  └────────────────┘  └────────────────┘                │
└─────────────────────────────────────────────────────────┘
```

### 1.2 模型职责

| 模型组 | 核心职责 | 关键实体 |
|--------|---------|---------|
| Identity | 终端身份与认证 | TerminalIdentity, TerminalCredential |
| Binding/Organization | 组织关系与绑定 | TenantRef, BrandRef, StoreRef, TerminalBindingContext |
| TCP 控制面 | 任务投递与上报 | TcpTaskDeliveryEnvelope, TerminalReportPayload |
| TDP 数据面 | 数据同步与投递 | TopicDefinition, ProjectionRecord, ChangeRecord |
| Runtime | 运行时状态 | DesiredConfigState, DesiredVersionState, DeliveryInboxItem |
| DerivedTree | 派生树视图 | DerivedTreeNode, DerivedTreeNodeType |

### 1.3 模型设计原则

**原则一：显式类型优先**
- 每个实体都有明确的 TypeScript 接口定义
- 不使用泛化的 `any` 或 `Record<string, unknown>` 作为核心类型
- 枚举类型用于表示有限状态集合

**原则二：不可变数据结构**
- 所有实体都是不可变的（immutable）
- 状态更新通过 Redux Toolkit 的 Immer 实现
- 避免直接修改对象属性

**原则三：领域语言一致性**
- 实体命名与服务端设计文档保持一致
- 字段命名遵循 camelCase 规范
- 避免使用缩写和简写

**原则四：可序列化**
- 所有实体都可以序列化为 JSON
- 支持本地持久化和跨进程同步
- 时间戳使用 number 类型（Unix timestamp）

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| terminalId | string | ✅ | 终端唯一标识，由服务端生成 |
| deviceId | string | ✅ | 设备唯一标识，由客户端采集 |
| deviceFingerprint | string | ✅ | 设备指纹，用于设备识别 |
| activationStatus | TerminalActivationStatus | ✅ | 激活状态 |
| activatedAt | number | ❌ | 首次激活时间戳 |
| lastActivatedAt | number | ❌ | 最近激活时间戳 |

**使用场景**：
- 终端激活流程
- 终端身份验证
- 终端状态查询
- 终端重新激活

**状态转换**：
```
UNACTIVATED → ACTIVATING → ACTIVATED
                    ↓
                 SUSPENDED
                    ↓
                DEACTIVATED
```

### 2.2 TerminalActivationStatus

**定义**：
```typescript
enum TerminalActivationStatus {
  UNACTIVATED = 'UNACTIVATED',     // 未激活
  ACTIVATING = 'ACTIVATING',       // 激活中
  ACTIVATED = 'ACTIVATED',         // 已激活
  SUSPENDED = 'SUSPENDED',         // 已暂停
  DEACTIVATED = 'DEACTIVATED',     // 已停用
}
```

**状态说明**：

| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| UNACTIVATED | 终端未激活，初始状态 | 可执行激活操作 |
| ACTIVATING | 终端激活中，临时状态 | 等待激活完成 |
| ACTIVATED | 终端已激活，正常运行 | 可执行所有操作 |
| SUSPENDED | 终端已暂停，临时禁用 | 可恢复激活 |
| DEACTIVATED | 终端已停用，永久禁用 | 需重新激活 |

**状态转换规则**：
```typescript
// 允许的状态转换
const allowedTransitions: Record<TerminalActivationStatus, TerminalActivationStatus[]> = {
  UNACTIVATED: [ACTIVATING],
  ACTIVATING: [ACTIVATED, UNACTIVATED],
  ACTIVATED: [SUSPENDED, DEACTIVATED],
  SUSPENDED: [ACTIVATED, DEACTIVATED],
  DEACTIVATED: [ACTIVATING],
}
```

### 2.3 TerminalCredential

**定义**：
```typescript
interface TerminalCredential {
  accessToken?: string
  refreshToken?: string
  expiresAt?: number
  tokenType?: 'Bearer'
  updatedAt: number
}
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| accessToken | string | ❌ | 访问令牌，用于 API 认证 |
| refreshToken | string | ❌ | 刷新令牌，用于获取新的访问令牌 |
| expiresAt | number | ❌ | 访问令牌过期时间戳 |
| tokenType | 'Bearer' | ❌ | 令牌类型，固定为 Bearer |
| updatedAt | number | ✅ | 凭证更新时间戳 |

**使用场景**：
- TCP API 认证
- Token 自动刷新
- Token 过期检测
- 本地凭证缓存

**Token 刷新策略**：
```typescript
// Token 刷新时机判断
function shouldRefreshToken(credential: TerminalCredential): boolean {
  if (!credential.accessToken || !credential.expiresAt) {
    return false
  }

  const now = Date.now()
  const expiresIn = credential.expiresAt - now
  const refreshThreshold = 5 * 60 * 1000 // 5 分钟

  return expiresIn < refreshThreshold
}
```

**安全要求**：
- accessToken 和 refreshToken 必须加密存储
- 不能在日志中输出完整 token
- Token 过期后必须立即清除
- 支持 Token 撤销机制

## 3. Binding / Organization 引用实体组

### 3.1 设计理念

**为什么使用显式引用实体**：

旧 `terminal` 包使用泛化的 `Unit` 类型表示所有组织实体，导致：
- 类型不明确，需要通过 `type` 字段判断
- IDE 无法提供准确的类型提示
- 容易混淆不同层级的实体

新设计使用显式引用实体：
- 每个组织层级都有独立的类型定义
- 类型安全，编译时检查
- 语义清晰，易于理解和维护

**显式引用 vs 泛化类型**：

```typescript
// ❌ 旧方式：泛化类型
interface Unit {
  id: string
  type: 'tenant' | 'brand' | 'store' | 'terminal'
  name: string
  data: any
}

// ✅ 新方式：显式类型
interface TenantRef {
  tenantId: string
  code?: string
  name: string
}

interface BrandRef {
  brandId: string
  tenantId: string
  code?: string
  name: string
}
```

### 4.3 TerminalPresenceStatus

**定义**：
```typescript

  OFFLINE = 'OFFLINE',     // 离线
  UNKNOWN = 'UNKNOWN',     // 未知
}
```

**状态判断规则**：
```typescript
function determinePresenceStatus(
  tdpSessionStatus: TdpSessionStatus,
  lastSeenAt?: number
): TerminalPresenceStatus {
  // 如果 TDP 会话已连接，则在线
  if (tdpSessionStatus === TdpSessionStatus.CONNECTED) {
    return TerminalPresenceStatus.ONLINE
  }

  // 如果最近 1 分钟内有心跳，则在线
  if (lastSeenAt && Date.now() - lastSeenAt < 60 * 1000) {
    return TerminalPresenceStatus.ONLINE
  }

  // 如果超过 5 分钟未上报，则离线
  if (!lastSeenAt || Date.now() - lastSeenAt > 5 * 60 * 1000) {
    return TerminalPresenceStatus.OFFLINE
  }

  return TerminalPresenceStatus.UNKNOWN
}
```

### 5.2 TdpScopeType

**定义**：
```typescript
enum TdpScopeType {
  TERMINAL = 'TERMINAL',   // 终端级别
  STORE = 'STORE',         // 门店级别
  BRAND = 'BRAND',         // 品牌级别
  TENANT = 'TENANT',       // 租户级别
  GLOBAL = 'GLOBAL',       // 全局级别
}
```

**Scope 层级关系**：
```
GLOBAL (全局)
  └─→ TENANT (租户)
        └─→ BRAND (品牌)
              └─→ STORE (门店)
                    └─→ TERMINAL (终端)
```

**Scope 解析规则**：
```typescript
function resolveScopeKey(
  scopeType: TdpScopeType,
  context: TerminalBindingContext
): string {
  switch (scopeType) {
    case TdpScopeType.TERMINAL:
      return context.terminalId
    case TdpScopeType.STORE:
      return context.store.storeId
    case TdpScopeType.BRAND:
      return context.brand.brandId
    case TdpScopeType.TENANT:
      return context.tenant.tenantId
    case TdpScopeType.GLOBAL:
      return 'global'
    default:
      throw new Error(`Unknown scope type: ${scopeType}`)
  }
}
```

### 7.2 DerivedTreeNode

**定义**：
```typescript
interface DerivedTreeNode {
  nodeId: string
  nodeType: DerivedTreeNodeType
  refId: string
  label: string
  parentNodeId?: string
  childrenNodeIds: string[]
  metadata?: Record<string, unknown>
}
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| nodeId | string | ✅ | 节点唯一 ID |
| nodeType | DerivedTreeNodeType | ✅ | 节点类型 |
| refId | string | ✅ | 引用实体 ID |
| label | string | ✅ | 节点标签 |
| parentNodeId | string | ❌ | 父节点 ID |
| childrenNodeIds | string[] | ✅ | 子节点 ID 列表 |
| metadata | Record<string, unknown> | ❌ | 元数据 |

**重要说明**：

> ⚠️ **派生树是客户端内部视图结构，不是服务端权威模型**
>
> - 服务端权威模型：显式实体引用（Tenant/Brand/Store/Terminal）
> - 客户端派生树：基于显式实体构建的树形视图
> - 派生树可以随时重建，不影响服务端模型

**使用场景**：
- UI 层树形展示
- Scope 解析辅助
- 组织关系浏览
- 快速查找父子关系

**派生树构建示例**：
```typescript
function buildDerivedTree(context: TerminalBindingContext): DerivedTreeNode[] {
  const nodes: DerivedTreeNode[] = []

  // 1. 创建 ROOT 节点
  nodes.push({
    nodeId: 'root',
    nodeType: DerivedTreeNodeType.ROOT,
    refId: 'root',
    label: 'Root',
    childrenNodeIds: [context.tenant.tenantId],
  })

  // 2. 创建 TENANT 节点
  nodes.push({
    nodeId: context.tenant.tenantId,
    nodeType: DerivedTreeNodeType.TENANT,
    refId: context.tenant.tenantId,
    label: context.tenant.name,
    parentNodeId: 'root',
    childrenNodeIds: [context.brand.brandId],
  })

  // 3. 创建 BRAND 节点
  nodes.push({
    nodeId: context.brand.brandId,
    nodeType: DerivedTreeNodeType.BRAND,
    refId: context.brand.brandId,
    label: context.brand.name,
    parentNodeId: context.tenant.tenantId,
    childrenNodeIds: [context.store.storeId],
  })

  // 4. 创建 STORE 节点
  nodes.push({
    nodeId: context.store.storeId,
    nodeType: DerivedTreeNodeType.STORE,
    refId: context.store.storeId,
    label: context.store.name,
    parentNodeId: context.brand.brandId,
    childrenNodeIds: [context.terminalId],
  })

  // 5. 创建 TERMINAL 节点
  nodes.push({
    nodeId: context.terminalId,
    nodeType: DerivedTreeNodeType.TERMINAL,
    refId: context.terminalId,
    label: `Terminal ${context.terminalId}`,
    parentNodeId: context.store.storeId,
    childrenNodeIds: [context.profile.profileId],
  })

  // 6. 创建 PROFILE 节点
  nodes.push({
    nodeId: context.profile.profileId,
    nodeType: DerivedTreeNodeType.PROFILE,
    refId: context.profile.profileId,
    label: context.profile.name,
    parentNodeId: context.terminalId,
    childrenNodeIds: [],
    metadata: {
      capabilities: context.profile.capabilities,
    },
  })

  return nodes
}
```

**树形查询辅助函数**：
```typescript
// 查找节点
function findNode(nodes: DerivedTreeNode[], nodeId: string): DerivedTreeNode | undefined {
  return nodes.find(n => n.nodeId === nodeId)
}

// 查找父节点
function findParent(nodes: DerivedTreeNode[], nodeId: string): DerivedTreeNode | undefined {
  const node = findNode(nodes, nodeId)
  return node?.parentNodeId ? findNode(nodes, node.parentNodeId) : undefined
}

// 查找子节点
function findChildren(nodes: DerivedTreeNode[], nodeId: string): DerivedTreeNode[] {
  const node = findNode(nodes, nodeId)
  return node ? node.childrenNodeIds.map(id => findNode(nodes, id)).filter(Boolean) : []
}

// 查找祖先路径
function findAncestorPath(nodes: DerivedTreeNode[], nodeId: string): DerivedTreeNode[] {
  const path: DerivedTreeNode[] = []
  let current = findNode(nodes, nodeId)

  while (current) {
    path.unshift(current)
    current = current.parentNodeId ? findNode(nodes, current.parentNodeId) : undefined
  }

  return path
}
```

### 8.7 tdpProjection Slice

**职责**：
- 保存 Projection 记录
- 保存 Change 记录
- 保存 Topic 目录

**状态定义**：
```typescript
interface TdpProjectionState {
  projections: Record<string, ProjectionRecord>  // key: `${topicKey}:${scopeKey}`
  changes: Record<string, ChangeRecord[]>        // key: `${topicKey}:${scopeKey}`
  topics: Record<string, TopicDefinition>        // key: topicKey
  lastSyncAt?: number
}
```

**初始状态**：
```typescript
const initialState: TdpProjectionState = {
  projections: {},
  changes: {},
  topics: {},
  lastSyncAt: undefined,
}
```

**Reducers**：
```typescript
const tdpProjectionSlice = createSlice({
  name: 'tdpProjection',
  initialState,
  reducers: {
    setProjection: (state, action: PayloadAction<ProjectionRecord>) => {
      const key = `${action.payload.topicKey}:${action.payload.scopeKey}`
      state.projections[key] = action.payload
      state.lastSyncAt = Date.now()
    },
    addChange: (state, action: PayloadAction<ChangeRecord>) => {
      const key = `${action.payload.topicKey}:${action.payload.scopeKey}`
      if (!state.changes[key]) {
        state.changes[key] = []
      }
      state.changes[key].push(action.payload)
    },
    setTopic: (state, action: PayloadAction<TopicDefinition>) => {
      state.topics[action.payload.topicKey] = action.payload
    },
    clearChanges: (state, action: PayloadAction<{ topicKey: string; scopeKey: string }>) => {
      const key = `${action.payload.topicKey}:${action.payload.scopeKey}`
      delete state.changes[key]
    },
    resetProjections: (state) => {
      state.projections = {}
      state.changes = {}
      state.topics = {}
      state.lastSyncAt = undefined
    },
  },
})
```

**持久化策略**：
- ✅ 需要持久化
- 持久化字段：`projections`, `topics`
- 不持久化字段：`changes`（可选，根据容量策略）

### 8.8 tdpInbox Slice

**职责**：
- 保存投递收件箱项
- 管理消费状态

**状态定义**：
```typescript
interface TdpInboxState {
  items: Record<string, DeliveryInboxItem>  // key: envelopeId
  pendingCount: number
  consumedCount: number
}
```

**初始状态**：
```typescript
const initialState: TdpInboxState = {
  items: {},
  pendingCount: 0,
  consumedCount: 0,
}
```

**Reducers**：
```typescript
const tdpInboxSlice = createSlice({
  name: 'tdpInbox',
  initialState,
  reducers: {
    addInboxItem: (state, action: PayloadAction<DeliveryInboxItem>) => {
      state.items[action.payload.envelopeId] = action.payload
      state.pendingCount += 1
    },
    updateItemStatus: (
      state,
      action: PayloadAction<{ envelopeId: string; status: DeliveryInboxItem['consumeStatus']; consumedAt?: number }>
    ) => {
      const item = state.items[action.payload.envelopeId]
      if (item) {
        item.consumeStatus = action.payload.status
        if (action.payload.consumedAt) {
          item.consumedAt = action.payload.consumedAt
        }
        if (action.payload.status === 'CONSUMED') {
          state.pendingCount -= 1
          state.consumedCount += 1
        }
      }
    },
    removeInboxItem: (state, action: PayloadAction<string>) => {
      const item = state.items[action.payload]
      if (item) {
        if (item.consumeStatus === 'PENDING' || item.consumeStatus === 'CONSUMING') {
          state.pendingCount -= 1
        }
        delete state.items[action.payload]
      }
    },
    clearConsumedItems: (state) => {
      Object.keys(state.items).forEach(key => {
        if (state.items[key].consumeStatus === 'CONSUMED') {
          delete state.items[key]
        }
      })
      state.consumedCount = 0
    },
    resetInbox: (state) => {
      state.items = {}
      state.pendingCount = 0
      state.consumedCount = 0
    },
  },
})
```

**持久化策略**：
- ✅ 需要持久化
- 持久化所有字段

## 9. 为什么不再用旧 UnitDataState

### 9.1 旧 UnitDataState 的问题

旧 `terminal` 包使用 `UnitDataState` 作为核心状态模型：

```typescript
// ❌ 旧方式
interface UnitDataState {
  [unitDataId: string]: {
    id: string
    groupKey: string
    data: any
    updatedAt: number
  }
}
```

**问题一：语义不清晰**
- `unitDataId` 是什么？需要查看代码才能理解
- `groupKey` 代表什么？与 Topic 的关系是什么？
- `data` 是什么类型？无法提供类型检查

**问题二：与新 TDP 模型不一致**
- TDP 的核心是 `ProjectionRecord` 和 `ChangeRecord`
- `unitData` 的 group 机制与 TDP 的 Topic/Scope 机制不同
- 无法直接映射到新的服务端模型

**问题三：树覆盖语义复杂**
- `unitData` 依赖 `rootPath` 优先级覆盖
- 覆盖规则难以理解和维护
- 容易出现数据不一致

### 9.2 新模型的优势

新 `terminal-data` 使用显式的状态模型：

```typescript
// ✅ 新方式
interface TdpProjectionState {
  projections: Record<string, ProjectionRecord>
  changes: Record<string, ChangeRecord[]>
  topics: Record<string, TopicDefinition>
}
```

**优势一：语义清晰**
- `ProjectionRecord` 明确表示数据快照
- `ChangeRecord` 明确表示增量变更
- `TopicDefinition` 明确表示 Topic 定义

**优势二：与服务端一致**
- 直接对应 TDP 服务端的概念
- 降低前后端沟通成本
- 便于理解和维护

**优势三：类型安全**
- 每个实体都有明确的类型定义
- TypeScript 提供完整的类型检查
- IDE 可以提供准确的类型提示

### 9.3 迁移路径

对于需要兼容旧 `unitData` 的项目，可以提供兼容层：

```typescript
// 兼容层：unitData 到 Projection 的转换
function convertUnitDataToProjection(unitData: UnitData): ProjectionRecord {
  return {
    projectionId: unitData.id,
    topicKey: unitData.groupKey,
    scopeType: inferScopeType(unitData),
    scopeKey: inferScopeKey(unitData),
    revision: 0,
    payload: unitData.data,
    updatedAt: unitData.updatedAt,
  }
}

// 兼容层：Projection 到 unitData 的转换
function convertProjectionToUnitData(projection: ProjectionRecord): UnitData {
  return {
    id: projection.projectionId,
    groupKey: projection.topicKey,
    data: projection.payload,
    updatedAt: projection.updatedAt,
  }
}
```

## 10. 状态持久化策略

### 10.1 持久化需求

| Slice | 持久化 | 原因 |
|-------|--------|------|
| terminalIdentity | ✅ | 终端身份需要跨会话保持 |
| terminalBinding | ✅ | 绑定关系需要跨会话保持 |
| terminalRuntime | ✅ | 运行时状态需要跨会话保持 |
| tcpClient | ⚠️ | 部分持久化（bootstrapData） |
| tdpSession | ❌ | 会话状态是瞬时的 |
| tdpProjection | ✅ | 数据缓存需要持久化 |
| tdpInbox | ✅ | 未消费的消息需要持久化 |
| terminalDerivedTree | ✅ | 派生树可以持久化以提升性能 |

const persistConfig = {

    'tdpProjection',
    'tdpInbox',
    'terminalDerivedTree',

  blacklist: [
    'tdpSession',

const persistedReducer = persistReducer(persistConfig, rootReducer)

## 11. 状态设计总结

### 11.1 核心变化

新状态设计的核心变化：

1. **从树覆盖式状态转向显式引用 + 投影缓存式状态**
   - 旧：`Unit` 树 + `rootPath` 覆盖
   - 新：显式实体引用 + `ProjectionRecord` 缓存

2. **从 kernel unitData group 转向 topic / projection / change / inbox**
   - 旧：`unitData` + `groupKey`
   - 新：`TopicDefinition` + `ProjectionRecord` + `ChangeRecord`

3. **从万能 terminal 包转向多子域状态**
   - 旧：所有状态混在一起
   - 新：identity / tcp / tdp / runtime / derived view 分域管理

### 11.2 设计优势

1. **类型安全**
   - 每个实体都有明确的 TypeScript 类型定义
   - 编译时类型检查
   - IDE 智能提示

2. **语义清晰**
   - 实体命名与服务端设计一致
   - 状态职责明确
   - 易于理解和维护

3. **可扩展**
   - 清晰的模块边界
   - 灵活的扩展点
   - 便于后续功能增强

4. **可测试**
   - 状态逻辑独立
   - 易于编写单元测试
   - 便于 mock 和 stub

### 11.3 与旧包对比

| 维度 | 旧 terminal 包 | 新 terminal-data 包 |
|------|---------------|-------------------|
| 核心模型 | Unit / unitData | TerminalIdentity / ProjectionRecord |
| 组织关系 | 树状覆盖（rootPath） | 显式引用（TenantRef/BrandRef/StoreRef） |
| 数据同步 | unitData group | Topic / Projection / Change |
| 任务投递 | 远程消息 | Task Delivery Envelope |
| 状态管理 | 混合在一起 | 分域管理（8 个 slices） |
| 类型安全 | 弱类型（any） | 强类型（TypeScript） |
| 持久化 | 部分支持 | 完整支持 |
| 跨进程同步 | 不支持 | 完整支持 |

## 12. 关键结论

### 12.1 领域模型设计

1. **显式实体优先**：使用 `TenantRef`/`BrandRef`/`StoreRef` 等显式类型，不使用泛化的 `Unit`
2. **客户端派生树**：允许构建派生树视图，但不作为权威模型
3. **控制面与数据面解耦**：TCP 和 TDP 分别建模

### 12.2 状态管理设计

1. **8 个 Redux slices**：identity / binding / runtime / tcpClient / tdpSession / tdpProjection / tdpInbox / derivedTree
2. **持久化策略**：大部分状态需要持久化，会话状态除外
3. **跨进程同步**：使用 MASTER_TO_SLAVE 模式

### 12.3 与旧包的关系

1. **替代范围**：身份、激活、Token、TDP 会话、数据同步、任务投递
2. **不替代范围**：unitData 模型、operatingEntity 树、项目特定逻辑
3. **迁移策略**：提供兼容层，逐步迁移
