# 01-架构与边界设计

---

## 1. 总体定位

`terminal-data` 的定位是：

> 面向终端侧的 TCP/TDP 客户端通用数据内核。

它位于 `1-kernel` 的 core 层，既不是业务模块，也不是 UI 模块，而是终端运行时最底层的数据与同步内核之一。

---

## 2. 与旧 `terminal` 包的关系

### 2.1 旧包本质定位
旧 `terminal` 包本质上是：
- 终端身份模块
- kernel http + ws 协议客户端
- unitData 本地状态机
- 老组织树语义容器

### 2.2 新包替代范围
新 `terminal-data` 首期替代旧包中的以下核心职责：
- 终端身份与激活状态
- token 与认证态
- TDP 会话与连接
- 终端本地数据快照与投递同步
- TCP 任务接收与结果回报

### 2.3 暂不直接替代范围
以下内容不在本次新内核首期中作为核心模型落入：
- 旧 `unitData` 的 group 模型
- operatingEntity / model 的历史树覆盖语义
- 老 kernel ws 特定消息协议
- 项目级远程命令语义

---

## 3. 新内核包含什么

### 3.1 Identity 子域
负责：
- 设备信息采集后形成终端身份上下文
- 终端激活态管理
- token / credential 管理
- 绑定上下文管理

### 3.2 TCP Client 子域
负责：
- 与 TCP 服务的 HTTP 通信
- 激活 / token refresh / bootstrap / 结果上报
- 配置、版本、任务结果等控制面交互

### 3.3 TDP Client 子域
负责：
- Session connect / heartbeat / disconnect
- 连接重建
- snapshot / change catch-up
- topic / projection 消费与本地缓存

### 3.4 Runtime State 子域
负责：
- 本地终端运行时数据
- 终端目标版本、目标配置、任务收件箱
- 最近同步时间、最近错误、最近会话结果

### 3.5 Derived View 子域
负责：
- 将显式实体引用构造成客户端派生树
- 构造 scope 解析辅助结构
- 提供 UI / selector 更友好的组织视图

---

## 4. 新内核不包含什么

### 4.1 不包含业务域解释器
例如：
- 菜品域解释
- 订单域解释
- 履约域解释
这些应由更上层业务模块处理。

### 4.2 不包含项目定制终端动作实现
例如：
- 某项目专属打印参数处理
- 某项目专属远控命令执行器

### 4.3 不包含服务端权威组织树
客户端可以派生树，但服务端不需要为了这个内核反向改造成树权威模型。

---

## 5. 为什么采用“显式实体 + 派生树”的混合模式

这是本设计的核心边界选择。

### 5.1 显式实体的好处
显式实体模型中：
- `tenant`
- `brand`
- `store`
- `profile`
- `terminal`
都是明确语义对象。

这样做的好处是：
- 语义清楚
- 接口清楚
- 与 TCP / TDP 服务端设计一致
- 不容易把组织关系和能力关系混淆

### 5.2 派生树的好处
客户端派生树保留的价值在于：
- 兼容旧终端包用户的心智
- 给 UI 提供树形浏览能力
- 给 scope 解析提供更直观视图
- 方便做按组织节点的选择与过滤

### 5.3 为什么不让树成为权威模型
如果树成为权威模型，会导致：
- 服务端领域模型复杂化
- TDP scope 解析复杂化
- profile 等非组织节点被强行塞进树
- 重新引入旧 `rootPath` 优先级体系

因此：
- **服务端权威模型保持显式实体**
- **客户端内部允许派生树视图**
是最优平衡。

## 6. 与 `1-kernel` 框架的对齐方式

根据旧 `terminal` 与其他 core 包的实现方式，新包必须遵循如下结构：

### 6.1 AppModule 结构
新包将以 `AppModule` 形式导出：
- `name`
- `version`
- `slices`
- `middlewares`
- `epics`
- `commands`
- `actors`
- `errorMessages`
- `parameters`
- `dependencies`
- `modulePreSetup`

### 6.2 slices 负责状态权威
所有终端客户端状态必须以 slice 为权威来源，而不是散落在 foundation 单例中。

### 6.3 commands 负责动作抽象
所有关键动作应使用 `createModuleCommands` 统一建模，避免上层直接调用散落函数。

### 6.4 actors 负责副作用执行
所有：
- http 调用
- session 建立
- projection 同步
- 上报结果
- 本地持久化协作
应由 actor 承担副作用编排。

### 6.5 foundations 负责底层抽象
底层组件例如：
- transport adapter
- storage adapter
- tree builder
- scope resolver
- sync coordinator
应归入 `foundations`。

## 7. 依赖建议

### 7.1 必选依赖
- `@impos2/kernel-core-base`
- `@impos2/kernel-core-interconnection`

### 7.2 谨慎依赖
- `@impos2/kernel-core-task`
- `@impos2/kernel-core-navigation`

建议原则：
- 新 `terminal-data` 不主动依赖 task / navigation 作为核心职责
- 若未来某项目需要把任务执行器或导航逻辑接入，应由上层模块扩展

这样可以保证新内核更通用、更纯粹。

---

## 8. 包内角色分层

建议形成如下五层：

### 第一层：types
定义所有：
- 实体
- 引用对象
- state
- api dto
- shared enum

### 第二层：supports
定义：
- apis
- parameters
- errors
- constants

### 第三层：foundations
定义：
- clients
- adapters
- coordinators
- builders
- resolvers
- repositories

### 第四层：features
定义：
- slices
- commands
- actors
- middlewares
- epics
- selectors

### 第五层：application
定义：
- modulePreSetup
- bootstrap 流程
- 初始化 wiring

---

## 9. 外部模块如何使用新内核

### 9.1 上层业务模块
可通过：
- hooks
- selectors
- commands
读取与驱动新内核。

### 9.2 RN / Electron 适配层
通过 foundation adapter 注入：
- storage
- ws transport
- device info provider
- network info provider
- app lifecycle provider

### 9.3 UI 层
通过 selectors 获取：
- identity
- session state
- projection state
- task inbox
- derived tree

---

## 10. 边界总结

### 应在内核内的
- 激活
- token
- binding context
- TDP session
- projection cache
- delivery inbox
- config/version target state
- derived tree view

### 不应在内核内的
- 旧 unitData 覆盖语义
- 项目业务域对象树
- 具体页面流程
- 具体业务远控动作解释
