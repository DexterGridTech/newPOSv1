# 00-设计背景与目标

---

## 1. 背景

当前老工程中的 `_old_/1-kernel/1.1-cores/terminal` 包，承担了大量历史职责：

- 终端激活与反激活
- operatingEntity 设置
- token 注入
- kernel API 接口封装
- kernel WS 长连接
- 心跳与远程消息接收
- `unitData` 组数据同步
- 远程命令确认与设备状态上报
- 终端、model、entity 的树状关系管理

这套实现对旧工程是可工作的，但它的问题也非常明显：

### 1.1 历史问题

#### 问题一：职责混杂
同一个包里同时包含：
- 终端身份
- 业务组织树
- 数据分发
- 连接管理
- 设备状态上报
- 业务单元数据覆盖

导致边界模糊，无法自然映射到现在已经明确设计好的 TCP / TDP 服务体系。

#### 问题二：旧模型语义太泛
旧 `Unit` / `unitData` 的通用树抽象能力很强，但副作用也大：
- 阅读代码时很难直接判断当前对象的真实业务语义
- 容易把组织层级、终端能力、配置覆盖、实体归属混在一起
- 与新 TCP / TDP 的显式模型不一致

#### 问题三：协议绑定过重
旧 `terminal` 直接围绕：
- old kernel http api
- old kernel ws
- old unitData group
展开实现，导致复用能力受限。

#### 问题四：难以支持新体系演进
现在新的生产设计已经拆出：
- TCP（Terminal Control Plane）
- TDP（Terminal Data Plane）

如果客户端内核还继续维持老 `terminal` 的思维方式，就会出现：
- 服务端一套概念
- 客户端另一套概念
- 中间需要大量翻译层
- 最终难以形成真正通用的内核

---

## 2. 新包设计目标

新包命名建议为：

- npm 包：`@next/kernel-core-terminal-data`
- 模块名：`kernel.core.terminal-data`
- 目录：`_old_/1-kernel/1.1-cores/terminal-data`

它的设计目标不是简单“把旧 terminal 包重命名”，而是要成为新的终端侧基础内核。

### 2.1 核心目标

#### 目标一：与 TCP / TDP 生产设计一致
新包的概念体系必须与：
- `design/4.终端数据平面服务`
- `design/5.终端控制平面服务`
保持一致。

#### 目标二：成为终端侧通用数据内核
它要能服务于：
- React Native 裸工程终端
- Electron 终端
- Web 调试终端
- 未来其他载体终端

#### 目标三：延续 `1-kernel` 的模块框架
必须遵守现有 `1-kernel` 的开发框架：
- `AppModule`
- `slices`
- `commands`
- `actors`
- `middlewares`
- `epics`
- `modulePreSetup`
- `foundations`
- `supports`
- `hooks`
- `selectors`

#### 目标四：支持显式实体 + 客户端派生树
服务端保持显式实体权威模型：
- tenant
- brand
- store
- profile
- terminal

客户端允许基于这些实体构建：
- 组织树视图
- 范围树视图
- 兼容导航树视图

但派生树仅作为客户端视图与辅助计算结构，不替代服务端权威模型。

#### 目标五：为老 terminal 的后续拆迁留出空间
本次设计不把老 terminal 包的全部内容直接吞进去，而是：
- 先建立通用 TCP/TDP 客户端内核
- 再把老 terminal 的剩余内容迁移到更合适的位置

---

## 3. 非目标

为了保证边界清晰，本次新内核明确不承担以下职责：

### 3.1 不以旧 `unitData` 为核心数据模型
可以提供迁移适配层，但不能让 `unitData` 再次成为内核中心。

### 3.2 不以旧 `Unit` 树作为权威领域模型
客户端可以派生树，但服务端和内核的正式领域模型必须是显式实体引用。

### 3.3 不承担项目级业务域数据解释
例如：
- 某个项目特有的菜单数据解析
- 某个项目特有的远控命令解释
- 某个项目特有的 operatingEntity 扩展逻辑

这些应该交由更上层模块处理。

### 3.4 不承担 UI 层逻辑
该包是 `1-kernel` 的 core 包，只负责：
- 状态
- 协议
- 命令
- actor
- foundation
不负责页面与组件。

## 4. 约束

### 4.1 必须遵守 monorepo 约束
- 使用 yarn workspace
- 版本统一遵循根目录 package 约定
- `dev` 与 `src` 必须分离
- 依赖导出时不得暴露 `dev`

### 4.2 必须使用 TypeScript 强类型
- 实体、状态、接口、scope、topic 类型必须显式建模
- 能用类型表示的，不用字符串拼凑

### 4.3 必须强调抽象与复用
- 连接、存储、日志、网络、设备能力必须抽象
- React Native / Electron 不应在核心逻辑中写死

### 4.4 必须面向 TCP / TDP 双体系
- TCP 负责治理与结果收集
- TDP 负责数据投递与同步
新内核必须同时支持两侧客户端能力。

---

## 5. 设计成功标准

新 `terminal-data` 设计完成后，应达到以下标准：

### 5.1 概念一致
阅读新包接口时，开发者能直接对照：
- TerminalIdentity
- TerminalBindingContext
- TopicDefinition
- ProjectionRecord
- TaskDeliveryEnvelope
- TdpSessionState

而不是再去猜某个 `Unit` 到底是什么。

### 5.2 结构清晰
开发者能快速知道：
- 哪些是 TCP 状态
- 哪些是 TDP 状态
- 哪些是运行时状态
- 哪些是派生视图状态

### 5.3 可替换老包
未来项目可逐步用新包替代旧 `terminal` 包的核心职责。

### 5.4 可扩展
后续可以基于新内核继续扩展：
- 热更新客户端模块
- 配置管理客户端模块
- 远控客户端模块
- 兼容老 `unitData` 的迁移适配模块
