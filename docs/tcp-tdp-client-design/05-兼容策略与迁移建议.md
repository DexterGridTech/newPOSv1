# 05-兼容策略与迁移建议

---

## 1. 设计结论

新 `terminal-data` 不是旧 `terminal` 的逐文件平移，而是一次概念重建。

因此迁移策略必须遵循：

- **先建立新内核**
- **再逐步迁移旧 terminal 职责**
- **不把旧 unitData / Unit 树原样搬进来**

---

## 2. 旧 `terminal` 包中各能力的归位建议

### 2.1 终端激活与 token
归入新 `terminal-data`。

### 2.2 kernel API token request interceptor
归入新 `terminal-data.modulePreSetup`，但仅面向 TCP client。

### 2.3 kernel WS 连接
不直接迁移。  
应重构为：
- `TdpTransportAdapter`
- `TdpSessionClient`
- `SessionCoordinator`

### 2.4 unitData 同步
不作为核心迁移。  
如确有兼容需求，应在后续设计 `terminal-data-legacy-unitdata-adapter` 一类扩展包。

### 2.5 operatingEntity / model / terminal 树
不迁移为权威模型。  
若某些项目仍需树状视图，应在客户端派生树中承接。

### 2.6 sendDeviceState / remoteCommandConfirm
需要拆分：
- 通用生命周期 / report 类可迁入 `terminal-data`
- 项目级远控语义不应直接内置

---

## 3. 为什么不建议直接兼容旧 `unitData`

如果直接兼容，会出现几个问题：

### 3.1 新旧语义冲突
新 TDP 的中心是：
- Topic
- Scope
- Projection
- ChangeLog

旧 `unitData` 的中心是：
- group
- unit path
- 覆盖优先级

两者不是一一对应关系。

### 3.2 客户端会重新陷入“树优先”思维
这会削弱：
- TCP/TDP 设计边界
- 显式实体设计优势
- 终端侧数据模型清晰度

### 3.3 迁移成本会被无限拖延
如果新内核里还继续以 `unitData` 为中心，就会导致：
- 大量兼容代码常驻
- 永远无法真正退掉老设计

---

## 4. 推荐迁移阶段

### 阶段一：建立新内核，不接旧 unitData
新建 `terminal-data`，完成：
- identity
- binding context
- tcp client
- tdp session
- projection / change / inbox
- runtime state
- derived tree

### 阶段二：让新项目直接接入新内核
所有新的 TCP/TDP 场景直接使用新包。

### 阶段三：对老项目做适配层
为旧 terminal 相关调用提供：
- facade
- mapper
- legacy adapter

### 阶段四：逐步迁出旧 terminal 剩余职责
例如迁出到：
- legacy-terminal-domain
- legacy-unitdata-adapter
- legacy-kernel-ws-bridge
等专门迁移包。

---

## 5. 客户端派生树的兼容价值

虽然我们不保留旧 `Unit` 树作为权威模型，但派生树仍然很有价值：

### 5.1 对开发者心智的兼容
旧 terminal 包用户仍然习惯：
- 从上到下看层级
- 用树来理解归属关系

### 5.2 对 UI 的兼容
很多管理界面、调试界面依然更适合：
- 展示 tenant -> brand -> store -> terminal
- 做树状选择器
- 做组织范围过滤

### 5.3 对 scope 解析辅助的兼容
虽然 scope 权威判断仍依赖显式实体，但树能帮助客户端更直观地理解范围关系。

---

## 6. 风险与控制点

### 风险一：为了兼容，把老设计整包搬回新内核
控制策略：
- 新包只吸收通用 TCP/TDP 客户端能力
- 老设计通过单独兼容层解决

### 风险二：派生树失控，重新变成权威模型
控制策略：
- 文档中明确声明：派生树不是权威模型
- 任何 scope 正式判断仍基于显式实体引用

### 风险三：新包又依赖太多上层包
控制策略：
- 保持依赖最小化
- 不把 navigation / task / 业务域包强行拉进内核

### 风险四：新旧状态同时存在导致混乱
控制策略：
- 迁移期明确“哪个状态是权威”
- 新项目禁止直接依赖旧 `terminal` 状态

---

## 7. 命名与接口兼容建议

### 7.1 命名建议
避免继续使用旧 `kernel` 语义命名，例如：
- `connectKernelWS`
- `kernelWSConnected`
- `kernelWSDisconnected`

建议替换为：
- `connectTdpSession`
- `tdpSessionConnected`
- `tdpSessionDisconnected`

### 7.2 状态命名建议
避免：
- `unitData_systemParameters`
- `unitData_errorMessages`

改为：
- `tdpProjection`
- `terminalRuntime`
- `tcpClient`

### 7.3 接口命名建议
避免围绕“设备”或“kernel”命名历史接口，改为围绕 TCP/TDP 语义命名。

---

## 8. 后续可追加的扩展包建议

在新内核稳定后，可以继续规划：

### 8.1 `@next/kernel-core-terminal-control`
承载更偏 TCP 控制域的专用扩展能力：
- 热更新
- 配置治理
- 远控治理

### 8.2 `@next/kernel-core-terminal-data-legacy`
承载旧 terminal / unitData 的兼容层。

### 8.3 `@next/kernel-core-terminal-runtime`
承载更运行时的设备感知与本地执行器能力。

这样可以保持：
- `terminal-data` 足够稳定、足够通用
- 兼容与业务扩展不污染核心

---

## 9. 最终迁移建议

如果以“替代旧 terminal 包”为目标，我建议最终路径是：

1. 新建并稳定 `terminal-data`
2. 新项目优先接新包
3. 旧项目通过 facade/adapter 逐步迁移
4. 把旧 `terminal` 拆成若干遗留兼容模块
5. 最终让 `terminal-data` 成为唯一的 TCP/TDP 客户端通用内核

---

## 10. 一句话总结

新 `terminal-data` 的迁移策略，不是“把旧 terminal 包复制一份”，而是：

> 用 TCP/TDP 的新生产体系重建终端客户端内核；旧 terminal 的历史能力按职责拆散、兼容、迁移。
