# HTTP 与 WS 通信体系评审与重设计方案

## 1. 本次修订的前提边界

这次修订基于一个非常重要的边界澄清：

> 本报告中的“API”，特指 **HTTP 服务通信**，不是泛指所有 kernel 对外能力。

也就是说：

- **HTTP 服务调用** 是一个独立通信体系。
- **WebSocket 通信** 也是一个独立通信体系。
- 二者都属于“通信基础设施”，应该放在统一设计视角下考虑。
- 但像 TurboModule、Native Bridge、Connector 这类非 HTTP / 非 WS 的本地能力，不属于本报告重设计的主体范围。

因此，这次我不再把“HTTP、WS、Native、Bridge”全部混成一个大而泛的“API 体系”，而是明确收敛为：

- **HTTP 服务通信体系**
- **WebSocket 通信体系**
- **二者共享的通信契约与运行时基础设施**

## 2. 本次重点关注的三个基础设施位置

你特别指出了工程中三个关键位置：

1. `1-kernel/1.1-cores/base/src/foundations/http`
2. `1-kernel/1.1-cores/interconnection/src/foundations/master-ws`
3. `1-kernel/1.1-cores/terminal/src/foundations/kernel-ws`

这三个位置非常关键，因为它们已经构成了当前 kernel 通信体系的雏形：

- `http`：负责 HTTP 服务调用
- `master-ws`：负责 interconnection 场景下的双端 WS 通信
- `kernel-ws`：负责 terminal 与 kernel server 之间的 WS 通信

所以，本次设计的核心目标不是“再设计一套单点 API 抽象”，而是：

> 重新设计一套 **统一的通信基础设施分层**，使 HTTP 与 WS 都有清晰职责、统一风格、强类型描述和可扩展边界。

## 3. 当前通信体系现状总览

### 3.1 当前 HTTP 通信体系

当前 HTTP 通信体系主要由以下部分组成：

- `Api<T, R>`：单个 HTTP 端点描述 + 调用入口
- `ApiManager`：server config、axios instance、retry、request queue、metrics、interceptor
- `server-config`：serverName 与 serverSpace 配置
- `supports/apis/index.ts`：各模块按业务聚合 HTTP 接口
- Actor：通过 `api.run({ request })` 使用 HTTP 服务

它本质上是一套：

- 以 `axios` 为执行底座
- 以 `serverName + path + method` 为端点描述方式
- 以 `RequestWrapper/ResponseWrapper` 为 transport envelope
- 以 `Actor` 为直接消费方

的 HTTP 服务调用体系。

### 3.2 当前 WS 通信体系

当前 WS 并不是一套统一体系，而是两套相互独立、用途不同的实现。

#### 3.2.1 `master-ws`

位置：

- `1-kernel/1.1-cores/interconnection/src/foundations/master-ws`

它的特点：

- 先 `HTTP POST /register`
- 再 `WebSocket /ws?token=...`
- 主要用于 master/slave 的 1:1 配对和双向转发
- 服务端发心跳，客户端被动 ACK
- 内部是一个完整的连接状态机 + 事件系统 + failover 逻辑

#### 3.2.2 `kernel-ws`

位置：

- `1-kernel/1.1-cores/terminal/src/foundations/kernel-ws`

它的特点：

- 通过 `ApiManager.getServerConfig(api.serverName())` 复用服务地址配置
- 使用 `api.path()` 拼接出 WS 地址
- 通过 query 参数传 `deviceId` 和 `token`
- 主要用于 terminal 与 kernel server 的连接、心跳和消息处理
- 也是完整的连接状态机 + queue + heartbeat + failover

### 3.3 当前现状的本质判断

如果抽象来看，当前其实已经存在两类通信：

1. **Request/Response 型通信** → HTTP
2. **Connection/Message Stream 型通信** → WS

问题不在于“有没有实现”，而在于：

- HTTP 和 WS 的基础设施没有从架构层统一规划
- 二者共享了一部分概念，但没有共享统一通信描述模型
- 二者又各自发展了一套实现，出现了一些重复和边界不清问题

## 4. 当前设计的亮点

### 4.1 HTTP 已经有一个相对成熟的统一执行底座

`1-kernel/1.1-cores/base/src/foundations/http` 这套能力，其实已经包含了不少成熟特征：

- 统一 server config 管理
- 多地址 failover
- retry
- timeout
- request queue / 限流
- interceptor
- metrics
- transport envelope

这说明当前 HTTP 通信并不是“散兵游勇式调用”，而是已经有了一个基础中台。

这个资产非常重要，我不建议推翻。

### 4.2 HTTP 接口按模块聚合，符合 Monorepo 分层复用思路

例如：

- `terminal/src/supports/apis/index.ts`
- `user-base/src/supports/apis/index.ts`

这种组织方式有利于：

- 模块边界清晰
- package 迁移方便
- API 可随模块一起复用

这个方向应该保留。

### 4.3 WS 不是“直接 new WebSocket 就完事”，而是已经有完整连接管理思路

无论是 `master-ws` 还是 `kernel-ws`，都已经具备：

- 状态机
- 心跳
- 自动重连/失败切换
- 事件机制
- 消息队列
- 错误处理

这说明团队已经把 WS 当成“通信系统”而不是“一个裸 socket”。

这同样是正确方向。

### 4.4 WS 已经开始复用部分 HTTP 基础设施思想

例如 `kernel-ws` 会通过 `ApiManager.getServerConfig(...)` 获取地址配置，这表明当前实现已经意识到：

- HTTP 和 WS 不应各自维护一套完全分离的 server 地址体系
- 至少 server addressing 可以共享

这说明统一通信基础设施的方向已经出现，只是还没抽象到位。

## 5. 当前设计的关键问题

## 5.1 HTTP 端点描述能力太弱

当前 HTTP 的核心描述是：

- `serverName`
- `path`
- `method`

这足以描述静态路径 RPC 风格接口，但不足以描述完整 HTTP 服务契约。

它无法天然表达：

- path params
- query params
- headers
- body
- endpoint meta

这会直接导致你指出的问题：

- `/api/device/{deviceId}/activate` 这种路径不好覆盖

因为当前 `path` 只是普通字符串，不是路径模板。

## 5.2 HTTP 请求结构没有被显式建模

现在是：

- GET → `request` 全塞 query
- 非 GET → `request` 全塞 body

问题在于，真实 HTTP 并不是这样。

真实 HTTP 一般需要区分：

- path
- query
- body
- headers

而当前没有这种结构拆分，所以只能靠约定和人工记忆，无法形成强约束。

## 5.3 HTTP 业务层直接处理 transport envelope，耦合过深

当前 Actor 需要直接处理：

- `ResponseWrapper`
- `APIResponseCode`
- `APIError`

这导致业务层看到的不是领域结果，而是网络包裹结构。

在分层架构中，这种耦合偏深，不利于长期演进。

## 5.4 WS 有两套实现，但没有统一抽象骨架

`master-ws` 和 `kernel-ws` 都是“完整的 WS 客户端实现”，但它们之间目前缺少一个统一抽象层。

它们有大量相似概念：

- server urls / failover
- connection state
- event manager
- message queue
- heartbeat
- error model

但这些共性没有被抽出来，而是各自实现。

这带来两个问题：

1. 维护成本会上升
2. 后续新增第三种 WS 客户端时，很可能继续复制一套

## 5.5 `kernel-ws` 复用了 `Api`，但方式不够纯净

`kernel-ws` 当前会把 `Api` 当成：

- 一个提供 `serverName()` 的配置来源
- 一个提供 `path()` 的路径来源

这在工程上“能用”，但从抽象上不够干净。

因为：

- `Api` 本质是 HTTP endpoint 描述
- WS connection descriptor 本质不是 HTTP 请求

如果继续这么演进，会把 HTTP 抽象和 WS 抽象糊在一起。

## 5.6 当前体系缺少统一的“通信描述层”

这是最核心的问题。

当前工程里有：

- HTTP endpoint 定义
- WS client config
- server config
- local route match
- retry/timeout/interceptor

但缺少一个统一的中间层，把这些概念组织起来。

我建议把这个中间层命名为：

- **Transport Contract** 或
- **Communication Contract**

这样“HTTP 和 WS”就可以在同一个设计视角下被统一治理，但又不会混成一个类。

## 6. 关于“只用 axios 不好吗”的明确结论

这次我明确给出更精准的结论：

### 6.1 对于 HTTP 服务通信：统一用 axios 是对的

我赞成以下原则：

> **所有 HTTP 服务通信，统一使用 axios 作为底层 transport。**

理由：

- 当前 `ApiManager` 已经基于 axios 构建了成熟能力
- axios 的拦截器、timeout、取消、实例管理更适合作为统一底座
- 没必要在 fetch 和 axios 之间来回摇摆

所以这次修订后，我不再建议把浏览器 fetch 作为一等设计对象。

### 6.2 但 HTTP 统一 axios，不等于“整个通信体系只有 axios”

因为 WS 不是 axios。

所以更准确的架构结论应该是：

- **HTTP 通信统一 axios**
- **WS 通信统一 WebSocket client framework**
- **HTTP 和 WS 在更高层共享通信契约、地址配置、元数据、错误模型、观测能力**

这才是符合你当前工程实际的说法。

## 7. 我建议的新命名与新分层

因为你这次明确说“API 特指 HTTP 服务”，所以我建议把旧的“Api”这个名字逐步淡化，改成更精确的命名。

### 7.1 推荐命名

#### HTTP 部分

旧命名：

- `Api`
- `ApiManager`
- `kernelCoreTerminalApis`

建议改为：

- `HttpEndpointDefinition`
- `HttpClient`
- `HttpTransport`
- `terminalHttpEndpoints`
- `userHttpEndpoints`

#### WS 部分

旧命名：

- `kernelDeviceWS`
- `DualWebSocketClient`
- `KernelWebSocketClient`

建议统一概念为：

- `SocketEndpointDefinition`
- `SocketClient`
- `SocketConnectionProfile`
- `masterWsProfile`
- `kernelWsProfile`

#### 更高一层总称

总称建议用：

- **Communication Foundation**
- 或 **Transport Foundation**

我个人更推荐：

- **Communication Foundation**

因为它更容易同时容纳 HTTP 和 WS。

## 8. 新设计的总目标

新体系目标不是把所有东西做成一个统一类，而是建立下面这个分层：

### 第一层：通信契约层（Communication Contract Layer）

定义：

- HTTP endpoint contract
- WS socket profile contract
- shared server addressing contract
- shared timeout / retry / auth / trace metadata

### 第二层：通信执行层（Communication Runtime Layer）

提供：

- `AxiosHttpTransport`
- `HttpClient`
- `MasterWsClient`
- `KernelWsClient`
- shared server config resolver

### 第三层：业务网关层（Gateway Layer）

对业务暴露：

- `TerminalGateway`
- `UserGateway`
- `InterconnectionGateway`

业务层不直接接触：

- axios
- endpoint path string
- ws url 拼接
- response wrapper

## 9. 新的 HTTP 设计

### 9.1 核心原则

我建议把当前 `Api<T, R>` 升级为：

- **HTTP Endpoint Definition**

它不是“可直接执行的对象”，而是“描述 HTTP 服务端点的契约对象”。

### 9.2 推荐结构

```ts
interface HttpEndpointDefinition<
  TPath,
  TQuery,
  TBody,
  TResponse,
  TError = DefaultHttpError
> {
  name: string
  server: ServerName
  method: HttpMethod
  pathTemplate: string
  request: {
    path?: TypeDescriptor<TPath>
    query?: TypeDescriptor<TQuery>
    body?: TypeDescriptor<TBody>
    headers?: TypeDescriptor<Record<string, string>>
  }
  response: TypeDescriptor<TResponse>
  error?: TypeDescriptor<TError>
  meta?: {
    auth?: 'none' | 'required' | 'optional'
    timeoutMs?: number
    retry?: number
    idempotent?: boolean
    tags?: string[]
  }
}
```

### 9.3 调用方式

```ts
httpClient.call(activateDeviceEndpoint, {
  path: { deviceId: 'D001' },
  body: { activeCode: 'ABC123' },
  extra: { traceId, requestId }
})
```

### 9.4 这如何解决动态路径问题

因为 `pathTemplate` 与 `path` 被显式拆开了。

例如：

```ts
const activateDeviceEndpoint = defineHttpEndpoint({
  name: 'terminal.activateDevice',
  server: SERVER_NAME_KERNEL_API,
  method: HttpMethod.POST,
  pathTemplate: '/api/device/{deviceId}/activate',
  request: {
    path: typed<{ deviceId: string }>(),
    body: typed<{ activeCode: string }>(),
  },
  response: typed<ActivateDeviceResponse>(),
})
```

然后统一由 `PathTemplateCompiler` 负责：

- render URL
- validate required path params
- reverse match URL

这样你指出的问题就从根上被解决了。

### 9.5 HTTP 执行层保留 axios

这次我明确建议：

- 保留 `ApiManager` 的成熟能力
- 但其角色重命名或收缩为：
  - `HttpClient`
  - 或 `AxiosHttpClient`

它仍然基于 axios，继续保留：

- server space
- server failover
- retry
- interceptor
- queue
- metrics

但上层不再直接拿 `Api` 去调用，而是：

- `endpoint definition` + `httpClient.call()`

### 9.6 HTTP 错误模型建议

建议 HTTP 层统一抛这两类错误：

- `HttpTransportError`
- `HttpBusinessError`

而不是让业务层看到 `ResponseWrapper`。

transport envelope 可以保留在线路层，但要在 `httpClient.call()` 内部解包。

## 10. 新的 WS 设计

### 10.1 核心原则

WS 不应该再伪装成 `Api`。

WS 应该有自己的契约对象，我建议命名为：

- `SocketEndpointDefinition`
- 或 `SocketConnectionProfile`

我更推荐：

- **SocketConnectionProfile**

因为 WS 的重点不是一次请求，而是一段连接生命周期。

### 10.2 推荐结构

```ts
interface SocketConnectionProfile<THandshake, TIncoming, TOutgoing> {
  name: string
  server: ServerName
  pathTemplate: string
  handshake: {
    query?: TypeDescriptor<Record<string, string | number | boolean>>
    headers?: TypeDescriptor<Record<string, string>>
  }
  message: {
    incoming: TypeDescriptor<TIncoming>
    outgoing: TypeDescriptor<TOutgoing>
  }
  meta?: {
    auth?: 'required' | 'optional' | 'none'
    connectionTimeoutMs?: number
    heartbeatIntervalMs?: number
    heartbeatTimeoutMs?: number
    maxQueueSize?: number
    tags?: string[]
  }
}
```

### 10.3 `kernel-ws` 如何重设计

目前 `kernel-ws` 需要：

- server name
- path
- deviceId
- token
- heartbeat config

那么可以定义为：

```ts
const kernelSocketProfile = defineSocketProfile({
  name: 'terminal.kernelSocket',
  server: SERVER_NAME_KERNEL_WS,
  pathTemplate: '/ws/connect',
  handshake: {
    query: typed<{ deviceId: string; token: string }>(),
  },
  meta: {
    auth: 'required',
    connectionTimeoutMs: 10000,
    heartbeatIntervalMs: 30000,
    heartbeatTimeoutMs: 60000,
    maxQueueSize: 100,
  }
})
```

然后 `KernelWsClient.connect(profile, input)` 即可。

### 10.4 `master-ws` 如何重设计

`master-ws` 比 `kernel-ws` 多一个注册阶段。

所以我建议把它拆成两个契约：

1. `MasterRegisterEndpoint`（HTTP）
2. `MasterSocketProfile`（WS）

也就是说：

- HTTP 注册属于 HTTP 通信体系
- 注册成功后的 socket 连接属于 WS 通信体系

这样职责最清楚。

### 10.5 WS 的统一抽象骨架

我建议把 `master-ws` 和 `kernel-ws` 共同的能力下沉成一个共享骨架，比如：

- `BaseSocketClient`
- `BaseConnectionManager`
- `BaseHeartbeatManager`
- `BaseEventManager`
- `SocketServerResolver`

然后：

- `KernelWsClient` 继承/组合共享骨架
- `MasterWsClient` 继承/组合共享骨架

这样能复用：

- failover
- state machine 基础结构
- queue
- event dispatch
- heartbeat timeout 基础能力
- cleanup 逻辑

而把各自特有逻辑留在子类/配置中：

- `master-ws`：register + pairing
- `kernel-ws`：kernel message protocol

## 11. HTTP 与 WS 如何统一，又如何分离

这是这次设计最重要的一点。

### 11.1 应该统一的部分

HTTP 与 WS 都应该共享：

- `serverName / server config`
- serverSpace/environment 选择
- timeout/retry/auth/trace 这类元数据结构
- metrics / logging / tracing 基础能力
- 路径模板能力
- 契约定义的组织方式

### 11.2 不应该统一成同一个类的部分

HTTP 与 WS 不应该共用同一个 `Api` 类，因为它们语义不同：

#### HTTP
- 一次调用
- request/response
- retry 主要围绕请求
- envelope 解包

#### WS
- 长连接
- stateful lifecycle
- message stream
- heartbeat
- reconnect
- queue flush

所以正确的做法是：

- **共享通信契约风格和底层公共设施**
- **分开 HTTP 和 WS 的协议对象与客户端实现**

## 13. 对现有 `ApiManager` 的处理建议

我不建议直接删除 `ApiManager`。

更合理的做法是：

### 阶段 1：保留实现，改造入口

- 新增 `HttpEndpointDefinition`
- 新增 `HttpClient.call(endpoint, input)`
- 内部仍然复用 `ApiManager` 的 axios、retry、server config

### 阶段 2：逐步重命名与瘦身

将：

- `ApiManager`

逐步演进为：

- `HttpClient`
- 或 `AxiosHttpClient`

将其中通用的 server resolving、metrics、queue 能力拆到 shared 层。

### 阶段 3：移除旧 `Api.run()` 风格

旧风格：

```ts
kernelCoreTerminalApis.activateDevice.run({ request })
```

新风格：

```ts
httpClient.call(activateDeviceEndpoint, {
  path,
  query,
  body,
  extra,
})
```

这样既能保留原有资产，也能建立新契约。

## 14. 对现有 `kernel-ws` 的处理建议

当前 `kernel-ws` 依赖 `Api` 提供 `serverName()` 和 `path()`。

我建议迁移成：

```ts
KernelWsClient.connect(kernelSocketProfile, {
  query: {
    deviceId,
    token,
  },
  options: {
    connectionTimeoutMs,
    heartbeatIntervalMs,
    heartbeatTimeoutMs,
  }
})
```

这样 `kernel-ws` 不再依赖 HTTP `Api`，但仍可以通过 shared `ServerResolver` 使用同一个 server config。

## 15. 对现有 `master-ws` 的处理建议

当前 `master-ws` 自己接收 `serverUrls`，而 `kernel-ws` 通过 `ApiManager` 取 server config。

这里不一致。

建议统一为：

- `master-ws` 也使用 `ServerResolver`
- 注册阶段定义为 HTTP endpoint
- WS 阶段定义为 socket profile

即：

```ts
await httpClient.call(masterRegisterEndpoint, {
  body: deviceRegistration,
})

await masterWsClient.connect(masterSocketProfile, {
  query: { token },
})
```

或者在 `MasterWsClient.connect()` 内部封装注册流程，但内部仍复用这两个契约。

## 16. 动态路径问题的新答案

你最初提到的：

> `/api/device/{deviceId}/activate` 覆盖不了

在修订后的设计里，答案非常明确：

这应该由 HTTP 的 `PathTemplate` 能力解决，而不是由 WS 或 Native runtime 解决。

设计上应该支持：

```ts
pathTemplate: '/api/device/{deviceId}/activate'
```

调用时：

```ts
httpClient.call(endpoint, {
  path: { deviceId },
  body: { activeCode },
})
```

运行时：

```ts
compilePath('/api/device/{deviceId}/activate', { deviceId })
```

本地 mock 或 HTTP 拦截时：

```ts
matchPath('/api/device/{deviceId}/activate', actualPathname)
```

这套能力只属于 HTTP endpoint / route 层，不应该再让业务层手动拼字符串。

## 17. 推荐迁移顺序

### 第 1 步：抽 shared communication 基础能力

先新增：

- `ServerResolver`
- `PathTemplate`
- `TransportMeta`
- `TransportError`

### 第 2 步：改造 HTTP endpoint 定义

先从 `terminal` 模块开始：

- `activateDevice`
- `deactivateDevice`
- `setOperatingEntity`
- `getUnitDataByGroup`

让它们从旧 `Api` 迁移到 `HttpEndpointDefinition`。

### 第 3 步：新增 `HttpClient.call()`

底层复用现有 `ApiManager` 能力。

这一步先不要大动 axios 逻辑。

### 第 4 步：改造 `kernel-ws` 的 profile 输入

把：

- `api: kernelDeviceWS.connectKernelWS`

替换为：

- `profile: kernelSocketProfile`

但 server config 仍通过 shared resolver 读取。

### 第 5 步：改造 `master-ws` 地址配置方式

让 `master-ws` 不再直接接收裸 `serverUrls`，而是通过 serverName / profile / resolver 获取地址。

### 第 6 步：抽取 WS 公共骨架

当 `kernel-ws` 和 `master-ws` 的 profile 形态稳定后，再抽：

- base state machine
- base event manager
- base heartbeat manager
- base failover connector

不要太早抽象，否则容易抽错。

## 18. 最终推荐结论

### 18.1 对 HTTP 的结论

HTTP 服务通信应该统一基于 axios。

但上层不要再直接暴露 `new Api(serverName, path, method)` 这种弱描述，而应升级为：

- `HttpEndpointDefinition`
- `HttpClient.call(endpoint, input)`

其中 input 必须显式区分：

- `path`
- `query`
- `body`
- `headers`

### 18.2 对 WS 的结论

WS 不应该继续借用 HTTP `Api` 抽象。

它应该升级为：

- `SocketConnectionProfile`
- `SocketClient.connect(profile, input)`

同时 `kernel-ws` 和 `master-ws` 应该共享：

- server resolving
- state machine 基础能力
- heartbeat 基础能力
- event 基础能力
- queue/failover 基础能力

### 18.3 对总体系的结论

最终体系应该叫：

> **Communication Foundation：HTTP + WS 的统一通信基础设施**

而不是把所有东西都叫 API。

其中：

- HTTP 是 request/response 协议，默认 axios
- WS 是 long-lived connection 协议，默认 WebSocket client
- 二者共享 server config、meta、path template、metrics、错误分类
- 二者不混用同一个类

## 19. 一句话总结

**你说得对：HTTP 服务 API 应该统一用 axios；我修正后的设计是把“API”收敛为 HTTP Endpoint，同时把 WS 作为同级通信协议纳入 Communication Foundation，共享配置和元数据，但协议对象与客户端实现彻底分开。**
