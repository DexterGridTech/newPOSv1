# API 体系评审与重设计方案

## 1. 背景与目标

本报告以 `1-kernel/1.1-cores/terminal/src/supports/apis/index.ts` 为入口，遍历 Monorepo 中与 API 定义、执行、封装、路由匹配、运行时拦截相关的实现，重点分析以下问题：

1. 当前 API 的定义方式是什么。
2. 当前 API 在业务层是如何被使用的。
3. 当前设计的亮点是什么。
4. 当前设计存在哪些结构性问题。
5. 已知问题“`/api/device/{deviceId}/activate` 这类 URL 无法覆盖”的根因是什么。
6. 如果基于当前整体分层架构重新设计一套 API 体系，应该如何设计，才能兼顾：
   - Monorepo 复用
   - 强类型
   - 多端运行时适配
   - 业务层解耦
   - RESTful / RPC 混合场景
   - WebSocket / HTTP 分离
   - 后续演进和维护成本

## 3. 当前 API 体系的整体结构

### 3.1 当前 Kernel 侧的核心抽象

当前 Kernel 的 API 体系以 `Api<TRequest, TResponse>` 为核心抽象。

典型定义方式如下：

- 在各模块的 `src/supports/apis/index.ts` 中导出 API 对象集合。
- 每个 API 通过 `new Api(serverName, path, method)` 定义。
- 业务层通过 `api.run({ request })` 触发请求。
- 请求真正执行由 `ApiManager` 统一负责。

也就是说，当前结构可以概括为：

1. **声明层**：模块本地 `supports/apis/index.ts`
2. **抽象层**：`Api<T, R>`
3. **执行层**：`ApiManager`
4. **配置层**：`server-config`
5. **消费层**：Actor / Command Handler

### 3.2 当前 HTTP 请求执行链路

以 `kernelCoreTerminalApis.activateDevice.run({ request })` 为例，请求链路大致如下：

1. Actor 组装请求 DTO。
2. 调用 `Api.run()`。
3. `Api.run()` 创建 `cancelTokenSource`，并将 `serverName / path / method / requestWrapper` 传给 `ApiManager.runApi()`。
4. `ApiManager` 基于 `serverName` 找到服务器配置。
5. `ApiManager` 通过请求队列进行并发控制和限流。
6. `ApiManager.executeWithRetry()` 根据配置在多个地址间轮询重试。
7. `ApiManager.sendOnce()` 使用 axios 实例发起请求：
   - `GET` 把 `request` 放到 `params`
   - 非 `GET` 把 `request` 放到 `data`
8. 后端返回 `ResponseWrapper<T>` 风格的数据结构。
9. Actor 根据 `result.code === APIResponseCode.SUCCESS` 判断成功失败。
10. 失败则抛 `APIError`，再交给上层命令/状态流处理。

## 4. 当前设计的亮点

虽然当前 API 体系存在明显短板，但它并不是“完全不可用”的设计。相反，它在你当前这种分层式 Monorepo 架构里，已经体现出了一些很有价值的思路。

### 4.1 API 被模块化归档，边界比较清晰

每个 kernel 核心或业务模块都有自己的 `supports/apis/index.ts`，这说明 API 定义没有散落在 Actor、UI、Adapter 中，而是被约束在模块支持层内。

这个设计的价值在于：

- 模块边界明确。
- API 资产可以随模块迁移与复用。
- 上层业务只依赖模块导出的 API 集合，不直接关心底层 axios 细节。

这和你希望的“四层架构”思路是一致的，说明当前方向本身没有偏掉。

### 4.2 请求与响应使用了泛型，具备基础类型安全

`Api<TRequest, TResponse>` 这一层虽然很轻，但至少做到了：

- 请求体有类型。
- 响应体有类型。
- Actor 在消费 API 时，不再直接写 `any` 风格的原始请求。

这对 Monorepo 来说非常关键，因为这意味着：

- DTO 可以跨 package 复用。
- 模块间调用可以在编译期发现一部分问题。
- 为以后做 schema / codegen 迁移留有空间。

### 4.3 服务发现与地址切换集中在 `ApiManager`

`ApiManager` 做了一件很重要的事：把“调用哪个服务地址”从业务逻辑中抽离了。

这带来几个好处：

- 支持同一个服务多地址配置。
- 支持失败轮询重试。
- 支持 dev / product 环境切换。
- 支持统一加拦截器。

这说明当前设计已经把“服务寻址”从“业务 API 描述”中拆出来了，这是对的。

### 4.4 请求拦截器机制是正确的扩展点

例如 `terminal` 模块在 `modulePreSetup.ts` 里给 `SERVER_NAME_KERNEL_API` 添加 token 拦截器，这说明：

- 鉴权逻辑没有硬编码在每个 API 里。
- 服务级别的横切逻辑已经有统一注入点。
- 这为未来增加 traceId、灰度头、设备标识、签名等提供了入口。

这是一个成熟 API 客户端必须具备的能力。

### 4.5 `ResponseWrapper` 统一了网络错误和业务错误的返回形态

当前服务端和客户端都倾向于用统一包裹结构返回结果，再由 `APIError` 统一处理。

它的优点在于：

- Actor 层不需要同时处理“HTTP transport error”和“业务 code error”两种分叉风格。
- 错误聚合更容易接入命令系统和状态系统。
- 在多端场景下更方便保证一致行为。

这个思路本身是有价值的，但当前实现还有不少细节问题，后文会展开。

### 4.6 WebSocket 连接也复用了“服务配置”这套基础设施

`kernelDeviceWS.connectKernelWS` 复用了 `Api` 上的 `serverName / path / method` 信息，让 WS 客户端可以从同一套 server config 获取地址。

这说明当前团队已经在尝试用统一的“端点描述”抽象 HTTP / WS 连接。这个方向没问题，但当前实现方式不够纯净，导致 `Api` 职责被混杂了。

## 5. 当前设计存在的核心问题

下面的问题不是“代码风格问题”，而是会直接影响体系可扩展性的结构性问题。

### 5.1 `Api` 抽象过于薄，只能表达“固定 path + method”

当前 `Api<T, R>` 的构造参数只有：

- `serverName`
- `path`
- `method`

这意味着它只能表达静态端点，例如：

- `/api/device/activate`
- `/api/device/deactivate`
- `/api/unit-data/by-group`

但无法自然表达这些更常见的真实场景：

- `/api/device/{deviceId}/activate`
- `/devices/{deviceId}/commands/{commandId}`
- `/users/{userId}?include=roles`
- `/orders/{orderId}/items/{itemId}`

因为当前 `path` 只是一个普通字符串，没有参数模板能力，也没有 query/path/body/header 的拆分能力。

这正是你已经观察到的已知问题：

> 如果 URL 是 `/api/device/{deviceId}/activate`，当前体系覆盖不了。

这是当前 API 体系最根本的问题之一。

### 5.2 请求模型没有区分 path / query / body / headers

当前 `run({ request })` 会在 `ApiManager.sendOnce()` 中被统一处理为：

- GET → `params: request`
- 非 GET → `data: request`

这带来几个直接问题：

1. **无法精确表达 REST 请求结构**
   - `path params` 应该进 URL
   - `query` 应该进查询串
   - `body` 应该进请求体
   - `headers` 应该单独控制

2. **GET 只能全部塞进 query**
   - 这意味着 path param 和 query param 没法区分

3. **POST / PUT / PATCH 无法同时拥有 path + body**
   - 比如 `/devices/{deviceId}/activate` + `{ activationCode }`
   - 当前只能把所有字段混在 body 里

4. **横切头信息难以做到 endpoint 级精细化控制**
   - 当前只能依赖拦截器统一加
   - 无法声明某个 API 的专用 header 要求

这说明当前体系实际上更接近“带路径字符串的 RPC 请求器”，而不是完整的 API 描述体系。

### 5.3 `Api` 既被用于 HTTP，又被用于 WS 配置承载，职责混杂

`kernelDeviceWS.connectKernelWS` 也是 `new Api(..., HttpMethod.WS)`。

这表面上统一了形式，实际上引入了职责混杂：

- HTTP API 关注请求/响应。
- WebSocket 连接关注握手、重连、事件流、心跳、消息收发。

二者的生命周期和语义完全不同。

当前只是因为 WS 也需要 `serverName` 和 `path`，才借用了 `Api` 这个类。

问题在于：

- `Api.run()` 对 WS 并没有统一意义。
- `HttpMethod.WS` 只是把一个协议差异硬塞进 HTTP 抽象里。
- 以后如果要支持 SSE、gRPC、IPC、TurboModule RPC，会进一步恶化抽象边界。

结论：**HTTP Endpoint Descriptor** 和 **WS Endpoint Descriptor** 应该是兄弟抽象，不应该是同一个类硬塞协议枚举。

### 5.4 业务层对 `ResponseWrapper` 和 `APIResponseCode` 耦合过深

当前 Actor 基本都是这个模式：

1. 调 `api.run()`
2. `if (result.code === APIResponseCode.SUCCESS)`
3. 否则 `throw new APIError(result)`

这导致几个问题：

- 每个 Actor 都在重复写同一套错误判断样板代码。
- 业务层直接依赖 transport 层返回包裹结构。
- 如果未来要切换错误模型，Actor 层几乎全部要改。
- `ResponseWrapper` 本应属于传输契约，但现在已经渗透到业务动作层。

理想状态下，业务层应该拿到：

- 成功时：干净的领域数据
- 失败时：统一抛出的领域化异常

而不是让每个业务动作自己手动解包 transport response。

### 5.5 `Api` 实例保存 cancelToken，有并发语义风险

`Api` 是被定义成单例对象并导出的，例如：

- `kernelCoreTerminalApis.activateDevice`
- `kernelUserBaseApis.loginWithPassword`

同时 `Api` 内部有一个实例级字段：

- `cancelTokenSource?: CancelTokenSource`

这意味着如果同一个 `Api` 实例被并发调用：

- 第二次调用会覆盖第一次的 `cancelTokenSource`
- `cancel()` 行为会失去精确性
- 超时取消也可能误伤/失效

这在单次串行调用时不明显，但在真实系统里是典型隐患。

结论：**请求取消句柄不应该挂在共享 endpoint 实例上，而应该挂在单次 request invocation 上。**

### 5.6 当前 endpoint 定义是“字符串散装”，没有可验证元数据

现在的定义大概是：

- serverName
- path string
- HttpMethod
- 泛型参数

它缺少以下关键描述能力：

- endpoint 的名字/标识
- path params 定义
- query 定义
- body 定义
- auth 要求
- timeout 策略
- retry 策略
- 幂等性声明
- mock / stub 配置
- 响应解码策略
- 版本信息

这会造成几个实际问题：

1. 无法做自动化校验。
2. 无法自动生成文档。
3. 无法做统一 mock。
4. 无法做 endpoint 级观测。
5. 无法在多端 adapter 中做稳定匹配。

### 5.7 运行时本地拦截采用“精确 pathname 匹配”，天然无法覆盖动态路径

这是你已知问题的一个非常关键的外部证据。

在 `3-adapter/electron/adapterV1/src/renderer/runtime/axios.ts` 中，本地路由识别逻辑是：

- 先把 URL 解析成 `new URL(url)`
- 然后判断：
  - hostname 是否为 `127.0.0.1` 或 `localhost`
  - pathname 是否 **严格等于** `/localServer/register`
  - method 是否为 `POST`

类似地，在 Electron 主进程和其他 mock 服务里，也大量使用：

- `url.pathname === '/xxx'`
- `pathname === {basePath}/register`

这种判断方式只适合：

- 固定静态路径

一旦变成：

- `/api/device/123/activate`
- `/devices/abc/commands/def`
- `/sessions/xyz/heartbeat`

就必须引入：

- 路由模板
- path params 解析
- matcher 编译

否则永远只能靠字符串拼接和硬编码分支。

所以，你提到的“`/api/device/{deviceId}/activate` 覆盖不了”，不是一个孤立 bug，而是**整个 endpoint 描述能力不足 + 路由匹配能力不足**共同导致的结果。

## 6. 进一步的问题拆解：为什么当前体系容易演化成“伪类型安全”

当前体系表面上使用了 TypeScript 泛型，因此看起来“强类型”。但从体系设计角度看，它更接近“伪类型安全”，原因如下：

### 6.1 类型只覆盖了 body，没有覆盖 endpoint 全部结构

真正强类型的 API 描述，应该至少覆盖：

- `path params`
- `query`
- `headers`
- `body`
- `response`
- `error`

而当前 `Api<T, R>` 实际只覆盖：

- `request`（且没有结构拆分）
- `response`

所以它只是“请求体和响应体有类型”，还不是“Endpoint 有完整类型”。

### 6.2 泛型参数无法约束 path 字符串与 request 结构的一致性

举例：

如果定义成：

- `new Api<ActivateDeviceRequest, ActivateDeviceResponse>(..., '/api/device/activate', POST)`

那么编译器不会知道：

- `ActivateDeviceRequest` 中哪些字段应该放 path
- path 里是否遗漏了某个参数
- path 模板和请求结构是否一致

所以这套类型系统没有能力检查：

- endpoint 结构是否合法
- endpoint 描述和请求 DTO 是否一致

### 6.3 `any` 使用仍然较多，削弱了整个体系的可信度

在现有 API 定义里仍大量存在：

- `Api<..., any>`
- `Promise<Record<string, any>>`

这会造成两个后果：

1. 上游错误被吞掉。
2. 类型系统对接口边界的保护被削弱。

如果未来要建立“行业通用、长期稳定”的产品级 API 体系，`any` 必须被逐步清理掉，尤其是在基础设施层。

## 7. 当前体系与整体四层架构之间的错位

你的目标架构是：

1. 业务逻辑层
2. UI 层
3. 适配层
4. 整合层

从这个目标往回看，当前 API 体系主要有三处错位。

### 7.1 业务逻辑层仍然直接感知“HTTP 细节”

Actor 层虽然不直接写 axios，但仍直接感知：

- `APIResponseCode`
- `ResponseWrapper`
- `server error`
- `network error`

这意味着业务层和 transport contract 绑定得过紧。

按理说，在你的分层架构里：

- 业务逻辑层应该只关心“能力接口”与“领域结果”
- 不应该感知 HTTP body 如何包裹、状态码如何映射

### 7.2 API 描述位于业务模块中，但缺少独立契约层

当前 `supports/apis/index.ts` 位于各业务模块内部。

这带来一个好处：模块聚合明确。

但同时也带来一个问题：

- “领域能力定义”
- “HTTP endpoint 细节”
- “运行时执行策略”

这三者没有清晰分层，揉在一起了。

理想上应该至少拆成：

1. 领域能力接口层
2. 传输契约层
3. 运行时客户端层

### 7.3 适配层缺少“基于 endpoint 契约的统一路由匹配能力”

无论是 Electron 本地服务、Mock 服务、还是未来 RN 裸工程本地桥接，都会遇到同一个问题：

- 需要知道请求命中了哪个 endpoint
- 需要知道 path params 是什么
- 需要把 URL 反解成结构化请求

但当前没有统一的 endpoint matcher/compiler，因此每个运行时都只能自己手搓字符串判断。

这会让不同 adapter 在长期演进中越来越不一致。

## 8. 我建议的新 API 体系设计目标

如果基于你当前 Monorepo 与四层架构重新设计一套 API 体系，我建议目标不是“把现有 `Api` 补丁补到能用”，而是建立一套**契约驱动、运行时可适配、业务层无感 transport 细节**的统一 API 体系。

### 8.1 设计目标

新体系建议满足以下目标：

1. **完整表达 endpoint 结构**
   - path / query / body / headers / response / error

2. **强类型**
   - path params 和 path template 编译期关联
   - 不再用 `any`

3. **支持 RESTful 和 RPC 并存**
   - 兼容你现在已经存在的 `/api/device/activate`
   - 同时自然支持 `/api/device/{deviceId}/activate`

4. **业务层不直接处理传输包裹结构**
   - 业务层直接拿领域结果
   - 错误自动抛出领域异常

5. **支持多运行时适配**
   - 浏览器 fetch
   - axios
   - Electron host bridge
   - 本地 mock / local web server
   - 未来 TurboModule / Native bridge

6. **HTTP 与 WS 分离建模**
   - Endpoint 统一风格，但协议职责独立

7. **具备元数据能力**
   - auth
   - timeout
   - retry
   - idempotent
   - tags
   - version
   - mockability

8. **具备 matcher / compiler 能力**
   - 能从模板渲染 URL
   - 也能从 URL 反向解析 params

## 9. 新体系推荐方案：契约层 + 端点定义层 + 运行时执行层

我建议把 API 体系拆成 4 层。

### 9.1 第一层：契约层（Contract Layer）

职责：定义纯类型契约，不包含运行时逻辑。

例如：

- 请求 DTO
- 响应 DTO
- 错误 DTO
- 路径参数 DTO
- 查询参数 DTO

建议位置：

- `1-kernel/*/src/contracts/apis/*`
- 或单独提炼到共享 package，例如 `@impos2/api-contracts-*`

这一层只负责“长什么样”，不负责“怎么请求”。

### 9.2 第二层：端点描述层（Endpoint Definition Layer）

职责：用一个统一的强类型描述对象，声明 endpoint 的全部结构。

示意结构：

```ts
export interface EndpointDefinition<
  TPathParams,
  TQuery,
  TBody,
  TResponse,
  TError = DefaultApiError
> {
  name: string
  server: ServerName
  protocol: 'http'
  method: HttpMethod
  pathTemplate: string
  path?: Schema<TPathParams>
  query?: Schema<TQuery>
  body?: Schema<TBody>
  response?: Schema<TResponse>
  error?: Schema<TError>
  meta?: {
    auth?: 'none' | 'optional' | 'required'
    timeoutMs?: number
    retry?: number
    idempotent?: boolean
    tags?: string[]
    version?: string
  }
}
```

这里最关键的是 `pathTemplate` 和结构拆分。

例如：

```ts
const activateDeviceEndpoint = defineEndpoint({
  name: 'terminal.activateDevice',
  server: SERVER_NAME_KERNEL_API,
  protocol: 'http',
  method: 'POST',
  pathTemplate: '/api/device/{deviceId}/activate',
  path: pathParams<{ deviceId: string }>(),
  body: bodySchema<{ activeCode: string }>(),
  response: responseSchema<ActivateDeviceResponse>(),
  meta: {
    auth: 'required',
    timeoutMs: 5000,
    retry: 1,
    idempotent: false,
    tags: ['terminal']
  }
})
```

这样一来：

- endpoint 能描述完整结构
- 动态路径天生支持
- 运行时可以统一编译/匹配

### 9.3 第三层：运行时客户端层（Runtime Client Layer）

职责：接收 endpoint definition 和结构化请求，产出真正网络请求。

统一请求输入建议改为：

```ts
client.call(endpoint, {
  path: { deviceId: 'D001' },
  body: { activeCode: 'ABC123' },
  query: undefined,
  headers: undefined,
  extra: { traceId, requestId }
})
```

而不是现在的：

```ts
api.run({ request })
```

这样做的价值是：

- path / query / body 不会混淆
- RESTful URL 能自然生成
- adapter/mock 能基于同一个 endpoint definition 做反向匹配

### 9.4 第四层：业务仓储/服务层（Domain Gateway Layer）

职责：把 transport 细节封装掉，给 Actor 暴露业务能力接口。

例如：

```ts
export interface TerminalApiGateway {
  activateDevice(input: {
    deviceId: string
    activeCode: string
  }): Promise<ActivateDeviceResponse>
}
```

其 HTTP 实现：

```ts
export class HttpTerminalApiGateway implements TerminalApiGateway {
  constructor(private client: ApiClient) {}

  async activateDevice(input: { deviceId: string; activeCode: string }) {
    return this.client.call(activateDeviceEndpoint, {
      path: { deviceId: input.deviceId },
      body: { activeCode: input.activeCode }
    })
  }
}
```

Actor 层就不再直接看到 `ResponseWrapper`：

```ts
const terminal = await terminalApiGateway.activateDevice({
  deviceId,
  activeCode,
})
```

这样业务层就真正和 transport 解耦了。

## 10. 新体系的关键设计细节

### 10.1 引入路径模板编译器与匹配器

这是解决你已知问题的核心。

新体系里必须具备两个能力：

1. **compile**：把结构化 path params 渲染成 URL
2. **match**：把 URL 反向解析成 path params

示意：

```ts
compilePath('/api/device/{deviceId}/activate', { deviceId: 'D001' })
// => '/api/device/D001/activate'

matchPath('/api/device/{deviceId}/activate', '/api/device/D001/activate')
// => { matched: true, params: { deviceId: 'D001' } }
```

有了这个能力之后：

- HTTP client 能正确构造动态 URL
- Electron 本地路由能正确命中动态 endpoint
- Mock server 能统一复用契约
- 测试和文档也可以共享这一层能力

### 10.2 将 transport envelope 解包下沉到 client 层

当前 `ResponseWrapper` 的问题不是“完全不能要”，而是它的位置放错了。

我建议：

- 保留 transport envelope 作为 wire contract
- 但在 `ApiClient.call()` 内部统一解包
- 对业务层只返回 `TResponse`
- 对失败统一抛出 `TransportError` / `BusinessError`

即：

```ts
const result = await rawHttpClient.request(...)
return unwrapResponse(result)
```

而不是把 `ResponseWrapper` 直接交给 Actor。

### 10.3 将 endpoint 元数据前置，支持统一策略控制

建议每个 endpoint 都能声明元数据：

- `auth`
- `timeout`
- `retry`
- `cache`
- `idempotent`
- `auditTag`
- `mockPolicy`

这样 `ApiClient` 就可以基于 endpoint meta 自动：

- 选择是否注入 token
- 设置 timeout
- 设置 retry 行为
- 记录埋点与日志
- 决定是否允许 mock / local intercept

这样做比“在模块 preSetup 中按 serverName 加拦截器”更细粒度，也更利于长期治理。

### 10.4 取消句柄必须请求级，不是 endpoint 实例级

推荐把调用结果改成：

```ts
const task = apiClient.createCall(endpoint, input)
const promise = task.run()
task.cancel()
```

或者：

```ts
const controller = new AbortController()
await apiClient.call(endpoint, input, { signal: controller.signal })
```

总之取消句柄应该与“单次调用”绑定，而不是挂在全局共享 endpoint 对象上。

### 10.5 HTTP 与 WS 要统一风格，但分离抽象

我建议：

- HTTP 使用 `EndpointDefinition`
- WS 使用 `SocketDefinition`

二者共享：

- `server`
- `pathTemplate`
- `meta`
- `auth`
- `version`

但不要强行共用同一个 `Api` 类。

例如：

```ts
const kernelSocket = defineSocket({
  name: 'terminal.kernelSocket',
  server: SERVER_NAME_KERNEL_WS,
  pathTemplate: '/ws/connect',
  auth: 'required',
  heartbeat: {
    intervalMs: 30000,
    timeoutMs: 60000,
  }
})
```

这样会更符合职责边界，也便于未来扩展到其他双向协议。

## 11. 新体系在你当前四层架构中的建议落位

### 11.1 业务逻辑层（1-kernel）

建议放：

- 领域能力接口（Gateway Interface）
- endpoint 契约定义
- 领域 DTO 类型
- transport 无感的 service/gateway

不要让业务层直接关心 axios / fetch / localWebServer。

### 11.2 UI 层（2-ui）

建议只依赖：

- 业务命令
- 业务状态
- 领域能力接口

UI 层不应该自己拼 URL，更不应该自己感知 path params / query/body 的拆分。

### 11.3 适配层（3-adapter）

建议负责：

- 具体运行时 client adapter
- Electron / Android / RN Web 的 transport 实现
- endpoint matcher / local intercept 实现
- mock / bridge / native request 执行

也就是说，适配层只实现“怎么发”，不定义“发什么”。

### 11.4 整合层（4-assembly）

建议负责：

- 实际装配哪一个 `ApiClient`
- 实际注入哪些 `Gateway` 实现
- 环境配置、server space、灰度切换、容灾策略

这样整体依赖关系会更清晰：

- 契约定义在上游共享层
- transport 实现在下游适配层
- assembly 负责绑定

## 13. 三种可选重构路线

### 方案 A：在现有 `Api` 上做渐进增强

做法：

- 给 `Api` 增加 `pathTemplate`
- 增加 `run({ path, query, body })`
- 逐步替换 `run({ request })`

优点：

- 改动最小
- 对现有调用兼容较强
- 可以快速解决动态路径问题

缺点：

- 仍会保留 `Api` 职责混杂问题
- 仍可能继续耦合 `ResponseWrapper`
- 后续很容易越改越重，形成“超级类”

我的评价：**适合作为过渡，但不适合作为最终形态。**

### 方案 B：保留 `ApiManager`，重写 endpoint 定义层与 client 层

做法：

- 新增 `defineEndpoint` + `ApiClient.call`
- `ApiManager` 继续作为底层请求执行器
- 现有 `Api` 逐步废弃

优点：

- 兼容你现有多地址、重试、拦截器、serverSpace 能力
- 重构成本可控
- 架构上能显著提升完整性

缺点：

- 需要一段时间同时维护新旧两套定义
- `ApiManager` 内部仍可能有历史包袱

我的评价：**这是当前项目最现实、性价比最高的路线，也是我的推荐方案。**

### 方案 C：契约驱动 + schema/codegen 全量重构

做法：

- 统一所有 endpoint 契约
- 用 schema / OpenAPI / zod 生成 client 和 server stub
- 前后端全面契约化

优点：

- 理论上最强、最标准化
- 最利于长期治理和自动化

缺点：

- 成本最高
- 对现有工程改造范围大
- 需要服务端、mock、adapter 一起跟进

我的评价：**适合中长期目标，不适合作为第一步。**

## 14. 我的推荐方案

### 14.1 总体建议

我建议采用：

- **短中期：方案 B**
- **长期：在方案 B 稳定后，再演进到部分契约生成能力**

也就是：

1. 保留 `ApiManager` 的地址管理、重试、拦截器、指标能力。
2. 引入全新的 `EndpointDefinition` 和 `ApiClient`。
3. 新 API 一律使用结构化输入：`path / query / body / headers`。
4. 新 API 一律通过 Gateway 暴露给业务层。
5. 旧 API 逐步迁移，不一次性推翻。

### 14.2 为什么我推荐这个方案

因为它最符合你项目当前的现实约束：

- 你是 Monorepo，大面积推翻风险高。
- 你有多层架构，需要保留抽象稳定性。
- 你已经有 `ApiManager`、serverSpace、interceptor 这些成熟资产，不应该浪费。
- 你现在最痛的点不是“没有请求能力”，而是“endpoint 描述能力不足”。

所以应该先补“契约表达能力”，而不是先推翻“底层执行能力”。

## 16. 推荐的迁移步骤

为了控制风险，我建议按下面顺序推进。

### 第 1 步：先抽 `EndpointDefinition` 和 `PathTemplate`

先只做两个基础能力：

- 结构化 endpoint 描述
- 路径模板 compile / match

这一步先不要大规模替换业务调用。

目标是先把“动态路径不可表达”这个核心问题解决。

### 第 2 步：新增 `ApiClient.call()`，但底层仍复用 `ApiManager`

即：

- 新 endpoint definition 走新 client
- 新 client 内部仍调用现有 `ApiManager`
- 先不动 `ApiManager` 重试、指标、serverSpace 逻辑

这样改动范围最可控。

### 第 3 步：新增 Gateway 层，先迁移最关键模块

优先迁移：

1. `terminal`
2. `user-base`

因为这两个模块已经有明确 API 定义和真实业务使用。

迁移目标不是“全部替换”，而是先建立新范式。

### 第 4 步：统一 Electron / Mock / LocalServer 的 endpoint matcher

把所有：

- `pathname === 'xxx'`
- `url.pathname === 'xxx'`

这种逻辑收敛到统一 matcher。

这样你提到的 `/api/device/{deviceId}/activate` 覆盖问题才能在各运行时真正被彻底解决，而不是只在 HTTP client 层解决一半。

### 第 5 步：逐步把 Actor 与 `ResponseWrapper` 解耦

当 Gateway 成熟后：

- Actor 层不再直接判断 `APIResponseCode.SUCCESS`
- 统一由 Gateway / Client 抛错
- 错误在更靠近 transport 的地方归一化

### 第 6 步：最后才考虑 schema 校验与 codegen

等 endpoint definition 稳定后，再决定是否引入：

- zod
- OpenAPI
- contract-to-doc
- contract-to-mock
- contract-to-client

不要一开始就引入重工具链，否则会增加实施阻力。

## 17. 风险与注意事项

### 17.1 不建议一次性推翻全部 `Api`

原因：

- 你当前工程层级多。
- 模块之间依赖复杂。
- 适配层运行时差异大。

一次性重写极容易把“设计升级”演变成“全工程不稳定”。

### 17.2 不建议继续把更多能力往现有 `Api` 类里硬塞

比如继续给 `Api` 增加：

- path params
- query
- body
- ws config
- mock config
- auth config

最后会得到一个职责过于庞杂的基础类。

这类类在前期看起来“好用”，后期会变成所有问题的汇合点。

### 17.3 不建议让 UI 层自己感知 endpoint 细节

即使 UI 侧有自己的 fetch 封装，也不建议未来让 UI 直接写 endpoint path template。因为你的架构目标本身就是：

- 业务能力沉到 kernel / gateway
- UI 尽量轻

### 17.4 WS 不要再继续伪装成 HTTP Api

这会短期省代码，长期坏边界。

未来如果你还会引入：

- native bridge RPC
- deep link passive event
- TurboModule command channel

那就更需要把“协议描述”分开建模。

## 18. 最终结论

### 18.1 对当前体系的总体评价

当前 API 体系属于：

- **方向基本正确**：模块聚合、服务发现、重试、拦截器这些都说明体系不是乱的。
- **抽象层级不够完整**：它只解决了“怎么发请求”的一部分问题，没有解决“如何完整描述 endpoint”。
- **适合 RPC 风格，不适合复杂 RESTful 风格**：固定 path 场景可用，动态 path 场景明显吃力。
- **业务层与 transport contract 耦合偏深**：Actor 仍直接处理 `ResponseWrapper` 和 `APIResponseCode`。
- **运行时 matcher 能力不足**：本地服务、Electron 拦截、本地 mock 都还停留在“字符串精确匹配”阶段。

### 18.2 对已知问题的明确结论

你指出的：

> `/api/device/{deviceId}/activate` 这种 URL 当前覆盖不了

结论是：**确实覆盖不了，而且这不是单点 bug，而是当前 API 体系在 endpoint 描述能力上的结构性缺陷。**

根因包含三层：

1. `Api` 只能表达静态 path。
2. `run({ request })` 没有 path/query/body 的结构拆分。
3. 各运行时本地路由采用 `pathname === '固定字符串'` 的方式匹配。

### 18.3 我对新体系的最终建议

如果让我基于当前整体架构重新设计一套 API 体系，我会采用以下原则：

1. **契约先行**：先定义 endpoint contract，而不是先写 axios 调用。
2. **结构化输入**：调用必须显式区分 `path / query / body / headers`。
3. **业务层解耦**：Actor 通过 Gateway 使用 API，不直接接触 `ResponseWrapper`。
4. **运行时分离**：HTTP / WS / NativeBridge 分别建模，统一风格但不混类。
5. **统一 matcher/compiler**：路径模板必须既能生成 URL，也能反向匹配 URL。
6. **渐进迁移**：保留 `ApiManager`，优先重构 endpoint definition 与 client 层。

### 18.4 一句话概括

**当前体系的问题，不在“没有请求能力”，而在“没有把 API 当作完整契约来建模”；下一代体系的关键，不是继续包装 axios，而是建立统一、强类型、可编译、可匹配、可适配的 endpoint contract。**
